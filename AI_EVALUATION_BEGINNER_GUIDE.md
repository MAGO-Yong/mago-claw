## 不到3分钟的产品认识

### 产品是做什么的？

想象你有一个AI机器人（比如ChatGPT）在帮助用户回答问题。问题来了：

**这个AI机器人的回答质量好吗？** 🤔

- 如果回答太多，用户会烦
- 如果回答太少，用户会不满
- 如果回答错了，用户会生气
- 如果不知道何时停止，用户会困惑

**AI应用评估系统就是来解决这个问题的！** 

它的工作是：
1. **自动评估** AI机器人的每个回答
2. **记录评分** 这个回答好不好（1-5分）
3. **分析趋势** 今天评分好还是差？
4. **找出问题** 哪些问题回答不好？
5. **持续改进** 帮助AI机器人越来越聪明

### 一句话总结

**AI应用评估 = 给AI机器人打分的自动系统**

## 5个核心名词解释

### 1️⃣ 链路（Trace）

**概念**：一个完整的"对话过程"

```
用户问：你好
AI回答：你好，有什么我可以帮助的吗？
用户问：今天天气怎么样
AI回答：根据最新数据...

↑ 整个过程叫一个"链路"
```

**类比**：像是电影中的一个完整场景，有开始、中间、结束。

**为什么重要**：
- 我们需要评估的就是每个链路中AI的表现
- 比如评估"今天天气怎么样"这个问题的回答质量

---

### 2️⃣ SPAN（跨度）

**概念**：链路中的某一个"环节"或"步骤"

```
链路：用户问 → AI思考 → 调用天气API → 组织回答 → 返回给用户
                ↑        ↑              ↑
             Span1     Span2           Span3
```

**类比**：像是电影场景中的一个镜头或片段。

**为什么重要**：
- 有时候我们只想评估某一个环节
- 比如只评估"AI思考"这个环节是否充分
- 或者只评估"调用API"是否成功

---

### 3️⃣ 评估器（Evaluator）

**概念**：评判AI回答好不好的"评委"

```
评估器的工作：
输入：AI的回答 + 用户的问题
处理：对比是否匹配、是否准确、是否完整
输出：分数（1-5分）+ 理由

评估器就像老师给学生打分一样
```

**有两种评估器**：

#### 💡 LLM评估器（用Prompt）
```
工作原理：让另一个AI来评价第一个AI

优点：快、便宜、易修改
缺点：可能不够准确

比如：
Prompt = "评价这个回答的相关性，1-5分"
LLM根据Prompt来评分
```

#### 🔧 HTTP代码评估器（用代码）
```
工作原理：用复杂的代码逻辑来评价

优点：准确、可靠、完全可控
缺点：慢、贵、需要写代码

比如：
代码检查：
- 是否包含关键词？
- 长度是否在1000-5000字？
- 是否有"很抱歉"这样的完整表达？
```

---

### 4️⃣ 自动评估任务（Auto Evaluation Task）

**概念**：一个"自动打分的工作计划"

```
任务配置：
- 每天凌晨00:30自动运行
- 对所有用户的提问进行评估
- 使用"质量评估器"给打分
- 保存评分结果

↑ 这就是一个"自动评估任务"
```

**类比**：像工厂的自动化流水线，每天自动工作。

**为什么重要**：
- 不用每次手动评估
- 自动定时运行
- 持续监控AI质量

---

### 5️⃣ 评估实验（Experiment）

**概念**：一次"自动评估任务的执行结果"

```
执行1：2026-03-09 凌晨00:30 
- 评估了9984条链路
- 其中8968个评分≥3分（好）
- 1016个评分<3分（需改进）
这次执行叫一个"实验"

执行2：2026-03-10 凌晨00:30
- 评估了10102条链路
- 其中9201个评分≥3分
- 901个评分<3分
这是另一个"实验"
```

**类比**：像是科学实验中每一次的操作记录。

**为什么重要**：
- 可以看到每天的质量变化
- 可以对比不同版本的质量
- 可以找出具体哪条链路有问题

## 两种评估模式详解

### 📡 模式1：在线链路模式（实时监控）

**什么是在线链路？**
```
用户实时提问
    ↓
AI实时回答
    ↓
这条链路被自动评估
    ↓
评分立即保存
```

**特点**：
- ✅ 数据最新（刚发生）
- ✅ 反应快（秒级反馈）
- ❌ 无法重复（链路已过期）
- ❌ 难以对比（每条链路都不同）

**场景**：
- 🎯 实时质量监控
- 🎯 快速问题发现
- 🎯 即时告警

**配置示例**：
```
每天00:15运行一次
评估所有包含"weather"关键词的链路
只评估"AI思考"这个环节
随机采样7%的数据
```

---

### 💾 模式2：离线数据模式（科学对比）⭐ 新功能

**什么是离线数据？**
```
历史数据集（之前保存的真实对话）
    ↓
复用这些数据再次评估
    ↓
可以用不同的评估器对比
    ↓
完整的AB测试
```

**特点**：
- ✅ 完全可重复（数据固定）
- ✅ 易于对比（相同条件）
- ✅ 支持AB测试（同一数据，不同服务）
- ✅ 科学严谨（受控环境）
- ❌ 需要提前准备数据

**场景**：
- 🎯 升级验证（新版本vs旧版本）
- 🎯 AB测试（服务A vs 服务B）
- 🎯 性能基准（长期对标）
- 🎯 回归测试（确保无性能下降）

**配置示例**：
```
使用"2026年3月历史数据集"（593条真实对话）
调用新版本的AI服务
使用3个评估器（相关性、准确性、完整性）
每周运行一次对比
```

## HTTP代码评估器深度教程

### 为什么要用HTTP代码评估器？

**场景对比**：

| 场景 | LLM评估 | HTTP评估 |
|------|--------|--------|
| 快速试验 | ✅ 好 | ❌ 复杂 |
| 高精度需求 | ❌ 中等 | ✅ 最佳 |
| 复杂规则 | ❌ 困难 | ✅ 容易 |
| 成本 | ✅ 便宜 | ❌ 贵 |
| 完全可控 | ❌ 否 | ✅ 是 |

**何时选用HTTP**：
- ✅ 内部有现成的评估服务
- ✅ 评估规则复杂且已定好
- ✅ 对准确度要求极高
- ✅ 有专业的技术团队

---

### HTTP评估器的工作原理

```
AI的回答
    ↓
系统发送HTTP请求给你的服务
    ↓
你的服务运行评估逻辑
    ↓
返回评分结果
    ↓
系统记录并展示
```

**类比**：像是在打电话给朋友问意见，而不是自己思考。

---

### 创建HTTP代码评估器的完整步骤

**步骤1：准备你的HTTP评估服务**

首先，你需要有一个HTTP服务来执行评估。这个服务需要：

```
监听地址：https://your-service.com/evaluate
接收请求：POST

请求格式（JSON）：
{
  "question": "今天天气怎么样",
  "answer": "根据最新数据..."
}

返回格式（JSON）：
{
  "score": 4,
  "reason": "回答准确且信息完整"
}
```

**示例代码**（Python Flask）：
```python
from flask import Flask, request, jsonify

app = Flask(__name__)

@app.route('/evaluate', methods=['POST'])
def evaluate():
    data = request.json
    question = data.get('question', '')
    answer = data.get('answer', '')
    
    # 你的评估逻辑
    # 示例：根据答案长度和关键词判断
    
    score = 1  # 默认评分
    reason = ""
    
    # 检查答案是否为空
    if not answer:
        score = 1
        reason = "答案为空"
    # 检查答案长度
    elif len(answer) < 20:
        score = 2
        reason = "答案过短"
    elif len(answer) > 1000:
        score = 2
        reason = "答案过长"
    # 检查是否包含关键信息
    elif any(keyword in answer for keyword in ['天气', '温度', '降雨']):
        score = 5
        reason = "答案包含关键信息"
    else:
        score = 3
        reason = "答案基本合格"
    
    return jsonify({
        "score": score,
        "reason": reason
    })

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000)
```

### HTTP代码评估器的高级用法

#### 案例1：复杂的评估逻辑

```python
# 更复杂的评估示例

def evaluate_answer(question, answer):
    score = 3  # 默认评分
    details = []
    
    # 1. 检查答案完整性
    if len(answer) < 50:
        score -= 1
        details.append("答案过短")
    elif len(answer) > 2000:
        score -= 1
        details.append("答案过长")
    
    # 2. 检查专业用语
    professional_terms = ['根据', '数据', '分析', '调查']
    if any(term in answer for term in professional_terms):
        score += 1
        details.append("包含专业表述")
    
    # 3. 检查是否有错误信息
    false_info = ['不可能', '绝对', '肯定']
    if any(info in answer for info in false_info):
        score -= 2
        details.append("可能包含绝对表述")
    
    # 4. 检查是否有礼貌用语
    polite = ['谢谢', '请问', '抱歉']
    if any(word in answer for word in polite):
        score += 1
        details.append("语气礼貌")
    
    # 确保分数在1-5之间
    score = max(1, min(5, score))
    
    return {
        "score": score,
        "reason": "; ".join(details) if details else "答案基本合格"
    }
```

#### 案例2：调用外部API进行评估

```python
import requests

def evaluate_with_external_api(question, answer):
    """
    调用外部API（比如情感分析、语法检查等）
    """
    score = 3
    details = []
    
    # 调用情感分析API
    try:
        sentiment_response = requests.post(
            'https://api.example.com/sentiment',
            json={'text': answer}
        )
        sentiment = sentiment_response.json().get('sentiment', 'neutral')
        
        if sentiment == 'positive':
            score += 1
            details.append("语气积极")
        elif sentiment == 'negative':
            score -= 1
            details.append("语气消极")
    except:
        details.append("情感分析失败")
    
    # 调用语法检查API
    try:
        grammar_response = requests.post(
            'https://api.example.com/grammar',
            json={'text': answer}
        )
        errors = grammar_response.json().get('errors', [])
        
        if len(errors) > 5:
            score -= 2
            details.append("语法错误过多")
        elif len(errors) == 0:
            score += 1
            details.append("语法完美")
    except:
        pass
    
    score = max(1, min(5, score))
    return {
        "score": score,
        "reason": "; ".join(details) if details else "答案基本合格"
    }
```

A：对于HTTP评估器，需要一些编码能力。

## 离线数据自动评估深度教程

### 什么是离线数据评估？

**核心概念**：
```
不用实时数据，而是用历史数据进行评估

```

### 离线数据评估的完整流程

#### 第一步：准备数据集

## 常见问题解答

### 🔧 技术问题

**Q1：如何选择采样率？**
```
A：
- 数据量少（<1000条/天）：100%采样
- 数据量中等（1000-10000条/天）：50%采样
- 数据量大（>10000条/天）：10%采样

更高采样率 = 更准确但更贵
```

**Q2：评分1-5分，各代表什么？**
```
A：一般来说
1分：完全不满意
2分：不太满意
3分：基本满足
4分：很满意
5分：非常满意

你可以根据需要自定义含义
```

**Q3：JsonPath是什么？怎么用？**
```
A：JsonPath是提取JSON数据的语言

常用表达式：
$.field           → 获取field字段
$.array[0]        → 获取数组第一个元素
$.array[*].name   → 获取所有数组元素的name
$.*.name          → 获取所有子对象的name
$[?(@.score>3)]   → 获取score>3的元素

例如：
如果数据是 {"user": {"name": "小明"}}
$.user.name       → 得到 "小明"
```

### 📊 使用问题

**Q4：为什么评估结果不稳定？**
```
A：可能原因：
1. LLM评估器本身不稳定（不同时刻结果不同）
   → 改用HTTP评估器
   
2. 评估数据不代表（样本太少）
   → 增加采样量或数据量
   
3. 没有清晰的评估标准
   → 优化Prompt或代码逻辑
   
4. 评估服务有bug
   → 测试并修复服务
```

**Q5：如何改进低分的链路？**
```
A：流程：
1. 找出低分链路（<2分的）
2. 查看失败原因
3. 分类低分原因
4. 针对性改进
5. 再次评估验证

例如：
- 如果是格式问题：修改输出格式
- 如果是内容问题：调整Prompt
- 如果是逻辑问题：修改代码
```

**Q6：多个评估器怎么综合评分？**
```
A：方法：
1. 简单取平均
   综合分 = (评估器1 + 评估器2) / 2

2. 加权平均（推荐）
   综合分 = 评估器1*0.5 + 评估器2*0.5
   
3. 取最低分（保守）
   综合分 = min(评估器1, 评估器2)

根据实际需要选择
```

### 💰 成本问题

**Q7：怎样控制成本？**
```
A：
1. 使用LLM评估（比HTTP便宜）
2. 降低采样率（但准确度下降）
3. 减少评估频率（但反应慢）
4. 使用便宜的LLM模型（但质量可能下降）

选择平衡点：
质量 vs 成本 vs 速度
```

**Q8：如何预估成本？**
```
A：
成本 = 链路数 × 采样率 × 评估频率 × 单位成本

例如：
- 链路数：10000条/天
- 采样率：10%（1000条）
- 频率：1天1次
- 单位成本：$0.01/条

日成本 = 1000 × $0.01 = $10
月成本 = $10 × 30 = $300
```
