## 产品概述

### 产品定位

AI 应用评估是一个**自动化、可扩展的AI服务质量评估系统**，支持：

- ✅ 多种评估方式（单环节、全链路、结果评估）
- ✅ 两种评估器类型（LLM Prompt、HTTP代码）
- ✅ 完全自动化的定时执行和数据回流
- ✅ 多维度的结果分析和可视化

### 主要价值

| 价值点 | 说明 |
|------|------|
| **自动化** | 配置后自动定时执行，无需人工干预 |
| **灵活性** | 支持多种评估方式和自定义评估规则 |
| **可观察性** | 链路级明细 + 整体分析双维度视图 |
| **循环改进** | 评估结果自动回流，支持持续迭代 |
| **易用性** | 四步向导简化复杂配置流程 |

## 核心概念

### 5个核心模块

```
┌─────────────────────────────────────────┐
│     AI 应用评估系统的5个核心模块         │
├─────────────────────────────────────────┤
│  1. LLM配置      → 模型和参数管理       │
│  2. 评估器        → 评估规则定义        │
│  3. 自动评估      → 任务配置执行        │
│  4. 评估实验      → 结果查看和分析      │
│  5. 数据集        → 评估数据管理        │
└─────────────────────────────────────────┘
```

### 4种核心概念

#### 1. 评估方式

| 评估方式 | 应用场景 | 数据源 | 精度 |
|--------|--------|------|------|
| **单SPAN** | 评估单个环节性能 | 在线链路 | 中 |
| **全链路评估** | 评估完整调用链 | 在线链路 | 高 |
| **结果评估** | 仅评估最终结果 | 在线/离线 | 低 |

#### 2. 评估器类型

**LLM评估器** - Prompt驱动
- 使用场景：快速初筛、灵活定制
- 配置方式：编写Prompt规则
- 执行性能：快（秒级）
- 评估精度：中

**HTTP代码评估器** - 代码驱动  
- 使用场景：精细化评估、复杂逻辑
- 配置方式：配置HTTP服务地址
- 执行性能：慢（需远程调用）
- 评估精度：高

#### 3. 数据流向

```
在线链路/数据集
     ↓
自动评估任务（定时或手动）
     ↓
评估器（LLM或HTTP代码）
     ↓
评估实验（结果记录）
     ├─→ 实验明细（链路级细节）
     ├─→ 实验分析（整体统计）
     └─→ 自动回流到数据集
           ↓
     [可用作下一轮数据源]
```

#### 4. 数据循环机制

- **自动回流**：满足条件的链路自动保存到数据集
- **版本管理**：每次回流创建新版本（带日期标签）
- **循环评估**：新数据集可作为下一轮评估的数据源
- **持续改进**：形成闭环的迭代改进流程

│
└─ 输出字段
   ├─ score (Number) - 得分
   └─ "+ 添加输出字段"
```

**Prompt编写指南**

```
## 任务
你是一个AI评估专家，需要评估以下内容...

## 评估规则
1. 相关性：回答是否与问题相关
2. 正确性：回答是否正确
3. 完整性：回答是否完整

## 输出格式
请输出一个JSON对象，包含：
- score (1-5)：整体评分
- reason (string)：评分理由

## 示例
输入：什么是机器学习？
输出：{
  "score": 4,
  "reason": "定义清晰，举例恰当，略缺深度"
}
```

**工作原理**：
1. 系统调用LLM API
2. 将链路数据和您的Prompt一起发送
3. LLM基于Prompt返回评估结果
4. 系统提取score、reason等字段

**适用场景**：
- 需要快速原型
- 评估规则不复杂
- 可以接受中等精度

#### 对比总结

| 维度 | HTTP代码 | LLM |
|------|---------|-----|
| 创建难度 | 中等（需要HTTP服务） | 简单（编写Prompt） |
| 执行速度 | 取决于服务 | 快（通常秒级） |
| 评估精度 | 高（完全可控） | 中等（LLM限制） |
| 成本 | 服务成本 | API调用成本 |
| 灵活性 | 低（服务黑盒） | 高（Prompt可调整） |

### 3️⃣ 自动评估模块 - 4步向导

**功能**：创建和管理自动评估任务

#### 步骤1：基本配置

```
任务名称*         文本输入    sug粗筛
评估类型*         单选        ⭕ 在线链路数据源
                             ☐ 离线数据源
执行周期*
├─ 周期类型       下拉        周期 | 按天 | 自定义
└─ 时间点         时间        00:15（每天00:15执行）
```

**关键字段说明**

- **在线链路数据源**：实时采集生产环境数据进行评估
- **离线数据源**：使用预先准备的数据集进行评估
- **执行周期**：支持每天、每周等定时执行

#### 步骤2：评估数据 - 模式A：在线链路数据源

```
评估方式*         单选        
├─ 📍 单SPAN
│  └─ 说明：基于每条中指定SPAN的数据进行评估，适用于单一环节效果
├─ ☐ 全链路评估
│  └─ 说明：评估完整的调用链
└─ ☐ 结果评估
   └─ 说明：仅评估最终结果

TraceId筛选      单选        
├─ 📍 日志（推荐）
└─ ☐ Jaeger

SPAN筛选         动态表      name | = | sug
                           "+ 添加筛选器"

链路采样率       数字        7%
```

#### 步骤2：评估数据 - 模式B：数据集数据源 🔄

**完全不同的配置模式！**

```
目标数据集*      下拉        测试回流数据集
是否开启回流     开关        ✓ 启用

对象类型*        下拉        自定义义模型

输入/输出配置
├─ Name*         文本        XPoint
├─ Domain*       文本        https://xray.devops.xiaohongshu.com
├─ Path*         文本        /iagent/api/v1/chatbot/chat
├─ Method*       下拉        POST
│
├─ IsStream      开关        ☑ 是否流式传输
│
├─ 请求Header    动态表
│  ├─ X-LangFuse-Source (String) → replay
│  └─ X-LangFuse-Eval (String) → cml69xr500159ib7j22
│  └─ "+ 添加Header"
│
└─ 请求Body      动态表
   ├─ messages (Object) → [{"content":"$.input[0]"}]
   └─ "+ 添加Body字段"
```

**关键差异对比**

| 维度 | 在线链路模式 | 数据集模式 |
|------|------------|---------|
| 数据源 | 实时生产环境 | 预先准备的数据集 |
| 配置复杂度 | 中等（需要TraceId筛选） | 高（需要HTTP映射） |
| SPAN筛选 | ✓ 需要 | ✗ 不需要 |
| 采样率 | ✓ 需要配置 | ✗ 全量数据 |
| HTTP配置 | ✗ 无 | ✓ 完整的HTTP配置 |
| 场景 | 实时监控、质量反馈 | 离线评估、回放对比 |

**数据集模式的配置说明**

1. **目标数据集**（必填）
   - 从现有数据集列表选择
   - 系统将使用该数据集的全部数据进行评估
   - 例：测试回流数据集（593条链路）

2. **是否开启回流**
   - 启用：评估完成后自动回流到新数据集版本
   - 关闭：仅评估不回流

3. **对象类型**（必填）
   - 定义评估的对象类型
   - 例：自定义义模型（代表自定义服务）
   - 用于在报告中标识

4. **HTTP配置信息**
   ```
   Name：这个配置的名称 (XPoint)
   Domain：API服务域名
   Path：API端点路径
   Method：HTTP方法 (POST/GET等)
   ```

5. **请求头配置**（动态表格）
   ```
   用于传递元数据到服务
   
   示例：
   - X-LangFuse-Source: replay (标记这是回放)
   - X-LangFuse-Eval: [评估ID] (标记评估ID)
   
   "+添加Header"支持无限添加
   ```

6. **请求体配置**（动态表格）
   ```
   映射数据集中的字段到HTTP请求
   
   示例：
   messages: [{"content":"$.input[0]"}]
   说明：从数据集的$.input[0]提取内容作为messages
   
   支持JsonPath表达式，参考字段参考文档
   ```

7. **流式传输选项**
   - 启用：支持流式响应
   - 用于长输出的场景（如生成长文本）

**评估方式详解**

1. **单SPAN** - 评估单个环节
   - 适合检查某个特定环节的质量
   - 例：检查LLM回复质量

2. **全链路评估** - 评估整个调用链
   - 适合检查端到端的完整流程
   - 例：检查从输入到输出的全过程

3. **结果评估** - 仅评估最终结果
   - 适合简单的终态评估
   - 例：评估最终回复是否正确

#### 步骤3：评估器

```
选择评估器*       下拉        sug粗筛
                           [删除]
                           
评估器Prompt      说明        展示Prompt文本
                           支持字段配置

输出字段映射      动态表
├─ response_content → $ response_content
└─ messages → $ messages[?(@.role == 'user')]
```

**字段映射说明**

- 使用JsonPath表达式
- `$` 表示根节点
- `$.field` 访问对象属性
- `$[0]` 访问数组元素
- `$[?(@.role == 'user')]` 条件过滤

#### 步骤4：数据回流（可选）

```
回流配置开关      开关        ✓ 已启用

回流配置1
├─ 目标数据集*    下拉        Sug粗筛数据集
├─ 版本策略       下拉        自动创建新版本
│
├─ 回流条件
│  ├─ 评估器      下拉        sug粗筛
│  ├─ 条件        表达式      score >= 2
│  ├─ "+ 添加条件"
│  └─ "+ 添加评估器组"
│
└─ 字段映射
   ├─ messages ← $ data.input.messages[?(@.role == 'user')]
   └─ raw_input ← $ data.input.raw_input
```

**回流条件说明**

- **单条件**：score >= 2（评分≥2的链路回流）
- **多条件AND**：score >= 2 AND relevance == 'high'
- **多条件OR**：score >= 3 OR is_important == true

**回流流程**

1. 评估完成后，系统检查是否满足回流条件
2. 满足条件的链路数据自动保存到目标数据集
3. 数据集自动创建新版本（附带日期标签）
4. 新数据集可用作下一轮评估的数据源

## 完整工作流

### 场景：建立AI服务质量评估系统

```
第1天：基础设置

## 字段参考

### LLM参数说明

| 参数 | 范围 | 说明 |
|------|------|------|
| Temperature | 0-1 | 越高越随机，越低越稳定 |
| Top P | 0-1 | 核采样，越小越专注 |
| Max Token | 1-4000 | 最大生成长度 |
| Frequency Penalty | -2-2 | 重复惩罚 |

### JsonPath表达式

| 表达式 | 说明 | 示例 |
|--------|------|------|
| $ | 根节点 | $ |
| $.field | 属性访问 | $.messages |
| $[0] | 数组索引 | $[0] |
| $[*] | 所有元素 | $[*].name |
| $[?(@.role=='user')] | 条件过滤 | $[?(@.role=='user')] |

### 回流条件操作符

| 操作符 | 说明 | 示例 |
|--------|------|------|
| >= | 大于等于 | score >= 2 |
| <= | 小于等于 | score <= 4 |
| == | 等于 | status == 'success' |
| != | 不等于 | level != 'error' |
| && | 逻辑与 | score >= 2 && is_valid == true |
| \|\| | 逻辑或 | score >= 4 \|\| is_important == true |

## FAQ

### Q1：在线链路评估 vs 离线数据集评估有什么区别？

**在线链路评估**
- 数据源：实时生产环境
- 优点：数据最新，代表当前质量
- 缺点：依赖生产流量，无法回放
- 使用场景：实时监控、快速反馈

**离线数据集评估**
- 数据源：预先准备的数据集
- 优点：可重复评估、支持不同规则
- 缺点：数据可能过期
- 使用场景：对比分析、回归测试

### Q2：应该选择哪种评估方式？

| 选择 | 条件 | 原因 |
|------|------|------|
| 单SPAN | 只关心某个环节质量 | 快速反馈 |
| 全链路 | 需要端到端评估 | 发现链路问题 |
| 结果评估 | 只关心最终输出 | 最快速度 |

### Q3：LLM评估器和HTTP评估器如何选择？

| 维度 | LLM | HTTP |
|------|-----|------|
| 时间 | 1分钟内推出 | 1天+ |
| 成本 | 低（API调用） | 中等 |
| 精度 | 中等 | 高 |
| 灵活性 | 高（可随时改Prompt） | 低（需修改代码） |

**建议**：先用LLM快速验证效果，再用HTTP优化精度

### Q4：回流条件应该怎么设置？

**原则**：回流的数据应该具有共同特征

```
条件设置建议
├─ 质量好的数据：score >= 3
├─ 需要人工审核的：1 < score < 3
└─ 错误数据（不回流）：score <= 1
```

### Q5：数据集版本能删除吗？

目前支持的操作：
- ✅ 编辑：修改字段配置
- ✅ 复制：创建副本
- ✅ 删除：删除整个版本

**建议**：删除前确认不再使用该版本

### Q6：评估任务出错了怎么办？

**排查步骤**
```
1. 检查LLM/HTTP配置是否正确
2. 检查评估器是否正常
3. 在实验列表查看失败原因
4. 手动触发一次看详细日志
5. 修改配置后重新执行
```

### Q7：可以同时运行多个自动评估任务吗？

**是的**，支持多个任务并行执行：
- 不同评估器
- 不同评估方式
- 不同执行周期

### Q8：评估结果准确吗？

**准确性取决于**：
- 🎯 评估器的质量（LLM Prompt 或 HTTP服务）
- 🎯 数据的代表性（采样率和样本量）
- 🎯 评估规则的合理性

**建议**：定期对比不同评估器的结果，持续优化

## 总结

AI 应用评估系统通过以下方式帮助你建立完整的质量保障体系：

1. **多种评估方式** - 灵活适应不同场景
2. **两种评估器** - LLM快速 + HTTP精准相结合
3. **完全自动化** - 配置后无需人工干预
4. **数据循环** - 评估结果自动回流再利用
5. **双维度分析** - 链路明细 + 整体分析

通过持续的评估 → 分析 → 优化，你可以不断提升AI服务的质量。
