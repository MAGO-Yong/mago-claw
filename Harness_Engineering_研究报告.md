# Harness Engineering 研究报告

**研究背景：** 随着公司业务全面 AI 化，AI 系统带来的稳定性不确定性加剧，探索 Harness Engineering 作为 AI 时代稳定性建设方法论的可行性

**研究范围：** 概念起源与本质、与传统稳定性的实质差异、具体落地能力、组织与工具演变、控制论理论根基

---

## 一、Harness Engineering 是什么——从控制论说起

### 1.1 一个 1948 年就有答案的问题

诺伯特·维纳在 1948 年出版《控制论——关于在动物和机器中控制和通信的科学》时，提出了一个跨越机器、生物和社会的统一框架：

> **控制论的核心**：系统根据实际输出与期望输出的误差，持续调整输入，消除误差，实现目标。这个过程的关键不是「执行力」，而是「反馈回路」。

维纳发现，从高射炮自动瞄准系统到人体神经调节，所有目标导向系统的本质都一样：**感知偏差 → 计算纠正 → 施加控制 → 再次感知**。

他还提出了两个对 AI 时代同样重要的概念：
- **熵（Entropy）**：系统趋向混乱的自然倾向。没有持续的控制输入，任何系统都会走向无序
- **信息**：「信息就是信息，不是物质也不是能量」——控制系统的本质是信息流动，而不是能量流动

### 1.2 同一个模式，三次出现

一篇分析 Harness Engineering 的文章（2026）揭示了这个历史规律：

```
第一次：瓦特离心调速器（1780s）
  问题：蒸汽机需要人站旁边手动调节阀门
  解法：机械装置自动感知转速 → 调节阀门
  人的角色变化：从「拧阀门」→「设计调速器」
  控制论本质：在机械层面闭合了反馈回路

第二次：Kubernetes（2014+）
  问题：微服务时代，人无法手动管理成百上千个 Pod
  解法：声明期望状态，控制器持续观察偏差并协调
  人的角色变化：从「重启服务」→「编写规格说明」
  控制论本质：在基础设施层面闭合了反馈回路

第三次：Harness Engineering（2025+）
  问题：AI Agent 写代码、做决策，但「好不好」只有人才能判断
  解法：工程师设计约束环境、构建反馈回路，Agent 在其中执行
  人的角色变化：从「写代码」→「设计控制系统的规则层」
  控制论本质：在认知/质量判断层面闭合了反馈回路
```

每一次，都是因为有人在新的层面造出了足够好的「传感器」（感知偏差）和「执行器」（纠正偏差），反馈回路才得以在那个层面闭合。LLM 是第一个能在「认知判断」层面同时充当传感器和执行器的技术。

### 1.3 Harness Engineering 的词义与定义

**Harness**（马具）：套在马身上让人能控制方向和力量的装备。AI Agent 是那匹动力十足但不守规矩的马，Harness 是让它跑得快又不跑偏的缰绳和马鞍。

它是一个概念演进的第三阶段：

| 阶段 | 时间 | 关注点 |
|------|------|------|
| Prompt Engineering | 2023-2024 | 怎么跟 AI 说话（单次输入-输出优化） |
| Context Engineering | 2025 | 给 AI 看什么信息（系统提示、记忆、RAG、工具） |
| Harness Engineering | 2026 | **构建什么环境让 AI 可靠工作**（整个执行环境的设计） |

**工作定义：** Harness Engineering 是为 AI Agent 设计执行环境的工程学科——通过构建上下文约束、行为边界、质量评估和反馈闭环，使非确定性的 AI 系统在生产环境中可靠、可控、可审计地运行。

## 二、为什么传统稳定性方法不够——一个根本性断层

### 2.1 传统方法失效的根本原因

传统软件工程的全套可靠性方法（单元测试、集成测试、告警规则、变更回滚）建立在一个核心假设上：

> **同样的输入 → 同样的输出。确定性。**

这个假设在 AI 系统里不成立。

**三个环节的具体失效：**

**① 测试失效**

传统测试的核心逻辑：定义输入，断言输出，完全一致才算通过。

AI Agent 的 LLM 大脑具有随机性——即使 temperature=0，在不同版本、不同平台下也可能有微小差异。更重要的是，Agent 的推理路径（Thought）和工具调用顺序（Action）在不同运行时可能完全不同，但两次都正确达成了目标。

传统「录制-回放」式测试在 Agent 上完全失效，必须引入**轨迹评估（Trajectory Evaluation）**——评估推理路径是否合理，而不只是最终结果是否一致。

**② 监控失效**

传统监控感知的：CPU 高、延迟高、HTTP 5xx 升高、Pod 崩溃。

这些指标全部正常，但 Agent 可能正在系统性地给出错误推荐、走偏的推理路径、质量下滑的输出——传统监控对此完全无感知。

**服务「活着」和服务「做对了」，在 AI 系统里是两件完全不同的事。**

这正是维纳控制论里「稳态（homeostasis）」概念的新挑战：传统系统的稳态是「服务可用」，AI 系统的稳态还包含「输出质量稳定」——后者传统监控无法感知。

**③ 变更管理失效**

传统变更：代码改了 → 触发构建 → 走发布流程 → 验证功能。

AI 系统里新增了三类传统流程完全感知不到的变更：
- **System Prompt 修改**：不在代码里，不触发 CI/CD
- **模型版本静默升级**：提供商无声更新，无任何通知
- **RAG 知识库更新**：内容变了但系统版本没变

从控制论角度看：这是系统的「干扰输入」（disturbance）——外部扰动改变了 Agent 的行为，但没有任何传感器能感知到这个扰动。

### 2.2 一个反直觉的数据

Catchpoint SRE Report 2025（调研 300+ 可靠性从业者）：**AI 工具引入后，运维 toil 不降反升**，连续五年下降后 2025 年首次反弹至 30%。

这不是「AI SRE 不管用」，而是在说：把 AI 工具叠加到旧的稳定性体系上，只会制造新的复杂度。需要的是新的方法论框架，而不是叠加工具。*(来源：Catchpoint，独立研究)*

### 2.3 「代码库为什么是最后的堡垒」——控制论视角的解释

代码库并非没有反馈回路，只是只在较低层面有：
- **编译器**：在语法层面闭合回路
- **测试套件**：在行为层面闭合回路  
- **Lint 工具**：在风格层面闭合回路

这些都是控制论意义上的控制——但它们只能检查「可以机械验证的属性」。

在这之上的一切——「这个改动符合架构吗」「这个抽象会不会埋坑」——既没有传感器，也没有执行器。只有人类能在那个层面运作，而且是两侧同时运作：感知偏差 + 执行纠正。

LLM 第一次让这两端都可以自动化。**反馈回路可以在「做决策」的层面闭合了。**

## 三、Harness Engineering 的核心能力体系

### 3.1 三层 Harness 架构

```
┌──────────────────────────────────────────────────────────┐
│  Layer 3: 评估层（Eval Harness）                          │
│  持续评估 Agent 输出质量，建立 AI SLO                    │
│  「系统运行正常」≠「Agent 做了正确的事」                  │
│  控制论本质：质量传感器，感知与期望输出的偏差             │
├──────────────────────────────────────────────────────────┤
│  Layer 2: 约束层（Constraints Harness）                   │
│  控制 Agent 的行为边界，限制失控风险                     │
│  权限管理、输出校验、人工审批、回退机制                  │
│  控制论本质：执行器，施加纠正控制                        │
├──────────────────────────────────────────────────────────┤
│  Layer 1: 上下文层（Context Harness）                     │
│  系统提示、工具描述、记忆管理、RAG 质量                  │
│  信息质量决定推理质量                                    │
│  控制论本质：参考输入，定义「期望状态」                  │
└──────────────────────────────────────────────────────────┘
```

**控制论的完整映射：**

| 控制论概念 | Harness Engineering 对应 |
|-----------|------------------------|
| 参考输入（期望状态）| System Prompt + 架构约束 + 黄金准则——定义「什么叫好」|
| 传感器（感知偏差）| Eval 评估器——感知 Agent 输出与期望的偏差 |
| 执行器（纠正控制）| 约束层——权限边界、输出校验、人工干预 |
| 反馈回路 | Eval → 发现问题 → 调整 Prompt/约束 → 再次 Eval |
| 稳态（homeostasis）| AI SLO——定义系统的「健康状态」 |
| 熵（无序趋势）| 模型漂移、Prompt 退化、知识库过时——持续对抗 |
| 噪声 | 幻觉、随机性、非确定性——需要被过滤或约束 |

### 3.2 Agent 系统 vs 传统在线服务：各环节实质差异

| 环节 | 传统在线服务 | Agent 系统 | 实质区别 |
|------|------------|-----------|---------|
| **变更是什么** | 代码 commit、配置文件 | 代码 + **Prompt 修改 + 模型版本 + RAG 更新 + 工具描述** | 变更对象扩大，大部分不在代码里 |
| **测试方法** | 给定输入，断言输出是否一致 | 功能测试 + **轨迹评估（推理路径是否合理）+ 质量评估（输出是否可信）** | 多了行为和质量两个维度，且不能用精确断言 |
| **判断「通过」标准** | Pass/Fail，自动判断 | 功能测试通过 + **Eval 评分不低于基线** | 准入标准从「对/错」变成「够不够好」 |
| **生产监控** | 5xx、延迟、Pod 崩溃 | 以上 + **持续 Eval 采样**：从生产流量抽样评估质量 | 多了「质量类告警」，不能只看基础设施指标 |
| **回滚什么** | 代码版本回滚 | 代码版本 + **Prompt 版本 + 模型版本指定 + RAG 快照** | 回滚对象变多，且工具链不成熟 |
| **审计内容** | 「谁调了什么接口，返回什么」 | 接口调用 + **推理步骤、工具调用、决策依据** | 审计粒度从接口细化到推理过程 |
| **高风险操作控制** | RBAC 预设权限 | 权限控制 + **Agent 动作实时审批**（因为 Agent 决定做什么是动态的）| 从「事前授权」变成「实时决策」 |
| **测试边界** | UI/接口限制了输入范围，可穷举 | 自然语言输入，**每句话都是边界情况**，无法穷举，需对抗性测试 | 攻击面从有限变成无限 |

**具体 Case 说明区别：**

> 传统场景：发布变更风险评估服务新版本 → 功能测试通过 → 上线 → 5xx 监控正常 → 认为成功。
>
> Agent 场景：修改变更风险评估 Agent 的 System Prompt → 功能没坏 → 上线 → 5xx 正常 → 但两周后发现：Agent 对某类变更的风险评估系统性偏低，若干本该拦截的变更被放行。传统监控完全无感知，只有在故障复盘时人工回查 Agent 历史记录才发现。
>
> Harness Engineering 的解法：上线前 Eval 基线比对（新 Prompt 在历史 case 集上的准确率不低于旧版本），生产环境持续 Eval 采样（每天抽 5% 做质量评分），质量下滑超阈值触发告警。

### 3.3 AI SLO——为什么必须新定义

**传统 SLO 衡量「服务活不活」，AI SLO 衡量「Agent 做没做对」。**

| 维度 | 传统 SLO | AI SLO |
|------|---------|--------|
| 核心问题 | 服务是否可用？是否够快？ | Agent 的输出是否正确？是否可信？ |
| 典型指标 | 可用率、P99 延迟、错误率 | 任务完成率、判断准确率、幻觉率、拒识率 |
| 数据来源 | 监控系统自动采集 | 需要评估器打分（LLM-as-Judge 或人工标注）|
| 违规感知 | 告警规则自动触发 | 持续 Eval 采样 → 评分下滑 → 告警 |
| 度量难度 | 容易，精确计算 | 困难，概率性评估 |
| 行业标准 | Google SRE Book 完整定义 | **无行业标准，各家自定义** |

**AI SLO 指标体系（当前行业最接近共识的定义）：**

```
一、事实性（Faithfulness）
  定义：输出内容是否忠实于来源，不乱编
  目标值示例：≥ 90%

二、任务完成率（Task Success Rate）
  定义：Agent 完成任务、结果正确的比例
  目标值示例：核心场景 ≥ 85%，长尾场景 ≥ 70%

三、答案相关性（Answer Relevance）
  定义：回答是否和用户意图相关，没有答非所问
  目标值示例：≥ 85%

四、拒识合理性（Appropriate Refusal Rate）
  定义：该拒绝的时候正确拒绝（越权/超范围指令）
  目标值示例：越权指令拒识率 ≥ 99%
```

## 四、控制论对 Harness Engineering 的五个关键启示

这是从维纳原著延伸出来的、对实践有直接指导价值的理论框架。

### 4.1 「把判断力变得机器可读」——校准传感器的本质

控制论的核心洞察之一：**传感器必须被校准**。调速器需要知道「正确转速是多少」，才能感知偏差并纠正。

Harness Engineering 的最大工程挑战，不是技术，而是同样的校准问题：

> 「智能体失败不是因为能力不够，而是因为它需要的知识——什么叫「好」、你的架构鼓励哪些模式——锁在你脑子里，你从没把它外化出来。」

OpenAI 案例：他们每周五花 20% 的时间清理「AI 垃圾代码」——直到他们把标准编进了 Harness 本身（自定义 Lint 规则、架构文档、黄金准则）。五个月后，一百万行代码，零手写。

**对稳定性 PM 的启示：** AI SLO 的定义就是「把判断力变得机器可读」。你对「这个变更有没有风险」「这次故障根因判断对没对」的判断，必须被外化为可量化的评估标准，Agent 才能被正确校准。

### 4.2 「熵与反熵」——为什么稳定性建设是持续工程而非一次性建设

维纳的熵概念：系统自然趋向无序。维持秩序需要持续的信息输入和控制。

在 Harness Engineering 的语境下，熵的表现是：
- **Prompt 退化**：随着业务变化，System Prompt 里的假设逐渐不再成立
- **模型漂移**：提供商静默升级模型，Agent 行为悄悄改变
- **RAG 知识库过时**：文档更新了，RAG 还在检索旧版本
- **测试 case 老化**：业务场景变了，原来的 Eval case 集不再有代表性

**关键推论：** Harness Engineering 不是一次性建设，而是持续的「反熵运动」——持续更新 Eval case 集、持续监控质量指标、持续迭代约束规则。这和传统稳定性建设「做完就稳定了」的预期完全不同。

### 4.3 「黑箱方法」——如何面对 LLM 的不可解释性

维纳提出了「黑箱方法」：当系统内部机制无法直接分析时，通过研究输入-输出关系来理解和控制系统。

LLM 的内部决策是黑箱，无法直接调试。但 Harness Engineering 的实践恰好对应黑箱方法：

- 不分析 LLM 内部为什么这样输出（无法做到）
- 通过大量输入-输出 pair 的 Eval，建立对系统行为的统计性理解
- 通过调整外部输入（Prompt、约束、上下文）来改变输出，而不是修改模型本身

**对 PM 的实践启示：** 当 Agent 表现不好，解法不是「让研究员去调模型」，而是「调整 Harness 的输入条件」——这是 PM 和 SRE 可以主导的工作，不依赖算法团队。

### 4.4 「目的论行为」——AI SLO 的哲学基础

维纳和罗森布吕斯 1943 年的经典论文提出：目标导向行为（purposeful behavior）不只是生物的专利，机器也可以有。关键是系统有一个「参考状态」，并持续根据与参考状态的偏差来调整行为。

AI SLO 正是这个参考状态的工程化表达：
- 定义「Agent 应该达到什么状态」（任务完成率≥85%，幻觉率≤10%）
- 持续感知实际状态与参考状态的偏差（Eval 采样）
- 根据偏差调整 Harness（Prompt 优化、约束调整）

**这是为什么没有 AI SLO，Harness Engineering 就无法运转的根本原因**——没有参考状态，就无法定义偏差，反馈回路就无从闭合。

### 4.5 「二阶控制论」——观察者本身也在系统内

1970 年后发展的二阶控制论（Second-Order Cybernetics）指出：观察者本身也是系统的一部分，不存在纯粹「客观外部」的观察者。

在 Harness Engineering 里，这个洞察有直接的实践含义：

**你定义的 Eval 标准，本身就在影响 Agent 的行为**——因为 Agent 会被优化来提高 Eval 评分。如果 Eval 设计得不好（只测试了「显而易见的好案例」），Agent 会在那些案例上表现好，在真实长尾场景里依然失控。

**推论：** Eval case 集的设计必须包含「对抗性测试」（你不希望 Agent 做的事）、「边界案例」（模糊地带的场景）、「退化监测」（专门用来发现质量下滑的 case）——而不只是「正向功能验证」。

## 五、工具生态全景

| 层级 | 要做的事 | 与传统区别 | 代表工具 | 成熟度 |
|------|---------|----------|---------|--------|
| **Layer 0 基础可观测** | 记录 Agent 完整推理链（Thought→Action→Observation）| 传统只记接口调用；Agent 需要记录推理步骤 | Langfuse / LangSmith / OpenTelemetry GenAI | ✅ 成熟 |
| **Layer 1 上下文管理** | Prompt 版本管理、RAG 知识库版本、模型版本登记 | 传统没有「Prompt 是变更」的概念 | Arize Phoenix Prompt Hub / LangWatch | 🟡 初步成熟 |
| **Layer 2 质量评估** | 定义 Eval case 集；持续采样评估；Eval 接入变更准入 | 传统测试是 Pass/Fail；Eval 是概率性评分 | Braintrust / LangSmith Evals / RAGAS / LangWatch | ✅ 成熟 |
| **Layer 3 变更准入** | Prompt/模型升级前，必须 Eval 基线比对，不退化才上线 | 传统准入=功能测试；Agent 准入=功能+质量不退化 | 无专用产品，各家自建 CI/CD + Eval API | 🔴 空白 |
| **Layer 4 权限与约束** | Agent 最小权限；高风险动作实时确认；输出结构化校验 | 传统 RBAC 是接口级；Agent 权限需语义级判断 | 各家自建，无成熟标准 | 🔴 空白 |
| **Layer 5 AI SLO 监控** | 生产流量持续 Eval 采样；质量下滑告警；AI SLO Dashboard | 传统 SLO 看可用率/延迟；AI SLO 看任务完成率/幻觉率 | LangWatch / Arize Online Eval | 🟡 初步可用 |
| **Layer 6 审计归档** | 完整记录推理过程 + 执行动作 + 审批记录 | 传统审计记「接口调用」；Agent 审计记「推理过程+决策依据」 | 无成熟方案，OTel GenAI 有底层格式 | 🔴 空白 |

**分层成熟度总结：**
- ✅ 可直接用：Layer 0（Trace）+ Layer 2（Eval）
- 🟡 需适配：Layer 1（Prompt 管理）+ Layer 5（AI SLO 在线评估）
- 🔴 需自建/主导定义：Layer 3（变更准入）+ Layer 4（语义权限）+ Layer 6（审计归档）

## 六、组织与角色演变

### 6.1 类比云原生转型

| 维度 | 云原生转型（2014-2020）| AI 全面化（2025-）|
|------|---------------------|----------------|
| 技术驱动力 | 容器化、微服务、K8S | LLM、Agent、SKILL |
| 旧方法失效 | ITIL「人工变更审批」跟不上发布速度 | 传统监控无法检测「系统健康但输出错误」 |
| 新方法论 | SRE（SLO + 错误预算 + 自动化）| Harness Engineering（Eval + 约束 + AI SLO）|
| 人的角色 | 从「运维机器」→「设计可靠系统的工程师」 | 从「响应故障」→「设计可靠 AI 系统 + 定义 AI 行为边界」|

**三个更根本的不同：**

1. **故障不可见性更高**：云原生的故障（Pod 崩溃）可被监控感知；AI 系统的故障（输出错误）可以在系统指标完全正常时发生
2. **整个生产链都在 AI 化**：云原生时代只有运维工具 AI 化；现在是需求、设计、开发、测试、部署整条链路都在被 AI 接管，系统复杂度叠加放大
3. **需要同时管两套 SLO**：系统 SLO（可用率/延迟）+ AI SLO（任务完成率/幻觉率），两者都不能缺

### 6.2 角色演变

| 角色 | 现在的工作 | AI 全面化后新增工作 |
|------|----------|-----------------|
| SRE | 监控配置、故障处置、容量规划 | Eval 体系设计、AI SLO 定义、Agent 权限治理 |
| 研发 | 编码、测试、架构设计 | Prompt 工程、Harness 设计、理解 AI 失效模式 |
| **稳定性 PM** | 稳定性规范、SLO 框架 | **AI SLO 定义、Agent 行为准入标准、Harness 产品设计、数据模型定义** |

### 6.3 PM 的不可替代价值

**「生成-验证不对称性」（P vs NP 的直觉）**：生成一个正确的解比验证一个解要难。你不需要在实现能力上超越机器，你需要在**评判能力**上超越它：

- 定义「正确」是什么样子 → **AI SLO 定义**
- 识别输出哪里不对 → **Eval case 集设计**
- 判断方向是否正确 → **Harness 产品设计**

这三件事，工程师做不了（不理解业务语义），算法团队不会做（不关心产品质量定义），只有 PM 能主导。

## 七、落地路径

### 7.1 为什么「实践没有变，代价变了」

文章的核心洞察：**文档、自动化测试、编码化的架构决策、快速反馈回路——这些一直是对的**，过去三十年每一本软件工程书都在推荐，但大多数人跳过，因为代价缓慢而弥散。

**Agent 化工程让这个代价变得极端：**
- 跳过文档 → Agent 在每一个 PR 上无视你的规范，以机器的速度，全天候
- 跳过测试 → 反馈回路根本无法闭合
- 跳过架构约束 → 漂移速度快过你修复的速度

**关键陷阱**：如果 Agent 不知道「干净」长什么样，你也没法用 Agent 来修复——没有校准，制造问题的机器同样无法解决问题。

### 7.2 分阶段落地

```
阶段一（Q2 2026）：建立质量感知能力
  ├── 接入 Trace 采集（Layer 0）——知道 Agent 做了什么
  ├── 建立 3 个核心场景的 Eval case 集——能评估 Agent 做对没有
  ├── 定义 AI SLO 指标和基线——知道「好」是什么标准
  └── 首批 SKILL 上线配套 Eval 流水线

阶段二（Q3 2026）：建立变更控制能力
  ├── Prompt 版本管理接入（Layer 1）
  ├── Eval-gated 变更准入（Layer 3）——质量不退化才能上线
  ├── AI SLO 持续监控（Layer 5）——质量下滑自动告警
  └── Agent 高风险动作审批流（Layer 4 初版）

阶段三（Q4 2026）：建立审计与治理能力
  ├── AgentActionRecord 数据模型落地（Layer 6）
  ├── 与 AI 平台打通链路数据
  ├── 与安审建立数据共享机制
  └── AI SLO Dashboard 上线
```

## 八、核心结论

**Harness Engineering 是真实的、值得作为方法论引领的方向，不是概念炒作**，理由：

1. **有历史规律支撑**：控制论 1948 年的框架完美解释了这个演变；瓦特调速器和 K8S 已经证明了这个模式的有效性
2. **有工程实证**：Datadog MCP GA、OpenAI 百万行代码案例、蚂蚁/腾讯/百度 AI Agent 运维实践
3. **有行业共识趋势**：Gartner 2025 年底建议「选择 AI SRE 工具并开始试点」
4. **有明确的空白地带**：Layer 3/4/6 是行业空白，是平台可以率先定义标准的位置

**Harness Engineering 里真正新的东西，集中在两点：**
- **质量维度的新增**：「系统活不活」→ 新增「Agent 做没做对」，这需要 Eval + AI SLO + 质量类告警
- **变更对象的扩大**：「代码变更」→ 扩展到「Prompt + 模型 + RAG + 工具描述」，这些新型变更需要纳入变更管理

其余（权限管理、回滚、审计）不是新概念，但在 Agent 系统里需要重新定义对象范围和判断粒度。

## 九、工程实践全景——可量化的 Harness 改进案例

> 本节是目前全网最完整的 Harness Engineering 实验证据汇编，均为可引用的公开来源。

### 9.1 概念起源：Mitchell Hashimoto 的定义（2026-02-05）

Harness Engineering 这个词由 HashiCorp 联合创始人 Mitchell Hashimoto（Terraform、Vagrant、Vault 的创造者）在 2026 年 2 月 5 日的个人博客中首次系统性提出。

他的原始定义非常简洁：

> 「每当你发现 Agent 犯了一个错误，你就花时间设计一个解决方案，使 Agent **从此永远不再犯同样的错误**。」

注意这个定义的核心——不是「修复那个错误」，而是「工程化地封闭那个错误路径」。这是一种防止熵增的持续工程思维，而不是救火。

六天后（2026-02-11），OpenAI 官方博客发表「Harness engineering: leveraging Codex in an agent-first world」，用同一个词描述了他们的内部实践，概念迅速在开发者社区引爆。

---

### 9.2 实验一：LangChain deepagents-cli（2026-02-17）

**来源：** LangChain Blog 官方发布，Terminal Bench 2.0 公开基准测试
**可信度：** ✅ 一手资料，数据可复现

**实验设定：**
- 底层模型：`gpt-5.2-codex`，全程固定不变
- 基准测试：Terminal Bench 2.0（89 个任务，覆盖机器学习/调试/生物学等）
- 评测系统：Harbor 编排沙盒，LangSmith 全量记录 Trace

**结果：**
```
初始基线（默认 Harness）：52.8%  → 全球 Top 30 之外
最终成绩（优化后 Harness）：66.5%  → 全球 Top 5
提升：+13.7 个百分点，模型一行代码未改
```

**四项关键优化及其逻辑：**

**① 自验证循环（Build-Verify Loop）**
- 问题：Agent 写完代码就停，不测试
- 解法：System Prompt 加入「规划→构建→验证→修复」强制循环；`PreCompletionChecklistMiddleware` 在 Agent 退出前拦截，强制执行验证步骤
- 原理：Agent 天然倾向于输出「第一个看起来合理的结果」——需要外部约束打破这个倾向

**② 环境上下文注入（LocalContextMiddleware）**
- 问题：Agent 不了解运行环境，常因找不到文件/工具失败
- 解法：Agent 启动时自动注入当前目录结构、可用工具列表、时间预算提醒
- 原理：「Harness 工程师的核心职责是准备并交付上下文，让 Agent 能自主完成工作」

**③ 循环检测（LoopDetectionMiddleware）**
- 问题：Agent 陷入「Doom Loop」，对同一文件反复小修改 10+ 次，无法跳出
- 解法：工具调用钩子跟踪每个文件编辑次数，超过阈值 N 时注入「考虑重新审视你的解决方案」
- 注意：这是针对当前模型局限性的临时护栏，随模型能力提升会逐渐不必要

**④ 推理预算三明治（Reasoning Sandwich）**
- 问题：全程 xhigh 推理导致超时（53.9%）；全程 high 损失深度（63.6%）
- 解法：规划和验证阶段用 xhigh，中间执行阶段用 high
- 最终得分：66.5%
- 原理：不同任务阶段对推理深度需求不同，统一配置是资源浪费

**Trace Analyzer Skill——用 Agent 优化 Agent 的闭环：**
```
运行 Agent → 收集 Trace（LangSmith）
     ↓
并行启动多个分析 Agent → 识别失败模式
     ↓
主 Agent 汇总建议 → 人工复核（可选）
     ↓
修改 Harness → 重新运行验证
```
这个「Boosting 式」改进循环是 LangChain 实验最有价值的方法论贡献——它把 Harness 改进从「人工经验积累」变成了「可重复执行的工程流程」。

**重要警示——过拟合风险：**
> 针对当前错误模式的 Harness 修改，可能导致在其他任务上性能退化。Eval case 集必须覆盖多样化场景，而不只是「已经出过问题的场景」。

### 9.3 实验二：CORE-Bench（arXiv 2603.05344，2026-03-05）

**来源：** arXiv 公开论文，第一篇开源终端原生编码 Agent 综合技术报告
**可信度：** ✅ 学术论文，可引用

**实验核心：**
- 底层模型：Claude Opus 4.5，固定不变
- 变量：Agent 能看到的上下文范围（Harness 的信息注入设计）

**结果：**
```
标准 Harness：42%
优化后 Harness（调整约束与上下文可见性）：78%
提升：+36 个百分点，几乎翻倍
```

LinkedIn 上的评论精准总结：「Same model. Same benchmark. 42% vs 78%. The only thing that changed was what the model could see.」

**与 LangChain 实验的共同启示：** 「模型能看见什么」比「模型有多聪明」更决定结果。这正是维纳「信息是控制的本质」的工程验证。

### 9.4 实验三：OpenAI Codex 内部实验（2026-02-11）

**来源：** OpenAI 官方博客（中文版）：https://openai.com/zh-Hans-CN/index/harness-engineering/
**可信度：** ✅ 一手原始资料，官方发布

#### 实验规模与结果

- **起点：** 2025 年 8 月下旬，向空代码库首次提交（连初始 AGENTS.md 都由 Codex 自己生成）
- **工程师：** 3 人起步 → 最终扩展到 7 人
- **时间：** 五个月
- **产出：** 约一百万行代码，零行人工手写，覆盖应用逻辑/基础设施/工具/文档/内部开发者工具
- **效率：** 平均每人每日 3.5 个 PR 合并，且团队规模扩大后吞吐量还在提升
- **产品状态：** 已有数百名内测用户，包括每天使用的内测高级用户

> **人类掌舵。智能体执行。**

#### 工程师在做什么——不是写代码，而是建造环境

OpenAI 原文关键总结：
> 「工程师工作的重点转向了**系统、架构和杠杆作用**。当事情进行不顺利时，解决方案基本上再也不会是'再努力一点'。唯一的方式是让 Codex 来完成工作，而人类工程师则追问：究竟还需要什么样的能力，又该如何让这个能力对智能体来说既清晰可读又可强制执行？」

#### 四大工程实践（原文细节）

**① 文档工程——给 Agent 一张地图，而不是一本说明书**

初期尝试「一个大型 AGENTS.md」的四个致命问题（原文）：
- **上下文是稀缺资源**：巨大的指令文件会挤掉任务和代码，Agent 要么错过关键约束，要么对错误的约束优化
- **过多指导变得无效**：当一切都「重要」时，一切都不重要了，Agent 最终只是本地模式匹配而非有意识地导航
- **立即腐烂**：庞杂的手册变成陈旧规则的坟场，Agent 无法判断哪些信息仍然有效
- **难以核实**：单个 blob 不适合机械检查（覆盖率/新鲜度/所有权/交叉链接），漂移不可避免

最终方案——**渐进式披露（Progressive Disclosure）**：

```
代码仓库根目录/
  AGENTS.md          ← 约 100 行「目录」，指向更深层的真实信息来源
  AGENTS.override.md ← 子目录级覆盖规则（例如 services/payments/ 有自己的测试命令）
  docs/
    ARCHITECTURE.md  ← 域和包分层的顶层地图
    DESIGN.md        ← 核心设计原则，含验证状态
    PLANS.md         ← 执行计划（活跃/完成/技术债务，全部版本控制）
    PRODUCT_SENSE.md ← 产品意图与用户旅程
    QUALITY_SCORE.md ← 各产品领域和架构层的质量评分，追踪差距
    RELIABILITY.md   ← 可靠性要求
    SECURITY.md      ← 安全约束
```

关键原则：
> 「从智能体的角度来看，它在运行时无法在上下文中访问的任何内容都**不存在**。存储在聊天记录或人们头脑中的知识，系统无法访问。」

**② 让 Agent「看见」运行时——可观测性直接接入**

- 本地可观测性栈对每个 git worktree 独立实例化，任务完成后连同日志/指标一起销毁
- Agent 可用 **LogQL** 查询日志，**PromQL** 查询指标
- 接入 **Chrome DevTools Protocol**——Agent 可操作浏览器，复现 Bug、验证修复、直接推理 UI 行为
- 效果：「确保服务启动在 800ms 内完成」这样的提示词变得可行

这是「让 Agent 既能写代码，又能看见代码运行后发生了什么」的关键一步。

**③ 自定义 Lint + 黄金原则——把标准编进 Harness**

初期：每周五花 20% 的时间清理「AI 残渣」（原文用词：AI slop）→ 不可扩展

最终方案——「黄金原则」直接编码到代码仓库：
- 带主观意见的机械规则，保持代码库对未来 Agent 运行的可读性和一致性
- 例：「在边界处解析数据形状，而不是验证」；「不使用 YOLO 式探测数据，依赖类型化 SDK」
- **Lint 错误信息里直接写着修复指令**，Agent 读到报错就知道该怎么改
- 严格架构模型：每个业务域划分固定层次（Types→Config→Repo→Service→Runtime→UI），依赖方向严格验证，违规自动拦截

原文评价：
> 「在以人为本的工作流程中，这些规则可能会让人感到迂腐或束缚。有了智能体，它们就成了**倍增器**：一旦编码，就能立即应用于所有地方。」

**④ 垃圾回收 Agent——持续反熵**

- 一组后台 Codex 任务定期扫描代码库偏差、更新质量评分、发起有针对性的重构 PR
- 大多数 PR 可在一分钟内完成审查并自动合并

原文类比：
> 「其功能类似于垃圾回收。技术债务就像一笔高息贷款：不断地以小额贷款的方式偿还债务，总比让债务不断累积再痛苦地一次解决要好得多。**人类的品味一旦被捕捉，就会持续应用于每一行代码。**」

#### 当前最大挑战（原文原话）

> 「**我们当前最棘手的挑战集中在设计环境、反馈回路和控制系统方面**，帮助智能体实现我们的目标：大规模构建和维护复杂、可靠的软件。」

#### 对稳定性平台的直接启示

| OpenAI 工程实践 | 技术风险平台对应动作 |
|---------------|------------------|
| AGENTS.md 渐进式披露 | SKILL 文档分层设计——能力描述精简（目录），具体规范在子文档 |

| 「Agent 看不到的等于不存在」 | Runbook、故障处置规范，必须结构化存储在 Agent 可检索的位置，否则等于没有 |

### 9.6 竞争格局：技术栈分层与 Harness 层的战略价值

从搜索结果综合来看，2026 年 AI 技术栈正在经历清晰的三层分化：

```
┌─────────────────────────────────────┐
│  Harness 层（2026年真正的竞争战场）  │
│  约束文档、记忆系统、上下文管理、     │
│  反馈循环——决定同一模型 42% vs 78%   │
├─────────────────────────────────────┤
│  协议层（正在标准化，趋向商品化）    │
│  MCP（工具接入标准）                │
│  A2A（Agent 间通信协议）            │
├─────────────────────────────────────┤
│  模型层（快速商品化）               │
│  API 价格 18 个月内下降 90%+        │
│  开源模型逼近闭源顶级水平           │
└─────────────────────────────────────┘
```

**关键判断（知乎综合多篇分析）：**
> 越底层的东西越容易被替代。模型可以换，协议会趋同，但一套在生产环境中跑了 6 个月、积累了数十万条 Agent 轨迹数据的 Harness 系统，是极难复制的。

**对稳定性平台的战略启示：** 你们正在建设的 Harness 体系（AI SLO、Eval 流水线、变更准入、Agent 审计），恰好处于技术栈里最难被替代的那一层。

### 9.7 Claude Code 生产级 Harness 解剖——7 个子系统

Claude Code 是目前可观察到的最完整的生产级 Harness 实现，包含 7 个子系统：

| 子系统 | 功能 | Harness 层级 |
|--------|------|-------------|
| 消息注入队列 | 在对话流中注入系统消息（环境信息/权限变更），用户不可见但模型参考 | 上下文层 |
| Prompt 组装引擎 | 根据任务阶段动态组装 Prompt，让同一模型在不同场景表现出「不同人格」 | 上下文层 |
| 工具注册表（动态） | 根据任务阶段决定哪些工具对模型可见——工具太多反而降低决策质量 | 约束层 |
| 安全系统（6 层冗余）| 文件权限/命令白名单/操作确认/沙盒隔离/git 保护/敏感信息过滤，任意 5 层失效仍能拦截 | 约束层 |
| 上下文压缩（5 阶段）| 超出 context window 时不是简单截断，而是 5 阶段智能压缩 | 上下文层 |
| 可观测性接入 | Agent 直接查询 LogQL/PromQL，通过 Chrome DevTools 验证 UI | 评估层 |
| 反馈闭环 | 「垃圾回收 Agent」定期扫描坏模式，自动发起修复 PR | 评估层 |

**「工具太多反而降低决策质量」** 的工程洞察：就像菜单有 200 道菜的餐厅，点菜效率一定不如只有 20 道菜的餐厅。工具注册表的动态可见性设计，解决了「工具泛滥」导致的 Agent 选择困难问题。
