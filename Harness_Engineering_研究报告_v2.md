# Harness Engineering 产品研究报告

**研究主题：** Harness Engineering 是否可作为 AI 时代稳定性建设的引领方法论，以及如何落地为具体工具和能力

**研究视角：** 技术风险平台 PM，负责公司全局稳定性治理，业务全面 AI 化背景下

**来源说明：** 本报告整合 11 篇 Tier 1 一手来源（AI 公司官方工程博客、官方实验报告）+ 控制论原著 + 行业综合分析，优先引用原始出处

---

## 一、核心结论（先看这里）

1. **Harness Engineering 是真实的方法论，不是概念炒作。** 有 3 个独立的可量化实验支撑，最高案例提升幅度达 36 个百分点，且模型本身未作任何修改。

2. **传统稳定性方法有结构性盲区。** AI 系统的核心失效模式（质量静默下滑、模型漂移、上下文腐化）传统监控完全无法感知，需要新的「传感器」和「执行器」。

3. **Harness Engineering 的本质是控制论。** 维纳 1948 年的框架精准描述了今天的工程挑战——负反馈、熵增对抗、稳态维持、黑箱方法，全部适用于 AI 系统可靠性设计。

4. **PM 在这个领域有不可替代的核心作用。** 不是「能写代码」，而是「把判断力变得机器可读」——定义 AI SLO、设计 Eval case 集、主导数据模型标准，这三件事工程师和算法团队都无法代劳。

5. **工具链成熟度分化明显。** Trace 采集和 Eval 评估已经成熟可用；变更准入门禁、语义级权限、Agent 审计归档是行业空白，是平台可以率先定义标准的位置。

## 二、概念起源与定义

### 2.1 词义：为什么叫「马具」

Harness（马具）——套在马身上让人能控制方向和力量的装备。AI Agent 是那匹动力十足但不守规矩的马，Harness 是让它跑得快又不跑偏的缰绳和马鞍。

### 2.2 概念起源时间线

```
2026-02-05  Mitchell Hashimoto 博客（HashiCorp 联合创始人，Terraform/Vagrant/Vault 作者）
            首次系统性提出「Harness Engineering」这个词
            原始定义：「每当你发现 Agent 犯了一个错误，你就花时间设计一个解决方案，
            使 Agent 从此永远不再犯同样的错误。」

2026-02-11  OpenAI 官方博客「Harness engineering: leveraging Codex in an agent-first world」
            引爆社区讨论，概念从个人博客升级为行业术语

2026-02-17  LangChain Blog「Improving Deep Agents with harness engineering」
            发布可量化的实验数据（52.8% → 66.5%），提供工程实践细节

2026-03-05  arXiv 2603.05344「Building AI Coding Agents for the Terminal」
            学术论文证实：仅改变 Harness，同一模型从 42% 跳到 78%
```

### 2.3 三个演进阶段，一个方向

| 阶段 | 时间 | 关注什么 | 局限 |
|------|------|---------|------|
| Prompt Engineering | 2023-2024 | 怎么写一条好指令 | 单次输入-输出，复杂任务失控 |
| Context Engineering | 2025 | 给 Agent 提供什么信息 | 好的上下文无法阻止 Agent 做不该做的事 |
| **Harness Engineering** | **2026** | **在什么样的环境里工作** | 当前仍在快速演进中 |

Anthropic（2025-09）原话：「Context Engineering 是 Prompt Engineering 的自然演进。当我们走向多轮推理、更长时间跨度的 Agent 时，需要管理整个上下文状态的策略。」

而 Harness Engineering 的出现，是因为「仅有好的上下文，Agent 依然会失控」。

### 2.4 工作定义（本报告采用）

> Harness Engineering 是为 AI Agent 设计执行环境的工程学科——通过构建上下文约束、行为边界、质量评估和反馈闭环，使非确定性的 AI 系统在生产环境中**可靠、可控、可审计**地运行。

关键区分（Anthropic 官方，2025-01-09）：
- **Agent Harness（执行 Harness）**：让模型能以 Agent 身份行动的系统——处理输入、编排工具调用、返回结果
- **Evaluation Harness（评估 Harness）**：运行 eval 的基础设施——提供指令和工具、并发运行任务、记录步骤、给输出评分、汇总结果

两者是独立的工程组件，都属于 Harness Engineering 的范畴。

## 三、控制论——为什么 Harness Engineering 在 2026 年出现是必然的

维纳 1948 年的《控制论》提供了理解这一切的底层框架。

### 3.1 同一个模式，三次出现

```
瓦特调速器（1780s）
  问题：蒸汽机需要人站旁边手动拧阀门
  解法：机械装置自动感知转速 → 调节阀门 → 负反馈闭合
  人的角色：从「拧阀门」→「设计调速器」

Kubernetes（2014+）
  问题：微服务时代，人无法手动管理数百个 Pod
  解法：声明期望状态，控制器持续感知偏差并协调
  人的角色：从「重启服务」→「编写规格说明」

Harness Engineering（2025+）
  问题：AI Agent 能写代码做决策，但「好不好」只有人能判断
  解法：工程师设计约束环境、构建质量反馈回路，Agent 在其中执行
  人的角色：从「写代码」→「设计控制系统的规则层」
```

每次这个模式出现，都是因为有人在新的层面造出了足够好的「传感器」（感知偏差）和「执行器」（纠正偏差），反馈回路才得以在那个层面闭合。LLM 是第一个能在「认知判断层面」同时充当传感器和执行器的技术。

### 3.2 七个控制论概念的 Harness Engineering 映射

| 控制论概念 | AI Agent 系统对应 |
|-----------|----------------|
| **负反馈** | Eval 发现质量偏差 → 调整 Prompt/约束 → 闭环纠正 |
| **信息是核心** | System Prompt 质量决定输出质量，不是算力 |
| **熵增** | AI 系统默认退化：Prompt 过时、模型漂移、RAG 知识库老化 |
| **稳态（Homeostasis）** | AI SLO 定义「行为质量健康范围」，维持而非静止 |
| **目的论行为** | Agent 有目标 → 稳定性必须包含「目的是否达成」 |
| **黑箱方法** | 调整外部 Harness 输入改变输出，不分析 LLM 内部 |
| **二阶控制论** | Eval case 设计必须含对抗性测试，否则 Agent 只是刷高了评分 |

**最重要的控制论推论：** AI 系统的默认状态是退化，不是稳定。没有持续投入的 Harness，就是在让系统熵增——这是为什么 Harness Engineering 必须是持续工程，而不是「做一次就行了」。

## 四、为什么传统稳定性方法不够——三个结构性失效

### 4.1 根本原因：确定性假设的崩塌

传统软件工程全套可靠性方法的隐含前提：

> **同样的输入 → 同样的输出。** 确定性。

AI Agent 的大脑是 LLM，这个假设不成立。

### 4.2 三个具体失效环节

**① 测试失效——「录制回放」不再有效**

传统测试：给定输入，断言输出，完全一致才算通过。

Agent 的推理路径（Thought）和工具调用顺序（Action）在不同运行时可能完全不同，但两次都正确达成了目标。用精确断言测试会产生大量误报；只看最终结果又太粗糙，无法发现中间走偏。

必须引入 **Trajectory Evaluation（轨迹评估）**——评估推理路径是否合理，不只是最终结果是否一致。

**② 监控失效——「活着」≠「做对了」**

传统监控能感知的：CPU 高、延迟高、5xx 升高、Pod 崩溃。

这些指标全部正常，但 Agent 可能正在系统性地给出错误推荐、走偏的推理路径、质量下滑的输出——传统监控完全无感知。

> **「服务活着」和「服务做对了」，在 AI 系统里是两件完全不同的事。**

**③ 变更管理失效——「什么叫变更」的定义变了**

传统变更：代码 commit → 触发构建 → 走发布流程。

AI 系统里三类新型变更，传统流程完全感知不到：
- **System Prompt 修改**：不在代码里，不触发 CI/CD
- **模型版本静默升级**：提供商无声更新，无任何告警
- **RAG 知识库更新**：内容变了但系统版本没变

### 4.3 反直觉的数据

Catchpoint SRE Report 2025（调研 300+ 可靠性从业者）：**AI 工具引入后，运维 toil 连续 5 年下降后首次反升，达到 30%。**

这不是「AI SRE 不管用」，而是在说：把 AI 工具叠加到旧的稳定性体系上，只会制造新的复杂度。*(来源：Catchpoint，独立研究)*

### 4.4 Context Rot——一个新发现的失效模式

**Context rot（上下文腐化）**：随着 context window 中 token 数量增加，模型准确回忆早期信息的能力下降。——Anthropic，2025-09-29

架构原因：Transformer 中每个 token 要 attend 到其他所有 token，n 个 token 产生 n² 的成对关系。context 越长，这些关系越被稀释，模型的注意力越分散。

**工程含义：** 重要约束（如「这个服务的 SLO 是 99.9%」「高风险操作必须人工审批」）不能只在会话开始时注入一次——随着 Agent 执行步骤增多，这些约束会被「遗忘」。需要通过 Middleware 在关键节点重新注入。

## 五、Harness Engineering 的核心能力体系

### 5.1 三层 Harness 架构

```
┌─────────────────────────────────────────────────────┐
│  评估层（Eval Harness）                              │
│  持续评估 Agent 输出质量，建立 AI SLO               │
│  控制论本质：质量传感器，感知与期望状态的偏差        │
├─────────────────────────────────────────────────────┤
│  约束层（Constraints Harness）                       │
│  控制 Agent 行为边界，限制失控风险                  │
│  权限管理、输出校验、人工审批、循环检测             │
│  控制论本质：执行器，施加纠正控制                   │
├─────────────────────────────────────────────────────┤
│  上下文层（Context Harness）                         │
│  System Prompt、工具描述、RAG 质量、记忆管理        │
│  信息质量决定推理质量                               │
│  控制论本质：参考输入，定义「期望状态」             │
└─────────────────────────────────────────────────────┘
```

### 5.2 Anthropic 总结的五种工作流模式

（来源：Building Effective Agents，2024-12-19）

这五种模式是构建 Harness 时的基本选择——从简单到复杂，越复杂越不确定：

| 模式 | 适合场景 | 对技术风险 SKILL 的应用 |
|------|---------|----------------------|
| **提示链（Prompt Chaining）** | 任务可被清晰拆解为固定子步骤 | 故障排查的多步推理（定位→分析→建议） |
| **路由（Routing）** | 输入类型不同，处理路径不同 | 按告警类型分发到不同 SKILL |
| **并行化（Parallelization）** | 独立子任务可同时运行 | 同时查多个数据源（指标+日志+变更记录）|
| **编排-执行（Orchestrator-Subagents）** | 复杂任务需要动态分解 | 复杂变更风险评估，主 Agent 协调多个子 Agent |
| **评估-优化（Evaluator-Optimizer）** | 质量判断需要独立视角 | Generator-Evaluator 分离，AI SLO 持续评估 |

**Anthropic 的核心建议：从最简单的解法开始，只在需要时增加复杂度。** 很多技术风险场景用确定性 Workflow 比 Agent 更可靠，不是所有场景都需要完整的 Agent。

### 5.3 Agent vs 传统在线服务：各环节实质差异

| 环节 | 传统在线服务 | Agent 系统 | 实质区别 |
|------|------------|-----------|---------|
| **什么叫变更** | 代码 commit、配置文件修改 | 代码 + Prompt + 模型版本 + RAG + 工具描述 | 变更对象扩大，大部分不在代码里 |
| **测试方法** | 给定输入，断言输出是否一致 | 功能测试 + 轨迹评估 + 质量评估 | 多了行为和质量两个维度 |
| **判断「通过」标准** | Pass/Fail，自动判断 | 功能测试通过 + Eval 评分不低于基线 | 从「对/错」变成「够不够好」 |
| **生产监控** | 5xx、延迟、Pod 崩溃 | 以上 + 持续 Eval 采样：从生产流量抽样评估质量 | 多了质量类告警 |
| **回滚** | 代码版本回滚 | 代码 + Prompt 版本 + 模型版本 + RAG 快照 | 回滚对象变多，工具链不成熟 |
| **审计内容** | 谁调了什么接口，返回什么 | 接口调用 + 推理步骤、工具调用、决策依据 | 审计粒度细化到推理过程 |
| **高风险操作控制** | RBAC 预设权限 | 权限控制 + Agent 动作实时审批 | 从「事前授权」变成「实时决策」 |

### 5.4 AI SLO——新增的质量稳态定义

**传统 SLO 衡量「服务活不活」，AI SLO 衡量「Agent 做没做对」。**

| 维度 | 传统 SLO | AI SLO |
|------|---------|--------|
| 核心问题 | 服务是否可用？是否够快？ | Agent 的输出是否正确？是否可信？ |
| 典型指标 | 可用率、P99 延迟、错误率 | 任务完成率、判断准确率、幻觉率、拒识率 |
| 数据来源 | 监控系统自动采集 | 评估器打分（LLM-as-Judge 或人工标注）|
| 违规感知 | 告警规则自动触发 | 持续 Eval 采样 → 评分下滑 → 触发告警 |
| 度量难度 | 容易，精确计算 | 困难，概率性评估 |
| 行业标准 | Google SRE Book 完整定义 | **无行业标准，各家自定义** |

**当前行业最接近共识的 AI SLO 指标（基于 RAGAS、LangWatch、Arize）：**

```
一、事实性（Faithfulness）
  定义：输出内容是否忠实于来源，不乱编
  建议目标值：≥ 90%

二、任务完成率（Task Success Rate）
  定义：Agent 完成任务、结果正确的比例
  建议目标值：核心场景 ≥ 85%，长尾场景 ≥ 70%

三、答案相关性（Answer Relevance）
  定义：回答是否与用户意图相关，没有答非所问
  建议目标值：≥ 85%

四、拒识合理性（Appropriate Refusal Rate）
  定义：越权/超范围指令时正确拒绝
  建议目标值：越权指令拒识率 ≥ 99%
```

**实践建议：** 不要从零定一个绝对值。先运行一个月建立 Baseline，再设 SLO 目标（如不低于 Baseline 的 90%），比拍脑袋定绝对值更可操作。

## 六、工程实践证据——三个独立实验

### 6.1 实验一：LangChain deepagents-cli（2026-02-17）

**来源：** LangChain Blog 官方，可复现的公开基准测试
**底层模型：** gpt-5.2-codex，全程固定

```
初始（默认 Harness）：52.8%  → Top 30 之外
最终（优化后 Harness）：66.5%  → Top 5
提升：+13.7 个百分点，模型未动一行
```

**四项关键 Harness 优化：**

**① 自验证循环**：在 System Prompt 里强制「规划→构建→验证→修复」循环；`PreCompletionChecklistMiddleware` 在 Agent 准备结束时拦截，强制执行验证步骤。

**② 环境上下文注入**：`LocalContextMiddleware` 在 Agent 启动时自动注入当前目录结构、可用工具列表、时间预算提醒，减少因信息缺失导致的失败。

**③ 循环检测（Doom Loop 防护）**：`LoopDetectionMiddleware` 跟踪每个文件的编辑次数，超过阈值 N 时注入「考虑重新审视你的解决方案」。*注：这是针对当前模型局限性的临时护栏，随模型能力提升会逐渐不需要。*

**④ 推理预算三明治**：

| 配置 | 得分 | 原因 |
|------|------|------|
| 全程 xhigh | 53.9% | 大量任务超时 |
| 全程 high | 63.6% | 平衡但未达最优 |
| **三明治（规划+验证 xhigh，执行 high）** | **66.5%** | 最优 |

**方法论贡献——Trace Analyzer Skill（用 Agent 优化 Agent）：**
```
运行 Agent → 收集 Trace（LangSmith）
       ↓
并行启动多个分析 Agent，识别失败模式
       ↓
主 Agent 汇总建议 → 人工复核（可选）
       ↓
修改 Harness → 重新运行验证
```

这个「Boosting 式」循环把 Harness 改进从「人工经验积累」变成了「可重复执行的工程流程」。

**警示：过拟合风险。** 只针对当前失败模式修改 Harness，可能导致其他场景性能退化。Eval case 集必须覆盖多样场景。

### 6.2 实验二：CORE-Bench（arXiv 2603.05344，2026-03-05）

**来源：** arXiv 学术论文，首篇开源终端原生编码 Agent 综合技术报告
**底层模型：** Claude Opus 4.5，固定不变

```
标准 Harness：42%
优化后 Harness（调整约束与上下文可见性）：78%
提升：+36 个百分点，几乎翻倍
```

LinkedIn 评论精准总结：「Same model. Same benchmark. 42% vs 78%. The only thing that changed was what the model could see.」

**核心启示：** 「模型能看见什么」比「模型有多聪明」更决定结果。这正是维纳「信息是控制的本质」的工程验证。

### 6.3 实验三：OpenAI Codex 内部实验（2026-02-11）

**来源：** OpenAI 官方博客（中英文版均已发布）
**规模：**
- 工程师：3 人起步 → 最终 7 人
- 时间：五个月（2025-08 至 2026-01）
- 产出：约 100 万行代码，零行人工手写
- 效率：平均每人每日 3.5 个 PR 合并，约为传统效率 10 倍
- 产品状态：已有数百名内测用户（含日常高活用户）

**那些工程师在做什么——不是写代码，是建造让 Agent 可靠工作的环境：**

**① 文档工程——渐进式披露（Progressive Disclosure）**

初期错误：把所有信息堆进一个大 AGENTS.md 文件。四个致命问题（原文）：
- 上下文是稀缺资源：庞大文件挤掉任务和代码，Agent 错过关键约束
- 过多指导变得无效：当一切都「重要」，一切都不重要，Agent 只是本地模式匹配
- 立即腐烂：庞杂手册变成陈旧规则坟场，无法判断哪些仍然有效
- 难以核实：单个 blob 不适合机械检查

最终方案：
```
AGENTS.md（目录，约 100 行）
docs/
  ARCHITECTURE.md   ← 域和包分层的顶层地图
  DESIGN.md         ← 核心设计原则，含验证状态
  PLANS.md          ← 执行计划（全部版本控制）
  PRODUCT_SENSE.md  ← 产品意图与用户旅程
  QUALITY_SCORE.md  ← 各领域质量评分，追踪差距
  RELIABILITY.md    ← 可靠性要求
  SECURITY.md       ← 安全约束
```

核心原则：**「从 Agent 的角度来看，它在运行时无法访问的任何内容都不存在。」** 存储在聊天记录或人脑中的知识，系统无法访问。

**② 让 Agent「看见」运行时**

- 每个 git worktree 有独立的本地可观测性栈实例
- Agent 可用 LogQL 查日志，PromQL 查指标
- 接入 Chrome DevTools Protocol——Agent 可操作浏览器，复现 Bug、验证修复

效果：「确保服务启动在 800ms 内完成」这样的提示词变得可行。

**③ 自定义 Lint + 黄金原则**

初期：每周五花 20% 时间清理「AI 垃圾代码」→ 不可扩展

最终方案：把代码规范直接编码进 Lint 规则，Lint 错误信息里直接写着修复指令，Agent 读到报错就知道怎么改。

> 「在以人为本的工作流程中，这些规则可能会让人感到迂腐。有了智能体，它们就成了**倍增器**：一旦编码，就能立即应用于所有地方。」

**④ 垃圾回收 Agent——持续反熵**

定期后台任务扫描代码库偏差、更新质量评分、发起有针对性的重构 PR。大多数 PR 一分钟内完成审查并自动合并。

> 「技术债务就像高息贷款：不断以小额方式偿还，总比让债务累积再痛苦一次解决要好。**人类的品味一旦被捕捉，就会持续应用于每一行代码。**」

**最重要的一句引用**（工程师 Ryan Lopopolo）：
> 「我们当前最棘手的挑战集中在**设计环境、反馈回路和控制系统**方面。」

## 七、Anthropic 工程博客系列——更深的技术细节

Anthropic 是目前发表最多 Harness Engineering 原创技术文章的公司，形成了完整的工程知识体系。

### 7.1 长任务 Harness 的两段式架构（2025-11-26）

**核心问题：** 跨多个 context window 的长任务，Agent 如何保持连贯性？

**两个典型失败模式：**
1. **一次性完成陷阱**：Agent 试图一口气干完，context 耗尽时任务只完成一半，下一个 Agent 不知道发生了什么
2. **提前认为完工陷阱**：后续 Agent 看到前面的进度，认为任务已完成，直接停止

**解法——两段式架构：**
```
初始化 Agent（第一个 context window 专用）
├── 创建 init.sh 初始化脚本
├── 创建 claude-progress.txt（唯一跨会话记忆）
└── 初始 git commit 留痕

编码 Agent（每个后续 context window）
├── 读取 claude-progress.txt 理解当前状态
├── 增量推进任务（feature by feature）
└── 留下干净的结构化更新（可以直接 merge 的状态）
```

**核心洞察：** 让每个 Agent 像优秀的软件工程师一样「交班」——离开时代码整洁、有文档、下一个人能立刻接手。

### 7.2 Context Anxiety 与 Context Reset（2026-03-24）

**新发现的失效模式——Context Anxiety（上下文焦虑）：**

> Agent 随着 context window 接近上限，会**提前开始收尾**，不是因为任务完成了，而是它预感自己快没空间了。

Compaction（压缩旧对话）无法解决这个问题——即使压缩了，Agent 仍然有「快撑不住了」的焦虑感。

**唯一有效解法：Context Reset**——完全清空 context，启动全新 Agent，通过结构化「交班文档」传递状态。

对技术风险 Agent 的直接含义：故障排查 Agent 在处理复杂故障时，如果没有 Reset 机制，它可能在任务未完成时提前「认为排查完了」给出不完整结论。这比 Doom Loop 更隐蔽、更危险。

### 7.3 Generator-Evaluator 分离（2026-03-24）

**新发现的失效模式——Self-Evaluation Bias（自评偏差）：**

> 让 Agent 评估自己生成的东西，它会系统性地高估质量——即使人类一眼就看出是垃圾，Agent 也会说「这很好」。

**解法：Generator + Evaluator 分离架构（受 GAN 启发）**

```
Generator Agent（生成者）
       ↓ 输出
Evaluator Agent（独立调校为「挑剔模式」）
       ↓ 具体批评
Generator Agent（根据批评迭代）
（循环 5-15 次）
```

关键：Evaluator 必须被**专门调校为怀疑论者**。把主观判断转换为可打分的具体维度（如：事实性/相关性/任务完成率），让评估有标准可依。

**对技术风险的直接应用：** 变更风险评估 Agent 不应该自己评价自己的判断质量——需要独立的 Evaluator 复核，这是把「主观判断变得可打分」的工程实现，也是 AI SLO 监控的底层机制。

### 7.4 Eval 体系设计（2025-01-09）

**Anthropic 的核心 Eval 概念定义（值得直接采用）：**

- **Task（任务）**：单个测试用例，包含输入和成功标准
- **Trial（试次）**：对一个任务的一次尝试（因为模型输出有随机性，需要多次 trial 才能稳定评估）
- **Grader（评估器）**：评分逻辑，一个任务可以有多个 grader
- **Transcript（轨迹）**：trial 的完整记录，包含输出、工具调用、推理步骤、中间结果
- **Outcome（结果）**：trial 结束时环境的最终状态（不是 Agent 说「完成了」，而是数据库里真的有记录）
- **Evaluation Harness**：运行 eval 的整套基础设施

**为什么 Eval 要早建：**

> 「团队在早期靠手动测试和直觉可以走得很远。但一旦 Agent 进入生产并开始扩展，不做 eval 就开始崩了。破点往往是：用户反馈 Agent 感觉变差了，但团队没有任何方式来验证，只能猜测和检查。」

## 八、OpenAI Codex 系列——Harness 作为平台架构

OpenAI 发布了三篇 Harness 系列文章，揭示了 Harness 的平台化演进路径。

### 8.1 三篇系列文章

| 文章 | 日期 | 核心内容 |
|------|------|---------|
| Unrolling the Codex agent loop | 2026-01-23 | Agent Loop 技术深度：模型+工具+提示的编排细节 |
| **Unlocking the Codex harness（App Server）** | **2026-02-04** | **Harness 平台化：多客户端共享同一 Harness** |
| Harness engineering | 2026-02-11 | 百万行代码实验（已在第六章详述）|

### 8.2 Harness 平台化——最重要的架构洞察

App Server 的设计思路：Codex 的 CLI、VS Code 插件、macOS 应用、JetBrains 插件——底层用的是**同一个 Harness**，通过 App Server（JSON-RPC 协议）暴露给不同客户端。

**架构洞察：** Harness 不只是单个 Agent 的运行环境，可以是一个平台服务：

```
Harness 平台服务
├── 工具注册表（统一管理，动态可见性）
├── 权限策略（统一执行）
├── 审批机制（高风险动作实时确认）
├── 上下文管理（跨会话持久化）
└── 线程/会话管理（可创建/恢复/分叉/归档）
    ↕
多个 Agent / 多个客户端共享以上能力
```

**对技术风险平台的直接意义：** 把平台的 SKILL 能力层设计成「Harness 服务」——不是每个 Agent 自带一套工具和约束，而是统一的平台 Harness 提供标准化的能力层，多个业务线的 Agent 都从这里获取能力。这是「技术风险平台 Agent-Native 化」的正确架构方向。

### 8.3 三个核心协议原语

App Server 定义的会话原语，可直接用于 AgentActionRecord 数据模型设计：

| 原语 | 定义 | 对应 AgentActionRecord |
|------|------|----------------------|
| **Item** | 原子输入/输出单元，有类型（消息/工具执行/审批请求/diff），有完整生命周期 | 单个动作记录 |
| **Turn** | 一次 Agent 工作单元（用户输入 → Agent 完成） | 一次任务执行记录 |
| **Thread** | 持久化的 Agent 会话容器（可创建/恢复/分叉/归档）| 完整对话/工单记录 |

## 九、工具生态全景

| 层级 | 要做的事 | 代表工具 | 成熟度 |
|------|---------|---------|--------|
| **Layer 0 基础可观测** | 记录 Agent 完整推理链 | Langfuse / LangSmith / OTel GenAI | ✅ 成熟 |

| **Layer 2 质量评估** | Eval case 集 + 评估器 + 接入变更准入 | Braintrust / LangSmith Evals / RAGAS | ✅ 成熟 |
| **Layer 3 变更准入** | Prompt/模型升级前 Eval 基线比对，不退化才上线 | 无专用产品，各家自建 | 🔴 空白 |
| **Layer 4 权限与约束** | 最小权限 + 高风险动作实时审批 + 输出结构化校验 | 无成熟标准 | 🔴 空白 |

## 十、组织与角色演变

### 10.1 类比云原生转型

| 维度 | 云原生（2014-2020）| AI 全面化（2025-）|
|------|-----------------|----------------|
| 旧方法失效 | ITIL 人工变更审批跟不上发布速度 | 传统监控无法检测「系统健康但输出错误」|
| 新方法论 | SRE（SLO + 错误预算 + 自动化）| Harness Engineering（Eval + 约束 + AI SLO）|
| 人的角色 | 从「运维机器」→「设计可靠系统的工程师」| 从「响应故障」→「设计可靠 AI 系统 + 定义 AI 行为边界」|

**三个更根本的不同：**
1. **故障不可见性更高**：云原生故障（Pod 崩溃）可被监控感知；AI 系统质量问题可以在系统指标完全正常时发生
2. **整个生产链都在 AI 化**：不只是运维工具，需求/设计/开发/测试/部署整条链路都被 AI 接管
3. **需要同时管两套 SLO**：系统 SLO（可用率/延迟）+ AI SLO（任务完成率/幻觉率），两者都不能缺

### 10.2 PM 的不可替代价值

**「生成-验证不对称性」（P vs NP 的直觉）：** 生成一个正确的解比验证一个解要难。

你不需要在实现能力上超越机器，你需要在**评判能力**上超越它：

| PM 核心工作 | 具体内容 | 为什么 PM 是不可替代的 |
|-----------|---------|---------------------|
| 定义「正确」 | AI SLO 定义、Eval 标准设计 | 工程师不理解业务语义，无法定义「对」的标准 |
| 识别「哪里不对」 | Eval case 集设计（含对抗性测试）| 算法团队不关心产品质量定义 |
| 判断「方向是否正确」| Harness 产品设计、变更准入标准 | 需要同时理解业务目标和技术约束 |

## 十一、对技术风险平台的落地建议

### 11.1 三个核心对齐

**与 AI 平台合作策略：**
- 不争「谁主导建设」，争「谁定标准 + 谁是数据汇聚中心」
- 找 AI 平台：订阅 Trace 数据，不增加对方建设量
- 找安审：提供已结构化的审计数据，对方不需要读 AI 链路原始数据
- **主导 AgentActionRecord 数据模型定义 = 掌握话语权**

**SKILL 设计原则（来自 Claude Code 工具动态注册表的启示）：**
- 工具太多反而降低决策质量——少而精，任务视角而非功能视角
- 不同任务阶段只暴露相关工具，而不是把所有 SKILL 一股脑给 Agent
- 参考 Copilot 从 40 个工具缩减到 13 个后性能提升的案例

**SKILL 文档设计原则（来自 OpenAI AGENTS.md 教训）：**
- 「Agent 看不到的等于不存在」——Runbook、故障处置规范必须结构化存储
- 渐进式披露：SKILL 描述精简为「目录」，具体规范在子文档
- 文档要可以机械检查（覆盖率、新鲜度、所有权），否则会腐烂

### 11.2 分阶段落地路径

```
阶段一（Q2 2026）：建立质量感知能力
  ├── 接入 Trace 采集（Layer 0）——知道 Agent 做了什么
  ├── 建立 3 个核心场景的 Eval case 集——能评估 Agent 做对没有
  ├── 定义 AI SLO 指标和 Baseline——知道「好」是什么标准
  └── 首批 SKILL 上线配套 Eval 流水线

阶段二（Q3 2026）：建立变更控制能力
  ├── Prompt 版本管理接入（Layer 1）
  ├── Eval-gated 变更准入（Layer 3）——质量不退化才能上线
  ├── AI SLO 持续监控（Layer 5）——质量下滑自动告警
  └── Agent 高风险动作审批流（Layer 4 初版）

阶段三（Q4 2026）：建立审计与治理能力
  ├── AgentActionRecord 数据模型落地（Layer 6）
  ├── 与 AI 平台打通链路数据
  ├── 与安审建立数据共享机制
  └── AI SLO Dashboard 上线（双视图：工程师 + PM）
```

### 11.3 复杂度警示

**不要低估以下几件事的难度：**

1. **Eval case 集的建设**：不是「写几个测试用例」。需要覆盖核心场景 + 边界场景 + 对抗性场景，还要避免过拟合（只测「熟悉的失败模式」）。这是一个持续维护的工程，而不是一次性交付物。

2. **LLM-as-Judge 的可靠性**：用 LLM 评估 LLM 的输出，评估器本身也可能有偏差（自评偏差问题）。需要通过 few-shot 标注例子校准评估器，并定期用人工抽样验证评估器的准确性。

3. **AI SLO 的度量成本**：传统 SLO 用 Prometheus 自动采集，几乎零成本。AI SLO 的每次评估都要调用 LLM-as-Judge（成本）或人工标注（时间）。这是为什么行业普遍采用「抽样评估」而非「全量评估」。

4. **Harness 随模型升级的维护**：模型能力提升后，原来的 Harness 护栏可能变成性能瓶颈。LangChain 的 LoopDetection 中间件就明确说了「这是针对当前模型的临时护栏」。需要建立定期 Harness 审查机制。

## 十二、核心结论

**Harness Engineering 值得作为引领 AI 稳定性建设的方法论，理由：**

1. **有历史规律支撑**：控制论 1948 年的框架完美解释了这个演变；瓦特调速器和 K8S 已经证明了这个模式的有效性

2. **有工程实证**：3 个独立可量化实验，提升幅度 13.7~36 个百分点，模型均未修改

3. **有行业一手来源**：OpenAI + Anthropic + LangChain 三家 AI 公司均发布了详细的工程实践报告，不是媒体分析，是工程师写的第一手经验

4. **有明确的空白地带**：Layer 3/4/6（变更准入/语义权限/审计归档）是行业空白，是技术风险平台可以率先定义标准的位置

5. **与现有稳定性工作的关系是「升维」而非「替代」**：传统 SRE 体系继续运作，Harness Engineering 是在其上新增「AI 行为质量」这个维度的治理

**Harness Engineering 里真正新的东西，只有两点：**
- **质量维度的新增**：「系统活不活」→ 新增「Agent 做没做对」
- **变更对象的扩大**：「代码变更」→ 扩展到 Prompt + 模型 + RAG + 工具描述

其余（权限管理、回滚、审计）不是新概念，但在 Agent 系统里需要重新定义对象范围和判断粒度。
