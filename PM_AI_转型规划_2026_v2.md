# 中台可观测性 PM 的 AI 时代转型规划
**作者：** 产品经理（可观测性平台 & 稳定性专项）
**时间：** 2026 年 Q1（v2，2026-03-23 更新）
**背景：** 组织推动全面 AI 化，PM 角色面临重构，寻求从"协同桥梁"向"AI 转型探路者"转变

---

## 一、背景与判断

### 1.1 行业正在发生什么

过去 12 个月，AI 对研发组织的冲击已从"工具提效"升级到"角色重构"，背后有几股清晰的底层力量：

- **研发链路在缩短**：AI 编码与需求生成，让"需求—设计—开发"的链路越来越短。Meta、Shopify 等头部公司已开始推行 PM vibe coding，要求 PM 能自己原型化产品并直接向高层汇报（Pendo, 2025）。原本大量依赖产品经理做的解释、拆解、转述、跟进，正在被更直接的协作方式替代
- **稳定性工作正在产品化**：SRE 借助 AI，把经验沉淀成 skill、runbook、agent、诊断流程。稳定性不再只是一组专项会议和推动动作，而越来越像标准化策略、自动化诊断、规则编排、风险评估面板、故障处置闭环。AIOps 从概念走向落地，AI 驱动的故障诊断、变更评估正在替代大量手工操作（Gartner 2025 Hype Cycle for SRE）
- **代码质量隐忧浮现**：CMU 2025 年研究对 807 个使用 Cursor 的开源项目分析发现，AI 编码后静态分析告警上升 ~30%，代码复杂度上升 40%+；CodeRabbit 研究显示 AI 生成代码引入 bug 数量是人工代码的 1.7 倍
- **组织默认逻辑正在改写**：Shopify 在 2025 年公开提出，团队申请新增人手和资源前，必须先证明为什么 AI 做不了。组织正在默认把 AI 放进岗位边界的第一道筛子——任何岗位，如果不能说清楚自己在 AI 时代的新增控制权与新增责任边界，就会自然滑向被压缩

### 1.2 组织正在发生什么

- 老板希望研发/SRE 全面拥抱 AI，实现模块级自闭环。研发角色全栈化，后端做前端、前端补后端；SRE 通过 AI 生成 skill 自闭环业务稳定性
- 产品经理原有的"需求桥梁"和"协同枢纽"职责被逐步分解——如果价值主要来自"把话说清楚"，价值会下降；如果价值来自"定义正确的问题、定义约束和指标"，价值反而会上升
- 老板期望产品经理：**①改变现有产研迭代模式，释放人力；②探索未知领域，为组织创造新的价值增量**

### 1.3 核心判断：旧桥梁消失，新控制面出现

> **旧桥梁会消失，但新桥梁不是"沟通桥梁"，而是"控制桥梁"。**

这些新的控制面问题，既不是纯研发问题，也不是纯 SRE 问题，更不是纯业务问题。它们正好落在**平台产品 + 稳定性认知 + 组织协同 + 机制设计**的交叉点——这就是今年最该深耕的位置。

**PM 被 AI 真正替代的，只是"信息传递"功能，不是"问题定义"和"价值判断"能力。当执行层变得更快、更廉价，占据控制面的人反而更值钱。**

## 二、转型三大方向

### 方向一：PM → Agent 工程师 —— 需求自闭环能力建设

**核心逻辑：** 用 vibe coding 替代画图和写文档去验证想法——旧流程一个需求从想法到验证需要数周，新流程是先快速跑出多个原型、筛选最优解再推进。这不是为了"成为会写代码的产品经理"，而是为了亲身走完迭代全流程，建立对 AI Native 研发规范的认知和说服力。PM 怎么写代码都写不过研发，但 PM 用代码快速验证想法，是研发替代不了的。

**目标：**
- 能独立用 AI 工具快速原型化想法，替代文档和原型图作为验证手段
- 优先消化长期积压的"研发不愿接、用户一直提"的存量需求（如大盘配置优化、告警规则模板），既有价值又不抢研发核心工作
- 释放出的研发资源聚焦在数据层、Agent/SKILL 层等高价值攻坚上

**量化指标：**
- 自闭环需求数量（过程指标）
- 释放的研发工时（结果指标，目标 Q2 末可量化）
- 用户满意度/产品体验提升（最终指标）

**执行要点：**

| 阶段 | 时间 | 动作 |
|------|------|------|
| 学习期 | M1（70% 精力） | 选 Cursor 等工具，聚焦 1-2 类需求（配置类、表单类），跑通完整链路 |
| 磨合期 | M2-M3（40% 精力） | 建立 prompt 库和代码模板库，提高复用效率 |
| 常态化 | M4+（20-30% 精力） | 自闭环成为日常验证方式，重心转移到方向二和方向三 |

**⚠️ 几个关键边界：**
- **范围边界：** 用户层功能可以闭环，核心数据层/引擎层不要碰，避免污染核心模块；先在内部工具小范围试跑，再给用户用
- **协作边界：** 提前跟研发同步，明确你是在"补位"而不是"抢活"——做的是他们不愿意接的，不是他们正在做的
- **认知边界：** 不要给人留下"会写代码的产品经理"的印象，这一步是为了更深入理解迭代全流程、为后续定义 AI Native 规范打基础

### 方向二：AI Native 迭代新范式 —— 定义团队的 AI 工作方式

**核心逻辑：** 当研发、SRE、PM 都开始用 AI 做事，最大的风险不是没人用，而是各自乱用。职责边界模糊、SKILL 野蛮生长、系统迭代质量失控，这些都是 AI 全面普及后必然出现的挑战。PM 最适合在这个时候站出来，定义"AI 时代团队怎么协作、怎么保证质量不失控"——这是产品思维在新范式下的天然延伸。

#### 2.1 做团队的"AI 氛围官"

这是推动方向二落地的个人行为抓手，不是独立目标，而是建立信任和影响力的前置动作——只有持续输出有价值的判断，才有资格牵头制定规范。

具体做法：
- **信息过滤与判断：** 持续跟踪 AI 工具和前沿方向（Agent、MCP、RAG、Harness Engineering 等），将海量信息过滤提炼，判断哪些值得团队跟进、哪些可以忽略，输出有决策价值的结论而非信息堆砌
- **认知基线建设：** 定期输出 AI 工具月报或内部分享，从工具使用、方法论、行业案例等维度帮团队建立统一认知基线，减少因信息不对称导致的重复探索
- **氛围带动与实践沉淀：** 主动发现和推广团队内的 AI 实践亮点，让好的探索被看见、被复制；识别团队 AI 使用薄弱环节，定向推动补强

#### 2.2 AI 时代各角色分工重定义

AI 让每个角色都能做更多事，但"能做"不等于"应该做"。需要清晰定义各角色在 AI 时代的主导边界和价值重心：

| 角色 | AI 赋能后能做什么 | 应该主导什么 | 从而释放什么 |
|------|----------------|------------|------------|
| PM | 原型验证、需求自闭环 | 方向定义、规范制定、价值判断 | 研发从需求解释中解放 |
| 前端 | 产品模块自闭环、AI 交互探索 | 体验创新、AI 交互新范式定义 | 从重复 UI 执行中解放 |
| 后端 | 技术模块闭环、Agent/SKILL 层创新 | 数据层与技术能力建设 | 从前端补位工作中解放 |
| SRE | 业务稳定性自闭环、稳定性策略编排 | 稳定性产品化设计与 SKILL 治理 | 从重复巡检响应中解放 |

这个分工图的核心价值不是限制边界，而是让每个角色的 AI 化往更高价值的方向聚焦，而不是各自往全栈方向无序蔓延。

#### 2.3 AI Native 全流程规范（含 SKILL 生态标准）

> **本节聚焦研发流程侧：什么 SKILL 值得做、怎么做好、上线前怎么验收。** SKILL 上线后的运行期可靠性治理，见方向三 3.3。

**为什么要做？**

AI 让大家都能调研、设计、写代码、孵化 SKILL，职责边界模糊带来两个风险：
- **系统风险：** AI 生成代码大量涌入，复杂度和质量难以把控，核心模块存在被污染的风险
- **生态风险：** SKILL 野蛮生长，没有准入标准，后续维护成本极高，形成"SKILL 灾难"

**做什么？**

两个层面：

① **研发全流程规范**：将调研、需求、设计、开发、上线每个环节的 AI 使用方式和输出质量标准化，做成可复用的 SKILL 嵌入流程

```
调研 → 需求 → 设计 → 开发 → 上线
 ↓      ↓      ↓      ↓      ↓
各环节对应 SKILL + 质量门禁标准
```

② **SKILL 生态标准**：制定 SKILL 准入准出机制和价值衡量标准，避免各自为战
- **准入：** 什么样的 SKILL 值得做（场景频次、价值可量化、可复用性）
- **准出：** SKILL 上线前的质量验收标准（功能完整性、边界清晰度、文档规范性）
- **度量：** SKILL 上线后如何持续衡量价值（调用量、问题拦截数、人工替代率）

**PM 的角色是什么？**

PM 不是独自制定规范，而是牵头联合研发、SRE 共同建立。PM 的独特贡献在于：定义"什么是好的输出"、设计度量体系、推动跨角色对齐——这些正是研发和 SRE 不擅长、也不会主动去做的事。

**分阶段落地：**
- **试点期（Q2）：** 选 1-2 个高频场景，跑通规范 + SKILL 样板的完整链路，验证可行性
- **团队推行（Q3）：** AI Native 规范 1.0 在团队内全面推行，积累优化
- **向外扩展（Q4）：** 整理方法论，探索向其他团队或公司层面输出

### 方向三：系统性风险的主动治理 —— 全局稳定性建设

**核心逻辑：** SRE 深入业务线，研发聚焦 SKILL 和技术模块，全局稳定性视角天然缺失——没有人从全局角度去看跨业务的风险、跨系统的依赖、以及 AI 化本身带来的新型稳定性挑战。这个位置正好落在平台产品 + 稳定性认知的交叉点，是 PM 可以主导的空间。

方向三分为两个层次：**A. 全局稳定性治理**（面向所有系统，不限 AI/非 AI）和 **B. AI 系统稳定性**（面向 Agent/SKILL 的新命题）。两者共享同一个核心资产——可观测性平台——这是 PM 能同时主导两个层次的独特优势。

---

#### A. 全局稳定性治理

##### 3.1 稳定性规范 SKILL 化 + 左移（Reliability Shift Left）

**现状问题：** 稳定性规范依赖人工评审，覆盖不全、响应滞后。

**目标状态：**

```
传统模式：变更上线 → 出现问题 → 故障复盘 → 完善规范（滞后）
新模式：  代码提交 → SKILL 自动扫描 → 拦截风险 → 阻断或告警（前置）
```

**落地路径：**
1. **规范梳理：** 整理现有稳定性规范，按标准化程度和触发频次分级
2. **样板先行：** 选 1-2 个高频且边界清晰的规范，跑通"规范→SKILL"转化路径
3. **流程嵌入：** 将 SKILL 嵌入 CI/CD 流水线，实现自动触发和拦截
4. **度量闭环：** 建立拦截数量、误报率、问题减少率等指标，持续优化
5. **逐步扩展：** 基于样板经验，按优先级覆盖更多规范场景

参考 DevSecOps "Security Shift Left" 的成熟路径，核心差异在于治理对象从安全漏洞扩展到稳定性风险。

##### 3.2 全局稳定性投入空洞

**问题本质：** AI 热潮下存在一个结构性矛盾——业务线资源向 AI 创新倾斜，稳定性投入被压缩；而 AI 编码和快速迭代带来的系统复杂度却在上升。投入减少与风险上升同步发生，但缺乏客观数据让这一矛盾显性化。

**为何以往推动方式失效：** 过去靠人工评审、专项会议、关系协调推动稳定性投入，本质是"人推人"——在团队全力投入 AI 创新的背景下更难为继。

**应对策略：** 从"人驱动"升级为"数据驱动"——用 AI 自动采集和分析全局稳定性数据，让风险客观可见，用数据替代人情施压。

**具体建设方向：**
- **全局容灾覆盖：** 自动扫描各业务线容灾能力，生成覆盖率报告，建立横向对比
- **架构依赖风险：** 持续梳理跨系统依赖，识别单点和强依赖风险，自动评级推送
- **可观测性盲区：** 度量监控覆盖缺失服务，建立接入标准，推动系统性补全
- **投入与风险关联分析：** 量化稳定性投入 vs 风险暴露的关联，为资源决策提供依据

#### B. AI 系统稳定性

##### 3.3 Skills Reliability Engineering —— SRE 的新命题

**背景：** 未来平台逐渐变薄，大家越来越多通过 AI 对话和自动执行 SKILL 完成任务。治理对象从"服务和系统"扩展到"SKILL 和 Agent"，传统稳定性体系不覆盖这个层面。

**新的稳定性命题：** 当公司核心业务流程越来越多由 SKILL 和 Agent 驱动，它们的可靠性、安全性、可控性就变成了新的稳定性议题——SKILL 质量参差不齐、Agent 执行链路不透明、AI 决策边界难以约束，这些风险在传统 SRE 体系里没有对应的解法。

**方法论支撑——Harness Engineering：** 行业已形成「给 AI Agent 构建运行约束系统」的工程学科。其核心三层能力与 Skills RE 的建设方向高度对应：

| Harness Engineering 能力层 | Skills RE 对应建设内容 |
|--------------------------|----------------------|
| **信息供给层**（Context Engineering）| Agent 执行前的上下文规范：AGENTS.md、业务文档、API 契约标准化 |
| **架构约束层**（Architectural Constraints）| SKILL 准入门禁：依赖规则、权限边界、自动化合规检查嵌入 CI/CD |
| **熵管理层**（Entropy Management）| SKILL/Agent 生命周期治理：文档与行为一致性扫描、执行成功率衰减监控、下线机制 |

> **与方向二 2.3 的分工：** 2.3 SKILL 生态标准关注**研发流程侧**（上线前，什么 SKILL 值得做、怎么做好）；3.3 Skills RE 关注**运行期可靠性**（上线后，SKILL 和 Agent 如何持续稳定运行）。

**需要建立的新能力：**
- **SKILL 可靠性：** SKILL 的质量准入、执行成功率监控、异常熔断机制
- **Agent 链路可观测：** Agent 决策路径、工具调用链、执行结果的全链路追踪
- **安全与权限管控：** AI 执行动作的权限边界定义、敏感数据访问控制、异常行为识别
- **生命周期治理：** 建立 SKILL 和 Agent 从孵化到下线的全流程规范，防止「认知熵」积累

**PM 的角色：** 联合 SRE 共同定义这套新体系的产品形态——哪些能力需要平台支撑、哪些通过规范约束、哪些需要自动化监控。这是行业都在摸索的方向，先跑通就是先发优势。

##### 3.4 Agent SLO —— 从基础设施层到认知层

**为什么 Uptime 在 AI 时代是残缺的度量：** AI Agent 引入了一类全新故障——**系统正常运行，但认知错误**。Agent 可以在 200ms 内响应、资源消耗正常、服务完全在线，同时输出错误的答案、执行越权的操作、或在长链路任务的第 300 步发生逻辑漂移。传统 SLO 对这类故障完全盲目。

**Agent 四层度量框架：**

```
Layer 4: 业务结果层 ── 任务完成率、结果准确率、用户满意度
         ↑ 建立在
Layer 3: 认知行为层 ── 指令遵循率、逻辑一致性、工具调用成功率、幻觉率
         ↑ 建立在
Layer 2: 链路追踪层 ── Trace 完整性、决策路径可追溯性、工具调用链覆盖率
         ↑ 建立在
Layer 1: 基础设施层 ── Uptime、Latency、Error Rate（传统 SRE 已覆盖）
```

目前大多数团队只有 Layer 1，Layer 2 刚起步，Layer 3/4 几乎空白。**可观测性平台已覆盖 Layer 1，向上扩展 Layer 2 是最直接的机会，定义 Layer 3/4 的度量标准是 PM 能主导的蓝海。**

**Agent 分级与对应稳定性要求：**

| 自主程度 | 典型场景 | SLO 重心 | 门禁要求 |
|---------|---------|---------|---------|
| L1 辅助型 | Copilot、代码补全 | 延迟、可用性 | 基础功能验收 |
| L2 工作流自动化 | 表单填写、数据处理 | 任务完成率、准确率 | 结果抽样评估 |
| L3 多 Agent 协同 | 跨系统任务编排 | 链路完整性、交接成功率 | 全链路 Trace + 回放测试 |
| L4 高度自主 | 自主决策、写入生产数据 | 指令遵循率、越权率 | 人工审核 + 熔断机制 |

**Trace 规范是基础：** 没有完整的 Trace，Layer 2-4 的度量都是空谈。PM 应推动建立统一的 Agent Trace 规范（参考 OpenTelemetry GenAI 语义约定），让可观测性平台成为所有 Agent 执行数据的汇聚点，进而支撑 Eval 闭环：

```
Agent 执行 → Trace 采集 → 数据集构建 → 质量评分 → 发布门禁决策
```

**PM 的角色：** 定义 Agent 分级标准和对应的 SLO 指标体系；推动 Trace 规范落地；联合 SRE 设计发布门禁机制——这套标准目前没有现成答案，先定义者拥有话语权。

## 三、整体规划节奏

```
2026 Q2（4-6月）：打地基
  ├── 个人：vibe coding 学习，聚焦 1-2 类需求跑通自闭环链路
  ├── 团队：输出第一份 AI 工具月报，建立 AI 氛围官角色
```

## 四、你的不可替代性在哪里

> **当所有人都在用 AI 加速执行的时候，你是那个在问"我们应该跑向哪里，以及跑起来之后不会翻车"的人。**

1. **Agent 工程师：** PM 自己能交付，不再只是提需求，同时建立对全流程的认知
2. **AI 氛围官：** 判断和引导团队 AI 演进方向，把信息转化为决策，稀缺认知
3. **规范守门人：** SKILL 孵化的准入准出（研发流程侧）+ AI 全流程的质量标准，没有产品把关就会失控
4. **全局稳定性视角：** 在所有人聚焦业务线和 AI 创新的时候，你在看全局——既治理传统系统的稳定性投入空洞，又定义 Agent/SKILL 运行期的新型可靠性标准
