# 技术风险 PM 的 AI 时代转型规划

**作者：** 产品经理（技术风险平台群）
**时间：** 2026 年 Q1（v4，2026-03-23 更新）
**背景：** 组织推动全面 AI 化，PM 角色面临重构，寻求从"协同桥梁"向"AI 转型探路者"转变

---

# 一、背景与判断

### 1.1 行业正在发生什么

过去 12 个月，AI 对研发组织的冲击已从"工具提效"升级到"角色重构"，背后有几股清晰的底层力量：

- **研发链路在缩短**：AI 编码与需求生成，让"需求—设计—开发"的链路越来越短。Meta、Shopify 等头部公司已开始推行 PM vibe coding，要求 PM 能自己原型化产品并直接向高层汇报（Pendo, 2025）。原本大量依赖产品经理做的解释、拆解、转述、跟进，正在被更直接的协作方式替代
- **稳定性工作正在产品化**：SRE 借助 AI，把经验沉淀成 skill、runbook、agent、诊断流程。稳定性不再只是一组专项会议和推动动作，而越来越像标准化策略、自动化诊断、规则编排、风险评估面板、故障处置闭环。AIOps 从概念走向落地，AI 驱动的故障诊断、变更评估正在替代大量手工操作（Gartner 2025 Hype Cycle for SRE）
- **代码质量隐忧浮现**：CMU 2025 年研究对 807 个使用 Cursor 的开源项目分析发现，AI 编码后静态分析告警上升 ~30%，代码复杂度上升 40%+；CodeRabbit 研究显示 AI 生成代码引入 bug 数量是人工代码的 1.7 倍
- **组织默认逻辑正在改写**：Shopify 在 2025 年公开提出，团队申请新增人手和资源前，必须先证明为什么 AI 做不了。组织正在默认把 AI 放进岗位边界的第一道筛子——任何岗位，如果不能说清楚自己在 AI 时代的新增控制权与新增责任边界，就会自然滑向被压缩

### 1.2 组织正在发生什么

- 老板希望研发/SRE 全面拥抱 AI，实现模块级自闭环。研发角色全栈化，后端做前端、前端补后端；SRE 通过 AI 生成 skill 自闭环业务稳定性
- 产品经理原有的"需求桥梁"和"协同枢纽"职责被逐步分解——如果价值主要来自"把话说清楚"，价值会下降；如果价值来自"定义正确的问题、定义约束和指标"，价值反而会上升
- 老板期望产品经理：**①改变现有产研迭代模式，释放人力；②探索未知领域，为组织创造新的价值增量**

### 1.3 核心判断：旧桥梁消失，新控制面出现

> **旧桥梁会消失，但新桥梁不是"沟通桥梁"，而是"控制桥梁"。**

这些新的控制面问题，既不是纯研发问题，也不是纯 SRE 问题，更不是纯业务问题。它们正好落在**平台产品 + 稳定性认知 + 组织协同 + 机制设计**的交叉点——这就是今年最该深耕的位置。

**PM 被 AI 真正替代的，只是"信息传递"功能，不是"问题定义"和"价值判断"能力。当执行层变得更快、更廉价，占据控制面的人反而更值钱。**

# 二、转型两大方向

> **两个方向的关系：** 方向一是方向二的认知基础——PM 只有亲身走完 AI 全流程、建立对前沿方向的判断力，才有能力主导方向二的系统建设。两者不是竞争关系，而是时序依赖：方向一精力投入随时间递减（前期重，后期轻），方向二是团队级空白，有时间窗口压力，越晚介入风险越大。

## 方向一：PM 在 AI 时代的角色转型

**核心逻辑：** AI 时代对 PM 的重构，不是让 PM 变成工程师，而是解决团队面临的两个结构性问题——

**问题一：人力结构性困境。** 团队不增 HC，但产品迭代不能停、新方向还要拓展。解法不是加人，而是 PM 通过 Vibe Coding 自闭环用户侧需求，把研发从"做页面"解放出来，专注 SKILL、数据等更深层的技术投入，实现同等资源下的更大产出。

**问题二：团队 AI 认知碎片化。** 现在团队每个人零散获取 AI 信息，没有人系统性判断和引导，导致信息不对称、方向各自为战、整体缺乏 AI 演进节奏。PM 作为 AI 先行者，核心价值是把碎片化信息转化为团队可对齐的方向判断，帮助团队在正确的时间押注正确的方向。

### 1.1 Vibe Coding：需求自闭环与原型验证

用 vibe coding 替代画图和写文档去验证想法——旧流程一个需求从想法到验证需要数周，新流程是先快速跑出原型、用真实交互筛选最优解再推进。这不是为了"成为会写代码的产品经理"，而是为了亲身走完迭代全流程，同时释放研发资源聚焦高价值工作。

**核心原则——Do the Simple Thing First：** AI 时代反而要警惕过度工程化。先用最简单的方式跑通一个闭环，验证假设，再决定是否值得深入。能用 prompt 验证的不要先写代码；能跑通 80% 的方案，不要先追求 100%。

**工作方式升级——Demo & Eval 替代文档：** Anthropic PM 内部的实践是，不再写详细 PRD 再等研发验证，而是直接用 Claude Code 做出可交互的原型或跑 eval 来说明问题和方向，几小时就能拿到真实反馈。想法到原型的距离从「数周」压缩到「数小时」——这不只是效率提升，而是让更多想法能真正经过用户验证的闭环，而不是死在文档里。

**目标：**
- 能独立用 AI 工具快速原型化想法，替代文档和原型图作为验证手段
- 优先消化长期积压的"研发不愿接、用户一直提"的存量需求（如大盘配置优化、告警规则模板），既有价值又不抢研发核心工作
- 释放出的研发资源聚焦在数据层、Agent/SKILL 层等高价值攻坚上

**执行节奏：**

| 阶段 | 时间 | 动作 |
|------|------|------|
| 学习期 | M1（70% 精力） | 选 Cursor 等工具，聚焦 1-2 类需求（配置类、表单类），跑通完整链路 |
| 磨合期 | M2-M3（40% 精力） | 建立 prompt 库和代码模板库，提高复用效率 |
| 常态化 | M4+（20-30% 精力） | 自闭环成为日常验证方式，重心转向认知侧升级 |

**⚠️ 关键边界：**
- **范围边界：** 用户层功能可以闭环，核心数据层/引擎层不要碰，先在内部工具试跑，再给用户用
- **协作边界：** 做的是研发不愿接的，不是他们正在做的
- **认知边界：** 这一步是为了理解全流程、建立规范制定的说服力，不是为了转型成"会写代码的 PM"

### 1.2 Side Quest：保持探索姿态，用小实验发现新方向

AI 时代方向变化极快，只靠规划内的工作会错过最重要的机会。需要刻意保留一部分精力做"不在 roadmap 上"的小实验——探索一个新工具、跑一个小 Agent、研究一个新概念——不设成果预期，只看能不能发现有价值的新方向。

这不是散漫，而是系统性地维持对前沿的感知。Anthropic PM 将这类实践称为"Side Quest"：有时候最重要的洞察，来自你本来没打算深入的那个角落。Anthropic 内部印证：Claude Code 桌面版、AskUserQuestion 工具、todo list 等多个核心功能，都是工程师和 PM 在 Side Quest 里跑出来的，而不是来自规划好的 roadmap。

**执行方式：**
- 每周留出固定时间（建议 2-4 小时）做非计划性探索
- 探索结果不要求完整，有一个清晰的"所以呢"就够
- 有价值的探索沉淀为团队分享或月报内容

### 1.3 AI 先行者：识别前沿方向，助力团队演进决策

AI 工具和方法论的迭代速度远超组织的消化速度。PM 需要承担信息过滤和判断的职责，而不只是信息搬运——把海量 AI 动态转化为「哪些值得团队跟进、哪些可以忽略、哪些需要现在就做准备」的决策建议。

**核心判断力——Building Chasm 识别：** AI 能力从"Demo 可用"到"Production Ready"之间有一道鸿沟。PM 需要持续判断：当前某个 AI 方向处于哪个阶段？是还在 Demo 阶段的概念，还是已经有稳定的工程实践可以押注？错误判断的代价是：要么过早投入一个还不成熟的方向，要么错过一个已经可以落地的机会。

**具体做法：**
- **信息过滤：** 持续跟踪 AI 工具和前沿方向（Agent、MCP、Harness Engineering 等），输出有决策价值的判断而非信息堆砌
- **认知基线：** 定期输出 AI 工具月报或内部分享，帮团队建立统一认知基线，减少重复探索
- **实践沉淀：** 发现和推广团队内的 AI 实践亮点；识别薄弱环节，定向推动补强

**落地形式——推动团队角色分工重定义：** AI 先行者的一个重要输出，是帮助团队厘清 AI 时代各角色的主导边界，防止全栈蔓延导致各自往低价值方向内卷：

| 角色 | AI 赋能后能做什么 | 应该主导什么 | 从而释放什么 |
|------|----------------|------------|------------|
| PM | 原型验证、需求自闭环 | 方向定义、规范制定、价值判断 | 研发从需求解释中解放 |
| 前端 | 产品模块自闭环、AI 交互探索 | 体验创新、AI 交互新范式定义 | 从重复 UI 执行中解放 |
| 后端 | 技术模块闭环、Agent/SKILL 层创新 | 数据层与技术能力建设 | 从前端补位工作中解放 |
| SRE | 业务稳定性自闭环、稳定性策略编排 | 稳定性产品化设计与 SKILL 治理 | 从重复巡检响应中解放 |

**AI 时代的角色融合：** Anthropic Claude Code 团队的实践印证了一个趋势——设计师在写代码，工程师在做产品决策，PM 在构建原型和跑 eval。角色边界在模糊，但这不是混乱，而是建立在清晰战略目标之上的自主协作。PM 的核心职责也因此在重新定义：不再是写文档、管流程，而是在模型能力快速演进制造的不确定性中**创造清晰度**——推动团队持续追问「如果当前的技术限制消失了，我们会怎么做」，并清扫快速交付路上的障碍。

**动态规划心态：** 方向一的三个子方向本身也需要持续迭代。AI 时代演进极快，每季度应回顾并基于新的探索和行业变化调整重心，而不是年初定了年底才看。

换一个视角：年初的 roadmap 本质上是一组假设，季度回顾是用真实结果来检验它——被推翻的假设不是失败，而是认知进步的证明。Side Quest 也是同样的逻辑：先提出「这个方向可能有价值」的猜想，用小实验来挑战它，扛住挑战的方向才值得押注。这比在会议室里反复讨论要可靠得多。

## 方向二：系统性风险的主动治理 —— 全局稳定性建设

**核心逻辑：** SRE 深入业务线，研发聚焦 SKILL 和技术模块，全局稳定性视角天然缺失。没有人从全局角度去看跨业务的风险、跨系统的依赖、以及 AI 化本身带来的新型稳定性挑战。这个位置正好落在平台产品 + 稳定性认知的交叉点，是 PM 可以主导的空间。

方向二分为两个层次：**A. 全局稳定性治理**（面向所有系统，不限 AI/非 AI）和 **B. AI 系统稳定性**（面向 Agent/SKILL 的新命题，以 Harness Engineering 为方法论主线）。两者共享同一个核心资产——可观测性平台，这是 PM 能同时主导两个层次的独特优势。

---

### A. 全局稳定性治理

#### 2.1 稳定性规范 SKILL 化 + 左移（Reliability Shift Left）

**现状问题：** 稳定性规范依赖人工评审，覆盖不全、响应滞后。

**目标状态：**

```
传统模式：变更上线 → 出现问题 → 故障复盘 → 完善规范（滞后）
新模式：  代码提交 → SKILL 自动扫描 → 拦截风险 → 阻断或告警（前置）
```

**落地路径：**
1. **规范梳理：** 整理现有稳定性规范，按标准化程度和触发频次分级
2. **样板先行：** 选 1-2 个高频且边界清晰的规范，跑通"规范→SKILL"转化路径
3. **流程嵌入：** 将 SKILL 嵌入 CI/CD 流水线，实现自动触发和拦截
4. **度量闭环：** 建立拦截数量、误报率、问题减少率等指标，持续优化
5. **逐步扩展：** 基于样板经验，按优先级覆盖更多规范场景

参考 DevSecOps "Security Shift Left" 的成熟路径，核心差异在于治理对象从安全漏洞扩展到稳定性风险。

#### 2.2 全局稳定性 AI 洞察

**问题本质：** AI 热潮下存在一个结构性矛盾——业务线资源向 AI 创新倾斜，稳定性投入被压缩；而 AI 编码和快速迭代带来的系统复杂度却在上升。投入减少与风险上升同步发生，但缺乏客观数据让这一矛盾显性化。

**为何以往推动方式失效：** 过去靠人工评审、专项会议、关系协调推动稳定性投入，本质是"人推人"——在团队全力投入 AI 创新的背景下更难为继。

**应对策略：** 从"人驱动"升级为"数据驱动"——用 AI 自动采集和分析全局稳定性数据，让风险客观可见，用数据替代人情施压。

**具体建设方向：**
- **全局容灾覆盖：** 自动扫描各业务线容灾能力，生成覆盖率报告，建立横向对比
- **架构依赖风险：** 持续梳理跨系统依赖，识别单点和强依赖风险，自动评级推送
- **可观测性盲区：** 度量监控覆盖缺失服务，建立接入标准，推动系统性补全
- **投入与风险关联分析：** 量化稳定性投入 vs 风险暴露的关联，为资源决策提供依据

---

### B. AI 系统稳定性（以 Harness Engineering 为方法论主线）

公司全面 AI 化后，SKILL 和 Agent 成为核心业务执行载体，随之带来传统稳定性体系未覆盖的新型风险——**系统在线，但认知错误**：监控绿灯、延迟达标，Agent 却在长链路任务中悄悄漂移、输出错误答案、越权执行。这是真实存在的空白，AI 化程度越深风险越大。

行业针对这一问题形成的 **Harness Engineering**（给 Agent 构建运行约束系统）与我们的挑战高度契合，以此为方法论主线，分解为以下三个方向。前期由 PM 主导探索定方向，联合 SRE/研发参与讨论；方向明确后共同落地。

#### 2.3 AI Native 全流程规范与 SKILL 生态标准（上线前）

**为什么要做：** AI 让大家都能调研、设计、写代码、孵化 SKILL，职责边界模糊带来两个风险：
- **系统风险：** AI 生成代码大量涌入，复杂度和质量难以把控，核心模块存在被污染的风险
- **生态风险：** SKILL 野蛮生长，没有准入标准，后续维护成本极高，形成"SKILL 灾难"

**做什么：**

① **研发全流程规范**：将调研、需求、设计、开发、上线每个环节的 AI 使用方式和输出质量标准化，做成可复用的 SKILL 嵌入流程

```
调研 → 需求 → 设计 → 开发 → 上线
 ↓      ↓      ↓      ↓      ↓
各环节对应 SKILL + 质量门禁标准
```

② **SKILL 生态标准**：制定 SKILL 准入准出机制和价值衡量标准
- **准入：** 什么样的 SKILL 值得做（场景频次、价值可量化、可复用性）
- **准出：** SKILL 上线前的质量验收标准（功能完整性、边界清晰度、文档规范性）
- **度量：** SKILL 上线后如何持续衡量价值（调用量、问题拦截数、人工替代率）

**PM 的角色：** 牵头联合研发、SRE 共同建立——定义"什么是好的输出"、设计度量体系、推动跨角色对齐。

**分阶段落地：**
- **试点期（Q2）：** 选 1-2 个高频场景，跑通规范 + SKILL 样板的完整链路
- **团队推行（Q3）：** AI Native 规范 1.0 在团队内全面推行
- **向外扩展（Q4）：** 整理方法论，探索向其他团队或公司层面输出

#### 2.4 Skills Reliability Engineering —— 运行期可靠性治理（上线后）

**定位：** 2.3 解决「什么样的 SKILL 值得上线、怎么上线好」；2.4 解决「上线后的 SKILL 和 Agent 如何持续稳定运行、出了问题怎么发现和处置」。治理对象从传统「服务和系统」扩展到「SKILL 和 Agent」，传统 SRE 体系对这层没有覆盖。

**Harness Engineering 对应能力层：**

| Harness 能力层 | 内部建设内容 |
|--------------|------------|
| **架构约束层** | SKILL 执行边界强制执行：权限管控、工具调用白名单、异常熔断机制 |
| **熵管理层** | 认知熵监控：文档与行为一致性扫描、执行成功率衰减告警、SKILL 生命周期治理 |

**需要建立的新能力：**
- **SKILL 可靠性：** 执行成功率监控、异常熔断、版本回滚机制
- **Agent 链路可观测：** 决策路径、工具调用链、执行结果的全链路追踪
- **安全与权限管控：** AI 执行动作的权限边界定义、敏感数据访问控制、越权行为自动识别
- **认知熵监控：** 跨 SKILL 的行为漂移检测、文档-行为一致性扫描、成功率衰减主动告警

**PM 的角色：** 前期主导探索，明确问题边界和可落地方向；后期联合 SRE 共同定义体系的产品形态——哪些需要平台支撑、哪些通过规范约束、哪些需要自动化监控。这是行业都在摸索的方向，先跑通就是先发优势。

#### 2.5 Agent SLO 与评估闭环

**定位：** Harness Engineering 解决「Agent 怎么稳定运行」，Agent SLO 与评估闭环解决「Agent 稳不稳定怎么量、怎么持续验证」。核心三件事：认知 SLO 体系（Layer 2-4）、Trace→Eval 评估链路、变更准入决策。没有这套闭环，Harness 优化是否有效无从验证，Agent 上线质量无从把控。

**为什么传统 SLO 不够：** 传统 SLO 问「系统在不在线」，Agent 的新型故障是「系统在线但认知错误」——延迟达标、资源正常、服务可用，同时输出错误答案。这类故障传统指标体系完全不可见。

**Agent 四层度量框架：**

```
Layer 4: 业务结果层 ── 任务完成率、结果准确率、用户满意度
         ↑ 建立在
Layer 3: 认知行为层 ── 指令遵循率、逻辑一致性、工具调用成功率、幻觉率
         ↑ 建立在
Layer 2: 链路追踪层 ── Trace 完整性、决策路径可追溯性、工具调用链覆盖率
         ↑ 建立在
Layer 1: 基础设施层 ── Uptime、Latency、Error Rate（传统 SRE 已覆盖）
```

目前大多数团队只有 Layer 1，Layer 2 刚起步，Layer 3/4 几乎空白。**可观测性平台已覆盖 Layer 1，向上扩展 Layer 2 是最直接的机会，定义 Layer 3/4 的标准是 PM 能主导的蓝海。**

**Agent 分级与对应稳定性要求：**

| 自主程度 | 典型场景 | SLO 重心 | 门禁要求 |
|---------|---------|---------|---------|
| L1 辅助型 | Copilot、代码补全 | 延迟、可用性 | 基础功能验收 |
| L2 工作流自动化 | 表单填写、数据处理 | 任务完成率、准确率 | 结果抽样评估 |
| L3 多 Agent 协同 | 跨系统任务编排 | 链路完整性、交接成功率 | 全链路 Trace + 回放测试 |
| L4 高度自主 | 自主决策、写入生产数据 | 指令遵循率、越权率 | 人工审核 + 熔断机制 |

**评估闭环是整套体系的收口：** Trace 规范是基础——没有完整的 Trace，Layer 2-4 的度量是空谈，Harness 的熵管理层也无从建立。PM 应推动统一的 Agent Trace 规范落地（参考 OpenTelemetry GenAI 语义约定），让可观测性平台成为所有 Agent 执行数据的汇聚点，进而驱动完整的评估闭环：

```
Agent 执行 → Trace 采集 → 数据集构建 → 质量评分（LLM-as-judge + 代码评分器）→ 变更准入决策
```

**PM 的角色：** 定义认知 SLO 指标体系（Layer 2-4）；推动 Trace 规范落地；设计 Eval 评估链路；联合 SRE 制定变更准入标准——这套闭环目前行业没有现成答案，先定义者拥有话语权。

### C. 技术风险平台 Agent-Native 化

**问题本质：** 当前所有技术风险平台都是面向人的功能和流程设计的。随着公司全面 AI 化，平台的主要「用户」正在从人变成 Agent——一个无法被 Agent 调用、没有对话入口的平台，在 Agent 时代将被绕过而边缘化。

Agent-Native 化有两个层次：**Agentic AIOps（浅层）**是在现有平台上加 AI 调用层，行业已有成熟案例；**真正的 Agent-Native（深层）**是重新审视现有功能的存在理由，哪些因「人需要」才存在、在 Agent 时代可以消解。两层都需要做，但深层才是平台真正的演进方向。

对现有功能按存在理由分三类：
- **A 类（纯为人类可视化）**：大盘、告警规则、日志查询 UI——Agent 时代大幅弱化。告警规则本质是「人把异常判断预编码成机器规则」，Agent 可持续推理自主判断，产品形态从「规则配置」重构为「意图配置」
- **B 类（分布式技术下的协同产物）**：日志采集→传输→集中存储这条链路，存在理由是「人没法逐台机器看日志」，Agent 可近源访问推理，集中存储从「主路径」收缩为「合规归档 + 跨服务关联」
- **C 类（数据有价值，消费者变了）**：Metric/Trace 原始数据——保留采集，接口从「人查询的 UI」重构为「Agent 可调用的 SKILL」

**应对策略：**

```
重构一：能力层 SKILL 化
  将核心能力（故障查询、变更评估、容灾检查、SLO 查询等）沉淀为 SKILL，
  天生可被任意 Agent 发现、调用和组合，形成技术风险能力的「可编程层」
  优先级：故障查询（P0）→ 变更风险评估（P0）→ SLO 查询（P0）→ 依赖图/容灾扫描（P1）

重构二：使用路径对话入口化
  提供统一对话入口，用自然语言描述意图，由 Agent 编排底层 SKILL 完成操作
  对话是主路径，原有 GUI 是辅助和兜底
```

**PM 的角色：** 主导平台 Agent-Native 化的产品设计——定义哪些能力优先 SKILL 化、设计 SKILL 的任务视角拆解、规划对话入口的用户动线、定义 Agent 推理结论的归档和审批数据模型。这是技术风险平台群在 AI 时代保持核心价值的关键，也是「平台不被绕过」的必要条件。

# 五、你的不可替代性在哪里

> **当所有人都在用 AI 加速执行的时候，你是那个在问"我们应该跑向哪里，以及跑起来之后不会翻车"的人。**

1. **执行侧升级：** 能自己动手验证，不再只是提需求；同时建立对 AI 全流程的第一手认知
2. **认知侧升级：** 持续识别 AI 前沿方向，判断 Building Chasm，让团队在正确的时间押注正确的方向
3. **规范守门人：** SKILL 生态标准（上线前）+ Skills RE（上线后）+ Agent SLO（度量层），构成完整的 AI 系统治理闭环，没有 PM 主导就会失控
4. **全局稳定性视角：** 在所有人聚焦业务线和 AI 创新的时候，你在看全局——既治理传统系统的稳定性投入空洞，又用 Harness Engineering 方法论定义 AI 时代的新型可靠性标准
