## 【第1页】行业正在发生什么

- **研发链路在缩短：** AI 编码让"需求—设计—开发"链路越来越短，PM 的解释、拆解、跟进正被更直接的协作方式替代
- **稳定性工作正在产品化：** SRE 借助 AI 把经验沉淀成 skill/runbook/agent，AIOps 从概念走向落地
- **代码质量隐忧浮现：** CMU 2025 研究显示 AI 编码后告警上升 30%，复杂度上升 40%+；AI 生成代码 bug 率是人工的 1.7 倍
- **组织默认逻辑正在改写：** Shopify 要求申请资源前先证明 AI 做不了——不能说清楚自己 AI 时代价值边界的岗位，会自然滑向被压缩

---

## 【第2页】核心判断：旧桥梁消失，新控制面出现

| 正在消失 ▼ | 价值上升 ▲ |
|-----------|-----------|
| 需求解释与拆解 | 定义正确的问题与约束 |
| 跨团队协同与跟进 | 制定规范与度量体系 |
| 信息传递与转述 | 占据新的控制面 |

> **旧桥梁会消失，但新桥梁不是「沟通桥梁」，而是「控制桥梁」。**
> 当执行层变得更快、更廉价，占据控制面的人反而更值钱。

## 【第3页】方向一：PM → Agent 工程师

**核心逻辑：** 用 vibe coding 替代画图和写文档验证想法——快速跑出多个原型、筛选最优解再推进。不是「会写代码的 PM」，而是亲身走完全流程，为定义 AI Native 研发规范建立认知和说服力。

**执行节奏：**
- M1（70% 精力）聚焦 1-2 类需求跑通链路
- M2-M3（40%）建立 prompt 库和模板库
- M4+（20-30%）常态化，重心转向方向二三

**关键边界：**
- 用户层可闭环，数据层/引擎层不碰
- 补位不抢活，提前跟研发对齐
- 不要留下「会写代码 PM」的印象

---

## 【第4页】方向二：AI Native 迭代新范式

**核心：** 最大风险不是没人用 AI，而是各自乱用。职责模糊、SKILL 野蛮生长、质量失控。PM 来定义 AI 时代协作方式和质量保障。

### 2.1 做团队的「AI 氛围官」
- **信息过滤与判断：** 跟踪 AI 前沿，过滤提炼，输出有决策价值的结论
- **认知基线建设：** 定期输出 AI 工具月报，建立团队统一认知基线
- **氛围带动与实践沉淀：** 推广 AI 实践亮点，定向推动补强

### 2.2 AI 时代各角色分工重定义

| 角色 | AI 赋能后能做什么 | 应该主导什么 | 从而释放什么 |
|------|----------------|------------|------------|
| PM | 原型验证、需求自闭环 | 方向定义、规范制定、价值判断 | 研发从需求解释中解放 |
| 前端 | 产品模块自闭环、AI 交互探索 | 体验创新、AI 交互新范式 | 从重复 UI 执行中解放 |
| 后端 | 技术模块闭环、Agent/SKILL 创新 | 数据层与技术能力建设 | 从前端补位中解放 |
| SRE | 业务稳定性自闭环、策略编排 | 稳定性产品化设计与 SKILL 治理 | 从重复巡检中解放 |

## 【第5页】2.3 AI Native 全流程规范

**为什么要做？**
- **系统风险：** AI 代码大量涌入，复杂度和质量难以把控
- **生态风险：** SKILL 野蛮生长，没有准入标准，形成「SKILL 灾难」

**全流程规范：**
```
调研 SKILL → 需求 SKILL → 设计 SKILL → 开发规范 → 上线门禁
```

**SKILL 生态标准：**
- 准入：场景频次 / 价值可量化 / 可复用性
- 准出：上线前质量验收标准
- 度量：调用量 / 问题拦截数 / 人工替代率

**PM 的角色：** 牵头联合研发、SRE 共建，定义好的输出标准、设计度量体系、推动跨角色对齐

**分阶段：** Q2 试点 → Q3 团队推行 → Q4 向外扩展

## 【第6页】方向三：全局稳定性建设

**核心：** SRE 聚焦业务线，研发聚焦 SKILL，全局稳定性视角天然缺失——这个交叉点是 PM 可以主导的空间。

### 3.1 稳定性规范 SKILL 化 + 左移（Reliability Shift Left）

| 传统模式（滞后）| 新模式（前置）|
|----------------|--------------|
| 变更上线 → 出现问题 → 故障复盘 → 完善规范 | 代码提交 → SKILL 自动扫描 → 拦截风险 → 阻断/告警 |

**落地五步：** 规范梳理 → 样板先行 → 流程嵌入 → 度量闭环 → 逐步扩展

---

## 【第7页】3.2 & 3.3 稳定性新命题

### 3.2 全局稳定性投入空洞

**结构性矛盾：** 业务线资源向 AI 创新倾斜，稳定性投入被压缩；同时 AI 编码带来系统复杂度上升。**投入减少 × 风险上升**，缺乏客观数据显性化。

**应对：人驱动 → 数据驱动**
- 全局容灾覆盖自动扫描
- 架构依赖风险自动评级
- 可观测性盲区度量
- 投入 vs 风险关联分析

### 3.3 Skills Reliability Engineering — SRE 的新命题

**背景：** 平台变薄，治理对象从「服务和系统」扩展到「SKILL 和 Agent」，传统体系不覆盖。

**需要建立的新能力：**
- SKILL 可靠性（准入 / 监控 / 熔断）
- Agent 链路可观测（决策路径 / 工具调用链追踪）
- 安全与权限管控（权限边界 / 敏感数据 / 异常识别）
- 生命周期治理机制
