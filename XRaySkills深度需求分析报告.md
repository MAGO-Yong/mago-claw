# XRaySkills 可观测平台 SKILL 需求深度分析报告

> 分析视角：产品经理  
> 数据来源：52条问卷，有效去重 46 条  
> 分析时间：2026-03-24

---

## 写在前面：为什么要做深度分析

第一轮分析停留在「频次统计」。但频次高不等于价值高——选项多因为展示在那里，用户会顺手勾选；真实痛点藏在问答里，需要还原场景、区分问题类型、判断用户分层，才能做出真正值得做的 SKILL。

本报告做四件事：
1. **用户分群**：谁在用 X-Ray，他们的使用模式有何本质差异
2. **场景还原**：把碎片化问答还原成完整的使用场景路径
3. **需求质量分级**：区分「功能诉求」vs「真实痛点」，后者更值钱
4. **SKILL 设计建议**：从用户旅程出发，定义每个 SKILL 的核心 MVP

## 一、用户分群分析：四类典型用户，诉求截然不同

通过对全量问答的阅读，用户可以分为 4 类，而不是简单的「按职位分」。

### 🔵 A 类：救火型用户（26人，57%，主力群体）

**特征：** 大量后端开发，被动触发——收到告警/客诉/报错后才用 X-Ray，核心诉求是「快速找到根因」。

**使用路径：**
```
收到告警/客诉 → 关键词查日志 → 找 traceId → 查链路 → 定位报错代码 → 关联变更 → 结案
```

**核心痛点（原话）：**
- *"先查询日志/根据日志查数据/看代码/定位问题，以上四个步骤，都可以自动化"*（凯恩/商业平台）
- *"遇到报警 sql 成功率低，xray 查异常堆栈，同时关联 ones/apollo 查变更——这一步可以通过 skill 自动化"*（灵梦/商业平台）
- *"半夜收到告警，想要进一步下钻监控，比如是什么接口、什么平台导致 error 突增"*（晦明/技术风险）
- *"通过关键词查日志 → 找 traceId → traceId 下所有日志查出来"*（山峰/研发三组）

**他们真正要的不是「日志查询 SKILL」，而是：告警 → 根因 的完整闭环自动化**

---

### 🟡 B 类：开发调试型用户（约 10 人，22%）

**特征：** 算法/策略开发者为主，主动触发——开发测试阶段频繁用 X-Ray 验证结果，AI 辅助开发诉求最强。

**使用路径：**
```
写代码/改模型 → 发布/测试 → 读 xray 日志确认是否符合预期 → 再改代码
```

**核心痛点（原话）：**
- *"目前 AI 在开发上已经很强了，现在卡点是每次测试需要去读 xray 日志，如果能让 AI 直接读 xray 日志，日常简单的需求就能完成开发测试闭环"*（向山/商业算法）
- *"agent 解决 bug 时，根据代码去搜日志，根据日志情况，自动规划解决方案"*（清隆/研发一组）
- *"debug skill，服务端/droom 算法同学发起请求，agent 自动打相应断点，输出分析报告/定位出问题"*（恋次/社区搜索算法）
- *"模型报错 100% 依赖 xray 日志"*（向山/商业算法）
- *"发版时观察特定日志量在 canary 和老 pod 是否有变化，核心服务有近十个集群，每次发版要一个集群一个集群查"*（朝风/风控工程）

**他们要的是：X-Ray 成为 AI Coding Agent 的眼睛——代码 ↔ 日志 双向联动**

---

### 🟢 C 类：运维配置型用户（约 8 人，17%，SRE 为主）

**特征：** SRE/平台工程师，主动触发——日常巡检、配置告警、压测保障，管理的服务多、重复操作多。

**使用路径：**
```
巡检服务 → 检查告警覆盖 → 验证告警有效性 → 配置/修改告警规则 → 创建观测看板
```

**核心痛点（原话）：**
- *"制定告警规则配置规范，检查服务告警覆盖情况，验证告警规则有效性——高频使用"*（小强/高可用架构）
- *"监控巡检、增删改告警、SLO 以及 RCA 添加修改、问题定位分析"*（萧峰/技术风险）
- *"压测时批量生成观测看板，对核心大盘 RCA 告警做静默处理"*（小强/高可用架构）
- *"仿照 url://xxx 规则写一个 for xx 的报警规则，阈值设为 yy/根据历史自动推荐阈值"*（新罗/商业算法）
- *"无权限时可对自己静默，告警人错误时避免自己被打扰"*（球棒/交易治理）

**他们要的是：告警/看板的批量配置自动化 + 覆盖率巡检**

---

### 🔴 D 类：数据驱动型用户（约 8 人，17%，算法/TL 为主）

**特征：** 技术负责人、算法同学，主动触发——关注业务指标趋势、成本分析、上线后数据观测，不在乎排障，在乎「看懂数据」。

**使用路径：**
```
上线/迭代后 → 持续观测指标趋势 → 导出数据 → 生成报表 → 决策
```

**核心痛点（原话）：**
- *"将 Xray 数据汇总定时更新到 RedBI 看板，需要用户自己二次给 RedBI 生产或通过 api 从 Xray 获取数据再插入 RedBI，一键配置看板场景非常普遍"*（无相/云原生存储）
- *"REDBI 和实验数据分析，帮我在 REDBI 里自动生成图表"*（道玄/交易算法）
- *"线上上线后的业务数据分析"*（魏婴/AIGC算法）
- *"GPU 利用率"*（桑铎/稠密平台）
- *"每周基于日志统计使用情况，需要每周下上一周固定字段的日志"*（孙瀚文/商业平台）
- *"日志结果直接导入在线 doc 和定时统计导入 OLAP 引擎用于统计报表"*（格飞/引擎架构）

**他们要的是：X-Ray 数据的自动化流转——Xray → RedBI/OLAP/Doc 的定时管道**

## 二、场景还原：从「我要功能」到「我有问题」

> PM 的核心判断：用户描述的「要什么」背后，「为什么要」才是真问题。

### 场景 1：告警驱动的根因定位（最核心场景，覆盖 A 类全部用户）

**完整旅程（现状）：**
1. 收到告警（通常半夜）
2. 打开 X-Ray，手动输入服务名、时间范围
3. 查 error 日志，找 traceId
4. 用 traceId 查链路，逐跳看哪一跳出了问题
5. 确认下游服务后，再去下游服务重复步骤 2-4
6. 同时打开 ones/apollo/gitlab，看有没有同期变更
7. 汇总分析，写根因

**用户最痛的点：**
- 步骤 4-5 的递归跳转：*"花了好几分钟才发现下游一个 cpu 利用率变为 100%"*（狸克）
- 步骤 6 需要跨系统：*"变更群或 gitlab/ones/apollo 查当前应用是否有变更"*（灵梦）
- 整个流程在半夜做，极度消耗心智

**SKILL 设计启示：**
- 核心动作不是「查日志」，而是**「给我一个告警，告诉我根因」**
- MVP 输入：服务名 + 时间点
- MVP 输出：异常聚类摘要 + 最可疑根因 + 同期变更列表
- 数据需要：日志 + 链路 + 指标 + 变更（跨系统）

---

### 场景 2：开发调试闭环（B 类特有，但需求强度最高）

**完整旅程（现状）：**
1. 写代码/改模型参数
2. 发布到测试环境
3. 打开 X-Ray，手动查服务日志
4. 把日志内容复制到 AI（Claude/Copilot）
5. AI 分析后给出修改建议
6. 回到步骤 1

**用户最痛的点：**
- 步骤 3-4 是纯手工中转，人肉当「数据管道」
- 模型报错时 100% 依赖 xray 日志，但每次都要手动操作
- 多集群发布时，一个集群一个集群查日志量对比

**SKILL 设计启示：**
- 核心是打通「代码修改 → 日志验证」的最后一公里
- 不是「让用户在 X-Ray 里用 AI」，而是「让用户在 Coding Agent 里直接用 X-Ray」
- 这个 SKILL 的主要消费者不是 X-Ray 用户，而是 **Coding Agent 本身**（被 Agent 调用）
- MVP：给定服务名/时间范围，返回结构化的 error 摘要（聚类 + 堆栈），供 Agent 消费

---

### 场景 3：批量告警配置与巡检（C 类特有，价值密度最高）

**完整旅程（现状）：**
1. 制定告警规则配置规范
2. 手动检查每个服务的告警覆盖情况
3. 发现未覆盖的服务，手动创建告警规则
4. 验证告警规则是否能正常触发
5. 压测前，手动静默相关告警

**用户最痛的点：**
- 管理服务多，每个都要手动配，**高频且重复**
- 压测场景需要批量静默，现在只能全局静默（影响范围太大）
- 告警阈值靠经验，没有历史数据辅助推荐

**SKILL 设计启示：**
- 告警「查询」和「管理」要一体化，不是两个独立 SKILL
- 关键差异化功能：**阈值智能推荐**（基于历史数据）、**批量操作**、**有效性验证**
- 静默颗粒度：支持「只对自己静默」是一个真实且具体的需求

---

### 场景 4：数据报表自动化流转（D 类特有，隐藏需求）

**完整旅程（现状）：**
1. 业务数据报到 X-Ray
2. 需要在 RedBI 配置统计看板
3. 手动通过 API 从 X-Ray 取数，再插入 RedBI
4. 每周/月重复一次

**用户最痛的点：**
- X-Ray 和 RedBI 是两个平台，数据流转靠人肉
- 这个需求非常普遍但没人关注（无相/云原生存储明确说「一键配置看板场景非常普遍，提效明显」）

**SKILL 设计启示：**
- 这个场景不是「看板创建」，是「数据管道配置」
- 可能超出 SKILL 范畴，更接近「X-Ray → RedBI 数据集成」产品特性
- 但作为 SKILL 也可以做：定时触发，自动拉取 X-Ray 数据写入 RedBI

## 四、关键洞察：选项设计本身的问题

> 这是很多频次统计分析会忽略的一点——问卷选项设计会影响结果

**发现 1：「成本分析」可能被高估了**
- 16人选，但仅 1 人（桑铎）提到具体场景（GPU利用率），其余问答完全没有提及
- 可能原因：选项名称宽泛，用户觉得「跟我相关就选了」
- 建议：做个小规模访谈，确认成本分析的真实场景

**发现 2：「Agent状态分析」是个黑盒**
- 12人选，但 0 人在问答中描述过「我用 Agent 状态分析做了什么」
- 整个 AI链路分析模块的选择率虽然 63%，但问答质量最低
- 可能原因：这些概念对用户来说比较新，选了但没有真实用过

**发现 3：日志查询的需求远不只是「查日志」**
- 36人选日志查询，但从问答内容看，真实需求有 5 个层次：
  1. 基础检索（关键词/时间范围/服务名）
  2. 结构化统计（SQL-like，group by，count）
  3. 跨服务联动（traceId 穿透多个服务的日志）
  4. 智能聚类（把重复报错归类，告诉我新增了哪条错误）
  5. AI 闭环（日志直接喂给 AI agent，不需要人中转）
- 这 5 个层次的实现难度和价值完全不同，需要分别规划

**发现 4：有一个高价值场景几乎被忽略——「运营/非研发人员查日志」**
- 丽达（商业平台）明确说：*"我们需要对外提供查询能力，对运营排查 case，需要可以直接指定环境和服务查询日志"*
- 这意味着日志查询 SKILL 的消费者不只是研发，还有运营——他们的操作门槛要求更低
- 这个方向如果做好，用户群体可以扩大 3-5 倍

**发现 5：填写时长与需求质量高度相关**
- 填写时长 > 300s 的用户（10人）：几乎每个人都有非常具体的场景描述，是最值得深挖的用户
  - 格飞 (372s)、小强 (528s)、球棒 (569s)、无相 (489s)、灵梦 (448s)、朝风 (438s)
- 填写时长 < 50s 的用户（8人）：大多只是勾选了选项，问答几乎为空
- 结论：**做后续访谈优先找那 10 个认真填写的用户，他们的需求最真实**

## 五、SKILL 重新排优先级（PM 视角）

基于以上深度分析，重新评估优先级。**核心逻辑：价值 × 需求清晰度 × 实现可行性**

### 第一优先级：应该先做这三个

#### 🥇 1. 告警根因诊断 SKILL（≠ 异常诊断）

- **为什么是第一**：覆盖 A 类用户（57%），是最高频的被动触发场景，且用户有非常清晰的期望
- **和「异常诊断」的区别**：异常诊断是事后分析；告警根因是实时响应——「有告警，立刻告诉我为什么」
- **MVP 定义**：
  - 输入：服务名 + 告警时间（或告警 ID）
  - 输出：①异常日志聚类摘要（Top3 错误）②指标下钻（哪个 pod/接口出问题）③同期变更关联（来自 ones/apollo/gitlab）④初步根因假设
- **关键技术**：需要跨系统数据聚合（X-Ray + 变更系统），这是难点也是壁垒

#### 🥈 2. 日志智能查询 SKILL（分层设计，先做 L1+L2）

- **为什么是第二**：78% 的用户选择，但需求有层次，不能一锅端
- **L1（立即做）**：自然语言查日志（服务名/关键词/uid/时间范围 → 结构化结果）
- **L2（跟进做）**：日志 SQL 统计（支持 group by/count/extract，解决 CAT 维度不足问题）
- **L3（后续规划）**：日志智能聚类（error 增量分析，告诉你新增了哪类错误）
- **L4（Coding Agent 专属）**：结构化日志输出接口，供 AI Agent 消费，无需人工中转
- **注意**：L3、L4 的价值远高于 L1，但需要 L1 的基础

#### 🥉 3. 链路追踪 SKILL（traceId 穿透，不是「链路查询」）

- **为什么是第三**：实际上是「告警根因」的关键子能力，且多个场景都依赖它
- **MVP 定义**：
  - 输入：traceId（或 uid + 接口名 + 时间，自动找 traceId）
  - 输出：全链路树形图（各跳延迟 + 错误 + 关键日志），自动标注最可疑节点
- **差异化**：现有 X-Ray 链路功能需要手动逐跳查，SKILL 要做到「输入 traceId，AI 帮你找出问题在哪跳」

---

### 第二优先级：价值高但需求待验证

#### 4. 告警配置管理 SKILL（SRE 专属，但价值密度高）

- **用户少但痛点深**：4个 SRE，但每个人的需求描述都非常具体
- **差异化设计**：阈值智能推荐是真正的护城河，别人做不了
- **MVP**：查询覆盖情况 + 自然语言创建告警 + 基于历史推荐阈值

#### 5. 大盘自动创建 SKILL

- **重新定义**：不只是「创建图表」，是「描述我要观测什么 → AI 帮我决定用什么指标、建什么图」
- **王坤的描述最有产品思路**：*"skill 询问需求方需要什么图表类型，然后自己生成一版——如果可以的话，根据需要的大盘图表倒推需要什么指标"*
- **这个 SKILL 的核心不是「填表单」，是「需求澄清 + 指标推荐 + 一键生成」**

---

### 第三优先级：先做调研再决策

#### 6. 成本分析 SKILL
- 需要先做 3-5 个用户访谈，明确「分析什么成本」「用来决策什么」

#### 7. X-Ray → RedBI 数据管道
- 无相的需求非常清晰，但这更像是产品集成功能而不是 SKILL
- 建议先调研有多少用户有这个需求，再决定做法

#### 8. Agent状态分析 SKILL
- 12人选但 0 人能描述场景，说明这是「用户以为自己想要的」而非「真实痛点」
- 建议暂缓，先做调研

## 六、三个值得深挖的「隐藏机会」

> 这些需求没有人明确说出来，但从数据里能推断出来

### 机会 A：X-Ray SKILL 作为 Coding Agent 的内置工具

- **信号**：至少 5 人（向山、清隆、恋次、新罗、灵梦）都在描述「AI 自动读日志帮我 debug」
- **洞察**：这些用户其实不想在 X-Ray 里用 AI，他们想在 Cursor/Claude/内部 Coding Agent 里直接调用 X-Ray 能力
- **机会**：把 X-Ray SKILL 设计成 MCP Server / Tool Call，让外部 AI Agent 可以直接调用——这比「X-Ray 内嵌 AI」的价值大得多
- **验证方式**：找那几个用 Coding Agent 比较深的同学做个 demo

### 机会 B：运营/非研发用户的日志查询

- **信号**：丽达说「需要对外提供查询能力，对运营排查 case」
- **洞察**：现有问卷只发给了研发，但实际上运营同学也有查日志的需求（客诉场景），只是他们不在这份问卷里
- **机会**：日志查询 SKILL 如果能做到「零门槛，运营也能用」，用户盘就不是 46 个，而是几百个
- **验证方式**：找 1-2 个业务运营同学了解他们的排查流程

### 机会 C：发布观测自动化（被低估的高频场景）

- **信号**：川岛说「每周 50 次」策略服务发布观测；朝风说「每次发版要一个集群一个集群查日志量」
- **洞察**：发布是研发最高频的操作之一，但「发布后的观测」完全靠人肉
- **机会**：「发布观测 SKILL」——发布触发 → 自动对比新老 pod 的日志/指标 → 异常自动报警
- **这个场景不在现有 SKILL 类型里，但需求非常真实**
