### 2.2 按使用模式分群（更有产品价值的分群方式）

职位分群只能说明"谁"，使用模式分群才能说明"怎么用"。

**A 类：救火型用户（约 26 人，53%）**

特征：被动触发，收到告警/客诉/报错后才用 X-Ray，核心诉求是快速找根因。

典型路径：`收到告警 → 查日志找 TraceId → 查链路定位报错 → 关联变更 → 结案`

代表原话：
- "先查询日志/根据日志查数据/看代码/定位问题，以上四个步骤，都可以自动化" —— 凯恩
- "遇到报警 sql 成功率低，xray 查异常堆栈，同时关联 ones/apollo 查变更" —— 灵梦
- "半夜收到告警，想要进一步下钻监控，是什么接口、什么平台导致 error 突增" —— 晦明

**B 类：开发调试型用户（约 10 人，20%）**

特征：主动触发，开发测试阶段频繁用 X-Ray 验证结果，AI 辅助开发诉求最强。

典型路径：`写代码/改模型 → 发布测试 → 读 xray 日志确认是否符合预期 → 再改代码`

代表原话：
- "现在卡点是每次测试需要我去读 xray 日志，然后把报错告诉 AI，如果能让 AI 直接读 xray 日志，开发测试可以完成闭环" —— 向山
- "agent 解决 bug 时，根据代码去搜日志，根据日志情况，自动规划解决方案" —— 清隆
- "模型报错 100% 依赖 xray 日志" —— 向山

**C 类：运维配置型用户（约 8 人，16%）**

特征：SRE 为主，日常巡检、配置告警、压测保障，管理服务多、重复操作多。

典型路径：`巡检服务 → 检查告警覆盖 → 验证有效性 → 配置/修改告警规则 → 创建观测看板`

代表原话：
- "检查服务告警覆盖情况，确保覆盖率达标；验证告警规则有效性，高频使用" —— 小强
- "压测时批量生成观测看板，对核心大盘 RCA 告警做静默处理" —— 小强
- "仿照 url://xxx 规则写一个报警规则，阈值设为 yy/根据历史自动推荐阈值" —— 新罗

**D 类：数据驱动型用户（约 8 人，16%）**

特征：TL/算法为主，关注业务指标趋势、成本分析、上线后数据观测。

典型路径：`上线/迭代后 → 持续观测指标趋势 → 导出数据 → 生成报表 → 决策`

代表原话：
- "将 Xray 数据汇总定时更新到 RedBI 看板，一键配置看板场景非常普遍，提效明显" —— 无相
- "每周基于日志统计使用情况，需要每周下上一周固定字段的日志" —— 孙瀚文

### 2.3 分群对产品设计的核心意义

同一个"日志查询"功能，四类用户需要的是四个完全不同的 Skill：

| 用户类型 | 真正需要的形态 |
| --- | --- |
| A 类救火型 | 告警触发后自动聚合上下文，零输入 |
| B 类调试型 | Coding Agent 的工具接口（API/MCP），结构化返回 |
| C 类运维型 | 批量操作 + 告警覆盖率分析 |
| D 类数据型 | 定时导出 + 流向 RedBI/OLAP 的数据管道 |

## 三、触发源分析

这是理解用户动线的核心维度——用户从哪里被触发来用 X-Ray，决定了 Skill 的交互起点。

| 触发源 | 人数 | 触发特征 | Skill 设计要求 |
| --- | --- | --- | --- |
| **告警触发** | **18 人（37%）** | 被动、紧急、常在非工作时间 | 响应速度第一，自动化程度最高，接近零输入 |
| **主动排障** | **13 人（27%）** | 有线索主动查，有初步判断 | 引导式交互，帮用户缩小排查范围 |
| **Coding/AI Agent 触发** | **13 人（27%）** | 机器调用，无人值守 | API 优先，返回结构化数据，MCP 协议支持 |
| 发布/上线触发 | 5 人（10%） | 固定时机，需要对比 | 基线对比能力，多集群并行查询 |
| 定时/周期触发 | 5 人（10%） | 预期结果，定时消费 | 调度 + 推送能力（不是 Skill 而是 Agent 任务） |
| 客诉触发 | 4 人（8%） | UID 已知，倒查链路 | 以用户视角聚合完整请求链路 |

**关键洞察**：13 位"Coding Agent 触发"的用户不是要在 X-Ray 里用 AI，他们要在 Cursor/内部 Coding Agent 里**直接调 X-Ray 数据**。如果 Skill 只做对话式界面，这 13 人的需求全部丢失。

## 五、用户动线还原

通过精读 49 条问答，还原出 5 条核心使用动线：

### 动线 1：告警驱动的根因定位（最高频）

**触发**：收到告警 → **目的**：定位根因止损

```
收到告警
  → 查指标确认异常范围（哪个服务/接口）
  → 查日志找报错堆栈
  → 用 TraceId 查链路定位根因服务
  → 切换到变更系统（ONES/Apollo/GitLab）看同期变更
  → 确定根因 → 修复或回滚
```

**最痛卡点**：步骤"切换变更系统"——用户必须离开 X-Ray 去另一个系统查变更，且这个步骤无法省略。

**原话证据**：
- "遇到报警 sql 成功率低，xray 查异常堆栈，同时变更群或者 gitlab/ones/apollo 查看当前应用是否有变更，这一步可以通过 skill 自动化" —— 灵梦
- "当 error 日志变多后，能帮我聚合出这条日志，并分析变多时对应的变更有哪些" —— 狸克
- "花了好几分钟才发现：下游一个 cpu 利用率变为 100%，导致调用下游 p99 变高很多" —— 狸克

---

### 动线 2：开发调试闭环（极高频，后端/算法为主）

**触发**：功能测试/上线后异常 → **目的**：快速定位代码报错

```
写代码/改模型参数
  → 发布测试环境
  → 手动打开 X-Ray 查日志（当前卡点）
  → 把日志内容复制给 AI（人工中转卡点）
  → AI 分析后给出修改建议
  → 回到第一步
```

**最痛卡点**："人工中转"——用户在 AI 和 X-Ray 之间手动搬运日志内容。

**原话证据**：
- "现在卡点是每次测试需要我去读 xray 日志，然后把报错告诉 AI，如果能让 AI 直接读 xray 日志，开发测试可以完成闭环" —— 向山
- "agent 解决 bug 时，根据代码去搜日志，根据日志情况，自动规划解决方案" —— 清隆
- "模型报错 100% 依赖 xray 日志" —— 向山

---

### 动线 3：发版健康观测（高频但被低估）

**触发**：服务发布/灰度 → **目的**：确认发版无异常

```
执行发版（多集群）
  → 逐个集群查日志量对比（canary vs 老 pod）
  → 观察关键指标变化
  → 判断是否需要回滚
```

**最痛卡点**："逐个集群"——核心服务有十个左右集群，每次发版都要一个一个查。

**原话证据**：
- "核心服务有将近十个集群，每次发版都要花很久一个集群一个集群查日志量，这个过程相当枯燥且重复" —— 朝风
- "完全可以自动化，期望能让 Agent 模仿人类搜索日志，能快速发现灰度的 Error" —— 川岛
- "策略服务发布观测 50 次/周" —— 川岛（频率极高）

---

### 动线 4：监控资产配置（配置类需求）

**触发**：新服务上线/需要监控 → **目的**：建大盘、配告警

```
描述监控需求（自然语言）
  → 手写 PQL / 填配置表单（当前卡点）
  → 创建大盘图表
  → 配置告警规则（靠经验设阈值是卡点）
  → 验证告警有效性
```

**最痛卡点**：
1. PQL 有门槛，非熟练用户写不了
2. 告警阈值靠经验，缺乏历史数据辅助

**原话证据**：
- "通过自然语言创建大盘、报警规则，而不是去平台上做繁琐的配置" —— 镜玄
- "告警规则的确定，很依赖于经验；二次调整告警规则的时间成本过高" —— 魏晋
- "仿照 url://xxx 规则写一个报警规则，阈值设为 yy/根据历史自动推荐阈值" —— 新罗

---

### 动线 5：数据汇总与定时分析（周期性）

**触发**：周期性任务 → **目的**：汇总数据、分析趋势、输出报表

```
（定时）从 X-Ray 提取数据
  → 手动写脚本/人工导出（当前卡点）
  → 写入 RedBI/OLAP
  → 生成报表看板
  → 分发给业务方
```

**最痛卡点**：X-Ray 和 RedBI 是两个平台，数据流转靠人肉，每周/月重复。

**原话证据**：
- "将 Xray 数据汇总定时更新到 RedBI 看板，一键配置看板场景非常普遍，提效明显" —— 无相
- "每周基于日志统计使用情况，需要每周下上一周固定字段的日志" —— 孙瀚文
- "每天自动分析一下当天的日志与前一天的日志，有哪些变化，分析是否有可能因变更引入的问题" —— 秋水

## 六、用户旅程卡点精确定位

### 卡点类型 1：批量操作缺失（最普遍）

用户需要在 N 个对象上重复相同操作（N 个接口/集群/服务）。

| 用户 | 原话 | 需要批量操作的对象 |
| --- | --- | --- |
| 秋水 | "只能每个接口都看一下监控" | 所有调用下游的接口 |
| 朝风 | "每次发版一个集群一个集群查日志量" | 10 个集群 |
| 小强 | "检查服务告警覆盖情况" | 所有服务 |

**SKILL 解法**：语义批量查询——用户描述"查哪类对象的什么指标"，Skill 自动并行查询并聚合结果。

### 卡点类型 2：信息分散跨系统（最影响根因效率）

排障过程中必须切换到另一个系统，且这步无法跳过。

| 用户 | 需要切换到 | 场景 |
| --- | --- | --- |
| 灵梦 | ONES/Apollo/GitLab | 查同期变更 |
| 狸克 | 变更系统 | error 增多时关联变更 |
| 无忆 | 变更系统 | 堆内存分析关联变更 |
| 朝风 | RPC 调用层（X-Ray 内部多层跳转） | RT 异常递归追踪下游 |

**SKILL 解法**：变更关联是告警根因诊断 Skill 的必要子能力，不是可选项。

### 卡点类型 3：数据最后一公里断裂

用户得到了数据，但还要手工搬运到下一个使用场景。

| 用户 | 当前手工操作 | 期望 |
| --- | --- | --- |
| 格飞 | 日志 select 结果下载失败 | 结果导入 Doc/OLAP |
| 文镜 | 下载日志文件放到项目里 | 直接给 AI 分析 |
| 向山 | 复制日志内容给 AI | AI 自己读 |
| 孙瀚文 | 每周手动下载固定字段日志 | 定时自动导出 |

**SKILL 解法**：数据导出 API + Agent 数据源接口（MCP），打通最后一公里。

### 卡点类型 4：高压时刻的认知负担

告警场景通常发生在非工作时间，用户需要在疲惫状态下做复杂推理。

> "半夜收到告警的时候，想要进一步下钻监控" —— 晦明

**SKILL 核心价值定位**：告警场景的 Skill 不是帮用户"更快完成操作"，而是**减少认知负担**——直接告诉我根因，不需要我来推理。

## 八、日志查询需求的五个层次

"日志查询"是选择率最高的功能（84%），但它背后是 5 个层次完全不同的需求：

| 层次 | 描述 | 代表用户 | 实现难度 | 价值 |
| --- | --- | --- | --- | --- |
| **L1 基础检索** | 关键词/时间/服务名查日志 | 大多数用户 | 低 | 基础 |
| **L2 结构化统计** | SQL-like 统计，group by/count/extract | 格飞 | 中 | 中 |
| **L3 跨服务联动** | TraceID 穿透多服务日志 | 山峰/镜玄/蛮骨 | 高 | 高 |
| **L4 智能聚类** | 把重复报错归类，标出新增了哪类错误 | 狸克/清隆 | 高 | 高 |
| **L5 Agent 消费接口** | 结构化日志输出，供 Coding Agent 直接调用 | 向山/清隆/恋次 | 高 | **最高** |

**产品策略**：先做 L1 快速交付；L4 和 L5 价值远高于 L1，要纳入路线图规划，不能做完 L1 就认为"日志查询 Skill 做完了"。

---

## 九、SKILL 消费主体分析

这决定 SKILL 的技术形态，前所有分析都忽略了这个维度。

**三类消费主体并存：**

**1. 人类直接对话（所有用户）**

需要：自然语言交互界面，容错，引导式追问。

**2. 业务 Bot/平台集成（至少 7 人明确提到）**

- 帕恩："离线引擎任务失败时，希望能查询 XRay Skills 获取错误信息，工作时间基本上业务会一直用"
- 十夜："希望通过 API 来查询日志，方便 bot 自动分析 larc 任务报错原因"
- 晦明："查询服务负载情况、日志等，不需要登到 xray 上"

需要：开放 API、鉴权、速率限制、标准化返回格式。

**3. Coding Agent（AI 调 AI，至少 13 人描述了这个场景）**

- 向山："让 AI 直接读 xray 日志，开发测试可以完成闭环"
- 清隆："agent 解决 bug 时，根据代码去搜日志"
- 恋次："有 agent 自动打相应断点，输出分析报告/定位出问题"

需要：MCP 协议/Tool Call 接口，低延迟，适合 Agent 消费的结构化输出。

> **架构决策**：不是"先做哪个"的问题，而是 Day 1 就要设计好三种接入方式。核心能力层（数据获取+分析）与消费方无关；接口层同时暴露对话式 + REST API + MCP Tool；输出层根据消费方返回不同格式。

## 十、需求时效性分类：SKILL vs Agent 调度任务

| 时效性 | 典型场景 | 技术形态 |
| --- | --- | --- |
| **实时（秒级）** | 告警触发根因诊断、测试后验证日志 | 低延迟 API，Skill 形态 |
| **按需（分钟级）** | 手动发起排障、创建大盘/告警规则 | 交互式 Skill，可等待 |
| **周期性（小时/天级）** | 每日日志变化对比、每周日志导出、定时写 RedBI | **Agent 调度任务，不是 Skill** |

> ⚠️ **规划警示**：秋水/格飞/孙瀚文/无相的"周期性需求"需要的是**Workflow/Agent 任务调度能力**，不是对话式 Skill。建议在产品规划中明确区分这两种形态，共享底层能力但产品呈现分开。

## 十一、跨系统数据流：X-Ray 的战略边界

| 外部系统 | 涉及人数 | 数据流方向 | 优先级建议 |
| --- | --- | --- | --- |
| **代码/项目仓库** | 9 人 | X-Ray 日志 → 关联代码行 | 🔴 优先，Coding Agent 集成的核心 |
| **变更系统**（ONES/Apollo/GitLab） | 4 人 | 告警时自动拉取同期变更 | 🔴 优先，根因诊断的必要子能力 |
| **RedBI**（报表系统） | 3 人 | X-Ray 数据定时写入 | 🟠 中期，需跨产品协调 |
| **OLAP/在线 Doc** | 3 人 | 数据导出到下游分析系统 | 🟠 中期，通用导出能力 |
| **CI/CD 流水线**（云效） | 1 人 | 发布时自动获取 X-Ray 日志 | 🟡 中期，与发布系统联动 |
| **CAT**（老打点系统） | 1 人 | 解决 CAT 维度不足问题 | 🟢 低优先 |

**战略判断**：用户真正期望的不是"更好用的监控平台"，而是：
- **往下游输出**：日志/指标/分析结果流向 RedBI、OLAP、Doc
- **往上游接入**：从变更系统、代码仓库获取上下文
- **横向穿透**：告警触发时自动聚合跨服务、跨系统的上下文

这意味着 X-Ray Skill 的产品定位应该是**可观测数据的 HUB**，而不只是"查询工具的 AI 化"。

## 十二、SKILL 优先级与设计建议

### 12.1 SKILL 分类体系重构

建议放弃问卷中"基础查询/操作/AI链路/智能诊断"的功能分类，改用**任务类型**：

| 任务类型 | 核心特征 | 代表 SKILL |
| --- | --- | --- |
| **响应型** | 告警/客诉触发，被动响应 | 告警根因诊断、客诉链路回溯 |
| **探索型** | 有线索主动查，逐步收敛 | 日志智能聚合、TraceID 全链路展开 |
| **构建型** | 零到一，创建监控资产 | 自然语言建大盘/告警、阈值智能推荐 |
| **集成型** | 数据从 X-Ray 流向其他系统 | 日志定时导出、RedBI 数据管道 |
| **接口型** | 供 Bot/Agent 调用 | X-Ray MCP Server、日志查询 API |

### 12.2 优先级排序

**🔴 P0：立即做**

**1. 告警根因诊断 Skill**
- 覆盖 A 类用户（53%），使用场景最紧迫
- MVP 输入：服务名 + 告警时间（或告警 ID）
- MVP 输出：①异常日志聚类摘要（Top 3 错误）②指标下钻（哪个 Pod/接口）③同期变更关联 ④初步根因假设
- 关键依赖：**变更系统联动是必做，不是加分项**

**2. 日志智能查询 Skill（分层交付）**
- L1（立即做）：自然语言查日志（服务名/关键词/时间 → 结构化结果）
- L4（跟进做）：日志智能聚类（error 增量分析，标出新增哪类错误）
- L5（关键）：结构化日志输出接口，供 AI Agent 消费（MCP/Tool Call）

**3. 链路追踪 Skill（不是"链路查询"）**
- MVP 输入：服务名 + 时间范围（自动找 TraceID）或 TraceID
- MVP 输出：全链路树形图（各跳延迟 + 错误 + 关键日志），自动标注最可疑节点
- 差异化：现有功能需要手动逐跳查，Skill 要做到"自动找到问题在哪跳"

---

**🟠 P1：重要，近期做**

**4. 监控资产配置 Skill（告警 + 大盘一体化）**
- 自然语言建大盘：描述想观测什么 → AI 决定用什么指标建什么图
- 告警规则配置：自然语言创建 + **基于历史数据智能推荐阈值**（差异化关键）
- 覆盖巡检：检查服务告警覆盖率，批量修复未覆盖服务

**5. 发版健康巡检 Skill**
- 触发：发版事件
- 能力：自动对比 canary/全量指标与日志，支持跨集群批量查询，输出发版健康报告
- 频率信号：川岛 50 次/周，是高价值高频场景

**6. RT/性能归因 Skill**
- 朝风原话："RT 异常上涨后，递归地看下游的下游，这个过程相当枯燥且重复"
- MVP：输入异常服务名 → 自动递归查 RPC 调用链路 → 定位 RT 上涨的根因服务

---

**🟡 P2：中期规划**

**7. X-Ray MCP Server / 数据管道**
- 供 Coding Agent 调用的 MCP 接口（最高战略价值，但实现复杂）
- X-Ray → RedBI 定时数据管道

**8. 成本分析 Skill**
- 先访谈（约桑铎等）确认具体场景再规划

---

**⏸️ 暂缓**

- **Agent 状态分析**：11 人选但 0 人描述场景，用户以为自己想要但没有真实痛点
- **事件查询独立 Skill**：13 人选但无场景描述，可作为其他 Skill 的子能力使用

## 十三、三个隐藏高价值机会

### 机会 A：X-Ray Skill 成为 Coding Agent 的内置工具（战略级）

至少 13 人描述的是"AI 自动读日志帮我 debug"，但他们想要的不是"X-Ray 内嵌 AI"，而是在 **Cursor/Claude/内部 Coding Agent 里直接调用 X-Ray 能力**。

- 产品形态：把 X-Ray Skill 设计为 MCP Server / Tool Call 规范接口
- 价值：这比"X-Ray 内嵌 AI"的价值大一个量级
- 验证方式：找向山、清隆做"Coding Agent 直接调用 X-Ray 日志"的 PoC

### 机会 B：运营/非研发人员的日志查询

丽达明确说："需要对外提供查询能力，对运营排查 case，需要可以直接指定环境和服务查询日志。"

- 现有问卷只发给了研发，但运营有客诉场景的查日志需求
- 如果能做到"零门槛，运营也能用"，用户盘从 49 人扩展到几百人
- 验证方式：找 1-2 个业务运营同学了解排查流程

### 机会 C：发版观测自动化（被低估的高频刚需）

- 川岛：策略服务发布观测 50 次/周（极高频）
- 朝风：核心服务 10 个集群，每次发版都要一个一个查
- 这个场景不在现有 Skill 分类里，但需求非常真实且频率极高

## 十四、高风险决策点

| 风险 | 信号 | 建议应对 |
| --- | --- | --- |
| 把"功能选择率高"当"需求明确" | 成本分析/Agent状态分析：高选择率但无场景描述 | 进入 P0 前必须做用户访谈确认 |
| 变更系统联动被当成加分项 | 灵梦/狸克/无忆都明确提到变更关联 | 视为告警根因诊断 Skill 的核心依赖，不打通则 Skill 没有价值 |
| 过度自动化导致用户不信任 | 恋次："参考文档，查了半天没查出来" | MVP 做"AI 建议+用户确认"，不上来做全自动决策 |
| 把周期性需求做成 Skill | 秋水/格飞/孙瀚文的定时场景 | 明确区分 Skill（按需触发）vs Agent 任务（调度触发） |
| 算法工程师需求被后端需求淹没 | GPU 利用率/Memory OOM/PS-worker 资源分析 | 单独设计算法工程师角色的 Skill |
