## 一、设计原则

### 1.1 为什么不能按功能页面封装 Skill

按 X-Ray 页面功能 1:1 映射 Skill，本质是把平台导航栏翻译成了 Skill 列表。Agent 拿到后面临的问题和用户一样——知道有哪些功能，但不知道面对具体任务应该怎么组合调用、中间结果怎么流转。

根本矛盾：用户/Agent 输入的是「任务」，功能型 Skill 描述的是「能力」，两者之间有翻译鸿沟，只能靠调用方自己填补。

### 1.2 Agent Native 的三个设计准则

准则一：入参是用户任务中自然会说出的词，不是平台内部 API 参数

用户自然语言里的词：服务名、时间范围、错误关键词、TraceID
平台内部概念（不应暴露给调用方）：logview、transaction、problem、endpoint pattern

准则二：输出是可直接决策的结论，不是原始数据

❌ 返回 200 条日志列表
✅ 「发现 3 类异常，主要是 NPE，集中在 14:32，对应本次发版的 commit xxx」

准则三：Skill 内部自己完成编排，不让 Agent 在多个 Skill 之间串联编排

### 1.3 三层 Skill 架构

```
场景 Skill  ──  特定角色/业务定制封装，内嵌领域知识
    ↑ 调用
复合 Skill  ──  编排多个原子，解决完整任务意图，有推理层，输出结论
    ↑ 调用
原子 Skill  ──  单一数据维度取数，零推理，输出结构化数据
```

用户和外部 Agent 只感知复合 Skill 和场景 Skill；原子 Skill 是内部构件，不对外单独暴露。

### 1.4 原子 Skill 的拆分依据

拆分依据是可观测数据的自然维度，不是 X-Ray 的页面结构：

信号类型（日志 / 链路 / 指标 / 告警 / 变更）× 聚合粒度（单条 / 列表 / 统计）

合格原子 Skill 的判断标准：
- 入参里没有 X-Ray 平台特有概念
- 输出里没有推理判断（不下「是否异常」的结论）
- 能被至少 3 种不同的复合 Skill 复用

## 二、原子 Skill（11 个）

### 日志域（3 个）

#### A1 · 按条件查日志 xray-log-query

解决什么：按指定条件从 X-Ray 取出原始日志列表

入参：serviceName · timeRange · keywords[]（可选）· level（可选：ERROR/WARN/INFO）· traceId（可选）· uid（可选）

出参：日志列表，每条含 timestamp / level / content / traceId / podName

不做什么：不分析日志含义，不判断是否异常

需求来源：山峰「提供uid和接口名称，直接获取最近的traceId」/ 杨青「输入uid和时间范围，给我接口日志」

现有状态：✅ 已有 xray-log-query，需补充 uid/keywords 入参，标准化出参格式

---

#### A2 · 日志聚类分析 xray-log-cluster

解决什么：对一批日志按错误 pattern 分组，识别新增异常类型和数量趋势

入参：serviceName · timeRange · baselineTimeRange（可选，用于识别新增 pattern）

出参：pattern 分组列表，每组含 pattern模板 / count / firstSeen / trend（新增/持续/下降）/ sampleLog

不做什么：不判断哪个 pattern 是根因，只返回聚类结果

需求来源：狸克「能帮我聚合出这条日志」/ 海格「自动筛选报错相关日志去除无关」/ 灵梦「对异常聚类然后定位对应代码」

现有状态：🔵 xray-log-query 聚类能力确认中，需明确拆为独立原子 Skill

---

#### A3 · 日志统计查询 xray-log-stats

解决什么：从日志中对指定字段做 group by / count 统计分析

入参：serviceName · timeRange · extractField（正则或字段名）· groupBy · aggregation（count/sum）

出参：统计结果表格（dimension / count / percentage）

不做什么：不做趋势解读，只返回统计数据

需求来源：格飞「从日志详情统计，解决 CAT 打点 label 太多不支持的情况」

现有状态：🔴 未建设

---

### 链路域（3 个）

#### A4 · 搜索链路列表 xray-trace-search

解决什么：按条件找出符合要求的 TraceID 列表

入参：serviceName · timeRange · endpoint（可选）· hasError（可选）· minDuration（可选，ms）· uid（可选）

出参：TraceID 列表，每条含 traceId / startTime / totalDuration / isError / entryEndpoint

不做什么：不分析链路内容，只返回符合条件的 trace 列表

需求来源：山峰「提供uid和接口名称直接获取最近的traceId」/ 孔明「查找访问redkv错误的trace」

现有状态：✅ 已有，需去除 X-Ray 内部概念（transaction/problem），改用通用入参

---

#### A5 · 查单条链路详情 xray-trace-detail

解决什么：给定 TraceID，返回该请求完整的跨服务调用树

入参：traceId

出参：调用树，每节点含 serviceName / endpoint / duration / statusCode / errorMessage / 关联日志片段

不做什么：不判断哪跳有问题，只返回原始调用树数据

需求来源：蛮骨「根据traceId查询链路日志」/ 萧瑟「输入traceId自动拉取链路，提示报错用户信息」/ 易奕「直接丢个traceID摘取关键request和response」

现有状态：✅ 已有 xray-logview-analyzer，需改名并标准化出参，去除「logview」概念

---

#### A6 · 查异常分布 xray-exception-stats

解决什么：查询某服务在某时段内各类异常/错误码的数量分布

入参：serviceName · timeRange · exceptionType（可选精确过滤）

出参：异常类型列表，每条含 exceptionClass / count / firstSeen / lastSeen / sampleTraceId

不做什么：不做根因判断，只返回异常分布数据

需求来源：豫津「一键总结错误原因并定位到错误代码行」（前置数据）/ 灵梦「诊断是否有大量异常」

现有状态：🔴 未建设（exception分析 todo 中，需将数据层拆出）

### 指标域（2 个）

#### A7 · 查接口指标曲线 xray-metric-query

解决什么：查询某服务/接口的成功率、RT、QPS 等指标时序数据

入参：serviceName · endpoint（可选）· timeRange · metrics[]（successRate/p99/p95/qps）· granularity（1min/5min）

出参：时序数据，每条含 timestamp / metricName / value，支持多指标并返

不做什么：不判断指标是否异常，不做趋势解读

需求来源：秋水「希望能自动生成包含上下游流量放大的文档」/ 川岛「50次/周策略服务发布观测」/ 王凯丽「收到告警后快速查看当前异常指标实时值」

现有状态：🔴 未建设（指标模块当前空白）

---

#### A8 · 查 Pod 与机器指标 xray-pod-metric

解决什么：查询某服务各 Pod 的 CPU/内存/状态等资源指标

入参：serviceName · timeRange · podName（可选，不传则返回所有）

出参：Pod 列表，每条含 podName / cpu / memory / status / restartCount / hostIp

不做什么：不判断 Pod 是否健康，只返回指标数据

需求来源：桑铎「查看指标异常的是哪个pod，在哪台机器上，联动日志查询为什么异常」/ 小强「通过指标分析定位问题pod」

现有状态：🔴 未建设

### 告警域（2 个）

#### A9 · 查告警事件列表 xray-alert-events

解决什么：查询某服务在某时段触发的告警事件列表

入参：serviceName · timeRange · severity（可选：P0/P1/P2）· status（可选：firing/resolved）

出参：告警事件列表，每条含 alertId / ruleName / severity / firedAt / resolvedAt / affectedMetric / description

不做什么：不判断告警是否合理，不做根因分析

需求来源：王凯丽「收到告警后快速查看当前异常指标实时值」/ 承太郎「收到告警，查原因」

现有状态：🔴 未建设（todo 中的「当前服务告警事件处理」需拆出数据层）

---

#### A10 · 查告警规则 xray-alert-rules

解决什么：查询某服务配置的告警规则详情

入参：serviceName 或 ruleId · ruleType（可选：pql/cat）

出参：规则列表，每条含 ruleId / ruleName / condition / threshold / notifyGroup / enabled / lastTriggeredAt

不做什么：不评估规则是否合理，不判断覆盖是否完整

需求来源：魏晋「告警太多，有时找不到谁负责？告警对应的监控在哪」

现有状态：✅ 已有 alarm-rule-query（入参只有规则id），需补充按服务名查询能力

### 外部系统域（1 个）

#### A11 · 查变更记录 xray-change-query

解决什么：查询某服务在某时段内的发布和配置变更记录

入参：serviceName · timeRange

出参：变更列表，每条含 changeTime / changeType（deploy/config/feature_flag）/ operator / description / commitId

数据来源：ONES + Apollo + GitLab（对接外部系统）

不做什么：不判断变更是否导致了异常

需求来源：灵梦「同时在gitlab/ones/apollo查看当前应用是否有变更，这一步可以通过skill自动化」/ 狸克「分析error变多时对应的变更有哪些」/ 无忆「关联近期业务服务的变更」

现有状态：🔴 未建设，⚠️ 这是告警根因诊断的关键依赖，不是可选项

## 三、复合 Skill（8 个）

设计原则：编排多个原子 Skill，内部完成数据获取 + 推理，对外只暴露任务意图和结论。调用方不需要知道内部调了哪些原子。

---

### 响应型（被动触发）

#### C1 · 告警根因诊断

解决的任务意图：「告警触发了，告诉我为什么，问题出在哪里」

触发方式：告警事件推送，或用户输入「xxx服务刚告警了是什么问题」

内部编排步骤：
1. A9 查告警事件详情（确认告警时间和影响范围）
2. A7 查该服务告警时间窗口的指标变化
3. A6 查异常分布，找 Top N 异常类型
4. A4 按异常取代表性 TraceID
5. A5 查单条链路详情，定位哪跳出问题
6. A2 日志聚类，归纳错误 pattern
7. A11 查同期变更记录
8. LLM 推理：综合以上数据输出根因假设 + 置信度

输出：
- 异常摘要（什么在报错、量级多少、从何时开始）
- 根因定位（哪个服务/接口/代码路径，置信度高/中/低）
- 变更关联（是否有同期变更，是否可能相关）
- 建议行动（修复建议或回滚建议）

需求来源（18人）：灵梦/魏晋/晦明/狸克/承太郎/王凯丽等

⚠️ 关键依赖：A11 变更记录是必做，没有变更关联则诊断价值降低 70%

---

#### C2 · 客诉/UID 链路回溯

解决的任务意图：「某个用户反馈有问题，查他当时的请求发生了什么」

触发方式：输入 uid + 时间范围 + 接口名（可选）

内部编排步骤：
1. A1 用 uid 查该用户的日志
2. A4 用 uid + 时间搜相关链路
3. A5 查有错误的链路详情
4. LLM 推理：总结用户请求的异常路径

输出：
- 该用户在该时间段的请求摘要
- 发现的异常（有无错误、在哪个服务、错误内容）
- 请求耗时分析

需求来源：库鲁「客诉排查，将xray日志查询和项目代码阅读结合」/ 丽达「需要对外提供查询能力，对运营排查case」/ 凯恩「定位客诉问题，一般会提供关键词、时间」

### 探索型（主动排查）

#### C3 · 报错快速定位

解决的任务意图：「我的服务有报错，帮我找到是什么错、在哪里」

触发方式：输入 serviceName + 时间范围 + 错误关键词（可选）

内部编排步骤：
1. A6 查异常分布，找最频繁的异常类型
2. A2 日志聚类，归纳错误 pattern
3. A4 取代表性 TraceID
4. A5 查链路，定位最深层报错节点
5. LLM 推理：定位根因代码路径，生成错误摘要

输出：
- Top N 错误类型（数量 + 首次时间）
- 最可疑的代码位置（service + 异常堆栈）
- 代表性 TraceID（供用户进一步查看）

需求来源：山峰「通过关键词查询最近日志，找到traceId并查看err/warn日志」/ 向山「模型报错100%依赖xray日志」/ 豫津「一键总结错误原因并定位到错误代码行」

---

#### C4 · RT/性能劣化归因

解决的任务意图：「我的服务 RT 变高了/成功率下降了，是哪里导致的」

触发方式：输入 serviceName + 异常指标类型 + 时间范围

内部编排步骤：
1. A7 查该服务指标曲线，确认异常时间窗口
2. A4 搜该时间段的慢链路（按 P99 基线过滤）
3. A5 递归查慢链路各跳耗时，找最慢的下游节点
4. A8 查 Pod 指标，排查资源瓶颈
5. A11 查变更记录
6. LLM 推理：定位性能瓶颈，输出归因

输出：
- 耗时分解（各跳贡献的延迟占比）
- 根因服务/接口
- 资源状态（是否有 Pod 异常）
- 变更关联

需求来源：朝风「RT异常上涨后，递归地看下游的下游，枯燥且重复」/ 狸克「花了好几分钟才发现下游cpu利用率变为100%」

---

#### C5 · 全链路诊断（TraceID 已知）

解决的任务意图：「给你这个 TraceID，帮我分析这次请求哪里有问题」

触发方式：输入 traceId

内部编排步骤：
1. A5 查完整链路树
2. A1 按 traceId 查关联日志
3. LLM 推理：标注异常节点，解释每跳状态，指出最可能的问题点

输出：
- 链路各节点的耗时和状态摘要
- 异常节点高亮（错误原因 + 关键日志片段）
- 问题结论

需求来源：易奕「直接丢个traceID，摘取关键request和response以及业务日志」/ 蛮骨「有告警时提供traceId，分析调用链路上的问题（端到端全链路）」/ 萧瑟「输入traceId通过skill自动拉取链路」

### 构建型（创建监控资产）

#### C6 · 自然语言建大盘

解决的任务意图：「我想观测 xxx 服务的 xxx 指标，帮我建个看板」

触发方式：自然语言描述监控需求

内部编排步骤：
1. LLM 理解意图：提取服务名、指标类型、时间粒度、图表偏好
2. A7/A8 查询现有指标，验证数据可用性
3. 自动生成 PQL / 图表配置
4. 调用 X-Ray 写入 API 创建大盘

输出：
- 已创建的大盘链接
- 包含的图表说明（每张图的意义和观测建议）

需求来源：镜玄「通过自然语言创建大盘，而不是去平台上做繁琐的配置」/ 小强「批量生成该场景观测看板」/ 王坤「以对话形式完成搭建，skill会询问需要什么图表类型」

---

#### C7 · 告警规则配置与优化

解决的任务意图：「帮我给 xxx 服务配个告警」或「帮我看看现有告警配得合不合理」

触发方式：自然语言描述告警需求，或输入服务名做巡检

内部编排（配置场景）：
1. LLM 理解意图：提取服务/接口、告警类型、阈值需求
2. A10 查现有同类规则作为参考模板
3. A7 查历史指标，基于历史波动范围智能推荐阈值
4. 调用 X-Ray 写入 API 创建告警规则

内部编排（巡检场景）：
1. A10 查该服务所有现有规则
2. A7 查近期指标，验证规则是否会触发（有效性校验）
3. A9 查近期告警事件，识别哑铃告警
4. LLM 分析：覆盖是否完整，阈值是否合理，给出优化建议

输出：
- 已创建的规则详情 + 推荐阈值理由
- 或：现有规则的有效性报告 + 优化建议清单

需求来源：魏晋「告警规则的确定很依赖于经验；二次调整成本过高」/ 新罗「根据历史自动推荐阈值」/ 小强「验证告警规则有效性」/ 球棒「支持静默告警规则，无权限时对自己静默」

### 观测型（周期/主动巡检）

#### C8 · 发版健康检查

解决的任务意图：「我刚发了版，帮我确认是否有新的异常引入」

触发方式：发版事件触发，或输入 serviceName + 版本时间点

内部编排步骤：
1. A6 查发版后时间窗口的异常分布，对比发版前基线
2. A7 查关键指标曲线（成功率/RT），对比发版前 delta
3. A8 查 Pod 状态（新 Pod 是否健康启动）
4. A2 日志聚类，识别是否出现新的错误 pattern
5. LLM 推理：判断是否有发版引入的新问题

输出：
- 健康 / 警告 / 异常 三级结论
- 指标对比（发版前后 delta）
- 新增错误 pattern（如有，附样本日志）
- 建议：继续放量 / 暂停观察 / 立即回滚

需求来源：朝风「每次发版一个集群一个集群查日志量，枯燥且重复」/ 川岛「50次/周策略服务发布观测」/ 灵梦「查看应用是否有异常激增」

## 四、场景 Skill（4 个）

设计原则：针对特定角色或业务场景，把复合 Skill 用领域语言重新包装，内嵌角色专属的上下文判断逻辑。

---

#### S1 · 算法任务异常诊断（算法/模型工程师专用）

与通用报错定位（C3）的区别：
- 理解 PS/Worker 架构，能区分是 PS 问题还是 Worker 问题
- 理解训练任务生命周期（初始化/训练/checkpoint/结束各阶段的正常日志模式）
- 能识别模型特有异常：OOM / GPU利用率异常 / 梯度爆炸 / checkpoint失败
- 输出包含模型配置参数的修改建议

内部调用：A1 + A8（含 GPU 指标）+ A2 + C3 + 领域知识推理

输出扩展：在通用诊断基础上，额外给出：资源使用分析（PS/Worker各自CPU/内存/GPU）+ 建议的配置参数修改

需求来源：海格「自动定位到PS/worker哪部分资源有问题，如memory oom，给出模型建议配置参数」

---

#### S2 · Flink 作业诊断（数据引擎团队专用）

与通用报错定位（C3）的区别：
- 理解 Flink Job/TaskManager/JobManager 概念和日志模式
- 联动 Pod 事件（TaskManager 挂掉）+ 机器指标 + 作业日志三路数据
- 识别 Flink 特有问题：背压 / Checkpoint 失败 / State Backend 异常 / Watermark 延迟
- 入参支持 jobName 而不只是 serviceName

内部调用：A1 + A8 + Pod事件查询 + A6 + 领域知识推理

需求来源：库奇「flink作业诊断需要查询flink pod的事件、日志，以及对应机器的监控」/ 帕恩「离线引擎任务在失败时，希望能查询XRay Skills获取错误信息」

---

#### S3 · 开发调试助手（Coding Agent 专用 MCP 接口）

与其他 Skill 的本质区别：这个 Skill 的消费者是 Coding Agent，不是人类

触发方式：Coding Agent 在 debug 流程中自动调用（MCP Tool Call）

输入：代码片段（可选）+ serviceName + timeRange

内部编排：
1. 若有代码片段，LLM 从代码中推断可能的报错关键词和日志模式
2. A1 按推断的关键词查日志
3. A2 聚类找主要错误 pattern
4. A4 找代表性 TraceID
5. A5 查链路定位具体报错节点

输出格式：JSON（不是自然语言），包含：
- errorType / count / firstSeen
- stackTrace / relevantClass / relevantMethod
- suggestedFix（基于错误类型的通用修复建议）
- relevantLogLines[]（供 Coding Agent 展示给用户的关键日志行）

暴露方式：MCP Tool，不做对话式界面

需求来源：向山「让AI直接读xray日志，开发测试可以完成闭环」/ 清隆「agent解决bug时，根据代码去搜日志，根据日志情况自动规划解决方案」/ 恋次「debug skill，有agent自动打断点，输出分析报告」/ 十夜「希望通过API来查询日志，方便bot自动分析larc任务报错原因」

---

#### S4 · SRE 服务巡检（SRE 专用）

与告警配置 Skill（C7）的区别：
- 面向 SRE 的日常例行检查，不是事件响应
- 支持批量：一次巡检多个服务，输出统一格式报告
- 输出适合定时发送到群（Markdown 格式，可直接阅读）
- 内嵌 SRE 视角的评判标准：告警覆盖率达标阈值、规则有效性判断逻辑、哑铃告警识别

内部调用：A9 + A10 + A7 + C7（有效性校验子流程）+ 批量并行执行

输出（巡检报告格式）：
- 服务健康总览（N个服务，X个正常，Y个需关注）
- 逐服务明细：告警覆盖率 / 有效性 / 近期触发情况 / 优化建议
- 需要立即处理的项目清单

需求来源：小强「检查服务告警覆盖情况确保覆盖率达标；验证告警规则有效性，高频使用」/ 萧峰「监控巡检、增删改告警、SLO添加修改」/ 球棒「支持对服务、策略、接口等配置告警规则」

## 五、架构关系与现有 Skill 归位

### 5.1 整体架构

```
┌────────────────────────────────────────────────────────────────┐
│  场景 Skill（领域语言封装，面向特定角色）                          │
│  S1 算法任务诊断   S2 Flink作业诊断                              │
│  S3 开发调试(MCP)  S4 SRE服务巡检                               │
└───────────────────────────┬────────────────────────────────────┘
                            │ 调用
┌───────────────────────────▼────────────────────────────────────┐
│  复合 Skill（任务意图 → 编排 → 结论）                             │
│  C1 告警根因诊断    C2 客诉链路回溯    C3 报错快速定位            │
│  C4 RT劣化归因     C5 全链路诊断      C6 自然语言建大盘           │
│  C7 告警规则配置   C8 发版健康检查                               │
└───────────────────────────┬────────────────────────────────────┘
                            │ 调用
┌───────────────────────────▼────────────────────────────────────┐
│  原子 Skill（数据维度取数，零推理，结构化输出）                     │
│  日志域：A1 按条件查日志  A2 日志聚类  A3 日志统计查询            │
│  链路域：A4 搜索链路列表  A5 单条链路详情  A6 异常分布            │
│  指标域：A7 接口指标曲线  A8 Pod机器指标                         │
│  告警域：A9 告警事件列表  A10 告警规则                           │
│  外部域：A11 变更记录（ONES/Apollo/GitLab）                     │
└────────────────────────────────────────────────────────────────┘
```

### 5.2 现有 Skill 归位与改造建议

| 现有 Skill | 归位 | 改造说明 |
| --- | --- | --- |
| xray-trace-search | → A4（改造） | 去除 endpoint/exception 等 X-Ray 内部入参，改用通用语言 |
| xray-logview-analyzer | → A5（改造） | 改名为「单条链路详情」，出参去除 logview 概念，标准化节点数据结构 |
| xray-single-trace-analysis | → C5（升级） | 在原有基础上加推理层，从「返回数据」变成「输出结论」 |
| xray-log-query | → A1（升级） | 补充 uid/keywords 入参，标准化出参，聚类能力独立为 A2 |
| alarm-rule-query | → A10（扩展） | 补充按服务名批量查询能力，现有按规则ID查询保留 |
| exception分析（todo） | → A6 + 并入C3 | 数据层做 A6，推理层并入 C3 报错快速定位，不单独暴露 |
| 链路聚合诊断（todo） | → C1 或 C4 | 不是原子能力，直接在复合层实现 |
| 接口/服务/异常分析（todo） | → C3 + C4 合并 | 三个功能页面合并为两个任务型复合 Skill |
| 告警事件处理（todo） | → A9 + C1 拆开 | 数据查询是 A9 原子，「是否有风险」的判断是 C1 复合 |
| 告警规则有效性校验（todo） | → C7 内部子流程 | 不单独暴露，在告警配置 Skill 内部执行 |
| 告警覆盖情况扫描（todo） | → C7 或 S4 内部 | 批量扫描逻辑内嵌在巡检 Skill 中 |

## 六、关键风险与约束

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| A11 变更记录对接外部系统 | C1 告警根因诊断的核心依赖，缺失则诊断价值大打折扣 | 优先排期与 ONES/Apollo 团队确认数据接口 |
| 指标域（A7/A8）当前空白 | C4/C8 均无法推进 | 指标模块建设需纳入 Q2 前置依赖 |
| 用户对 AI 诊断结论的信任度 | 过度自动化可能导致用户不敢用 | MVP 阶段所有结论附置信度 + 原始证据链，采用「AI建议+用户确认」模式 |
| 复合 Skill 的 LLM 推理成本 | 每次调用成本较高，高频场景可能受限 | 对高频原子查询做结果缓存；推理层只在明确需要结论时触发 |
| S3 MCP 接口的安全鉴权 | Coding Agent 直接调用需要控制数据访问范围 | MCP 接口需接入现有鉴权体系，按服务维度做数据隔离 |
