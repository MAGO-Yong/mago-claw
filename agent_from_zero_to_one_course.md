## 第一章【认知篇】Agent 到底是什么

### 1.1 一句话定义

**Agent = 大模型 + 感知 + 工具 + 记忆 + 规划**

传统软件是"你告诉它做什么，它照着做"。Agent 是"你告诉它目标，它自己想方法"。

### 1.2 用一个类比理解

| 角色 | 比喻 |
|------|------|
| **大模型（LLM）** | 一个知识渊博但只会说话的人，没有手和脚 |
| **工具（Tools）** | 给它的手——可以操作电脑、查数据库、发邮件 |
| **记忆（Memory）** | 它的笔记本——记住你说过的话和你的偏好 |
| **规划（Planning）** | 它的大脑前额叶——遇到复杂任务，先想步骤再动手 |
| **感知（Perception）** | 它的眼睛和耳朵——看到系统告警、日志变化、用户输入 |

> 💡 **关键点**：没有工具的大模型只是个聊天机器人；有了工具，它才变成了 Agent。

### 1.3 Agent vs 传统软件 vs 聊天机器人

| 维度 | 传统软件 | 聊天机器人 | Agent |
|------|---------|-----------|-------|
| 输入 | 结构化操作（点按钮/填表单） | 自然语言 | 自然语言或事件触发 |
| 输出 | 固定结果 | 文本回复 | 文本 + 行动结果 |
| 执行方式 | 预设代码路径 | 生成回复 | 自主规划→调用工具→迭代 |
| 记忆 | 数据库记录 | 单次对话上下文 | 跨会话持久记忆 |
| 出错时 | 报异常 | 胡编乱造 | 尝试其他路径 |

### 1.4 三种 Agent 形态

**形态一：对话式 Agent（Chat Agent）**
- 你说话 → 它回答 → 继续聊
- 典型：ChatGPT、Claude

**形态二：任务式 Agent（Task Agent）**
- 你给目标 → 它规划步骤 → 调用工具 → 交付结果
- 典型：帮你写报告的研究 Agent、帮你 debug 的诊断 Agent

**形态三：自主 Agent（Autonomous Agent）**
- 你设规则 → 它持续监控 → 发现问题自动处理 → 汇报给你
- 典型：24h 自动巡检系统、自动发布 Agent

> 🔍 **产品经理视角**：你正在做的 XRay 告警诊断 Agent 属于"形态二 + 形态三"——既能接受用户对话进行诊断，也能自动监控告警并执行诊断。

### 1.5 延展概念：什么是 Agent 框架？

**Agent 框架 = 把"连接大模型、管理工具、维护记忆、实现规划"这些事封装成可复用的组件**。

想象你要盖房子：
- **大模型 API** = 砖头（原材料）
- **Agent 框架** = 预制构件 + 施工图纸（帮你快速组装）
- **你自己写的 Agent** = 最终的房子

没有框架，你需要自己写：HTTP 请求怎么调大模型、工具结果怎么喂回去、对话历史怎么管……有了框架，你只需要定义：用什么模型、有哪些工具、系统提示词是什么。

> 📌 **常见框架**：LangChain Deep Agents、OpenClaw、Claude Agent SDK、CrewAI、AutoGen

## 第二章【架构篇】Agent 的五层架构

### 2.1 整体架构图

```
┌─────────────────────────────────────────────────────┐
│                   用户交互层                          │
│  对话界面 / 告警触发 / 定时任务 / API 调用             │
└──────────────────────┬──────────────────────────────┘
                       │ 输入（用户请求 or 事件）
┌──────────────────────▼──────────────────────────────┐
│                   大脑层（LLM）                       │
│  ┌─────────┐  ┌─────────┐  ┌──────────────────────┐ │
│  │  理解   │→│  规划   │→│  决策（调用哪个工具）    │ │
│  └─────────┘  └─────────┘  └──────────────────────┘ │
└──────────────────────┬──────────────────────────────┘
                       │ 工具调用指令
┌──────────────────────▼──────────────────────────────┐
│                   工具层（Tools）                     │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐      │
│  │搜索  │ │读文件 │ │查DB │ │发邮件 │ │执行  │ ...  │
│  └──────┘ └──────┘ └──────┘ └──────┘ └──────┘      │
└──────────────────────┬──────────────────────────────┘
                       │ 工具执行结果
┌──────────────────────▼──────────────────────────────┐
│                   记忆层（Memory）                    │
│  ┌────────────┐ ┌────────────┐ ┌─────────────────┐  │
│  │ 短期记忆   │ │ 长期记忆   │ │ 技能库(Skills)  │  │
│  │ (对话历史) │ │(偏好/规则) │ │ (专业知识/SOP)  │  │
│  └────────────┘ └────────────┘ └─────────────────┘  │
└─────────────────────────────────────────────────────┘
```

### 2.2 每一层详解

#### 第一层：用户交互层（入口）

Agent 需要"被唤醒"。唤醒方式有多种：

| 唤醒方式 | 例子 |
|---------|------|
| 用户对话 | 在聊天界面输入"帮我查一下这个告警" |
| 事件触发 | 监控平台发现 CPU 异常 → 自动触发诊断 Agent |
| 定时任务 | 每天早上 8 点自动跑一次健康检查 |
| API 调用 | 其他系统通过接口调用 Agent |

> 🔍 **产品视角**：你的 XRay Agent 至少有三种唤醒方式——用户在界面提问、告警自动触发、定时巡检。

#### 第二层：大脑层（LLM 推理引擎）

这是 Agent 的"大脑"，负责三件事：

1. **理解**：用户的意图是什么？"帮我查告警" vs "这个服务怎么了" → 需要不同的处理
2. **规划**：要完成这个任务，需要哪些步骤？先查日志 → 再看指标 → 最后出结论
3. **决策**：下一步该调用哪个工具？是查日志工具还是查指标工具？

**关键概念：ReAct 模式**

ReAct = Reasoning（推理）+ Acting（行动），是 Agent 的核心工作模式：

```
用户问题："服务 A 为什么响应慢？"

大脑思考过程（ReAct 循环）：

📋 Thought 1: 服务响应慢，先看 CPU 指标
🔧 Action 1: 调用"查CPU指标"工具 → 结果：CPU 正常
📋 Thought 2: CPU 正常，再看内存
🔧 Action 2: 调用"查内存指标"工具 → 结果：内存使用率 95%
📋 Thought 3: 内存很高，再看有没有内存泄漏的日志
🔧 Action 3: 调用"查日志"工具 → 结果：发现 OOM 警告
📋 Thought 4: 找到了！内存不足导致响应慢
📝 Final Answer: 服务 A 响应慢的原因是内存使用率 95%，日志显示 OOM 警告，建议扩容或排查内存泄漏
```

> 💡 **理解关键**：Agent 不是一次性给出答案，而是"想一步 → 做一步 → 看结果 → 再想下一步"，反复循环直到得出结论。

#### 第三层：工具层（Tools）

工具是 Agent 的"手"，让它能操作外部世界。

**工具的本质**：一段代码 + 一段描述（告诉大模型这个工具是干嘛的）

```
┌─────────────────────────────────────┐
│  工具：查CPU指标                     │
│                                     │
│  描述：查询指定服务在指定时间段的     │
│        CPU 使用率                    │
│                                     │
│  输入参数：                          │
│    - service_name（服务名）          │
│    - time_range（时间范围）          │
│                                     │
│  返回值：CPU 使用率百分比            │
└─────────────────────────────────────┘
```

大模型看到工具描述后，会判断："用户问 CPU，我有这个工具，调用它！"

> 🔍 **产品视角**：XRay Agent 的工具可能包括——查指标、查日志、查变更、查拓扑、调诊断 SOP、发送通知……

#### 第四层：记忆层（Memory）

记忆让 Agent 不是"金鱼脑"（7 秒忘）。

| 记忆类型 | 类比 | 作用 | 例子 |
|---------|------|------|------|
| **短期记忆** | 工作记忆 | 当前对话的上下文 | 刚才用户问了什么、Agent 做了什么 |
| **长期记忆** | 笔记本 | 跨会话持久保存 | 用户偏好、项目规范、历史经验 |
| **技能记忆** | 操作手册 | 按需加载的专业知识 | 某个服务的排障 SOP、API 文档 |

#### 第五层：安全与治理层

Agent 能做很多事情，但需要约束：

- **权限控制**：这个 Agent 能看哪些数据？能执行哪些操作？
- **人在回路**：敏感操作（删数据、重启服务）需要人工审批
- **沙箱隔离**：让 Agent 在隔离环境里操作，不影响生产

## 第三章【大脑篇】大模型如何"思考"

### 3.1 大模型是什么

简单理解：一个**读了互联网上几乎所有文本的超级语言学家**。

给它一段文字，它能预测下一个最可能的词是什么。通过不断预测下一个词，它能生成连贯的回复。

### 3.2 大模型的能力边界

| 它能做 | 它不能做 |
|--------|---------|
| 理解自然语言 | 访问实时数据（需要工具） |
| 逻辑推理 | 记住上次对话（需要记忆） |
| 代码生成 | 操作文件系统（需要工具） |
| 知识问答（基于训练数据） | 知道今天发生了什么（需要搜索工具） |

> 💡 **核心理解**：大模型的"智商"很高，但"手脚"被绑着。Agent 框架就是给它解绑，让它能动手。

### 3.3 关键概念：Prompt（提示词）

Prompt = 你告诉大模型该怎么做的一段话。

**好 Prompt 的结构**：

```
你是一个【角色】，你的任务是【目标】。
请按照【步骤】来完成。
注意【约束条件】。
输出格式要求【格式】。
```

**Agent 的系统提示词示例**：

```
你是一个运维诊断专家。你的职责是：
1. 当收到告警时，先分析告警类型和严重级别
2. 按以下顺序排查：指标异常 → 日志错误 → 最近变更 → 拓扑依赖
3. 每一步排查后，给出置信度和下一步建议
4. 最终输出：根因分析 + 处置建议

注意：
- 不要编造数据，所有结论必须基于工具返回的结果
- 如果数据不足以得出结论，明确说"数据不足"
- 高风险操作必须建议人工审批
```

### 3.4 关键概念：Tool Calling（工具调用）

Tool Calling 是大模型的一个核心能力——**它不仅能生成文本，还能"决定调用某个工具"**。

工作流程：

```
1. 你把工具描述（名称、参数、用途）告诉大模型
2. 大模型分析用户问题
3. 大模型决定："我需要调用 X 工具，参数是 Y"
4. 框架执行工具，拿到结果
5. 把结果喂回给大模型
6. 大模型继续推理或给出最终答案
```

> 💡 **类比**：你让一个专家帮你解决问题，告诉他"你可以用这 5 个工具"，专家会自己判断该用哪个。

### 3.5 关键概念：上下文窗口（Context Window）

上下文窗口 = 大模型一次能"记住"的最大文字量。

| 模型 | 上下文窗口 | 相当于 |
|------|-----------|--------|
| Claude Sonnet 4 | 200K token | 约 15 万字，一本中等小说 |
| GPT-5.2 | 128K token | 约 10 万字 |
| Gemini 3 Pro | 1M token | 约 75 万字，几本书 |

**为什么上下文窗口很重要？**

Agent 的所有信息——系统提示、对话历史、工具结果——都要塞进上下文窗口。如果信息太多，就会：
- 丢失早期信息
- 推理质量下降
- 成本飙升（按 token 计费）

> 🔍 **产品视角**：这就是为什么需要"上下文工程"——管理 Agent 看到什么信息，而不是一股脑全塞进去。

### 3.6 模型选择指南

| 场景 | 推荐模型 | 原因 |
|------|---------|------|
| 复杂推理/诊断 | Claude Opus/Sonnet | 推理能力最强，工具调用稳定 |
| 编码任务 | GPT-5 Codex | 代码生成优化 |
| 超长上下文 | Gemini 3 Pro | 1M token，适合分析大量日志 |
| 成本控制 | GPT-5 Nano / Haiku | 便宜 10-50 倍 |
| 企业合规 | AWS Bedrock / Azure | 数据不出境 |

## 第四章【工具篇】Agent 的手和脚

### 4.1 工具的本质

工具 = **函数 + 描述**。

| 组件 | 说明 |
|------|------|
| 函数 | 实际执行逻辑（代码） |
| 描述 | 告诉大模型"我是干嘛的、怎么用我" |
| 参数 | 调用时需要传什么 |
| 返回值 | 执行后返回什么 |

### 4.2 工具分类

| 类别 | 工具举例 | 说明 |
|------|---------|------|
| **信息查询** | 查指标、查日志、查数据库 | 只读，安全 |
| **操作执行** | 重启服务、扩容、回滚 | 有副作用，需谨慎 |
| **内容生成** | 写报告、生成图表 | 基于已有信息产出新内容 |
| **外部交互** | 发邮件、发通知、调用 API | 与外部系统通信 |
| **自我工具** | 写文件、读文件、写 TODO | Agent 自己管理工作进度 |

### 4.3 关键概念：MCP（Model Context Protocol）

MCP = **Agent 连接外部工具的"USB 接口"标准**。

**没有 MCP 之前**：每个工具都要单独写对接代码，像一个设备一个专用接口。

**有了 MCP 之后**：所有工具都通过统一接口连接，像 USB 一样即插即用。

```
Agent ←─── MCP ────→ 工具服务器
                     ├── GitHub
                     ├── 数据库
                     ├── 监控平台
                     └── 邮件服务
```

> 💡 **类比**：MCP 就是 Agent 世界的 USB-C 接口，任何符合 MCP 标准的工具都能直接插上用。

### 4.4 关键概念：Skill（技能）

Skill = **打包好的专业能力**。

**工具 vs 技能的区别**：

| 维度 | 工具 | 技能 |
|------|------|------|
| 粒度 | 单一操作（如"查日志"） | 复合流程（如"Java OOM 排障 SOP"） |
| 内容 | 函数 + 描述 | 指令文档 + 模板 + 参考脚本 |
| 加载方式 | 始终可用 | 按需加载（Agent 判断需要时才读） |
| 类比 | 一把扳手 | 一整套"更换轮胎"的操作手册 |

**技能的"渐进式加载"机制**：

```
Step 1: Agent 启动时 → 只读每个技能的"标题和简介"（很省空间）
Step 2: 用户输入问题 → Agent 判断"这个问题需要 Java OOM 排障技能"
Step 3: Agent 加载完整的 Java OOM 排障技能文档（SOP 步骤、命令模板等）
Step 4: Agent 按照技能文档的 SOP 执行排查
```

> 🔍 **产品视角**：你的 XRay Agent 可以把每个服务的排障 SOP 做成一个 Skill。Agent 收到告警时，自动加载对应服务的 SOP，按步骤排查。

## 第五章【记忆篇】Agent 的记忆系统

### 5.1 记忆的三种类型

| 类型 | 类比人脑 | 持久性 | 用途 |
|------|---------|--------|------|
| **情景记忆** | "昨天我帮小明修了个 bug" | 跨会话 | 搜索历史经验，避免重复排查 |
| **语义记忆** | "Python 的列表是可变的" | 永久 | 项目规范、技术知识、用户偏好 |
| **程序记忆** | "我知道怎么弹钢琴" | 永久 | SOP、操作流程、技能 |

### 5.2 记忆的技术实现

**实现方式一：文件记忆（推荐）**

Agent 的记忆存储为文件（如 Markdown），每次启动时自动加载。

```
┌─────────────────────────────────────┐
│  AGENTS.md（Agent 的长期记忆文件）   │
│                                     │
│  ## 用户偏好                        │
│  - 正一喜欢简洁的结论优先           │
│  - 架构图用蓝紫渐变配色              │
│                                     │
│  ## 项目规范                        │
│  - XRay 告警诊断 SLO：MTTR < 5min   │
│  - P0 告警必须 5 分钟内响应          │
│                                     │
│  ## 历史教训                        │
│  - 2026-03-15：服务 A 的 CPU 告警   │
│    是日志采集进程导致的，已优化       │
└─────────────────────────────────────┘
```

**实现方式二：数据库记忆**

用数据库存储，支持按用户/场景隔离。适合多用户 Agent 平台。

```
用户 A 的记忆 → namespace: user-A → 只加载 A 的偏好
用户 B 的记忆 → namespace: user-B → 只加载 B 的偏好
```

### 5.3 关键概念：Namespace（命名空间）

Namespace = **记忆的"文件夹"，用来隔离不同用户/场景的数据**。

```
Store
├── user-alice/
│   └── preferences.md
├── user-bob/
│   └── preferences.md
└── my-agent/
    └── AGENTS.md（所有用户共享）
```

### 5.4 记忆的读写权限

| 场景 | 读 | 写 | 说明 |
|------|----|----|------|
| 个人偏好 | ✅ | ✅ | Agent 可以学习并更新 |
| 项目规范 | ✅ | ❌ | 只读，由管理员维护 |
| 历史经验 | ✅ | ✅ | Agent 可以新增记录 |
| 组织政策 | ✅ | ❌ | 只读，不可修改 |

### 5.5 关键概念：情景记忆（Episodic Memory）

情景记忆 = **记住过去的对话和经历**，而不只是结论。

**为什么重要？** 有时候解决方案不在最终结论里，而在排查过程中。比如：

```
上次排查服务 A 慢的问题：
- CPU 正常 → 内存 95% → 以为是内存泄漏
- 但进一步查发现是日志采集进程占用了大量内存
- 最终解决：优化日志采集配置

结论：服务 A 慢 ≠ 一定是应用本身的问题，也可能是基础设施
```

这种"排查路径 + 转折点 + 最终结论"的完整记录，就是情景记忆。

## 第六章【规划篇】Agent 如何拆解任务

### 6.1 规划的本质

规划 = **把一个复杂目标拆成一系列可执行的步骤**。

### 6.2 两种规划方式

**方式一：内置 TODO 列表**

Agent 自动维护一个任务清单：

```
□ 分析告警类型和严重级别
☐ 查询相关指标数据
☐ 检查最近变更
□ 查看服务日志
☐ 分析拓扑依赖
☐ 生成诊断结论
☐ 输出处置建议
```

**方式二：子 Agent 委派**

主 Agent 把子任务委派给专门的子 Agent：

```
主 Agent（指挥官）
├── 指派给"指标分析 Agent"：查 CPU/内存/网络指标
├── 指派给"日志分析 Agent"：查错误日志和异常堆栈
└── 指派给"变更分析 Agent"：查最近有没有发布/配置变更
```

### 6.3 关键概念：子 Agent（Subagent）

子 Agent = **主 Agent 创建的临时 Agent，用于隔离工作和并行执行**。

**为什么需要子 Agent？**

1. **上下文隔离**：子 Agent 做了 100 步排查，主 Agent 只收到最终结论，不会被中间过程撑爆
2. **专业化**：不同子 Agent 有不同工具和知识
3. **并行执行**：多个子 Agent 同时工作，加速诊断

```
┌──────────────┐
│  主 Agent     │ ← 只看到最终结果
│  （指挥官）    │
└──────┬───────┘
       │ task 工具
       ├─────────────┐
       ▼             ▼
┌──────────┐  ┌──────────┐
│ 研究 Agent │  │ 编码 Agent │
│           │  │           │
│ 做了 50 步 │  │ 做了 30 步 │
│ 搜索...   │  │ 编写...   │
└─────┬────┘  └─────┬────┘
      │ 最终报告     │ 最终代码
      ▼             ▼
  返回给主 Agent    返回给主 Agent
```

### 6.4 关键概念：异步子 Agent

**同步子 Agent**：主 Agent 等子 Agent 做完才能继续。

**异步子 Agent**：主 Agent 启动子 Agent 后，继续和用户对话，子 Agent 在后台跑。

```
异步模式：

用户："帮我分析一下服务 A"
主 Agent："好的，我已经启动了分析任务，ID: abc123，你先说别的"
用户："对了，服务 B 怎么样？"
主 Agent："服务 B 我也帮你看看..."
（后台：服务 A 的分析 Agent 在独立运行）
用户："服务 A 的分析结果出来了吗？"
主 Agent："出来了，结果是..."
```

> 🔍 **产品视角**：对于耗时较长的诊断任务（如全链路分析），异步子 Agent 是关键——用户不用傻等。

## 第七章【安全篇】如何防止 Agent 闯祸

### 7.1 安全问题的三个层次

| 层次 | 问题 | 解决方案 |
|------|------|---------|
| **数据访问** | Agent 能看到不该看的数据吗？ | 权限控制、命名空间隔离 |
| **操作执行** | Agent 能执行危险操作吗？ | 人在回路、审批机制 |
| **环境影响** | Agent 的操作会影响生产吗？ | 沙箱隔离、只读模式 |

### 7.2 关键概念：人在回路（Human-in-the-loop）

人在回路 = **Agent 执行敏感操作前，先暂停等人审批**。

```
Agent："我准备执行以下操作，请审批：
        操作：重启服务 recommend-service-prod-01
        原因：检测到内存泄漏，OOM 频繁
        影响：服务将中断约 30 秒"

用户：[✅ 批准] [✏️ 修改] [❌ 拒绝]
```

三种审批模式：
- **批准**：按原计划执行
- **修改**：改参数后执行（比如"等凌晨再重启"）
- **拒绝**：不执行

### 7.3 关键概念：沙箱（Sandbox）

沙箱 = **隔离的执行环境，Agent 在里面操作不会影响主机**。

```
┌──────────────────────────────┐
│  主机系统（你的电脑/服务器）   │
│  ┌────────────────────────┐  │
│  │     沙箱环境            │  │
│  │                        │  │
│  │  Agent 在这里执行代码    │  │
│  │  读文件、写文件、跑命令  │  │
│  │                        │  │
│  │  ── 边界 ──            │  │
│  │  Agent 无法触达主机文件  │  │
│  └────────────────────────┘  │
│                              │
│  主机的 .env / 私钥等         │
│  Agent 看不到也碰不到         │
└──────────────────────────────┘
```

**常见沙箱提供商**：

| 提供商 | 特点 |
|--------|------|
| LangSmith | 零配置，与 LangChain 生态集成 |
| Daytona | 快速启动，可自定义环境 |
| Modal | 支持 GPU，弹性扩展 |
| 本地 VFS | 轻量级虚拟文件系统 |

### 7.4 关键概念：文件权限（Permissions）

声明式权限 = **用规则定义 Agent 能读/写哪些路径**。

```
权限规则（从上到下匹配，第一条匹配的生效）：

规则1: 拒绝写 .env 文件（保护密钥）
规则2: 允许读/写 /workspace/ 下所有文件
规则3: 拒绝读/写其他所有路径
```

> ⚠️ **规则顺序很重要**：具体规则放前面，宽泛规则放后面。

### 7.5 安全最佳实践

| 场景 | 推荐做法 |
|------|---------|
| 开发调试 | 本地文件系统 + 人在回路 |
| 生产环境 | 沙箱隔离 + 只读权限 |
| 敏感操作 | 必须人工审批 |
| 多用户 | 按用户隔离命名空间 |

## 第八章【组合篇】多 Agent 协作

### 8.1 为什么需要多 Agent？

单个 Agent 能力有限，就像一个人干不了一个团队的事。

**场景举例**：一个完整的告警诊断

```
如果只有一个 Agent：
- 查指标、查日志、查变更、分析拓扑、出报告
- 步骤太多，上下文撑爆，容易遗漏

如果有多个 Agent：
- 主 Agent 协调
- 指标 Agent 查指标
- 日志 Agent 查日志
- 变更 Agent 查变更
- 各自并行，最后主 Agent 汇总
```

### 8.2 多 Agent 架构模式

**模式一：指挥官模式（Supervisor）**

```
           ┌──────────┐
           │ 指挥官   │
           └────┬─────┘
                │ 分派任务
         ┌──────┼──────┐
         ▼      ▼      ▼
      ┌───┐  ┌───┐  ┌───┐
      │ A │  │ B │  │ C │
      └───┘  └───┘  └───┘
```

- 指挥官负责协调和汇总
- 子 Agent 各自执行，互不干扰
- **适合**：复杂诊断、多步骤任务

**模式二：流水线模式（Pipeline）**

```
用户输入 → Agent A → Agent B → Agent C → 输出
           (理解)    (分析)    (总结)
```

- 每个 Agent 处理一个阶段
- 上一个的输出是下一个的输入
- **适合**：标准化流程

**模式三：投票模式（Voting）**

```
问题 → Agent A ↘
       Agent B  → 汇总决策 → 输出
       Agent C ↗
```

- 多个 Agent 独立给出答案
- 比较答案的一致性
- **适合**：高风险决策（如是否要回滚发布）

### 8.3 与 OpenClaw 多 Agent 的关系

OpenClaw 的多 Agent 是**平台级别**的多 Agent——不同 Agent 服务不同场景：

```
OpenClaw Gateway
├── Work Agent（XRay/XPILOT）
├── Life Agent（行程/万豪）
└── Invest Agent（投资）
```

Agent 框架的多 Agent 是**任务级别**的多 Agent——一个 Agent 内部为了完成某个任务创建多个子 Agent：

```
Work Agent
├── 指标诊断子 Agent
├── 日志分析子 Agent
└── 变更分析子 Agent
```

> 💡 **理解**：OpenClaw 管的是"哪个 Agent 服务谁"，Agent 框架管的是"一个 Agent 怎么干活"。

## 第九章【工程篇】从原型到生产

### 9.1 构建 Agent 的六个阶段

```
阶段1: 定义目标    → 这个 Agent 解决什么问题？
阶段2: 设计能力    → 它需要哪些工具、记忆、技能？
阶段3: 开发原型    → 用最少的组件跑通端到端流程
阶段4: 测试评估    → 它做得好不好？哪里需要改进？
阶段5: 安全加固    → 加权限、加审批、加沙箱
阶段6: 部署上线    → 接入生产环境，持续运营
```

### 9.2 阶段一：定义目标

**关键问题**：

| 问题 | 例子（XRay 告警诊断） |
|------|----------------------|
| Agent 解决什么问题？ | P1/P2 告警的人工排查效率低 |
| 用户是谁？ | SRE、值班工程师 |
| 成功标准是什么？ | MTTR 从 30min 降到 5min |
| 有哪些约束？ | 不能影响生产、需要人工审批 |

### 9.3 阶段二：设计能力

**工具设计**：

| 工具名 | 输入 | 输出 | 说明 |
|--------|------|------|------|
| query_metrics | 服务名、时间范围 | 指标数据 | 查 CPU/内存/延迟 |
| query_logs | 服务名、时间范围、关键词 | 日志片段 | 查错误日志 |
| query_changes | 服务名、时间范围 | 变更记录 | 查发布/配置变更 |
| get_topology | 服务名 | 拓扑图 | 查上下游依赖 |
| run_sop | SOP 名称、参数 | 诊断步骤和结果 | 执行标准排障流程 |

**记忆设计**：

| 记忆文件 | 内容 | 谁维护 |
|---------|------|--------|
| AGENTS.md | 诊断规范、SLO、用户偏好 | 管理员 |
| history/ | 历史诊断案例 | Agent 自动更新 |
| services/ | 各服务的基本信息 | 管理员 |

**技能设计**：

| 技能名 | 触发条件 | 内容 |
|--------|---------|------|
| CPU 异常排查 | CPU 指标异常 | SOP 步骤、排查命令模板 |
| OOM 排查 | 内存告警/OOM 日志 | JVM 分析命令、Heap Dump 解读 |
| 慢接口排查 | 延迟指标异常 | 拓扑分析、慢查询排查 |

### 9.4 阶段三：开发原型

**最小可行 Agent（MVP）需要三样东西**：

1. **模型**：选一个支持工具调用的大模型
2. **工具**：至少 2-3 个能用的工具
3. **系统提示词**：告诉 Agent 它是谁、该干什么

跑通端到端流程：
```
用户提问 → Agent 理解 → 调用工具 → 拿到结果 → 给出回答
```

> 💡 **产品视角**：先用最简单的方式跑通，不要一开始就搞完美的架构。

### 9.5 阶段四：测试评估

**评估 Agent 的三个维度**：

| 维度 | 评估方法 | 例子 |
|------|---------|------|
| **正确性** | 给定问题，看答案对不对 | 100 个历史告警，Agent 能正确诊断多少个？ |
| **效率** | 看完成时间、调用次数 | 平均诊断需要几轮工具调用？耗时多少？ |
| **安全性** | 看会不会越权/乱操作 | 给 Agent 危险权限，它会滥用吗？ |

**关键概念：LangSmith Tracing**

LangSmith = **记录 Agent 每一步操作的追踪系统**。

```
一次诊断的完整 Trace：
├── 用户输入："服务 A CPU 告警"
├── LLM 调用 1（理解）
├── 工具调用 1：query_metrics → CPU 95%
├── LLM 调用 2（分析）
├── 工具调用 2：query_logs → 发现异常进程
├── LLM 调用 3（推理）
├── 工具调用 3：query_changes → 最近无变更
├── LLM 调用 4（结论）
└── 输出："CPU 95% 由异常进程引起，建议..."
```

有了 Trace，你可以：
- 看到 Agent 的每一步推理
- 发现哪里做了错误决策
- 针对性地优化工具或提示词

### 9.6 阶段五：安全加固

| 安全措施 | 实施内容 |
|---------|---------|
| 权限控制 | 定义文件读写权限，隔离敏感路径 |
| 人在回路 | 危险操作（重启/回滚）必须审批 |
| 沙箱隔离 | Agent 代码执行在隔离环境 |
| 输入校验 | 防止恶意提示词注入 |
| 输出审计 | 检查 Agent 回复是否合规 |

### 9.7 阶段六：部署上线

**部署方式**：

| 方式 | 说明 | 适合 |
|------|------|------|
| 本地运行 | 在自己电脑上跑 | 开发调试 |
| 容器部署 | 打包成 Docker 部署 | 团队内部使用 |
| 云平台部署 | 部署到 LangSmith/K8S | 生产环境 |

**接入方式**：

| 接入方式 | 协议 | 适合 |
|---------|------|------|
| 对话界面 | Web UI / IM 集成 | 用户交互 |
| API 调用 | HTTP API | 其他系统集成 |
| MCP 服务器 | MCP 协议 | 编辑器集成（VS Code / Zed） |
| ACP 协议 | ACP 标准 | IDE 内嵌 Agent |

## 第十一章【实战篇】以告警诊断 Agent 为例

### 11.1 场景描述

**背景**：XRay 平台收到告警——服务 recommend-service-prod CPU 使用率 95%

**目标**：Agent 自动诊断根因并给出处置建议

### 11.2 Agent 设计

```
┌──────────────────────────────────────────────────┐
│               XRay 告警诊断 Agent                 │
│                                                  │
│  大脑：Claude Sonnet 4                            │
│                                                  │
│  工具：                                           │
│    ├── query_metrics    查指标（CPU/内存/网络）    │
│    ├── query_logs       查日志（错误/异常堆栈）    │
│    ├── query_changes    查变更（发布/配置）        │
│    ├── get_topology     查拓扑（上下游依赖）       │
│    ├── run_sop          执行标准排障流程           │
│    └── send_notification 发送诊断报告              │
│                                                  │
│  记忆：                                           │
│    ├── AGENTS.md          诊断规范、SLO           │
│    └── history/           历史诊断案例            │
│                                                  │
│  技能：                                           │
│    ├── cpu_anomaly        CPU 异常排查 SOP        │
│    ├── memory_leak        内存泄漏排查 SOP        │
│    ├── slow_api           慢接口排查 SOP          │
│    └── topology_analysis  拓扑分析 SOP            │
│                                                  │
│  安全：                                           │
│    ├── 只读权限（不修改生产配置）                  │
│    └── 人在回路（处置建议需确认）                  │
└──────────────────────────────────────────────────┘
```

### 11.3 诊断流程（Agent 视角）

```
第1步【理解告警】
  → Agent 读取告警信息：服务 recommend-service-prod，CPU 95%，持续 5 分钟
  → Agent 加载记忆：这个服务之前有过类似问题吗？（查历史）

第2步【规划排查】
  → Agent 生成 TODO 清单：
    ☐ 查 CPU 指标趋势
    ☐ 查最近变更
    ☐ 查错误日志
    ☐ 查拓扑依赖

第3步【执行排查】
  → 并行派发给三个子 Agent：
    ├── 指标子 Agent：查 CPU/内存/网络 → CPU 95%，内存 80%，正常
    ├── 日志子 Agent：查最近 5 分钟日志 → 发现大量 GC 日志
    └── 变更子 Agent：查最近变更 → 昨天发布了 v2.3.1

第4步【分析推理】
  → Agent 综合三个子 Agent 的结果：
    - CPU 高 + 大量 GC = 可能是内存问题
    - 昨天刚发布 v2.3.1 = 可能是引入的回归
  → Agent 加载"内存泄漏排查 SOP"技能
  → 执行 SOP 步骤：查 Heap 使用趋势 → 确认内存泄漏

第5步【输出结论】
  → 根因：v2.3.1 引入内存泄漏，导致频繁 GC，CPU 飙升
  → 建议：回滚到 v2.3.0，同时通知开发排查 v2.3.1 的内存问题
  → 发送诊断报告到 Hi 群

第6步【记录经验】
  → Agent 更新历史记忆：recommend-service-prod CPU 告警 → v2.3.1 内存泄漏 → 回滚解决
```

### 11.4 关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 模型选择 | Claude Sonnet 4 | 工具调用最稳定，推理能力强 |
| 子 Agent | 异步模式 | 指标/日志/变更可并行，加速诊断 |
| 记忆方式 | 文件系统（Markdown） | 可读可编辑，方便管理员维护 |
| 技能方式 | SKILL.md 渐进式加载 | 只在需要时加载，节省上下文 |
| 安全策略 | 只读 + 人在回路 | Agent 不能直接操作生产，只能给建议 |

### 11.5 评估指标

| 指标 | 目标值 | 如何测量 |
|------|--------|---------|
| 诊断准确率 | > 80% | 用历史告警做测试集 |
| 平均诊断时间 | < 5 分钟 | 从告警触发到输出报告 |
| 工具调用次数 | < 10 次/次诊断 | LangSmith Trace 统计 |
| 上下文利用率 | > 70% | 上下文窗口使用 vs 有效信息占比 |

## 第十二章【未来篇】Agent 的演进方向

### 12.1 当前 Agent 的局限

| 局限 | 说明 |
|------|------|
| 上下文有限 | 再大的窗口也有上限，复杂任务仍会丢失信息 |
| 工具调用成本高 | 每轮工具调用都要调大模型，费用不菲 |
| 可靠性不稳定 | 同样的输入，偶尔会给出不同结果 |
| 难以处理超长任务 | 步骤越多，出错概率越高 |
| 缺乏真正的学习 | 每次对话独立，没有从经验中自动改进 |

### 12.2 正在演进的方向

| 方向 | 说明 | 代表 |
|------|------|------|
| **异步子 Agent** | 非阻塞并行，主 Agent 可继续交互 | LangChain Deep Agents v0.5 |
| **后台记忆合并** | 对话间自动总结，更新长期记忆 | Deep Agents Memory |
| **模型动态路由** | 根据任务复杂度自动选择模型 | OpenClaw 路由 |
| **Skill 生态** | 社区共享可复用技能包 | AgentSkills.io |
| **自评估/自改进** | Agent 评估自己的表现并优化 | 研究中 |

### 12.3 对产品经理的启示

| 启示 | 行动建议 |
|------|---------|
| Agent 框架趋同 | 选框架不要过度纠结，核心是工具/SOP/评估 |
| 安全是生产关键 | 没有安全设计，Agent 永远停留在 Demo |
| 评估体系是壁垒 | 谁能量化 Agent 表现，谁就能持续改进 |
| Skill 生态是趋势 | 未来的 Agent 竞争力在于 Skill 数量和质量 |
| 上下文工程是核心 | 管理上下文比堆能力更重要 |

## 附录：核心概念速查表

| 概念 | 一句话解释 |
|------|-----------|
| **Agent** | 能自主完成复杂任务的 AI 系统 |
| **LLM / 大模型** | Agent 的"大脑"，负责推理和决策 |
| **Tool / 工具** | Agent 的"手"，让它能操作外部世界 |
| **Memory / 记忆** | Agent 的"笔记本"，跨会话保存信息 |
| **Planning / 规划** | Agent 把复杂目标拆成可执行步骤 |
| **Skill / 技能** | 打包好的专业知识和操作流程 |
| **Subagent / 子Agent** | 主 Agent 创建的临时 Agent，用于隔离和并行 |
| **Context Window / 上下文窗口** | 大模型一次能处理的文字量上限 |
| **Token** | 大模型处理文字的基本单位（约 0.5 个汉字） |
| **ReAct** | "想一步 → 做一步 → 看结果 → 再想"的循环模式 |
| **MCP** | Agent 连接外部工具的 USB 接口标准 |
| **Sandbox / 沙箱** | 隔离的执行环境，保护主机安全 |
| **Human-in-the-loop / 人在回路** | 敏感操作需要人工审批 |
| **Namespace / 命名空间** | 隔离不同用户/场景的记忆存储 |
| **Progressive Disclosure / 渐进式披露** | 按需加载信息，不一次性全塞进去 |
| **Context Engineering / 上下文工程** | 管理 Agent 看到的信息，而不是堆砌 |
| **Tracing / 追踪** | 记录 Agent 每一步操作的日志系统 |
| **ACP** | Agent 与 IDE 通信的标准协议 |
| **异步子Agent** | 非阻塞的子 Agent，主 Agent 可继续交互 |
