# 技术风险平台 Agent-Native 化 — 产品研究报告

**研究问题：** 技术风险平台如何从「面向人的 GUI 流程」转型为「Agent 可原生调用 + 对话主路径」

**研究范围：** 以可观测性平台为核心切入，覆盖日志、指标、Trace、告警、大盘五大功能域

**研究轮次：** Round 1（场景对比）+ Round 2（功能存在理由拆解）+ Round 3（PM 产品设计决策）

---

## 一、核心判断

这个方向**是真实发生的转变，不是概念**。Datadog 在 2026 年 3 月已将其能力层工具对外 GA，行业最领先的平台正在经历「平台从被人用，变成被 Agent 用」的结构性转型。

**平台被边缘化的方式，不是被替换，而是被绕过。** Agent 时代，研发想查某服务稳定性状况，会直接问 IDE 或飞书里的 AI，而不是打开平台点菜单。如果平台能力无法被 Agent 调用，Agent 就会用文档、RAG、甚至猜测来回答，平台沉淀的真实数据被闲置。

**两种不同的 Agent-Native 化：**
- **Agentic AIOps**：在现有平台上加 AI 调用层，让 Agent 更聪明地使用现有平台能力（行业已有生产案例）
- **真正的 Agent-Native**：重新审视现有功能的存在理由，哪些是因为「人需要」才有的，哪些在 Agent 时代可以消解或重构（这是更深层的命题）

本报告同时覆盖两个层次。

---

## 二、三类功能的不同命运

```
A 类：纯为人类可视化存在 → Agent 时代大幅弱化，但不消失
B 类：分布式技术下的协同工程产物 → 存在理由需重新审视，部分可消解
C 类：数据本身仍有价值，消费者变了 → 保留但重构接口
```

| 功能/链路 | 类型 | Agent-Native 后命运 |
|---------|------|-------------------|
| 日志查询 UI | A | 弱化为兜底，Agent 直接读结构化数据 |
| 大盘创建/维护 | A | 弱化，Agent 按需生成，人看「结论」不看图表 |
| 告警规则配置 | A+B | 消解 → 重构为「意图配置」 |
| 日志采集/传输/集中存储 | B | 部分消解，主路径从「全量归档」→「近源推理+结论归档」 |
| Trace 瀑布图 | A | 弱化，Agent 直接读 Span 结构 |
| Metric 时序采集 | C | 保留，消费者从人变成 Agent |
| Trace Span 采集存储 | C | 保留，Agent 直接消费 |
| 结构化查询 API | C | 升格为主要接口 |
| 近源日志访问接口 | 新增 | Agent 直接访问实例日志，无需先上报 |
| Agent 推理结论归档 | 新增 | 替代原始日志归档成为核心数据层 |
| 人类监督/审批界面 | 新增 | 看 Agent 决策过程，不是看原始数据 |

## 三、日志：最值得颠覆的链路

### 现有链路存在的真实理由

```
应用产生日志 → Filebeat/Agent 采集 → Kafka 传输 → ES/ClickHouse 存储 → 查询 UI → 人工分析
```

- **采集/传输/集中存储**：日志散落在 N 台机器，人没法逐台看 → **分布式协同产物（B 类）**
- **查询 UI**：人通过界面交互式检索 → **纯为人存在（A 类）**
- **人工分析**：看日志推理根因 → 将被 Agent 替代

### Agent-Native 的近源推理路径

Amazon 在 2026 年 3 月发表论文（VIGIL）验证了一个方向：**Edge-Extended Agent**——在每台服务器/容器上部署本地 Agent，直接访问近源日志推理，不需要先全量上报到集中存储。

实测数据（100 个端点，10 周）：
- 诊断速度提升 **4x**
- 交互轮次减少 **39%**
- **82% 的案例实现自助解决**（不需要人工介入）

```
Agent-Native 日志路径（近源推理模式）：

问题发生
→ 本地 Agent 直接访问该实例日志（无需先上报）
→ 近源推理定位根因
→ 只上报「推理结论 + 关键证据片段」，不上报全量日志
→ 集中层只存「结论归档」，不存「原始日志」
```

**成本结构的根本变化：**

| | 现在 | Agent-Native |
|--|------|-------------|
| 数据量 | 全量日志上报（PB 级） | 结论 + 证据摘要（GB 级） |
| 存储成本 | 极高 | 大幅降低 |
| 查询延迟 | 跨网络查询（秒级） | 本地推理（毫秒级） |
| 覆盖能力 | 已采集的才能查 | 任何本地可访问数据都能分析 |

### 当前约束（近两年内不会完全消解集中存储）

1. **合规审计**：金融、安全等场景要求日志留存 N 年
2. **跨实例关联**：排查分布式问题需要跨服务日志关联
3. **Agent 推理成本**：每台机器跑本地推理 Agent，LLM 成本仍高

**现实判断：** 集中存储从「主路径」降为「合规归档 + 跨实例关联」的补充；主路径逐步向「按需拉取→近源推理→结论归档」迁移。

## 四、告警：最有可能被消解的产品功能

### 现有告警规则存在的本质

告警规则的存在理由：**人类把「什么是异常」的判断，预先编码成机器可执行的规则**——因为机器过去不会自己理解什么叫异常。

这个理由在 Agent 时代正在失效。

### 从「规则」到「意图」的范式转变

**现在（规则配置）：**
```
人预设规则（CPU > 80% 持续 5 分钟）
→ 规则引擎扫描 → 触发告警 → 通知人 → 人排查
```
问题：只能发现「人已知的异常模式」；规则维护成本极高，容易产生告警风暴。

**Agent-Native 后（意图配置）：**
```
Agent 持续观测原始数据流（不依赖预设规则）
→ Agent 多源融合判断异常（指标 + 日志 + Trace + 变更关联）
→ Agent 完成初步 RCA，形成结论和置信度
→ 仅在需要人决策时通知，带着「根因初判 + 建议动作」
```

### 意图配置的产品形态设计

**核心交互：用自然语言描述「我关心什么」，而不是「触发条件是什么」**

```
旧范式（规则配置）：
  指标：http_requests_total{status=~"5.."}
  条件：rate > 0.01
  持续：5 分钟
  严重级别：P1

新范式（意图配置）：
  我关心：结账服务的用户下单成功率
  目标：99.9% 的请求在 3 秒内成功完成
  容忍度：误差预算消耗超过 10%/小时时介入
  处置偏好：Agent 先自动排查，有结论再通知我
```

**三层产品结构：**

```
Layer 1 — 意图表达层（人操作）
  - 关心的业务目标（SLO 目标值）
  - 误差预算消耗速率阈值
  - 处置偏好（Agent 先处理 / 直接通知 / 静默到特定时间段）

Layer 2 — Agent 推理层（自动持续运行）
  - 自主判断哪些信号与业务目标相关
  - 多源融合综合判断（指标 + 日志 + Trace + 变更）
  - 完成初步 RCA，输出结论和置信度

Layer 3 — 人类决策层（按需介入）
  - Agent 置信度低时：「我判断是 DB 问题，但不确定，需要你确认」
  - 需要人授权执行：「建议回滚 v2.3.1，是否确认」
  - 误差预算消耗超阈值：「过去 1 小时消耗了 15% 的月度预算」
```

**Agent 推理结论卡片设计（建立信任的关键）：**

```
异常判断：结账服务错误率上升（置信度 87%）

判断依据：
  ① 过去 15 分钟 HTTP 5xx 从 0.1% → 1.8%（↑ 18x）
  ② 同期 DB 连接池使用率从 45% → 92%
  ③ 17:43 有一次 DB schema 变更（关联度：高）

推理链路：指标异常 → 关联变更 → 缩小根因范围

建议动作：回滚 17:43 的 schema 变更
风险评估：回滚操作影响范围小，建议直接执行

[确认执行]  [我来自己处理]  [需要更多信息]
```

**设计要点：**
- **意图模板库**：提供「用户可用性守护」「变更稳定性守护」等场景化模板，降低表达成本
- **规则配置作为高级模式保留**：大多数工程师用意图配置，少数精细控制场景仍可写规则
- **透明化推理过程**：工程师不会信任「我感觉有问题」的 Agent，需要看到完整的证据链

> 工程理论基础：SLO-based alerting（Google SRE Book）已经是意图驱动的雏形——基于「错误预算消耗速率」而非单一阈值。Agent 进一步将其推向「持续推理 + 按需通知」。

## 五、大盘：从「主要产品」到「监督界面」

### 大盘存在的理由与变化

大盘的存在前提：**人是主要的数据消费者**。Agent 不需要图表，直接读结构化数据。

Datadog DASH 2025 已经实现：用自然语言描述，Bits AI 自动生成 Dashboard——这意味着「手动创建大盘」这个工作正在消失。

**会消失的大盘工作：**
- 手动创建和维护大盘（Agent 按需生成）
- 「值班看大盘」的工作模式（Agent 持续观测替代）
- 大盘随系统变化的手动更新

**会保留的大盘：**
- **人监督 Agent 的界面**：不是原始指标图表，而是「Agent 当前判断 + 置信度 + 历史动作记录」
- **高管/跨团队全局视图**：定期汇报、架构决策仍需人类可读的可视化
- **异常回溯复盘**：出了问题需要人工复盘，仍然需要可视化

**产品重心转变：**
从「数据展示」→「Agent 决策的监督和审批」

---

## 六、Trace：变化最小，接口需重构

Trace 是三大支柱中最接近「Agent 友好」的——Span 天然结构化，Agent 可直接读。

**不会消失：** Span 采集、结构化存储（Agent 推理依赖完整调用链数据）

**会变化：**
- 采样策略：从「人工配置固定采样率」→「Agent 动态决定哪条链路值得完整采集」
- 瀑布图/火焰图等可视化弱化（Agent 直接读 Span，不需要图）
- 主要消费者：从「工程师查问题」→「Agent 做 RCA 的数据来源」

## 七、Agent 推理结论归档——新数据层设计

### 为什么这是当前空白

现有平台归档的是**原始数据**（日志文本、指标数值、Span 详情）。Agent 大规模上线后，需要归档的是**推理过程和结论**——这一层目前完全空白。

「谁授权了 Agent 这么做？Agent 基于什么数据做的判断？做了什么？结果如何？」——监管、审计、复盘都需要回答。

### AgentActionRecord 数据模型

基于 OpenTelemetry GenAI Semantic Conventions（2025 年正式发布）的 Agent Span 标准，结合技术风险平台业务场景：

```
AgentActionRecord（Agent 行动记录）
├── 基础标识
│   ├── record_id          # 唯一 ID
│   ├── agent_id           # 哪个 Agent
│   ├── agent_version      # Agent 版本（可追溯）
│   ├── trigger_event_id   # 由什么触发（告警 ID / 用户请求）
│   └── timestamp          # 行动时间
│
├── 输入上下文（Agent 看到了什么）
│   ├── intent_config_id   # 关联的意图配置
│   ├── data_sources[]     # 使用了哪些数据源
│   ├── data_snapshot      # 关键数据快照（不是全量，是 Agent 认为关键的）
│   └── context_summary    # 输入上下文摘要
│
├── 推理过程（Agent 怎么想的）
│   ├── reasoning_steps[]  # 推理步骤链（结构化）
│   ├── hypotheses[]       # 考虑过的假设
│   ├── evidence[]         # 支撑结论的证据
│   └── confidence_score   # 置信度 0-1
│
├── 输出结论（Agent 判断什么）
│   ├── anomaly_detected   # bool
│   ├── root_cause_summary # 根因摘要（自然语言）
│   ├── affected_services[]# 影响范围
│   └── impact_level       # L1/L2/L3/L4
│
├── 建议与执行（Agent 做了什么）
│   ├── recommended_action # 建议动作
│   ├── action_taken       # 实际执行的动作
│   ├── human_approval     # 是否经过人工审批（bool + 审批人 + 时间）
│   └── execution_result   # 执行结果
│
└── 复盘反馈（结果怎样）
    ├── outcome_verified    # 问题是否被解决
    ├── human_feedback      # 人工反馈（Agent 判断对不对）
    └── learning_signal     # 是否作为训练信号
```

**这个数据模型解决四个核心问题：**

1. **审计合规**：完整可查证的 Agent 行动链路
2. **信任建立**：工程师可回溯「Agent 上次判断对没对」，随记录积累建立信任
3. **Agent 质量度量**：统计判断准确率、置信度校准、MTTR 缩短程度——Agent SLO 的数据基础
4. **持续学习闭环**：`human_feedback` 字段为 Agent 能力改进提供数据

> 目前 OpenTelemetry GenAI Semantic Conventions 只定义了底层 Span 格式，不涉及业务语义层（根因、影响范围、审批流、复盘反馈）。这是 PM 可以直接定义的空白地带。

## 八、SKILL 孵化优先级

| SKILL | 调用频次 | 场景价值 | 难度 | 优先级 |
|-------|---------|---------|------|--------|
| 故障查询（告警+Trace+日志聚合） | 极高 | 极高（缩短 MTTR） | 中 | **P0** |
| 变更风险评估 | 高 | 极高（防故障引入） | 中 | **P0** |
| SLO 状态查询 | 高 | 高（替代人工查询） | 低 | **P0** |
| 服务依赖图查询 | 高 | 高（影响范围判断） | 低 | **P1** |
| 容灾覆盖扫描 | 中 | 高（全局风险发现） | 中 | **P1** |
| 可观测性接入指引 | 中 | 中（减少人工指导） | 低 | **P1** |
| Runbook 智能匹配 | 中 | 高（加速故障处置） | 低 | **P1** |
| 全局稳定性 AI 洞察报告 | 低 | 中（管理层视角） | 高 | **P2** |

**SKILL 设计三原则（来自 GitHub Copilot/Block 工具重构经验）：**
- **少而精**：Copilot 从 40 个工具→13 个，基准分显著提升；工具太多让 Agent 迷失
- **任务视角**：SKILL 名应是「评估变更风险」而非「查询变更记录」
- **输出结构化**：便于 Agent 继续处理，不是大段非结构化文本

## 九、落地路径

### 阶段一（Q2）：PM 可立即主导

1. **`AgentActionRecord` 数据模型定义**——先定义，推动工程落地，建立 Agent 行为归档基础
2. **意图配置 UX 原型**——从「规则配置」到「意图表达」的产品设计，优先覆盖「用户可用性守护」「变更稳定性守护」两个模板
3. **3 个核心 SKILL 试点**——故障查询 + SLO 查询 + 变更风险评估，接入内部 AI 助手小范围验证
4. **Agent 推理结论卡片设计**——工程师怎么看 Agent 的判断（透明化推理过程）

### 阶段二（Q3）：SKILL 扩展 + 协同工具嵌入

5. SKILL 库扩展到 6-8 个
6. 嵌入飞书机器人，研发和 on-call 在工作上下文直接使用
7. 意图配置 v1.0 上线，替代部分简单告警规则配置

### 阶段三（Q4）：对话门户 + 长期方向探索

8. 统一对话门户 MVP，支持跨平台复合查询
9. 近源日志访问接口——与基础架构团队合作探索（依赖底层技术，长期方向）

## 十、主要风险

1. **把「加 AI 助手」当 Agent-Native 化**：前者是功能点，后者是架构重构——核心差异是平台能力有没有被拆解为可被外部 Agent 调用的原子能力
2. **SKILL 边界不清，工具爆炸**：克制拆解，一个 SKILL 解决一个明确任务
3. **Agent 推理质量不稳定**：先聚焦 3-5 个核心场景跑通再扩展，避免因质量参差损害信任
4. **数据结构化程度不够**：SKILL 需要结构化 API 数据源，散落在 Wiki/Excel 的数据无法支撑高质量 SKILL
5. **近源日志的合规边界**：哪些日志有留存合规要求，决定了集中存储能压缩到什么程度
