## 六、AgentOps 深度规划

### 6.1 方向一：OPS for Agent

> 方向一解决"AgentOps 平台能做什么"——五个能力柱子，三条最佳实践流程，全局 13 个卡点。

#### 6.1.1 五个能力柱子总览

AgentOps 覆盖五个核心能力领域，**AI 评估是整体的质量传感器**，驱动其他四个柱子的决策：

| 柱子 | 核心分工 | 关键能力 |
|---|---|---|
| **发布变更** | 管 Agent 上线前后的"变" | 变更语义感知 / Eval Gate / 灰度管控 / 版本回滚 |
| **AI 评估** | 整体"质量传感器" | 离线 Eval / 在线评估 / 评估器体系 / 数据集管理 |
| **AI 可观测** | 管运行期的"看" | 全链路 Trace / 失败模式聚类 / SLO 监控 |
| **成本治理** | 横切所有柱子 | Token 计量 / 成本归因 / 预算熔断 |
| **安全护栏** | 横切所有柱子 | 输入/输出/行为三层实时拦截 |

---

#### 6.1.2 柱子一：发布变更（细化）

**核心命题**：传统发布只管"服务有没有上"，AgentOps 的发布还要管"AI 行为有没有变好"。变更的定义从"镜像/配置变更"扩展到"Prompt/模型/工具/知识库变更"。

**能力 1.1：Agent 元数据注册**

Agent 创建时自动注册，建立"身份档案"。输入 ALLIN workflow.created 事件，校验必填字段后初始化，自动关联 ONES 服务和 XRay App。SLO 未定义时发布申请给出警告（Phase 2 升级为阻断）。

关键字段：`agent_id / owner / intent / capability_boundary / slo_definition / xray_app_id / ones_service_id`

**能力 1.2：变更语义感知**

解析每次发布携带的 AI 语义变更：

| 变更类型 | 风险等级 | 后续流程 |
|---|---|---|
| Prompt 措辞微调（语义相似度 > 90%）| 低 | 精简 Eval（50 条）|
| Prompt 逻辑调整（语义相似度 < 90%）| 中 | 标准 Eval（200 条）|
| 工具增减 | 中 | 标准 Eval + 工具专项评估 |
| 模型版本升级（同厂商）| 中 | 标准 Eval + 回归用例 |
| 模型替换（跨厂商）| 高 | Eval Gate + 人工审批 |
| RAG 知识库全量更新 | 高 | Eval Gate + 召回专项评估 |
| 高危工具（exec_shell 等）| 高 | 强制人工审批，不可绕过 |

**卡点**：agent_meta 字段缺失时拒绝创建发布单（硬阻断）。

**能力 1.3：Eval Gate（评估门禁）**

用历史数据集回放新版本，评估分达标才放行，不达标不让上线。

**Eval Gate 决策矩阵：**

| 评估结果 | 处置 |
|---|---|
| 全部指标达标 | ✅ 放行 |
| 任务完成率不达标 | ❌ 驳回（核心指标，一票否决）|
| 幻觉率超限 | ❌ 驳回（安全类，一票否决）|
| vs 基线 delta < -5%（显著退化）| ❌ 驳回 |
| 工具调用合规率不达标（工具变更时）| ❌ 驳回 |
| 格式合规率不达标 | ⚠ 警告（不阻断，但记录）|

加速机制：低风险变更走精简数据集（50 条 vs 标准 300 条）；未涉及的节点无需全量回放。

**能力 1.4：灰度流量管控**

| 策略 | 切流比例 | 观察窗口 |
|---|---|---|
| 低风险变更 | 1% → 10% → 100% | 15min / 30min |
| 高风险变更 | 1% → 5% → 20% → 100% | 30min / 1h / 2h |

灰度期异常处置：错误率暴涨 >5x → 自动暂停；质量下滑 >10% → 自动暂停（均不自动回滚，需人工决策）。

**能力 1.5：版本回滚 + 热修复快速通道**

回滚：确认目标版本历史 Eval Gate 结果 → 创建回滚发布单 → 快速灰度通道（1%→100%，窗口 5min）。

热修复（仅 Prompt 变更）：精简数据集 50 条 → 快速灰度通道。每个 Agent 每天最多触发 3 次。

#### 6.1.3 柱子二：AI 评估（细化）

**核心命题**：评估是整个 AgentOps 的"质量传感器"。没有评估，"Agent 是否健康"就只是感觉；有了评估，质量才有数字、有基线、有趋势。

**四类评估模式：**

| 评估模式 | 触发时机 | 目标 |
|---|---|---|
| Benchmarking（基准测试）| 首次上线后 | 建立绝对质量基线，跨版本可比较 |
| Unit Tests（单元测试）| 每次 Prompt/代码改动 | 快速验证特定行为未被破坏 |
| Regression Tests（回归测试）| 发布前 Eval Gate | 检测新版本是否比上一版本退化 |
| Backtesting（回溯测试）| 发现线上质量问题后 | 用历史生产 Trace 重放，验证修复方案 |

**评估器体系：**

| 类型 | 实现方式 | 适用维度 | 特点 |
|---|---|---|---|
| LLM-as-Judge | GPT-4o/Claude 作裁判 | 任务完成率、推理质量 | 准确率高、成本较高 |
| 规则评估器 | 正则/关键词/结构校验 | 格式合规、敏感词过滤 | 速度快、成本低 |
| 语义相似度 | Embedding 计算 | 输出与参考答案的语义一致性 | 适合有参考答案场景 |
| 自定义评估器 | 业务方自定义函数 | 业务特定规则 | 最灵活、需业务方维护 |
| 人工标注 | 人工队列 | 高置信度校准 | 质量最高、不可实时 |

**评估器组合策略：**

- 发布前离线评估（追求准确）：LLM-as-Judge（主）+ 规则评估器（辅）+ 自定义评估器
- 运行期在线评估（追求速度）：规则评估器（实时）+ 语义相似度（实时）+ LLM-as-Judge（异步采样 5%）
- 人工标注（定期校准）：抽取 LLM-as-Judge 评分边界案例 → 人工确认 → 更新评估器权重

**在线评估自进化闭环：**

生产 Trace → 在线评估 → 失败 Trace 一键加入数据集 → 创建针对性评估器 → 离线 Backtesting 验证修复 → 发布修复版本

数据集管理必须支持"从生产 Trace 一键加入数据集"，否则在线↔离线评估的数据闭环打不通。

**数据集结构**：正常用例（60%）+ 边界用例（20%）+ 对抗用例（20%）；版本化管理，每季度更新。

#### 6.1.4 柱子三：AI 可观测（细化）

**核心命题**：传统可观测看"服务活没活"，AI 可观测要多看一层："服务做对没有"。且 AI 链路和基础设施链路必须打通。

**全链路 Trace 设计：**

每个 Span 类型及记录内容：

| Span 类型 | 额外记录内容 |
|---|---|
| LLM Span | system_prompt（hash）/ 完整 messages / completion / 模型 / temperature / token_usage |
| Tool Span | 工具名 / 调用参数 / 返回结果 / 耗时 / 成功/失败 |
| RAG Span | 查询向量 / 召回文档列表 / 召回分数 / 实际使用文档 |
| Sub-Agent Span | 子 Agent ID / 输入输出 / 递归深度 |
| Memory Span | 读写操作 / Key / 命中/miss 情况 |

采样率：开发期 100%；生产期默认 10%；失败请求 100%；用户主动打标请求 100%。

**核心价值：AI × 基础设施打通**

把"AI 质量下滑"和"下游 rca-service 超时 815ms"关联起来，精准定位根因。这是业界集体盲区——我们通过 XRay 同时拥有 AI 链路和 HTTP/RPC/中间件数据，可以真正实现联合分析。

对齐 OTel GenAI SemConv 后，AI Span 和 HTTP Span 在同一 Trace 树中共存（TraceContext 传播），而不是两套独立系统。

**失败模式聚类**：规则分类（错误码）+ Embedding 聚类（相似错误消息）→ 自动发现系统性问题；某模式频率超阈值时附代表性 Trace 告警。

**SLO 监控**：滑动窗口（1h/6h/24h）均值对比；连续 N 个窗口低于阈值才告警（避免抖动误报）。

#### 6.1.5 柱子四：成本治理

**核心命题**：AI 系统的成本是非确定性的——同样的 Agent，不同请求的 Token 消耗可能差 10 倍。传统"资源配额"管不住，需要精细化的感知、归因和干预。

**Token 计量粒度**：Agent 级 / 版本级 / 请求级 / LLM Span 级；按模型单价换算为费用（¥）。

**成本归因维度**：按团队 / 按 Agent / 按模型 / 按功能（哪个 Tool 最耗 Token）/ 按版本（升级前后自动对比）。

**预算熔断五级机制：**

| 干预级别 | 触发条件 | 处置方式 |
|---|---|---|
| L1 优化提示 | 单次请求 tokens > 30K | 提示研发检查 context 压缩策略 |
| L2 自动截断 | 单次请求 tokens > 50K | 自动压缩历史对话，保留最近 N 轮 |
| L3 模型降级 | 单 Agent 日消耗 > 日预算 80% | 自动从高价模型降为低价模型 + 告警 |
| L4 限流 | 月消耗 > 月预算 90% | 请求限流至 50% + 告警 + 通知 owner |
| L5 硬熔断 | 月消耗达到月预算上限 | 拒绝新请求，强制人工审批后恢复 |

#### 6.1.6 柱子五：安全护栏

**核心命题**：Agent 有工具调用能力，能查数据库、发接口、写文件——比普通 LLM 的风险大一个量级。安全护栏是运行时三层实时拦截：输入层 + 输出层 + 行为层。

**三层拦截设计：**

| 层级 | 检测类型 | 处置方式 |
|---|---|---|
| 输入层 | Prompt 注入攻击 | 拒绝 + 告警 + 记录 |
| 输入层 | 越权意图识别 | 拒绝 + 解释 + 记录 |
| 输入层 | 敏感信息识别（PII）| 脱敏处理后传入，不传原文 |
| 输出层 | 敏感数据泄露 | 脱敏/拦截 + 告警 |
| 输出层 | 幻觉内容检测 | 添加不确定性声明 / 拦截 |
| 行为层 | 工具白名单校验 | 非白名单工具调用直接拒绝 |
| 行为层 | 高危操作审批 | exec_shell / 写数据库等强制人工确认 |
| 行为层 | 调用深度/频率限制 | 防止递归死循环 |

**上线策略**：Monitor-Only（观察误报率）→ Selective Block（核心高危场景）→ Full Block（全量）。不要一上来全量 block，先观察误报率再开启。

**OWASP LLM Top 10 覆盖**：Prompt Injection ✅ / Insecure Output Handling ✅ / Training Data Poisoning 规划中 / Model Denial of Service ✅ / Supply Chain Vulnerabilities 规划中。

#### 6.1.7 三条最佳实践流程

**流程 A：发布流（完整版）**

```
发起发布申请
  → agent_meta 完整性校验（缺字段直接拒绝）
  → 变更语义解析 + 风险定级
  → Eval Gate（低50条/中200条/高300条+）
      ├── 不达标 → 驳回 + 报告，研发修复后重测
      └── 达标 → 继续
  → 人工审批（高风险变更，与 Eval Gate 并行）
  → 灰度发布（按风险等级分步切流）
      ├── 异常 → 自动暂停 + 推送回滚建议
      └── 正常 → 继续扩量
  → 全量上线
  → 更新元数据 + 建立新基线 + 推送事件到 ALLIN
```

关键设计：Eval Gate 和人工审批**并行**，不串行——缩短等待时间；评估不通过不需要重走人工审批。

**流程 B：质量降级处置流（完整版）**

```
持续质量感知（常态化）
  → 异常判定：
      轻度（下滑 10-20%）→ 通知 owner + 附分析报告
      中度（下滑 20-30%）→ 推送回滚建议 + 等待人工审批（超时 1h 自动升级）
      重度（下滑 >30%）   → 自动回滚（需提前配置 P0 授权）+ 创建 RCA 事件
  → 根因自动关联（质量下滑时间点 vs 变更时间 + 基础设施异常）
  → 处置执行（回滚 / Prompt 热修复 / 继续观察）
  → 闭环归档（MTTR 记录 + 知识库沉淀）
```

**流程 C：Agent 全生命周期主线**

| 阶段 | AgentOps 自动执行 | 关键卡点 |
|---|---|---|
| 创建期 | 注册元数据 / 关联工程资产 / 引导 SLO 定义 | SLO 未定义→警告 |
| 开发期 | 开发态 Trace 实时上报 / 开发期 Eval 快照 | 评估分低于基线→提示 |
| 发布期 | 走流程 A | Eval Gate 不达标→拒绝 |
| 运行期 | 在线评估 / SLO 监控 / 成本监控 / 安全护栏 | 重度降级→自动回滚 |
| 退役期 | 归档 Trace / 沉淀知识库 / 评估数据永久保留 | 有未处理告警→提示确认 |
