# 六、AgentOps 深度规划

<redoc-highlight>
本章节围绕 AgentOps（运维态）展开两个核心方向的深度规划：**方向一：OPS for Agent**（能力柱子与最佳实践流程），**方向二：Agent Native**（AI 原生服务形态）。两个方向相辅相成，前者定义"做什么"，后者定义"以什么姿态提供服务"。
</redoc-highlight>

## 6.1 方向一：OPS for Agent

### 6.1.1 五个能力柱子总览

AgentOps 覆盖五个核心能力领域，AI 评估是整体的质量传感器，驱动其他四个柱子的决策：

| 柱子 | 核心分工 | 关键能力 |
|------|---------|---------|
| **发布变更** | 管 Agent 上线前后的"变"——改了什么、能不能上、上了之后还能不能回 | 变更语义感知、Eval Gate、灰度管控、版本回滚、热修复通道 |
| **AI 评估** | 整体质量传感器——发布前判断好坏、运行中持续感知、数据驱动其他四柱决策 | 离线评估、在线评估、评估器体系、数据集管理、基线管理 |
| **AI 可观测** | 管运行期的"看"——Trace 看行为、指标看健康、告警看异常 | 全链路 Trace、AI×基础设施联合分析、失败模式聚类、SLO 监控 |
| **成本治理** | 横切所有柱子——每一笔 Token 消耗看得见、算得清、管得住 | Token 计量、成本归因、异常检测、预算熔断（五级干预）|
| **安全护栏** | 横切所有柱子——输入/输出/行为三层防线，不该做的不能做 | 输入检测、输出过滤、工具调用控制、审计日志 |

### 6.1.2 柱子一：发布变更（细化）

**核心命题：** 传统发布只管"服务有没有上"，AgentOps 的发布还要管"AI 行为有没有变好"。变更的定义从"镜像/配置变更"扩展到"Prompt/模型/工具/知识库变更"。

**变更类型 × 风险等级 × 处置方式：**

| 变更类型 | 风险等级 | 处置方式 |
|---------|---------|---------|
| Prompt 措辞微调 | 低 | 标准 Eval Gate（精简数据集 50 条）|
| Prompt 逻辑调整 | 中 | 标准 Eval Gate（200 条）|
| 工具增减 | 中 | Eval Gate + 工具专项评估 |
| 模型版本升级 | 中 | Eval Gate（必须含回归用例）|
| 模型跨厂商替换 | 高 | Eval Gate + 人工审批 |
| 核心流程节点增删 | 高 | Eval Gate + 人工审批 |
| RAG 知识库全量更新 | 高 | Eval Gate + 召回专项评估 |
| 高危工具（exec\_shell 等）| 高 | 强制人工审批，不可绕过 |

**Eval Gate 决策矩阵：**

| 评估结果 | 处置 |
|---------|------|
| 全部指标达标 | ✅ 放行 |
| 任务完成率不达标 | ❌ 驳回（核心指标，一票否决）|
| 幻觉率超限 | ❌ 驳回（安全类，一票否决）|
| vs 基线 delta < -5%（显著退化）| ❌ 驳回 |
| 格式合规率不达标 | ⚠ 警告（不阻断，但记录）|
| 工具调用合规率不达标（工具变更时）| ❌ 驳回 |

**灰度策略：**
- 低风险变更：1% → 10% → 100%（观察窗口 15min/30min）
- 高风险变更：1% → 5% → 20% → 100%（观察窗口 30min/1h/2h）
- 灰度期异常率超阈值 → 自动暂停（不自动回滚，需人工决策）

### 6.1.3 柱子二：AI 评估（细化）

**评估器体系：**

| 评估器类型 | 实现方式 | 适用维度 | 特点 |
|-----------|---------|---------|------|
| LLM-as-Judge | GPT-4o/Claude 作裁判 | 任务完成率、推理质量、输出相关性 | 准确率高，成本较高 |
| 规则评估器 | 正则/关键词/结构校验 | 格式合规、敏感词过滤、必填字段 | 速度快，成本低 |
| 语义相似度 | Embedding 计算 | 输出与参考答案的语义一致性 | 中等成本，适合有参考答案场景 |
| 自定义评估器 | 业务方自定义函数 | 业务特定规则（如退款逻辑合规性）| 最灵活，需业务方维护 |
| 人工标注 | 人工队列 | 高置信度校准、对抗样本判定 | 质量最高，不可实时 |

**评估器组合策略：**
- 发布前离线评估（追求准确）：LLM-as-Judge（主）+ 规则评估器（辅）+ 自定义评估器
- 运行期在线评估（追求速度）：规则评估器（实时）+ 语义相似度（实时）+ LLM-as-Judge（异步采样 5%）
- 人工标注（定期校准）：抽取 LLM-as-Judge 评分边界案例 → 人工确认 → 更新评估器权重

**数据集要求：** 正常用例（60%）+ 边界用例（20%）+ 对抗用例（20%），版本化管理，每季度更新。

### 6.1.4 柱子三：AI 可观测（细化）

**全链路 Trace 设计：**

每个 Span 类型及记录内容：

| Span 类型 | 额外记录内容 |
|----------|------------|
| LLM Span | system\_prompt hash、完整 messages、completion、模型版本、token\_usage |
| Tool Span | 调用参数、返回结果、工具名称、调用耗时 |
| RAG Span | 查询向量、召回文档列表、召回分数、实际使用文档 |
| Memory Span | 检索 key、命中内容、miss 情况 |
| Sub-Agent Span | 子 Agent ID、输入输出、递归深度 |

**核心价值：AI × 基础设施打通**——把"AI 质量下滑"和"下游 rca-service 超时 815ms"关联起来，精准定位跨层根因，这是业界集体盲区，我们通过 XRay 同时拥有 AI 链路和 HTTP/RPC/中间件数据实现。

**失败模式聚类：** 自动发现和归组类似失败请求，暴露系统性问题（规则分类 + Embedding 聚类），触发告警时附代表性 Trace。

### 6.1.5 柱子四：成本治理（细化）

**五级成本干预机制（预算熔断）：**

| 级别 | 触发条件 | 处置方式 |
|------|---------|---------|
| L1 优化提示 | 单次请求 tokens > 30K | 提示研发检查 context 压缩策略 |
| L2 自动截断 | 单次请求 tokens > 50K | 自动压缩历史对话，保留最近 N 轮 |
| L3 模型降级 | 单 Agent 日消耗 > 日预算 80% | 自动从 GPT-4o 降为 GPT-3.5 + 告警 |
| L4 限流 | 月消耗 > 月预算 90% | 请求限流至 50% + 告警 + 通知 owner |
| L5 硬熔断 | 月消耗达到月预算上限 | 拒绝新请求 + 强制人工审批后恢复 |

**成本归因维度：** 团队 / Agent / 模型 / 功能（哪个 Tool 最耗 Token）/ 版本（升级前后对比）

### 6.1.7 三条最佳实践流程

#### 流程 A：发布流（开发完 → 上生产）

<redoc-highlight>
核心卡点链：发布申请校验 → 变更语义解析（风险定级）→ Eval Gate（质量门禁）→ 人工审批（高风险）→ 灰度发布 → 全量上线 → 建立新基线
</redoc-highlight>

**关键设计：**
- Eval Gate 和人工审批**并行**，不串行——缩短等待时间
- 评估不通过不需要重走人工审批——研发修复后直接重测
- 灰度期在线评估与离线 Eval Gate 数据集不同——前者看真实流量，后者看准备好的用例
- 每次发布产生的评估报告永久关联到该版本——可随时回溯

#### 流程 B：质量降级处置流（线上发现问题 → 闭环）

<redoc-highlight>
核心链路：持续质量感知（10% 采样）→ 异常判定 → 根因自动关联变更 → 分级处置 → 处置执行 → 闭环归档 → 沉淀知识库
</redoc-highlight>

**分级处置规则：**

| 异常级别 | 触发条件 | 处置方式 |
|---------|---------|---------|
| 轻度 | 评估分下滑 10-20% | ALLIN 打标 ⚠ + 通知 owner |
| 中度 | 评估分下滑 20-30% | 推送回滚建议单 + 等待人工审批（超时 1h 升级）|
| 重度 | 评估分下滑 >30% 或 SLO Emergency 级别 | P0 自动回滚（需提前授权）+ 自动创建 RCA 事件 |

**处置三选项：**
- A. 立即回滚：走快速通道，回滚后验证评估分是否恢复
- B. Prompt 热修复：走缩减版 Eval Gate（50 条核心用例），通过后快速灰度
- C. 继续观察：延长监控窗口，增加在线评估采样率至 30%

#### 流程 C：Agent 全生命周期主线（创建 → 退役）

| 阶段 | AgentDev 侧 | AgentOps 侧 | 核心卡点 |
|------|------------|------------|---------|
| **创建期** | 创建 Workflow，触发 workflow.created | 自动注册元数据 / 关联工程资产 / 下发埋点 / 引导 SLO | SLO 未定义时警告"发布将被阻断" |
| **开发期** | 迭代 Workflow，触发 workflow.updated | 开发态 Trace 实时上报 / 建立评估基线 | 变更幅度 >30% 时提示重建基线 |
| **发布期** | 发起发布申请 | 走流程 A 完整执行 | Eval Gate（核心卡点）|
| **运行期** | 接收质量/告警事件 | 持续评估采样 / SLO 监控 / 成本监控 / 安全护栏 | SLO 违规 / 成本超限 / 安全事件 |
| **退役期** | 下线 Workflow，触发 workflow.deprecated | 归档元数据 / 保留 Trace / 沉淀知识库 | 有未处理告警时提示确认 |

### 6.1.8 全局卡点清单

| 卡点位置 | 触发条件 | 处置方式 | 可绕过？ |
|---------|---------|---------|---------|
| 发布申请 | agent\_meta 字段缺失 | 拒绝创建发布单 | 否 |
| 发布申请 | SLO 未定义 | Phase1 警告；Phase2 拒绝 | Phase1 可 |
| 变更解析 | 高危工具（exec\_shell 等）| 升级人工审批 | 否 |
| 变更解析 | 模型跨厂商替换 | 升级人工审批 | 否 |
| Eval Gate | 任务完成率不达标 | 驳回（一票否决）| 人工审批覆盖（留记录）|
| Eval Gate | 幻觉率超限 | 驳回（一票否决）| 否 |
| Eval Gate | vs 基线退化 >5% | 驳回 | 人工审批覆盖（留记录）|
| 灰度期 | 错误率暴涨 >5x | 自动暂停灰度 | 否（需人工恢复）|
| 灰度期 | 质量下滑 >10% | 自动暂停灰度 | 否（需人工恢复）|
| 运行期 | 质量下滑 >30% | P0 自动回滚（需授权）| 提前配置授权范围 |
| 运行期 | 成本超月预算 | 硬熔断（需人工恢复）| 人工审批后恢复 |
| 运行期 | 高危工具调用 | 实时拦截 | 否 |
| 退役期 | 有未处理告警 | 提示确认 | 强制确认后可继续 |

## 6.2 方向二：Agent Native

### 6.2.1 核心命题

<redoc-highlight>
不只是做一个给人点的 Web 平台。当业务 Agent 是用 Claw / vibe coding 开发出来的，它们的 OPS 也应该能被 Agent 直接消费——对话触发、Skill 调用、事件驱动。
</redoc-highlight>

三种服务形态：

| 形态 | 定位 | 面向对象 |
|------|------|---------|
| **Ops Agent** | 有推理能力的对话式 OPS 入口，聚合所有 AgentOps 能力，有上下文记忆 | 研发/SRE 在 Claw 里直接对话 |
| **Skill 化能力** | 原子操作封装，无状态，有明确 Schema，可被任意 Agent 调用 | 其他 Agent 程序化调用 |
| **事件总线** | 系统间自动驱动，双向事件闭环，不依赖人主动触发 | AgentDev ↔ AgentOps 自动联动 |

### 6.2.2 形态一：Ops Agent（对话式治理）

**核心能力——7 类意图：**

<redoc-columns>
<redoc-column ratio="0.5">

**查询类（直接返回，无需确认）**

- **看状态**：汇总 health / SLO / 活跃告警 / 最近变更，给出建议
- **分析根因**：评估分趋势 + 关联变更 + Trace 失败模式 + 知识库历史案例，输出置信度排序的根因分析
- **查 Trace**：精确查询单条或过滤查询，定位失败节点，关联变更版本
- **查成本**：消耗数据 + 对比基线 + 分析异常原因

</redoc-column>
<redoc-column ratio="0.5">

**变更类（必须二次确认）**

- **触发发布**：预检查 → 确认 → 触发 Eval Gate → 实时播报灰度进度
- **触发回滚**：确认目标版本 → 触发快速通道 → 播报结果
- **触发评估**：指定数据集 → 运行 → 推送报告 → 询问是否发布

**高危操作（不执行，引导正规流程）**

- 跳过 Eval Gate → 说明原因 + 提供热修复通道 / 人工审批覆盖两个合规选项

</redoc-column>
</redoc-columns>

**上下文记忆：** 记住当前操作 Agent、本轮已确认的操作结果、用户偏好（跨会话写入数据底座）。敏感操作每次必须重新确认，不因"记住"而跳过。

**集成方式：** 作为 Claw Skill 安装（`agentops-skill`），支持频道订阅（告警直推研发 Claw 频道），支持主动推送（Eval Gate 结果、灰度状态、质量恢复）。

### 6.2.3 形态二：Skill 化能力（原子能力包）

共 10 个 Skill，覆盖 AgentOps 全能力域：

| Skill | 核心功能 | 典型调用方 |
|-------|---------|----------|
| `agentops.meta.get` | 获取 Agent 完整元数据画像（服务/仓库/配置/依赖/SLO）| Ops Agent / vibe coding AI / CI-CD |
| `agentops.health.snapshot` | 轻量健康快照，适合决策前快速判断（调用前"看一眼"）| 任意 Agent |
| `agentops.trace.query` | 查询 Trace 链路，支持精确查和过滤查，可返回完整 Span 树 | Ops Agent / 排障 Agent |
| `agentops.eval.run` | 触发离线评估（standard/quick/full 三种模式，支持异步）| Ops Agent / CI-CD |
| `agentops.eval.gate.check` | 查询 Eval Gate 历史记录（有记录直接返回，无需重跑）| 发布 Agent / CI-CD |
| `agentops.deploy.trigger` | 触发发布/回滚（需 confirm\_token，支持三种灰度策略）| Ops Agent（用户确认后）|
| `agentops.cost.query` | 查询成本数据（支持多维 group\_by，含趋势和预测）| Ops Agent / 报表 Agent |
| `agentops.slo.status` | 查询 SLO 各维度达成情况及趋势 | 监控 Agent / 报表 Agent |
| `agentops.alert.query` | 查询告警事件（含关联变更和建议操作）| Ops Agent / 告警诊断 Agent |
| `agentops.knowledge.query` | 查历史案例/排障 SOP/Prompt 调优记录（自然语言查询）| Ops Agent / 诊断 Agent |

**通用规范：**
- 所有 Skill 支持自然语言参数（Skill 内部做意图解析转结构化）
- 查询 Skill 返回结构化 JSON + `summary` 自然语言摘要字段
- 错误返回：`{error_code, message, suggestion, retry_able}`
- 变更 Skill 通过 `confirm_token` 保证幂等

### 6.2.4 形态三：事件总线（双向事件驱动）

**上行事件（AgentDev → AgentOps）：**

| 事件 | 触发时机 | AgentOps 响应 |
|------|---------|--------------|
| `workflow.created` | Agent 创建 | 注册元数据 / 关联工程资产 / 下发埋点 / 引导 SLO 定义 |
| `workflow.updated` | Workflow 迭代 | 更新元数据 / 评估变更幅度 / 大幅变更时提示重建基线 |
| `workflow.deprecated` | Agent 下线 | 检查告警 / 归档元数据 / 触发知识沉淀 |
| `deploy.requested` | 发起发布 | 变更语义解析 / 触发 Eval Gate / 高风险触发人工审批 |

**下行事件（AgentOps → AgentDev）：**

| 事件 | 触发条件 | 业务 Agent 侧响应 |
|------|---------|-----------------|
| `eval.gate.passed` | Eval Gate 通过 | 继续发布流程 / 通知申请人 |
| `eval.gate.failed` | Eval Gate 不通过 | 暂停发布 / 附评估报告和修复建议 |
| `deploy.progress` | 灰度各阶段变化 | 实时更新发布单状态 |
| `quality.degraded` | 在线评估检测质量下滑 | ALLIN 打标 ⚠ / 通知 owner（含根因关联）|
| `rollback.suggested` | 质量异常已关联变更 | 推送回滚建议卡片（含一键回滚按钮）|
| `slo.violated` | SLO 持续违规 | 告警 / critical 时触发 oncall |
| `cost.threshold.exceeded` | 成本超阈值触发自动干预 | 通知 owner 已采取的降级措施 |
| `security.alert` | 安全护栏触发 | 推送安全告警 / critical 时自动暂停对该用户服务 |

**技术演进路径：**
- Phase 1：HTTP Webhook（快速落地，无重试保证）
- Phase 2：内部 MQ（高可靠，支持重试和顺序消费）
- Phase 3：统一 Agent 事件总线（跨平台，CloudEvents 规范，事件可回放）

### 6.2.5 三种形态的典型组合场景

<redoc-columns>
<redoc-column ratio="0.5">

**场景 A：vibe coding 全流程**<br/>
研发用 Claw 写完 Agent → 问 Ops Agent"可以发布吗"→ Ops Agent 调 eval.run Skill → 返回结果 → 研发确认 → Ops Agent 调 deploy.trigger → 实时播报灰度状态。**全程无需打开 Web 平台。**

**场景 B：告警自动诊断**<br/>
