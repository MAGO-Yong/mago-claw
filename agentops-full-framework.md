## 方向一：OPS for Agent — 能力柱子 × 流程卡点

### 1.1 能力领域全景（5 个柱子）

```
┌─────────────────────────────────────────────────────────────┐
│                     AgentOps 平台                           │
├──────────┬──────────┬──────────┬──────────┬─────────────────┤
│ 发布变更  │ AI 可观测 │  AI 评估  │ 成本治理  │   安全护栏      │
│          │          │          │          │                 │
│ 变更语义  │ Trace链路 │ 离线评估  │ Token计量 │ 输入过滤        │
│ 元数据   │ 异常聚类  │ 在线评估  │ 成本归因  │ 输出过滤        │
│ 灰度管控  │ 告警中心  │ 数据集    │ 预算熔断  │ 行为护栏        │
│ 回滚中心  │ 指标大盘  │ 评估器    │ 模型降级  │ 权限审计        │
│ Eval门禁  │ SLO看板   │ 标注队列  │           │                │
└──────────┴──────────┴──────────┴──────────┴─────────────────┘
                           ↑
              AI 评估是所有领域的数据基础
              （质量传感器，驱动其他四个柱子的决策）
```

### 1.2 三条核心最佳实践流程

---

#### 流程 A：发布流（开发完 → 上生产）

```
ALLIN/Darwin 完成开发
        │
        ▼
① 发起发布申请
   · 携带 agent_meta（workflow版本、prompt_hash、模型、工具列表、变更diff）
   · 卡点：agent_meta 必须完整，否则拒绝创建发布单
        │
        ▼
② 变更语义解析
   · 解析 prompt_hash 变化（Prompt改了什么）
   · 解析 flow_diff（新增/删除哪些节点）
   · 解析模型/工具变更
   · 写入变更记录（关联发布单）
   · 卡点：高危工具（exec_shell等）自动升级人工审批
        │
        ▼
③ Eval Gate（评估门禁） ← 核心卡点
   · 用历史数据集回放新版本
   · 评估维度：任务完成率 / 幻觉率 / 格式合规率 / vs基线delta
   · 全部达标 → 放行
   · 任一不达标 → 驳回 + 附评估报告 + 通知研发
   · 卡点：不达标不放行，不可跳过（高优Agent可配置人工审批覆盖）
        │
        ▼
④ 灰度发布
   · 默认 1% 流量先上
   · 监控：异常率 / task_success_rate / 延迟
   · 正常 → 自动扩量（1% → 10% → 100%）
   · 异常 → 自动暂停 + 告警
   · 卡点：灰度期异常率超阈值自动暂停，不自动回滚（需人工）
        │
        ▼
⑤ 全量上线
   · 更新 Agent 元数据中心（current_version / health / last_deploy_at）
   · 建立新基线（当前评估分作为下次对比基准）
   · 写入变更历史（关联 prompt_hash / 评估分 / 发布时间）
```

**流程 A 的卡点清单：**
| 卡点 | 触发条件 | 处置方式 |
|------|---------|---------|
| agent_meta 不完整 | 缺少必要字段 | 拒绝创建发布单 |
| 高危工具 | exec_shell等高权限工具 | 升级人工审批 |
| Eval Gate 不达标 | 任一指标低于阈值 | 驳回 + 评估报告 |
| 灰度期异常 | 异常率/质量超阈值 | 自动暂停灰度 |

---

#### 流程 B：质量降级处置流（线上发现问题 → 闭环）

```
持续 Eval 采样发现质量下滑
        │
        ▼
① 质量异常检测
   · 在线评估：生产流量 10% 实时打分（LLM-as-Judge）
   · 对比 SLO 基线：task_success_rate / 幻觉率 / P99延迟
   · 触发条件：连续 N 个采样周期低于阈值
        │
        ▼
② 自动关联变更
   · 查找质量下滑时间点前的最近变更记录
   · 关联 prompt_hash / 模型版本 / 工具变更
   · AI Trace × 基础设施异常联合分析（XRay）
   · 输出：疑似根因 + 置信度
        │
        ▼
③ 分级告警
   · 轻（质量下滑 < 10%）：在 ALLIN 打标 ⚠ 质量异常，通知研发
   · 中（质量下滑 10-30%）：推送回滚建议单，等待人工审批
   · 重（质量下滑 > 30% 或 SLO 严重违规）：P0 自动触发回滚（需提前授权）
        │
        ▼
④ 人工决策
   · 研发查看评估报告 + 关联变更 + Trace 样本
   · 选择：A. 立即回滚  B. 热修复（Prompt级别）  C. 继续观察
        │
        ▼
⑤ 处置执行
   · 回滚：发布平台一键回滚到上一版本，重新走灰度
   · 热修复：Prompt 修改后重新触发 Eval Gate（走快速通道）
   · 继续观察：延长监控窗口，降低告警阈值
        │
        ▼
⑥ 闭环归档
   · 记录：问题版本 / 根因 / 处置方式 / 恢复时间（MTTR）
   · 更新元数据：last_anomaly_at / anomaly_type / recovery_action
   · 写入知识库：沉淀为同类 Agent 的参考案例
```

**流程 B 的卡点清单：**
| 卡点 | 触发条件 | 处置方式 |
|------|---------|---------|
| 质量下滑检测 | 连续N期低于基线 | 分级告警 |
| 根因关联 | 自动关联最近变更 | 疑似根因 + 置信度 |
| 自动回滚授权 | P0级质量崩溃 | 需提前配置授权范围 |
| 热修复快速通道 | Prompt级别修改 | Eval Gate 缩减版（更快） |

---

#### 流程 C：Agent 全生命周期主线（创建 → 退役）

```
① 创建期
   ALLIN 创建 Workflow → 触发 workflow.created
   AgentOps：自动注册元数据 / 配置埋点 / 引导定义 SLO
   卡点：SLO 未定义不允许发布

② 开发期
   ALLIN 迭代 Workflow → 触发 workflow.updated
   AgentOps：开发态 Trace 实时上报 / 建立评估基线
   卡点：基线评估分过低时给出警示（不阻断）

③ 发布期
   → 走流程 A（发布流）
   核心卡点：Eval Gate

④ 运行期
   AgentOps：持续 Eval 采样 / SLO 监控 / 成本监控 / 安全护栏
   异常时→走流程 B（质量降级处置流）
   卡点：SLO 违规 / 成本超限 / 安全事件

⑤ 退役期
   ALLIN 下线 → 触发 workflow.deprecated
   AgentOps：归档元数据 / Trace数据保留N天 / 沉淀知识库
   卡点：有未处理告警时提示确认
```

## 方向二：Agent Native — AgentOps 的 AI 原生服务形态

### 2.1 核心命题

> 不只是做一个给人点的 Web 平台。
> 当业务 Agent 是用 Claw / vibe coding 开发出来的，
> 它们的 OPS 也应该能被 Agent 直接消费。

**三个 Agent Native 服务形态：**

```
┌────────────────────────────────────────────────────────────┐
│              AgentOps Agent Native 服务层                   │
├──────────────────┬─────────────────┬────────────────────── ┤
│  Ops Agent       │  Skill 化能力   │  事件总线              │
│ （对话式治理）    │ （原子能力包）   │ （双向事件驱动）        │
└──────────────────┴─────────────────┴───────────────────────┘
```

---

### 2.2 形态一：Ops Agent（对话式治理）

**是什么：**
AgentOps 平台自己有一个 Agent，可以被业务研发通过 Claw 对话直接调用，完成所有 OPS 操作，不需要打开 Web 平台。

**核心能力：**

| 对话意图 | Ops Agent 做什么 |
|---------|----------------|
| "帮我看看客服 Agent 线上状态" | 查元数据中心 + SLO看板 + 最近告警，汇总回答 |
| "最近质量下滑是什么原因" | 查 Trace + 关联变更 + 评估分趋势，输出根因分析 |
| "我想发布 v1.9.0" | 触发发布流程，实时播报每步状态，Eval Gate 结果直接回消息 |
| "这个版本 Eval Gate 为什么没过" | 查评估报告，列出失败样本，解释根因 |
| "帮我回滚到上一个版本" | 确认后触发回滚，播报灰度状态 |
| "这个 Agent 今天花了多少钱" | 查成本明细，对比基线，判断是否异常 |
| "帮我跑一次离线评估" | 指定数据集，触发评估，完成后推送报告 |

**交互设计原则：**
- 所有查询类操作：直接返回，不需要确认
- 变更类操作（发布/回滚）：必须二次确认，明确告知影响范围
- 高危操作（强制跳过 Eval Gate）：不执行，说明原因，引导走人工审批

**与 Claw 的集成方式：**
- 作为内部 Skill 安装到 Claw：`agentops-skill`
- 业务研发在 Claw 里直接问，Ops Agent 响应

---

### 2.3 形态二：Skill 化能力（原子能力包）

**是什么：**
AgentOps 的每个核心能力，都封装成独立的 Skill，可以被其他 Agent 直接调用。不是给人用的 UI，是给 Agent 用的 API 封装。

**Skill 清单：**

| Skill 名 | 对应能力 | 典型调用方 |
|---------|---------|----------|
| `agentops-trace-query` | 查询 Trace 链路 | 诊断 Agent、Ops Agent |
| `agentops-eval-run` | 触发离线评估 | CI/CD Agent、Ops Agent |
| `agentops-eval-gate` | 查询/触发 Eval Gate | 发布 Agent |
| `agentops-deploy` | 触发发布/回滚 | Ops Agent、CI/CD Agent |
| `agentops-cost-query` | 查询成本数据 | 报表 Agent、Ops Agent |
| `agentops-slo-status` | 查询 SLO 达成情况 | 监控 Agent、Ops Agent |
| `agentops-alert-query` | 查询告警事件 | 诊断 Agent、Ops Agent |
| `agentops-meta-get` | 查询 Agent 元数据 | 任意 Agent |

**Skill 设计规范：**
- 每个 Skill 输入/输出结构化（JSON Schema）
- 支持自然语言参数（Skill 内部做意图解析）
- 错误返回结构化（error_code + message + suggestion）
- 无副作用的查询 Skill 无需鉴权，变更 Skill 需 token + 二次确认

---

### 2.4 形态三：事件总线（双向事件驱动）

**是什么：**
AgentOps 和业务 Agent（ALLIN/Darwin）之间不靠人工触发，靠事件自动驱动，形成闭环。

**事件清单（完整版）：**

#### AgentDev → AgentOps（业务 Agent 侧发出）
| 事件 | 触发时机 | AgentOps 响应 |
|------|---------|--------------|
| `workflow.created` | Agent 创建 | 注册元数据 + 配置埋点 + 引导定义 SLO |
| `workflow.updated` | Workflow 迭代 | 更新元数据 + 触发基线重建（变更幅度大时）|
| `workflow.deprecated` | Agent 下线 | 归档元数据 + 保留 Trace N 天 |
| `deploy.requested` | 发起发布 | 触发 Eval Gate + 写入变更记录 |

#### AgentOps → AgentDev（AgentOps 侧发出）
| 事件 | 触发条件 | 业务 Agent 侧响应 |
|------|---------|-----------------|
| `eval.gate.passed` | Eval Gate 通过 | 继续发布流程 |
| `eval.gate.failed` | Eval Gate 不通过 | 暂停发布 + 附评估报告 |
| `quality.degraded` | 在线评估质量下滑 | 打标 ⚠ + 通知研发 |
| `rollback.suggested` | 质量异常关联到变更 | 推送回滚建议单 |
| `slo.violated` | SLO 指标违规 | 告警 + 降级建议 |
| `cost.threshold.exceeded` | 成本超预算 | 告警 + 触发降级模型 |
| `security.alert` | 安全护栏触发 | 实时中断 + 人工介入 |

**事件总线技术形态（建议）：**
- 短期：HTTP Webhook（AgentOps 推送到 ALLIN 注册的 URL）
- 中期：内部消息队列（MQ）解耦，支持重试和顺序保证
- 长期：统一的 Agent 事件总线（跨平台、跨团队）

---

### 2.5 三种形态的关系

```
业务研发（用 Claw）
      │ 自然语言
      ▼
  Ops Agent ──── 调用 ────► Skill 化能力（原子操作）
      │                          │
      │                          ▼
      │                    AgentOps 平台核心
      │                   （5个能力柱子）
      │                          │
      └──────── 订阅 ────────► 事件总线
                                 │
                                 ▼
                          ALLIN / Darwin
                         （业务 Agent 响应）
```

**三者分工：**
- **Ops Agent**：面向研发的对话入口，聚合多个 Skill，有上下文记忆
- **Skill 化能力**：原子操作，可被任意 Agent 调用，无状态
- **事件总线**：系统间自动驱动，不依赖人主动触发

## 落地优先级

| 阶段 | 方向一（OPS能力） | 方向二（Agent Native） |
|------|----------------|----------------------|
| Phase 1（近期） | 元数据中心 + 变更语义 + Eval Gate + 基础Trace | 事件总线（Webhook版）+ 核心 Skill × 3 |
| Phase 2（中期） | 在线评估 + 质量处置流 + 成本治理 | Ops Agent（Claw接入）+ Skill 全集 |
| Phase 3（远期） | 安全护栏 + 热修复快速通道 + 知识库 | 事件总线升级（MQ）+ 跨团队 Agent 接入 |
