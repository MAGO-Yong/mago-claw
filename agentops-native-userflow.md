### 6.3 用户动线设计：Agent Native 如何贴近真实使用场景

> 方向二的三种形态（Ops Agent / Skill / 事件总线）是能力视角。本节从**用户视角**出发，定义三类核心角色的关键使用动线，说明每个时刻 Native 体验是什么样的。

---

#### 6.3.1 三类用户角色

| 角色 | 主要诉求 | 当前痛点 |
| --- | --- | --- |
| **研发（Agent 开发者）** | 快速交付，质量可控 | 发布前不知道 Agent 是否 OK，出问题了定位慢 |
| **SRE / Oncall** | 快速感知异常，有效处置 | 告警信息碎片化，根因要跨多个系统手动拼 |
| **Agent 自身（程序化调用）** | 调下游前知道对方是否健康 | 下游 Agent 质量不透明，盲调后才知道有问题 |

---

#### 6.3.2 角色一：研发的发布动线

**场景：研发完成 Prompt 修改，想发布新版本**

研发在 Vibe Coding 环境（ALLIN / Darwin）中完成 Agent 修改后，**不需要打开任何额外的 Web 平台**，整条动线在 Claw 对话里闭环。

**关键决策时刻 × Native 体验：**

**时刻 1：我准备好发布了吗？**

研发在 Claw 问 Ops Agent："客服退款助手可以发布吗？"

Ops Agent 执行：预检 agent_meta 完整性 → 变更语义风险定级 → 查当前 SLO 达成 → 查数据集是否就绪

```
Ops Agent 回复：
  ⚠️ 发布预检：有 1 项需要确认

  ✅ agent_meta 完整（5 类变更载体已声明）
  ⚠️ 变更风险：Prompt 修改，语义相似度 0.63，定级「中」
     建议跑完整评估集（200 条），当前仅有 47 条
  ✅ 当前 SLO：任务完成率 87.2%，幻觉率 3.1%（均达标）

  建议先补全数据集再触发 Eval Gate。需要我帮你从历史 Trace
  里找 200 条相关样本吗？
```

**时刻 2：评估通过了，可以发布吗？**

研发："好，帮我发布 v1.9.0"

Ops Agent 调用 `agentops.deploy.trigger`，二次确认后触发发布：

```
Ops Agent 回复：
  📋 发布确认

  · Agent：客服退款助手
  · 版本：v1.8.3 → v1.9.0
  · 变更摘要：Prompt 修改（退款流程约束增强）
  · Eval Gate：✅ 通过（任务完成率 89.1%，幻觉率 2.7%）
  · 灰度计划：1% → 5% → 20% → 全量

  确认发布？（回复「确认」继续）
```

研发回复"确认"→ 灰度开始，后续进度通过事件总线主动推送到 Claw 消息，无需研发主动查询。

**时刻 3：灰度过程中质量有问题，怎么办？**

事件总线检测到质量下滑 → 推送到研发 Claw 消息：

```
⚠️ 客服退款助手 · 灰度质量告警

当前阶段：5% 灰度
评估分：74.3%（基线 87.2%，下滑 14.8%）
失败模式：61% 的失败样本是「退款前未查订单」

高度关联：昨天 14:02 的 Prompt 修改

建议：
  [一键回滚至 v1.8.3]   [查看失败 Trace 详情]   [手动继续灰度]
```

研发点击「一键回滚」→ Ops Agent 调 `agentops.deploy.trigger`（rollback 模式）→ 执行 → 播报结果。**全程不需要打开 Web 平台，决策 + 执行都在 Claw 里完成。**

---

#### 6.3.3 角色二：SRE / Oncall 的异常处置动线

**场景：凌晨收到告警，需要快速定位并处置**

**时刻 4：告警来了，根因是什么？**

SRE 收到 Claw 消息推送（事件总线 `quality.degraded` 触发）：

```
🚨 SLO 违规告警

· Agent：商品推荐助手
· 违规指标：任务完成率 71.3%（SLO 85%），持续 23 分钟
· 影响面：约 3,200 次请求/小时

正在自动诊断中…
```

30 秒后追加推送（Ops Agent 后台完成诊断）：

```
📊 根因分析（置信度 82%）

  变更关联：
  · 今日 02:17 知识库更新（RAG 内容版本 3.1 → 3.2）
  · 失败率在 02:23 开始上升，时间高度吻合

  失败模式（聚类分析）：
  · 78% 失败：「商品价格字段返回 null」
  → 推测：3.2 知识库中价格查询 Prompt 模板有变更

  历史参考：2025-09 同类案例，根因是 RAG 内容格式变化

建议：
  [回滚知识库至 3.1]   [查看 Trace 详情]   [联系知识库负责人]
```

**时刻 5：处置执行 + 验证**

SRE 点击「回滚知识库至 3.1」→ Ops Agent 二次确认（高风险操作，需 oncall 权限）→ 执行 → 5 分钟后主动推送验证结果：

```
✅ 回滚完成 · 质量恢复确认

· 知识库已恢复至 3.1（耗时 2 分 18 秒）
· 恢复后评估分：86.1%（恢复正常）
· SLO 违规已解除

本次事件已自动记录至知识库，下次同类 RAG 变更前会主动提醒评估。
```

**时刻 6：不想让同类问题再发生，怎么沉淀？**

SRE 在 Claw 说："把这次的根因和处置步骤存下来"

Ops Agent 自动生成 SOP 草稿 → 写入知识库 Layer 3 → 下次同类 Agent 发布前自动关联：

```
✅ 已写入知识库

案例：RAG 内容版本升级导致质量下滑
根因：知识库格式变更，价格查询字段为 null
处置：回滚知识库版本
预防：RAG 内容变更后必须重跑价格查询场景评估集（已加入 SLO 门禁）
```

#### 6.3.4 角色三：Agent 自身的程序化调用动线

**场景：推荐 Agent 需要调用 NLP Agent 处理文本，调用前判断是否安全**

**时刻 7：我要调下游，对方靠谱吗？**

推荐 Agent 在调用 NLP Agent 前，先通过 `agentops.health.snapshot` Skill 检查：

```json
// 调用示例
{
  "skill": "agentops.health.snapshot",
  "params": { "agent_id": "nlp-sentiment-v2" }
}

// 返回结果
{
  "health": "degraded",
  "eval_score": 71.3,
  "slo_status": { "task_completion": "violated" },
  "alert_count": 2,
  "summary": "NLP 情感分析服务当前质量下滑，不建议正常调用"
}
```

推荐 Agent 根据 `health: degraded` 自动执行降级策略（不调用，改用规则处理），**而不是盲目调用后发现下游返回结果不可信。**

**时刻 8：新 Agent 上线，如何自动纳入治理？**

研发在 ALLIN 创建新 Workflow → 自动触发事件 `workflow.created` → AgentOps 自动完成：

1. 注册 Agent 元数据（绑定工程资产 + 下发 Trace 埋点）
2. Ops Agent 主动推送 Claw 消息给研发：

```
🎉 新 Agent 已纳入 AgentOps 治理

· Agent：营销文案助手（workflow_id: mkt-copy-001）
· 已完成：元数据注册、Trace 埋点下发

📋 建议你完成以下配置（点击即可）：
  [ ] 设置 SLO 阈值（任务完成率 / 幻觉率 / 延迟）
  [ ] 创建初始评估数据集（推荐：100 条）
  [ ] 指定 Owner 和 Oncall

未完成配置前，Agent 无法通过 Eval Gate 发布。
```

研发在 Claw 对话里完成配置，不需要进入任何 Web 平台。

#### 6.3.5 用户动线与三层形态的对应关系

| 动线时刻 | 用户感知 | 底层形态 |
| --- | --- | --- |
| 研发问"可以发布吗" | 对话式，有上下文记忆 | **Ops Agent** |
| 评估结果驱动发布决策 | 系统自动关联，无需手动 | **Skill（eval.gate.check）** |
| 灰度质量告警推送 | 消息卡片 + 一键操作 | **事件总线（quality.degraded）** |
| SRE 收到告警 + 自动诊断 | 推送 + 延迟追加根因 | **事件总线 + Ops Agent** |
| SRE 一键回滚 | 高危操作二次确认 | **Skill（deploy.trigger）** |
| Agent 调用前预判 | 程序化，无人参与 | **Skill（health.snapshot）** |
| 新 Agent 自动纳入治理 | 消息引导 + 对话配置 | **事件总线（workflow.created）+ Ops Agent** |

**核心设计原则：用户在需要决策时才出现，不需要决策时系统自动驱动。**

- 查询类：Ops Agent 直接返回，无确认步骤
- 低风险操作（评估、查 Trace）：一步确认
- 高风险操作（发布、回滚）：明确二次确认 + 审计记录
- 自动化动作（告警诊断、基线更新）：事件驱动，完成后主动通知
