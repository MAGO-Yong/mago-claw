# 方向一：OPS for Agent — 五个柱子 × 三条流程（细化版）

---

## 总览：五柱子的分工逻辑

```
┌──────────────────────────────────────────────────────────────────┐
│                    问题发现链路                                    │
│   发布变更 ──感知变更──► AI 评估 ──质量判断──► AI 可观测 ──异常定位─►│
│        ↑                    │                     │              │
│        └────────────────────┼─────────────────────┘              │
│                             ▼                                    │
│               成本治理（横切）+ 安全护栏（横切）                    │
└──────────────────────────────────────────────────────────────────┘
```

**各柱子的核心分工：**
- **发布变更**：管 Agent 上线前后的"变"——改了什么、能不能上、上了之后还能不能回
- **AI 评估**：是整体的"质量传感器"——发布前判断好坏、运行中持续感知、输出数据驱动其他四柱决策
- **AI 可观测**：管运行期的"看"——Trace 看行为、指标看健康、告警看异常
- **成本治理**：横切所有柱子——每一笔 Token 消耗都要看得见、算得清、管得住
- **安全护栏**：横切所有柱子——输入/输出/行为三层防线，不该做的不能做

---

## 柱子一：发布变更

### 核心命题
> 传统发布只管"服务有没有上"，AgentOps 的发布还要管"AI 行为有没有变好"。
> 变更的定义从"镜像/配置变更"扩展到"Prompt/模型/工具/知识库变更"。

### 能力 1.1：Agent 元数据注册与管理
**做什么：** Agent 创建时自动在 AgentOps 注册，建立 Agent 的"身份档案"。

| 项目 | 内容 |
|------|------|
| **输入** | ALLIN workflow.created 事件：agent_id、owner、intent、framework、工具列表、模型 |
| **输出** | Agent 元数据记录（写入数据底座 Layer 1） |
| **核心逻辑** | 接收注册请求 → 校验必填字段 → 初始化 health=unknown → 自动关联 ONES 服务（按命名规范匹配）→ 下发埋点配置 → 引导 SLO 定义 |
| **卡点** | SLO 未定义时，发布申请给出警告（Phase 2 升级为阻断） |
| **关键字段** | agent_id / owner / intent / capability_boundary / slo_definition / xray_app_id / ones_service_id |

### 能力 1.2：变更语义感知
**做什么：** 解析每次发布携带的 AI 语义变更，让发布平台真正理解"改了什么 AI 行为"。

| 项目 | 内容 |
|------|------|
| **输入** | 发布申请携带的 agent_meta 字段：workflow_version / prompt_hash / model / tools / flow_diff |
| **输出** | 结构化变更记录，写入发布单和数据底座 Layer 2 |
| **核心逻辑** | diff prompt_hash（对比上一版本）→ 解析 flow_diff（新增/删除/修改哪些节点）→ 识别模型变更 → 识别工具集变更 → 识别 RAG 知识库版本变更 → 生成变更摘要（人可读 + 机器可读） |
| **卡点** | agent_meta 字段缺失时拒绝创建发布单；Prompt 变更幅度 > 阈值时标记为"大幅变更"，触发更严格的 Eval Gate |
| **变更类型分级** | 低风险（仅措辞优化）/ 中风险（逻辑调整/工具增减）/ 高风险（模型替换/核心流程重构）|

**变更类型 → 风险等级 → 处置方式：**
```
Prompt 措辞微调         → 低风险 → 标准 Eval Gate
Prompt 逻辑调整         → 中风险 → 标准 Eval Gate + 更大数据集
工具增减                → 中风险 → 标准 Eval Gate + 工具专项评估
模型版本升级            → 中风险 → 标准 Eval Gate（必须含回归用例）
模型替换（跨厂商）       → 高风险 → Eval Gate + 人工审批
核心流程节点增删         → 高风险 → Eval Gate + 人工审批
RAG 知识库全量更新       → 高风险 → Eval Gate + 召回专项评估
高危工具（exec_shell等） → 高风险 → 强制人工审批，不可绕过
```

### 能力 1.3：Eval Gate（评估门禁）
**做什么：** 用历史数据集回放新版本，评估分达标才放行发布，不达标不让上线。

| 项目 | 内容 |
|------|------|
| **输入** | 新版本 Agent、绑定的评估数据集、SLO 中定义的各项阈值 |
| **输出** | 评估报告（通过/驳回 + 各维度得分 + 失败样本列表）；回传 eval.gate.passed / failed 事件 |
| **核心逻辑** | 选取数据集（全量或抽样，按变更风险等级决定规模）→ 并行运行多个评估器 → 汇总各维度得分 → 对比阈值 → 全部达标放行，任一不达标驳回 |
| **评估维度** | 任务完成率（核心）/ 幻觉率 / 格式合规率 / vs 基线 delta（回归检测）/ 工具调用合规率（工具变更时必评）|
| **卡点** | 不达标不放行，不可跳过；高优 Agent 可配置"人工审批覆盖"（需留审计记录）|
| **加速机制** | 未涉及的节点无需全量回放（增量评估）；低风险变更走精简数据集（50条 vs 标准300条）|

**Eval Gate 决策矩阵：**
```
全部指标达标                    → ✅ 放行
任务完成率不达标                → ❌ 驳回（核心指标，一票否决）
幻觉率超限                      → ❌ 驳回（安全类，一票否决）
vs 基线 delta < -5%（显著退化） → ❌ 驳回
格式合规率不达标                → ⚠ 警告（不阻断，但记录）
工具调用合规率不达标（工具变更时）→ ❌ 驳回
```

### 能力 1.4：灰度流量管控
**做什么：** 新版本不直接全量上线，分阶段切流，发现异常及时止损。

| 项目 | 内容 |
|------|------|
| **输入** | Eval Gate 通过的新版本；灰度策略配置 |
| **输出** | 分阶段流量切换；异常时自动暂停并告警 |
| **核心逻辑** | 按策略切 1% 流量到新版本 → 观察窗口期（默认 15min）→ 监控异常率/质量/延迟 → 全部正常则自动扩量 → 直到 100% |
| **监控指标** | 新版本 vs 旧版本：错误率对比 / task_success_rate 对比 / P99 延迟对比 |
| **卡点** | 灰度期异常率超阈值 → 自动暂停（不自动回滚，需人工决策）；延迟超阈值 → 告警 + 暂停 |
| **灰度策略** | 默认：1% → 10% → 50% → 100%；高风险变更：1% → 5% → 20% → 100%（更长观察窗口）|

**灰度异常的处置分支：**
```
灰度期发现异常
      │
      ├── 错误率暴涨（>5x 基线）→ 自动暂停 + P0 告警 + 推送回滚建议
      ├── 质量下滑（>10%）      → 自动暂停 + 告警 + 附评估对比报告
      ├── 延迟劣化（>1.5x 基线）→ 告警，不自动暂停（成本容忍）
      └── 成本暴涨（>2x 基线）  → 告警 + 记录，不暂停
```

### 能力 1.5：版本回滚
**做什么：** 线上发现问题时，快速回退到上一个稳定版本。

| 项目 | 内容 |
|------|------|
| **输入** | 回滚指令（人工触发或自动触发）+ 目标版本（默认上一版）|
| **输出** | 回滚发布单；重走灰度流程（1% 先切，确认无问题再全量）|
| **核心逻辑** | 确认目标版本的 Eval Gate 历史结果（有历史记录则不重跑）→ 创建回滚发布单 → 走灰度流程（可配置为快速通道：1%→100%，窗口缩短到5min）|
| **卡点** | P0 授权的自动回滚：质量下滑 >30% 且根因已关联到当前变更 → 自动触发（需提前配置授权范围）|
| **版本保留策略** | 保留最近 10 个版本的完整快照（含 Prompt/模型/工具/评估报告）|

### 能力 1.6：热修复快速通道
**做什么：** 线上紧急问题需要修改 Prompt（不涉及代码），走缩减版流程快速上线。

| 项目 | 内容 |
|------|------|
| **输入** | 仅 Prompt 级别变更（prompt_hash 变更，其他不变）|
| **输出** | 缩减版 Eval Gate 报告；快速灰度（1%→100%，窗口 5min）|
| **核心逻辑** | 识别为 Prompt-only 变更 → 用精简数据集（50条核心用例）跑快速 Eval → 通过则直接走快速灰度 |
| **约束** | 仅限 Prompt 变更，涉及工具/模型变更不走此通道；每个 Agent 每天最多触发 3 次 |

## 柱子二：AI 评估

### 核心命题
> 评估是整个 AgentOps 的"质量传感器"。
> 没有评估，"Agent 是否健康"就只是感觉；有了评估，质量才有数字、有基线、有趋势。
> 评估数据驱动发布卡点、运行期告警、回滚决策、知识库沉淀。

### 能力 2.1：离线评估（发布前质量门）
**做什么：** 用固定数据集对新版本进行确定性质量评测，生成基线快照。

| 项目 | 内容 |
|------|------|
| **输入** | 新版本 Agent + 评估数据集（正样本/负样本/边界样本）+ SLO 阈值 |
| **输出** | 评估报告：总分 / 各维度分 / 失败样本列表 / vs 上一版本 delta |
| **核心逻辑** | 并发运行新版本 Agent 处理数据集 → 多评估器并行打分 → 汇总 → 对比基线 → 生成报告 |
| **评估维度** | 任务完成率 / 幻觉率 / 工具调用合规率 / 格式合规率 / 安全合规率 |
| **评估数据集要求** | 正常用例（60%）+ 边界用例（20%）+ 对抗用例（20%）；版本化管理；每季度更新 |

### 能力 2.2：在线评估（运行期持续感知）
**做什么：** 对生产流量进行实时采样打分，持续感知 Agent 质量，不依赖用户反馈。

| 项目 | 内容 |
|------|------|
| **输入** | 生产流量（采样率默认 10%）+ 评估器配置 |
| **输出** | 实时质量得分；SLO 达成状态；质量趋势图 |
| **核心逻辑** | 旁路采样（不影响主链路）→ 异步送评估器打分 → 滑动窗口聚合（默认 1h）→ 对比 SLO 基线 → 异常时触发告警 |
| **采样策略** | 默认 10% 随机采样；失败请求 100% 采样；高价值用例（人工标注过的）100% 采样 |
| **延迟要求** | 采样 + 打分异步，不阻塞主链路；评估结果延迟 ≤ 5min |

### 能力 2.3：评估器体系
**做什么：** 提供多种类型的评估器，支持不同维度的质量判断。

**评估器分类：**

| 类型 | 实现方式 | 适用维度 | 优缺点 |
|------|---------|---------|--------|
| **LLM-as-Judge** | GPT-4o/Claude 作裁判 | 任务完成率、推理质量、输出相关性 | 准确率高、成本较高、延迟较高 |
| **规则评估器** | 正则/关键词/结构校验 | 格式合规、敏感词过滤、必填字段 | 速度快、成本低、覆盖面有限 |
| **语义相似度** | Embedding 计算 | 输出与参考答案的语义一致性 | 中等成本、适合有参考答案场景 |
| **自定义评估器** | 业务方自定义函数 | 业务特定规则（如退款逻辑合规性）| 最灵活、需业务方维护 |
| **人工标注** | 人工队列 | 高置信度校准、对抗样本判定 | 质量最高、成本最高、不可实时 |

**评估器组合策略：**
```
发布前离线评估（追求准确）：
  LLM-as-Judge（主）+ 规则评估器（辅）+ 自定义评估器（业务规则）

运行期在线评估（追求速度）：
  规则评估器（实时）+ 语义相似度（实时）+ LLM-as-Judge（异步，采样5%）

人工标注（定期校准）：
  抽取 LLM-as-Judge 评分的边界案例 → 人工确认 → 更新评估器权重
```

### 能力 2.4：数据集管理
**做什么：** 版本化管理评估数据集，确保评估的可重复性和持续有效性。

| 项目 | 内容 |
|------|------|
| **数据集来源** | 人工编写（冷启动）/ 生产流量挖掘（人工审核后入库）/ 对话上下文自动提取（需审核）|
| **版本管理** | 每个数据集有版本号；变更数据集需重跑基线评估 |
| **数据集结构** | input（用户输入）+ expected_output（期望输出，可选）+ tags（用例类型标签）+ weight（权重）|
| **更新机制** | 季度人工审核更新；发现新的失败模式时即时增补；生产流量自动挖掘（人工确认后入库）|
| **覆盖度检测** | 自动检测数据集是否覆盖主要业务场景；覆盖度不足时给出提示 |

### 能力 2.5：评估基线管理
**做什么：** 维护每个 Agent 的质量基线，作为后续所有对比的参照物。

| 项目 | 内容 |
|------|------|
| **基线建立时机** | 首次上线后（用生产流量建立）/ 每次大版本发布后（更新基线）|
| **基线内容** | 各评估维度得分 + 对应的数据集版本 + 建立时间 |
| **基线更新规则** | 新版本 Eval Gate 通过且上线稳定 N 天后，自动将新版本的评估分设为基线 |
| **基线失效检测** | 数据集大幅更新时，旧基线标记为"需重建"；提醒 owner 重新触发基线评估 |

## 柱子三：AI 可观测

### 核心命题
> 传统可观测看的是"服务活没活"，AI 可观测要多看一层："服务做对没有"。
> 且 AI 链路和基础设施链路必须打通——Agent 质量下滑可能是 LLM 问题，也可能是下游 RPC 超时。

### 能力 3.1：全链路 AI Trace
**做什么：** 完整追踪一次 Agent 运行的所有节点，包括 LLM 调用、工具调用、RAG 检索、子 Agent 调用。

| 项目 | 内容 |
|------|------|
| **Trace 结构** | 树状 Span：Agent(root) → LLM / Tool / RAG / Memory / Sub-Agent |
| **每个 Span 记录** | 输入/输出（含 Prompt + LLM 原始输出）/ 耗时 / Token 数 / 状态（成功/失败/超时）/ 工具调用参数和结果 |
| **RAG Span 额外记录** | 查询向量 / 召回文档列表 / 召回分数 / 实际使用文档 |
| **LLM Span 额外记录** | system_prompt（hash）/ 完整 messages / completion / 模型 / temperature / token_usage |
| **与基础设施打通** | AI Span 关联同一 Trace 的 HTTP/RPC/Redis/DB Span（通过 TraceContext 传播）|
| **采样率** | 开发期 100%；生产期默认 10%；失败请求 100%；用户主动打标的请求 100% |

**Trace 查询场景：**
```
按 Trace ID 精确查    → 排查单次失败请求
按 Agent 聚合查       → 看某个 Agent 的整体运行情况
按失败类型过滤        → 快速找到同类失败请求
按耗时过滤            → 找慢请求，定位性能瓶颈
按工具调用过滤        → 找某个工具调用失败的所有请求
```

### 能力 3.2：AI × 基础设施异常联合分析
**做什么：** 把 AI Trace 数据和基础设施监控数据关联起来，联合定位根因。

| 项目 | 内容 |
|------|------|
| **关联维度** | 同一 Trace ID / 同一时间窗口 / 同一服务 |
| **联合分析逻辑** | AI 质量下滑时，自动查同时段的：HTTP 错误率 / RPC 超时 / DB 慢查询 / Redis 命中率 / GC 停顿 |
| **输出** | 根因候选列表（按置信度排序）：AI 模型问题 / 工具调用超时 / 下游服务异常 / 知识库数据质量 |
| **关键价值** | 把"AI 质量下滑"和"下游 rca-service 超时 815ms"关联起来，精准定位 |

### 能力 3.3：失败模式聚类
**做什么：** 自动发现和归组类似的失败请求，暴露系统性问题（而不是看单个失败）。

| 项目 | 内容 |
|------|------|
| **聚类维度** | 失败类型（超时/幻觉/工具调用失败/格式错误）/ 失败节点 / 错误信息相似度 |
| **实现方式** | 规则分类（错误码）+ Embedding 聚类（相似错误消息）|
| **输出** | 失败模式列表：模式名称 / 出现频率 / 代表性样本 / 首次出现时间 / 趋势（上升/下降/稳定）|
| **触发告警** | 某个失败模式频率超阈值时触发告警，附代表性 Trace |

### 能力 3.4：SLO 监控与告警
**做什么：** 持续对比 Agent 实际运行指标和 SLO 定义，违规时告警。

| 项目 | 内容 |
|------|------|
| **监控指标** | 在线评估分 / P99 延迟 / 错误率 / 工具调用成功率 / 每会话 Token 数 |
| **告警策略** | 滑动窗口（1h/6h/24h）均值对比；连续 N 个窗口低于阈值才告警（避免抖动误报）|
| **告警分级** | Warning（接近阈值）/ Critical（超过阈值）/ Emergency（严重违规，触发自动处置）|
| **告警内容** | 当前值 vs 阈值 / 下滑趋势图 / 最近变更关联 / 建议处置操作 |
| **降噪机制** | 相同 Agent 同类告警 N 分钟内合并；已知问题（人工确认处置中）不重复告警 |

### 能力 3.5：指标大盘
**做什么：** 提供多层次的指标可视化，从 Agent 整体健康到单次请求细节。

**大盘层次：**
```
L1 全局大盘  → 所有 Agent 健康状态概览，异常 Agent 高亮
L2 Agent 大盘 → 单个 Agent 的完整指标：评估分趋势/SLO 达成/成本/告警
L3 版本对比   → 新版本 vs 旧版本的各项指标对比（发布时自动生成）
L4 请求明细   → 单次 Trace 详情（瀑布图 + Span 属性 + LLM IO）
```

## 柱子四：成本治理

### 核心命题
> AI 系统的成本是非确定性的——同样的 Agent，不同请求的 Token 消耗可能差 10 倍。
> 传统的"资源配额"管不住，需要精细化的感知、归因和干预机制。

### 能力 4.1：Token 精细化计量
**做什么：** 精确追踪每一笔 Token 消耗，建立可计量的成本账本。

| 项目 | 内容 |
|------|------|
| **计量粒度** | Agent 级 / 版本级 / 请求级 / LLM Span 级 |
| **计量维度** | Prompt tokens / Completion tokens / 嵌入 tokens（RAG）/ 总 tokens |
| **成本换算** | 按模型单价换算为费用（¥）；支持多模型混用场景 |
| **计量时机** | 每个 LLM Span 完成时实时记录；异步汇总到 Agent / 团队 / 月度维度 |

### 能力 4.2：成本归因
**做什么：** 把成本精确归因到团队、Agent、模型、功能，让每笔消耗都有人负责。

| 归因维度 | 说明 |
|---------|------|
| 按团队归因 | 哪个业务团队的 Agent 消耗了多少 |
| 按 Agent 归因 | 单个 Agent 的历史成本趋势 |
| 按模型归因 | GPT-4o vs GPT-3.5 的成本占比 |
| 按功能归因 | 哪个 Tool 调用最耗 Token（RAG 检索 / 长上下文 / 多轮对话）|
| 按版本归因 | 版本升级前后成本变化（自动对比）|

### 能力 4.3：成本异常检测与告警
**做什么：** 主动发现成本异常，在烧掉大量预算之前告警。

| 项目 | 内容 |
|------|------|
| **异常类型** | 单次请求 Token 暴涨（context 过长）/ 单 Agent 日消耗超基线 N 倍 / 月度消耗超预算 |
| **检测逻辑** | 实时：单次请求超阈值立即告警；小时级：对比同比/环比；日级：预测当月消耗 |
| **告警内容** | 异常幅度 / 疑似原因（context 过长 / 高频调用 / 模型升级）/ 建议操作 |
| **预测告警** | 按当前趋势预测月底消耗，提前 N 天告警（不只是超了才说）|

### 能力 4.4：成本干预（预算熔断）
**做什么：** 主动干预控制成本，而不只是看着烧。

| 干预级别 | 触发条件 | 处置方式 |
|---------|---------|---------|
| L1 优化提示 | 单次请求 tokens > 30K | 提示研发检查 context 压缩策略 |
| L2 自动截断 | 单次请求 tokens > 50K | 自动压缩历史对话，保留最近 N 轮 |
| L3 模型降级 | 单 Agent 日消耗 > 日预算 80% | 自动从 GPT-4o 降为 GPT-3.5 + 告警 |
| L4 限流 | 月消耗 > 月预算 90% | 请求限流至 50% + 告警 + 通知 owner |
| L5 硬熔断 | 月消耗达到月预算上限 | 拒绝新请求 + 强制人工审批后恢复 |

## 柱子五：安全护栏

### 核心命题
> Agent 有工具调用能力，能查数据库、发接口、写文件——比普通 LLM 的风险大一个量级。
> 安全护栏不是事后审计，而是运行时三层实时拦截：输入层 + 输出层 + 行为层。

### 能力 5.1：输入层检测
**做什么：** 在用户输入进入 Agent 之前，检测恶意意图和越权请求。

| 检测类型 | 说明 | 处置方式 |
|---------|------|---------|
| Prompt 注入攻击 | 用户输入试图覆盖系统 Prompt | 拒绝 + 告警 + 记录 |
| 越权意图识别 | 用户要求执行超出 capability_boundary 的操作 | 拒绝 + 解释 + 记录 |
| 敏感信息识别 | 输入包含密码、密钥、身份证等 | 脱敏处理后传入，不传原文 |
| 高频异常请求 | 同一用户短时间内大量异常输入 | 限流 + 告警 |

### 能力 5.2：输出层过滤
**做什么：** 在 Agent 输出返回给用户之前，过滤不合规内容。

| 过滤类型 | 说明 | 处置方式 |
|---------|------|---------|
| 敏感数据泄露 | 输出包含不应展示的数据（其他用户信息等）| 脱敏/拦截 + 告警 |
| 幻觉内容检测 | 输出包含明显虚假信息（低置信度）| 添加不确定性声明 / 拦截 |
| 违规内容过滤 | 输出违反内容安全政策 | 拦截 + 返回标准拒绝语 |
| 格式硬校验 | 输出不符合约定格式（影响下游系统）| 重试一次 / 返回错误 |

### 能力 5.3：行为层护栏（工具调用控制）
**做什么：** 控制 Agent 的工具调用行为，防止越权操作和危险动作。

| 项目 | 内容 |
|------|------|
| **工具白名单** | 每个 Agent 只能调用 capability_allow 中声明的工具 |
| **工具参数校验** | 工具调用参数是否在合理范围内（如 SQL 不允许 DELETE、资金金额不超过上限）|
| **调用深度限制** | 防止 Agent 递归调用导致无限循环（max_tool_call_depth）|
| **高危操作审批** | exec_shell / 写数据库 / 发送外部请求等高危工具，调用前强制人工确认 |
| **调用频率限制** | 同一工具在单次对话中调用次数上限，防止死循环 |

### 能力 5.4：审计日志
**做什么：** 全量记录 Agent 的所有操作，不可篡改，支持事后审计和溯源。

| 项目 | 内容 |
|------|------|
| **记录范围** | 所有输入 / 所有工具调用（含参数和结果）/ 所有输出 / 所有安全拦截事件 |
| **存储要求** | 不可篡改（写入即锁定）/ 保留 N 年（按合规要求）|
| **查询能力** | 按 agent_id / user_id / 时间范围 / 操作类型查询 |
| **审计报告** | 定期生成 Agent 行为审计报告（工具调用分布 / 拦截事件统计 / 异常行为列表）|

## 三条最佳实践流程（细化版）

---

### 流程 A：发布流（完整版）

```
开发完成
   │
   ▼
【A1】发起发布申请
   输入：agent_id + agent_meta（完整字段）
   校验：
     ✓ agent_meta 字段完整性检查
     ✓ SLO 已定义检查
     ✓ 数据集绑定检查
   失败处理：缺字段 → 返回缺失清单，拒绝创建
   │
   ▼
【A2】变更语义解析
   解析 prompt_hash diff → 工具集 diff → 模型变更 → flow_diff
   输出：变更摘要（人可读）+ 风险等级（低/中/高）
   影响：风险等级决定 Eval Gate 的数据集规模和评估器组合
   │
   ├── 高风险变更 ──► 触发人工审批（并行，不阻塞 Eval Gate）
   │
   ▼
【A3】Eval Gate
   数据集规模：低风险50条 / 中风险200条 / 高风险300条+
   评估器：规则评估器（全量）+ LLM-as-Judge（采样or全量）
   并行运行多个评估维度
   │
   ├── 不达标 ──► 驳回 + 评估报告 + 事件 eval.gate.failed → ALLIN
   │              研发修复后重新提交（不需要重走人工审批）
   │
   ▼（达标）
【A4】人工审批（高风险变更）
   高风险变更需等待人工审批通过
   审批内容：变更摘要 + Eval Gate 报告 + 影响面评估
   超时未审批（默认 24h）：自动撤销发布单，通知申请人
   │
   ▼
【A5】灰度发布
   低风险：1% → 10% → 100%（窗口 15min/30min）
   高风险：1% → 5% → 20% → 100%（窗口 30min/1h/2h）
   监控：错误率 / 评估分（在线评估采样）/ P99延迟
   │
   ├── 异常 ──► 自动暂停 + 告警 + 推送回滚建议
   │
   ▼（正常）
【A6】全量上线
   更新 Agent 元数据（current_version / last_deploy_at / health）
   建立新基线（评估分写入 baseline_eval_score）
   写入变更历史（关联 Trace 可查）
   推送事件 deploy.completed → ALLIN
```

**关键设计点：**
- Eval Gate 和人工审批并行，不串行——缩短等待时间
- 评估不通过不需要重走人工审批——研发修复后直接重测
- 灰度期的在线评估和离线 Eval Gate 的数据集不同——前者看真实流量，后者看准备好的用例
- 每次发布产生的评估报告永久关联到该版本——可随时回溯

### 流程 B：质量降级处置流（完整版）

```
【B1】持续质量感知（常态化运行）
   在线评估：10% 生产流量采样 → LLM-as-Judge 异步打分
   滑动窗口：1h / 6h / 24h 三个粒度
   对比基线：当前窗口均分 vs baseline_eval_score
   │
   ▼
【B2】异常判定（触发条件）
   连续 2 个 1h 窗口均分低于基线 10%  → 轻度异常
   连续 1 个 6h 窗口均分低于基线 20%  → 中度异常
   单个 1h 窗口均分低于基线 30%       → 重度异常（快速劣化）
   │
   ▼
【B3】根因自动关联
   查最近 N 次变更记录（default: 最近 3 次）
   时间对齐：质量下滑起始时间 vs 变更时间
   关联逻辑：下滑时间点在某次变更后 30min 内 → 高置信度关联
   补充信息：同期基础设施异常（XRay：RPC超时/DB慢查询/模型接口异常）
   输出：疑似根因列表（按置信度排序）
   │
   ▼
【B4】分级处置
   ┌──────────────────────────────────────────────────────┐
   │ 轻度（下滑 10-20%）                                   │
   │  → ALLIN 打标 ⚠ 质量异常                             │
   │  → 通知 owner（消息 + 邮件）                          │
   │  → 附：当前评估分 / 下滑趋势图 / 建议排查方向          │
   ├──────────────────────────────────────────────────────┤
   │ 中度（下滑 20-30%）                                   │
   │  → 推送回滚建议单（带关联变更信息）                    │
   │  → 等待人工审批（超时 1h 自动升级为重度处理）           │
   │  → 告警：owner + oncall                              │
   ├──────────────────────────────────────────────────────┤
   │ 重度（下滑 >30% 或 SLO Emergency 级别）               │
   │  → 自动触发回滚（需提前配置 P0 授权）                  │
   │  → 同时通知 owner + oncall + 管理层                  │
   │  → 创建 RCA 事件（自动填充：影响范围/起始时间/根因）   │
   └──────────────────────────────────────────────────────┘
   │
   ▼
【B5】处置执行
   A. 立即回滚
      → 走回滚快速通道（灰度窗口缩短到 5min）
      → 回滚完成后验证评估分是否恢复
      → 若未恢复：根因可能不是这次变更，升级排查

   B. Prompt 热修复
      → 走热修复快速通道（缩减版 Eval Gate，50 条核心用例）
      → 通过后快速灰度（1%→100%，窗口 5min）
      → 监控恢复情况

   C. 继续观察（轻度异常时常见）
      → 延长监控窗口
      → 增加在线评估采样率至 30%
      → 设置升级告警（N 小时内未改善自动升级处置级别）
   │
   ▼
【B6】闭环归档
   记录：问题版本 / 根因 / 处置方式 / 发现时间 / 恢复时间（MTTR）
   更新：last_anomaly_at / anomaly_count / mttr_avg（元数据中心）
   沉淀：写入知识库（案例类型 / 症状 / 根因 / 处置方法）
         供下次同类问题时 Ops Agent 直接引用
```

### 流程 C：Agent 全生命周期主线（完整版）

```
【C1】创建期 — 注册即治理
   ALLIN 创建 Workflow → 发送 workflow.created 事件
   
   AgentOps 自动执行：
   ① 注册 Agent 元数据（Layer 1 完整字段）
   ② 关联工程资产（自动匹配 ONES 服务 / XRay App）
   ③ 下发埋点配置（SDK 参数：采样率100% / XRay App ID）
   ④ 初始化评估配置（引导绑定数据集 / 选择评估器）
   ⑤ 引导 SLO 定义（提供同类 Agent 的 SLO 参考值）
   
   卡点设计：
   · SLO 未定义 → 允许开发，警告"发布时将被阻断"
   · 数据集未绑定 → 允许开发，警告"无法跑 Eval Gate"
   · ONES 服务未关联 → 允许，但发布时无法自动写入变更记录

   ─────────────────────────────
【C2】开发期 — 基线先行
   ALLIN 迭代 Workflow → 发送 workflow.updated 事件
   
   AgentOps 自动执行：
   ① 开发态 Trace 实时上报（采样率 100%，不等上线）
   ② 更新元数据（prompt_hash / flow_diff）
   ③ 变更幅度大时触发基线重建提示（Prompt 改动 > 30%）
   ④ 开发者触发离线评估时，记录结果为"开发期快照"
   
   能力支持：
   · 研发可以在开发期主动触发 Eval，对比不同 Prompt 版本的质量
   · 开发期 Trace 和生产期 Trace 同构，可直接对比
   · 开发期评估分低于基线时给出警示（不阻断）

   ─────────────────────────────
【C3】发布期 — 质量门禁
   → 走流程 A（发布流）完整执行
   核心卡点：Eval Gate（不达标不放行）

   ─────────────────────────────
【C4】运行期 — 持续治理
   常态化运行（7×24）：
   · 在线评估采样（10% 生产流量）
   · SLO 监控（滑动窗口对比）
   · 成本监控（实时计量 + 异常检测）
   · 安全护栏（输入/输出/行为三层实时拦截）
   · 变更-异常自动关联
   
   异常时 → 走流程 B（质量降级处置流）
   
   运行期事件：
   · AgentOps → ALLIN：quality.degraded / slo.violated / rollback.suggested / cost.threshold.exceeded
   · AgentOps → 研发：告警通知（消息 / 邮件 / oncall）

   ─────────────────────────────
【C5】退役期 — 知识沉淀
   ALLIN 下线 → 发送 workflow.deprecated 事件
   
   AgentOps 执行：
   ① 检查是否有未处理告警（有则提示确认）
   ② 元数据状态更新为 deprecated
   ③ Trace 数据归档（保留 N 天，默认 90 天）
   ④ 评估数据归档（永久保留，数据集可复用）
   ⑤ 知识沉淀：
      · 哪些 Prompt 版本效果最好 → 写入最佳实践
      · 历次异常和处置记录 → 写入案例库
      · 成本基线数据 → 供同类 Agent 创建时参考
```

## 汇总：所有卡点清单

| 卡点位置 | 触发条件 | 处置方式 | 可绕过？ |
|---------|---------|---------|---------|
| 发布申请 | agent_meta 字段缺失 | 拒绝创建发布单 | 否 |
| 发布申请 | SLO 未定义 | Phase1 警告；Phase2 拒绝 | Phase1 可 |
| 变更解析 | 高危工具（exec_shell等）| 升级人工审批 | 否 |
| 变更解析 | 模型跨厂商替换 | 升级人工审批 | 否 |
| Eval Gate | 任务完成率不达标 | 驳回（一票否决）| 人工审批覆盖（留记录）|
| Eval Gate | 幻觉率超限 | 驳回（一票否决）| 否 |
| Eval Gate | vs 基线退化 >5% | 驳回 | 人工审批覆盖（留记录）|
| 灰度期 | 错误率暴涨 >5x | 自动暂停灰度 | 否（需人工恢复）|
| 灰度期 | 质量下滑 >10% | 自动暂停灰度 | 否（需人工恢复）|
| 运行期 | 质量下滑 >30% | P0 自动回滚（需授权）| 提前配置授权范围 |
| 运行期 | 成本超月预算 | 硬熔断（需人工恢复）| 人工审批后恢复 |
| 运行期 | 高危工具调用 | 实时拦截 | 否 |
| 退役期 | 有未处理告警 | 提示确认 | 强制确认后可继续 |
