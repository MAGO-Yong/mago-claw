# AgentOps 平台规划

> 内部对齐文档 · 2026 Q2
> 定位：运维态（Ops），不含 Dev 态（框架、Memory、RAG、编排）

---

## 一、为什么要做 AgentOps

### 1.1 Agent 规模化带来的新挑战

2026 年基础技术各团队大规模孵化 Agent，运维侧面临三个结构性问题：

**① 传统监控感知不到 Agent 失效**
传统监控能感知 CPU 高、延迟高、5xx 升高，但无法感知 Agent 正在系统性地给出错误推荐、走偏的推理路径、质量下滑的输出。**"服务活着"和"服务做对了"，在 AI 系统里是两件完全不同的事。**

**② 变更的定义变了**
AI 系统存在三类传统流程完全感知不到的变更：
- Prompt 修改：不在代码里，不触发 CI/CD
- 模型版本静默升级：提供商无声更新，无任何告警
- RAG 知识库更新：内容变了但系统版本没变

**③ 没有评估就没有质量基线**
Agent 输出是非确定性的，不跑评估就不知道"健康状态"是什么样的，也就无法判断上线后是变好还是变差。

### 1.2 我们的独特优势

业界 AgentOps 平台（LangSmith、Langfuse、Arize Phoenix 等）普遍只解决"看得见"的问题，存在三个集体盲区：

| 盲区 | 业界现状 | 我们的优势 |
|------|---------|-----------|
| 发布变更 | 没有平台真正接管企业发布流水线 | **发布入口在手**，ALLIN/Darwin 必须调我们接口创建发布单 |
| AI × 基础设施打通 | 所有平台的观测止步于 AI 链路层 | XRay 同时拥有 AI Trace + HTTP/RPC/中间件数据 |
| 评估→发布硬性闭环 | 评估结果只能"看"，没有平台做到不达标不让上线 | 评估结果可直接驱动发布卡点 |

> **核心定位**：业界能做"看得见 + 测得准"，我们能做"**管得住**"。

## 二、AgentOps 的完整领域

AgentOps 覆盖五个领域，评估是所有领域的数据基础：

```
发布变更 → AI 可观测 → AI 评估 → 成本治理 → 安全护栏
              ↑           ↑
         （依赖评估作为质量传感器）
```

## 三、差异化定位

| 领域 | 业界现状 | 我们的切入点 |
|------|---------|------------|
| 发布变更 | 各平台极薄，三段断裂（CI/CD评测 → Runtime → 监控各管一段） | 发布入口在手 + 变更数据（RCM）+ AI 语义层，三合一 |
| AI 可观测 | AI 链路完整，但止步于 AI 层 | AI Trace × 基础设施（HTTP/RPC/中间件）打通，结构性做不到 |
| AI 评估 | 能力同质，缺硬性卡点 | 评估结果直接驱动发布门禁，形成硬约束 |
| 成本治理 | 只能看，不能干预 | 成本超限熔断 + 团队级归因 |
| 安全护栏 | 集体空白 | 输入/输出/行为三层防护，企业级治理 |

## 四、与 AgentDev 平台的 AI Native 联动

### 4.1 现状与问题

当前两个平台的关系：

```
ALLIN/Darwin（AgentDev）
    ↓ 调接口创建发布单
发布平台（你们）
    ↓
服务跑在你们这里
```

问题：**变更粒度停在"服务级"**，Workflow 内部发生了什么（Prompt 改了、模型版本、工具集变化）发布平台完全不知道。

| 层级 | 谁知道 | 谁不知道 |
|------|--------|---------|
| 服务维度（镜像/实例/配置） | 发布平台 ✅ | ALLIN ❌ |
| Workflow 语义（Prompt/模型/工具） | ALLIN ✅ | 发布平台 ❌ |
| 运行期行为（Trace/质量/异常） | XRay ✅ | 两边都不完整 |

### 4.2 联动核心：Agent 元数据中心

两个平台共享的唯一数据源，贯穿 Agent 全生命周期持续更新：

```
创建时写入          发布时更新              运行时刷新
─────────           ──────────              ──────────
agent_id            current_version         slo_status
owner               prompt_hash             last_eval_score
intent              model                   last_anomaly_at
capability_boundary tools                   health: normal/degraded/critical
slo_definition      deploy_at
embed_config        change_diff
```

不是发布时一次性传递，而是**两平台持续共写、持续共读的活数据**。

> 对应文档中的 Agent MetaData 模块（当前 ❌ 未建设），需从"静态注册表"升级为"动态状态中心"。

### 4.3 全生命周期联动地图

```
┌──────────┬────────────────────────────┬────────────────────────────┐
│  阶段    │  ALLIN 做什么              │  AgentOps 做什么           │
├──────────┼────────────────────────────┼────────────────────────────┤
│ 创建期   │ 创建 Workflow              │ 自动注册 Agent 元数据      │
│          │ 定义节点/工具/模型         │ 配置观测埋点（采样率/App） │
│          │                            │ 引导定义 SLO               │
├──────────┼────────────────────────────┼────────────────────────────┤
│ 开发期   │ 迭代 Workflow              │ 开发态 Trace 已开始上报    │
│          │ 测试环境调试               │ 建立评估基线（测试集）     │
│          │                            │ 离线 Eval 随时可触发       │
├──────────┼────────────────────────────┼────────────────────────────┤
│ 发布期   │ 调接口创建发布单           │ 接收 AI 语义变更包         │
│          │ 传入 agent_meta            │ 触发 Eval Gate             │
│          │                            │ 管理灰度流量               │
│          │                            │ 写入变更记录               │
├──────────┼────────────────────────────┼────────────────────────────┤
│ 运行期   │ 接收质量反馈               │ 持续 Eval 采样             │
│          │ 查看异常归因               │ 质量下滑关联变更           │
│          │ 响应回滚建议               │ 异常触发反馈回流           │
│          │                            │ SLO 违规告警               │
├──────────┼────────────────────────────┼────────────────────────────┤
│ 退役期   │ 下线 Workflow              │ 元数据归档                 │
│          │                            │ 历史数据保留               │
│          │                            │ 沉淀为后续参考案例         │
└──────────┴────────────────────────────┴────────────────────────────┘
```

### 4.4 三层联动机制

**Layer 1：变更语义感知**（Phase 1，近期）

ALLIN 创建发布单时，新增 `agent_meta` 字段传入 Workflow 语义快照：

```
workflow_version: "v2.3.1"
prompt_hash:      "a3f9..."
model:            "gpt-4o"
tools:            ["search", "code_exec", "db_query"]
flow_diff:        {新增节点/删除节点/修改节点}
```

能做到：发布单里看得到"改了什么"，出问题时变更和异常自动关联。

**Layer 2：观测埋点前置**（Phase 1，近期）

当前问题：埋点在上线后才配置，上线前没有数据，无法建基线。

新机制：ALLIN 创建 Workflow 时，AgentOps 自动下发埋点配置（采样率、上报目标 XRay App ID、关键节点埋点建议）。开发期和生产期的 Trace 同构，天然可对比。

**Layer 3：双向反馈总线**（Phase 3，远期）

不是点对点接口，而是事件驱动：

```
AgentDev → AgentOps：
  workflow.created   → 注册元数据，配置埋点
  workflow.updated   → 更新语义快照，触发基线重建
  deploy.requested   → 触发 Eval Gate
  workflow.deprecated → 归档

AgentOps → AgentDev：
  eval.gate.passed   → 放行发布
  eval.gate.failed   → 驳回 + 附评估报告
  quality.degraded   → 质量下滑告警 + 关联变更
  rollback.suggested → 回滚建议（人工审批）
  slo.violated       → SLO 违规通知
```

## 五、分阶段落地路径

```
Phase 1（近期）：数据层打通
─────────────────────────────
目标：两个平台"认识彼此"
· ALLIN 发布时传 agent_meta 字段（变更语义感知）
· AgentOps 建 Agent 元数据中心（只读入，先不回写）
· 观测埋点配置前置到创建期（开发期即有 Trace）
· 评估基线在测试期建立
交付物：发布单有语义、创建即有 Trace、有质量基线

Phase 2（中期）：评估层接入
─────────────────────────────
目标：AgentOps "能判断好坏"
· 发布期 Eval Gate 上线（评估结果作发布门禁）
· SLO 定义写入元数据，AgentOps 持续对齐
· 在线评估：生产流量实时打分，持续监控质量漂移
· AI Trace × 基础设施异常关联
交付物：发布有门禁、运行有质量基线、异常有归因

Phase 3（远期）：治理闭环
─────────────────────────────
目标：AgentOps 从"观测"变"治理"
· 双向事件总线建设
· 质量下滑自动关联变更 + 回流 ALLIN
· 回滚建议 / 自动回滚（分级）
· 安全护栏：输入/输出/行为三层实时拦截
· 成本超限自动熔断 + 团队级归因
交付物：真正的负反馈闭环，管得住
```

## 六、核心主张

> **评估是入场券，发布卡点才是护城河。**

业界做评估的都在做，我们的差异化在于：**评估结果 × 发布入口 × 基础设施数据**，三者合一，形成别人结构性做不到的治理闭环。

AgentOps 的终极目标：

```
传感器（评估）+ 执行器（发布卡点/护栏）= 负反馈闭环
```

让 Agent 不只是"跑起来"，而是"**跑得稳、跑得好、跑得省**"。
