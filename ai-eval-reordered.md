# AI 应用评估产品使用手册

## 1. 认识 AI 应用评估

### 1.1 AI 应用评估是什么
AI 应用评估 是 X-RAY 中面向 AI 应用效果验证、问题发现和持续优化的一套产品能力。它关注的不是某个基础模型的通用能力强不强，而是一个更贴近真实业务的问题：这个智能体（Agent、Workflow）在真实场景里到底好不好用。对业务方来说，关键不只是模型“会不会答”，而是它在用户真实请求、复杂上下文、工具调用、RAG 检索和多轮交互中，能不能稳定给出符合预期的结果；对研发和算法同学来说，关键则是版本更新后到底有没有变好、有没有劣化、问题出在哪一层、能不能持续回归验证。
因此，AI 应用评估不是一个单纯“给结果打分”的工具，而是一套围绕真实样本、评分规则、实验结果和问题沉淀构建起来的工作流。它的目标不是替代人工判断，而是帮助团队建立统一、可重复、可持续的评估方法，让效果验证从“凭感觉”变成“有依据、有记录、能追溯、能优化”的过程。换句话说，AI 应用评估就是基于真实链路或离线数据集，对 AI 应用进行持续评估、分析和优化的一整套机制。

### 1.2 AI 应用评估解决什么问题
在真实业务里，AI 应用往往并不缺数据，而是缺少把数据转化成“可判断、可比较、可优化结果”的能力。AI 应用一旦上线，系统里会快速积累大量真实链路，但团队常常并不知道应该用什么标准来系统性判断效果好坏。模型升级后，某些场景的输出悄悄波动了；Agent 策略改动后，线上回答开始偏离预期；上下游变更后，工具调用、RAG 检索或最终回复中的某一环节逐渐失效。这些问题并不是完全不可见，而是缺少统一、持续、自动化的方式去发现。
另一个高频问题发生在版本迭代阶段。很多团队能感知“新版好像更好了”，但往往缺少统一的评估口径，最终很难回答一个最关键的问题：这次改动到底是优化了，还是劣化了。于是版本上线经常靠经验判断，bad case 靠人工记忆，回归验证靠临时抽样，导致评估结果不稳定，也无法长期积累成可复用的方法论。
第三个问题是问题样本沉淀不下来。即使线上已经发现过 bad case，如果没有机制把它沉淀成数据集，那么下一版是否修复、同类问题是否再次出现、哪些边界场景仍未覆盖，就很难被稳定验证。这样一来，线上真实问题无法持续反哺评估体系，团队也无法建立真正的优化闭环。

### 1.3 适合哪些角色使用
这套产品适合业务同学、算法同学、应用研发和平台同学共同使用。对业务同学来说，它帮助回答“AI 功能是否真的满足用户需求、哪里表现好、哪里表现差”；对算法同学来说，它帮助验证 Prompt、策略和模型迭代前后的效果差异；对应用研发来说，它帮助确认 Agent、workflow、工具调用等关键链路是否稳定；对平台同学来说，它帮助把线上链路、自动评估、实验分析和数据回流串成统一流程。因此，这个产品不是只服务某一类角色，而是服务整条 AI 应用生产链路上的协作。

### 1.4 AI 应用评估的核心流程
从产品心智上，可以把整套流程理解成：
观测 -> 数据 -> 评估 -> 优化 -> 再观测
如果从用户实际操作出发，最常见的路径是下面这条：
![图片](https://xhs-doc.xhscdn.com/104004dg31uddmmpbj40akp4c8o?imageview2/2/w/1600)
这条流程可以拆成四个连续的阶段来理解。第一阶段是准备规则，也就是先确定“谁来判分”。如果使用 LLM 评估器，就要先准备 LLM 配置，然后创建评估器，把评估标准、输出字段和评分方式定义清楚；如果使用 HTTP 评估器，则需要先有可对外调用的业务服务，通过代码实现复杂规则匹配逻辑。第二阶段是准备数据，也就是确定“这次到底评什么”。这里通常有两种来源：一种是已经上线的真实线上链路，另一种是提前准备好的离线数据集。第三阶段是组织执行，这一步由 自动评估任务 承担，它负责把数据来源、评估方式、评估器、调度周期和回流策略组合成一份真正可运行的任务。第四阶段是查看结果并继续优化，任务执行后会生成 评估实验，用户真正消费的是对应的 实验报告，在报告中查看整体表现、异常样本和评分原因，并决定哪些样本要继续回流沉淀。
这里有一个很容易被忽略、但对整套产品都非常关键的前提：评估不是在“真空”里发生的，它始终建立在输入之上。评估器之所以能判断一个结果是否合理，是因为链路里已经采集到了足够完整的输入、输出和中间过程；实验报告之所以能解释问题，是因为这些输入在进入任务前已经被结构化、被筛选、被映射。如果前面的采集和输入定义不清楚，后面的评估、分析和回流都会失真。所以从产品逻辑上看，评估数据 这一步并不只是“选样本”，而是在决定整套评估到底能看见什么、能判断什么、不能判断什么。

第一次使用时，不建议一上来就直接新建任务，因为这样很容易在“数据来源、评估器、任务配置、实验结果”之间来回跳转，最后对整体关系依然不清楚。更推荐的理解顺序是：先理解 评估器，再理解 数据集，接着看 自动评估任务，最后再去看 实验报告。这样做的好处非常实际：如果不理解评估器，你就不知道系统到底怎么判分；如果不理解数据来源，你就不知道任务到底在评什么；如果不理解实验报告，你也很难判断这个任务是否真的有价值。

### 1.5 在线评估与离线评估
当前产品主要支持两种评估方式：在线链路评估 和 离线数据集评估。在线链路评估适合AI 应用已经上线、希望基于真实用户请求持续监控效果的场景，它会直接基于真实 trace、observation、输入输出、工具调用、RAG 等链路信息来做评估，因此更适合发现线上问题、观察周期波动和持续沉淀 bad case。这里的关键不只是“线上数据更新更快”，而是这些链路数据本身已经被采集和结构化，能够支撑后续按 trace、SPAN、最终结果等不同粒度去做判断。离线数据集评估则更适合应用仍在开发或迭代阶段，或者已经准备了一批可重复使用的样本，希望基于这些样本做稳定版本对比和回归验证的场景。两者的差别，本质上不是“线上更高级、离线更简单”，而是数据来源和目标不同：在线更贴近真实运行，离线更适合稳定对比。

### 1.6 LLM 评估器与 HTTP 评估器
当前产品中的评估器主要分为两类：LLM 评估器 和 HTTP 评估器。LLM 评估器 通过 Prompt + LLM 的方式，对输出进行语义判断和打分，更适合相关性、正确性、幻觉、query 合理性这类需要理解上下文和业务语义的问题。HTTP 评估器 则通过代码实现复杂规则匹配逻辑，以业务服务形式对外提供评估接口，更适合规则较复杂、需要精确控制判断逻辑的场景。用户在这一步通常不需要一开始就纠结“哪一种更高级”，而是先看自己要解决的问题到底更接近语义判断，还是更接近规则判断。


### 1.8 如何选择适合自己的评估方案
如果你是第一次接触这个产品，最容易卡住的通常不是“不会点页面”，而是不知道自己到底应该从哪条路径开始。很多用户会在“我是应该先建数据集，还是先建评估器”“我到底该选在线还是离线”“我是不是应该直接上 LLM 评估器”这几个问题之间来回犹豫。下面这张图的作用，就是先帮你选路，而不是直接教你点按钮。
![图片](https://xhs-doc.xhscdn.com/104004dg31uddn7441m0e8kv7pc?imageview2/2/w/1600)
如果你的 AI AI 应用已经上线，希望基于真实用户请求做持续监控，希望自动发现线上 bad case，并把真实问题沉淀下来，那么更适合优先选择 在线链路评估。如果你已经有一批结构化样本，希望做版本效果对比、回归验证，或者想在上线前先尽可能把问题暴露出来，那么更适合优先选择 离线数据集评估。如果你的判断问题主要依赖语义理解，需要结合上下文去判断结果是否合理，比如相关性、幻觉、query 合理性这类问题，那么通常优先选择 LLM 评估器；如果你已经有业务 checker，或者规则已经被封装成服务，希望判定逻辑更稳定、更可控，那么通常更适合选择 HTTP 评估器。


### 1.9 快速开始：跟着 sug 粗筛走一遍完整流程

> 这一节不是功能说明，而是一条可以直接跟着操作的完整路径。读完前面 1.1-1.8 后，建议先跟着这条路径走一遍，建立真实的操作感，再去看后面各章的模块细节。
>
> **前提**：你的 AI AI 应用已经接入 X-RAY，并且在 AI 链路分析页面可以看到稳定的 Trace 数据。如果还没有接入，先参考 [AI 应用链路埋点上报规范](https://docs.xiaohongshu.com/doc/ba8f8236ae557122bcc71af185d1ad82) 完成接入后再回来。

### 1.9.1 第一步：配置 LLM（评估器的底层模型）

进入 **AI 应用评估 → LLM 配置**，点击「添加 LLM 配置」，填写：
- 配置名称：随意，建议体现模型名称，例如 `gpt-4o-eval`
- 模型提供商、模型名称、API Key、Endpoint：填写你们环境中真实可用的模型信息。模型资源需提前在 [QS 平台](https://qs2.devops.xiaohongshu.com/modelrepo) 申请，评估器运行时消耗的 token 都来自这里申请的模型配额

填完后点「测试链接」，确认状态为「连通」后保存。

> **为什么先做这步**：LLM 评估器依赖底层模型才能运行，没有这一步，评估器创建了也无法执行。

### 1.9.2 第二步：创建 LLM 评估器

进入 **评估器**，点击「新建评估器 → LLM 评估器」，填写：

- **名称**：例如 `sug query 相关性评估`
- **评估方式**：选择与你的场景匹配的方式（全链路 / 单 SPAN / 结果评估）
- **LLM**：选择第一步创建的配置
- **Prompt**：这是评估器的核心，它告诉模型按什么标准来判断。以 sug 粗筛场景为例：

```
**任务**
你会收到一段笔记内容（message）和 AI 生成的搜索推荐词（query）。
请判断每个 query 是否与笔记内容相关，是否能吸引用户点击。

**判断标准**
- query 必须与笔记内容直接相关
- query 长度不超过 10 个中文字符
- 不能出现重复 query
- 对于图文笔记，不能出现"总结视频"

**输入**
笔记内容：{{message}}
待判断的 query：{{query}}

**输出格式**
对每个 query 输出"是"或"否"，格式如下：
<query>
  <q>是/否</q>
  ...
</query>
```

- **字段映射**（告诉系统从 SPAN 的哪个字段取值注入到 Prompt 变量中）：

| Prompt 变量 | 来源 | JsonPath |
|-------------|------|----------|
| `query` | output | `$.response_content` |
| `message` | input | `$.messages[?(@.role == 'user')]` |

- **输出字段**：至少配置 `score`（数值型）和 `reason`（字符串型）

填完后保存。

### 1.9.3 第三步：创建自动评估任务

进入 **自动评估**，点击「新建自动评估任务」，按 4 步填写：

**第一步 基本配置**
- 任务名称：例如 `sug粗筛-在线评估`
- 评估类型：选「在线链路数据」
- 执行周期：先选「按天」，后续可以调整

**第二步 评估数据**
- **评估方式**：LLM 评估器只支持「单 SPAN 评估」或「结果评估」，不支持全链路评估。sug 粗筛场景选「单 SPAN 评估」（对单个 LLM 调用 SPAN 进行评估）
- Trace 筛选：填入你的服务名或 scene 字段，例如 `scene=sug`
- 采样率：第一次建议设 1%-5%，控制规模，先跑通再扩大

**第三步 评估器**
- 选择第二步创建的评估器
- 确认字段映射已自动带入，检查变量和 JsonPath 是否正确

**第四步 数据回流（可选）**
- 暂时跳过，先把主流程跑通

保存后，在列表页点击「触发」，手动执行一次。

### 1.9.4 第四步：查看评估实验

进入 **评估实验**，找到刚才触发的那条实验，等状态变为「已完成」后点进去。

重点看两个地方：

1. **实验明细**：按评估分筛选，找得分为 0（"否"）的样本，点开单条详情，看评估器给出的 reason——这是判断评估器是否在按你预期工作的最直接方式
2. **实验分析**：看整体得分分布，判断当前线上 query 质量的整体水位

如果发现 reason 不符合预期（例如把明显相关的 query 判成"否"），回去调整评估器的 Prompt，再手动触发一次，对比两次实验结果。

### 1.9.5 下一步：开启数据回流

当你确认评估器的判断基本符合预期之后，可以回到自动评估任务里，编辑任务，在「数据回流」步骤中：
- 设置回流目标数据集（提前创建好）
- 回流条件：例如 `score = 0`（即被判为"否"的 query）
- 字段映射：把 SPAN 的 input / output 映射到数据集字段

这样，每次任务运行后，被判为低质量的 query 样本就会自动沉淀进数据集，成为后续人工复盘和离线回归验证的基础。

---

走完这条路径，你就完成了一次完整的「在线链路 → LLM 评估 → 实验报告 → 数据回流」闭环。后面想做版本对比、离线回归或扩展更多评估器，都可以在这个基础上继续迭代。

## 2. 典型使用场景

### 2.1 场景一：离线数据集评估

### 2.1.1 什么时候适合离线评估
离线数据集评估最适合“你已经有样本，想稳定验证效果”的场景。很多团队在做应用迭代时，最先遇到的问题不是没有线上数据，而是不知道怎样把已经遇到过的问题稳定复现出来。比如你已经整理了一批 bad case，希望每次改 Prompt、调策略或换模型后都能重新跑一遍，确认这些样本到底有没有被修复；又或者你已经有一批人工确认过的高质量样本，希望把它们作为回归集，在每个版本上线前先做一次离线验证。这些场景的共同特点是：样本相对稳定，评估目标明确，希望多次复用同一批数据来比较不同版本的表现。
相对来说，如果你的主要目标是观察真实线上效果、发现最近一周才出现的新问题，或者你想持续抽样真实用户请求，那就不应该优先从离线评估开始。离线评估更像一条“可重复验证”的路径，它不一定最接近实时线上情况，但非常适合做版本回归、效果比对和上线前兜底。

### 2.1.2 离线评估的典型目标
从目标上看，离线数据集评估最常见的用途有四类。第一类是版本回归验证，也就是在每次发布前确认旧问题没有重新出现；第二类是方案对比，例如比较不同 Prompt、不同策略或不同模型配置在同一批样本上的表现差异；第三类是 bad case 复验，把之前已经发现的问题样本沉淀下来，作为后续迭代的固定检验集；第四类则是上线前效果兜底，在真实流量进入之前，先用一批已知样本做一次稳定检查。对很多刚开始建设评估体系的团队来说，离线评估也是最容易起步的一条路径，因为它不依赖实时线上抽样，只要样本准备得足够清楚，就可以先建立最基础的一套评分和对比机制。

### 2.1.3 离线评估的推荐使用路径
离线评估的使用路径通常比较清晰。第一步是创建数据集，并根据你的评估对象设计字段，例如输入内容、上下文、标准答案、模型输出、标签信息等；第二步是把样本放进数据集，样本可以来自人工整理、历史问题沉淀，也可以来自之前的链路回流结果；第三步是创建评估器，明确这次要评的是相关性、正确性、幻觉，还是某种业务规则；第四步是在 自动评估任务 中选择 离线数据集 作为评估数据来源，并把数据集字段映射给评估器；最后再运行任务、查看 评估实验 和对应的 实验报告，确认哪些样本得分偏低、问题集中在哪些维度。
如果用一句话概括，离线评估更像是“先把题库准备好，再反复测试不同方案”。它不强调实时性，强调的是复用、稳定和可比较。
![图片](https://xhs-doc.xhscdn.com/104004dg31uddnjjp360eg7ff14?imageview2/2/w/1600)

### 2.1.4 离线评估适合搭配哪些评估器
离线评估既可以搭配 LLM 评估器，也可以搭配 HTTP 评估器，关键要看你希望评估器扮演什么角色。如果你的问题更接近语义判断，比如“这个回答是否切题”“这个结果是否正确”“这个回复是否存在明显幻觉”，那么更适合使用 LLM 评估器，因为它更擅长结合上下文做整体判断。如果你的业务已经有比较明确的规则，或者某类判断已经被沉淀成稳定的服务逻辑，那么也可以使用 HTTP 评估器，把业务自己的判分逻辑接入统一实验体系。对第一次搭建离线评估的团队来说，通常会从一个或几个简单的 LLM 评估器 开始，因为这条路径更容易快速跑通；而当业务判断越来越复杂时，再逐渐补充 HTTP 评估器 作为更可控的自定义能力。

### 2.1.5 离线评估结果怎么看
离线评估跑完之后，不建议只盯着某个总分看。更有效的做法是从两层结果同时判断：一层是 实验明细，也就是看单条样本为什么得这个分，哪些样本被判低分、评估器给出的原因是什么、是否存在误判；另一层是 实验分析，也就是看整体聚合结果和分布情况，判断这次版本在整体上有没有明显变好、是否只优化了头部样本、是否在某个维度上出现新的劣化。对离线评估来说，这两层信息结合起来才有意义，因为单看整体分布容易忽略具体问题，单看个别样本又很难判断是否真的值得上线。

### 2.2 场景二：在线链路评估

### 2.2.1 什么时候适合在线评估
在线链路评估适合AI 应用已经上线，并且你已经能在 X-RAY 中看到稳定链路数据的场景。和离线评估相比，在线评估关心的不是“同一批样本能不能稳定复测”，而是“真实用户请求最近到底表现怎么样”。如果你希望持续观察线上效果、自动发现低分样本、监控某类问题是否在最近一段时间集中出现，或者希望把真实 bad case 自动沉淀下来，那么在线链路评估通常会比离线评估更有价值。它的优势不在于控制变量，而在于足够接近真实环境，因此更适合做持续监控和问题发现。
当然，在线评估也有前提。它要求你的 AI AI 应用已经接入 X-RAY，并且链路上报、输入输出记录、observation 结构等信息比较完整。如果链路本身还不稳定，或者样本还没有形成基本结构，那么这时直接做在线评估，往往会把“链路问题”和“评估问题”混在一起，不利于初期搭建。
从输入视角看，在线评估之所以成立，依赖的正是这套结构化链路数据。你可以把一次完整请求理解成一条 trace，把链路中的某个具体执行环节理解成一个 SPAN / observation。因此，当你选择 全链路评估 时，本质上更接近对整条请求过程的整体判断；当你选择 单span评估 时，更接近对某一个中间环节，例如一次模型调用、工具调用或子流程节点的局部判断；而 结果评估 则通常更聚焦于最终输入和最终输出。这组边界如果理解清楚，后面在选择评估方式时就不会只停留在名字层面。

### 2.2.2 在线评估的典型目标
在线链路评估最常见的目标可以概括成三类。第一类是线上效果监控，也就是持续了解真实用户请求在最近一段时间里的整体表现；第二类是问题发现与样本沉淀，通过自动评估把低分样本、异常样本或重点样本筛出来，并进一步回流到数据集；第三类是长期迭代观察，也就是在版本持续发布的过程中，观察某些关键指标是否稳定、是否存在周期波动、某次调整后是否带来了明显改善或劣化。对已经进入稳定运营期的 AI 应用来说，在线评估往往会成为“日常看盘”的一部分，因为它能帮助团队持续回答“最近到底有没有变差”“哪些问题正在增长”“下一轮应该优先处理什么”。

### 2.2.3 在线链路的 LLM 评估
在线链路中的 LLM 评估 更适合那些需要语义理解的问题。比如相关性判断、回答正确性、回复是否存在明显幻觉、query 是否值得点击，这些问题往往很难仅靠硬规则覆盖，而是需要结合上下文、输入意图和最终输出一起判断。在这类场景里，LLM 评估器 本质上是在扮演“语义裁判”的角色：它不是简单看某个字段是否命中规则，而是按照你定义好的 Prompt 和输出结构，对真实链路样本做更接近人工判断的分析。
当前比较典型的代表路径就是 sug粗筛。这条路径的价值不在于“它用了 LLM”，而在于它展示了一种很典型的在线语义评估方式：先从真实链路中抽取样本，再用 LLM 评估器 对这些样本做自动初筛，最后把命中的样本继续下钻分析或沉淀回数据集。对于很多以推荐、问答、对话、搜索理解为核心的业务来说，这类方案通常是最容易建立直觉、也最容易快速见效的一条路径。

### 2.2.4 在线链路的 HTTP 评估
在线链路中的 HTTP 评估 更适合另一类问题：业务规则已经比较明确，或者你们已经有现成 checker 服务，希望把已有的业务判断能力接入统一评估体系。在这种场景下，重点不是让大模型重新理解一遍上下文，而是复用你们已经沉淀好的判分逻辑，让平台负责统一调度、统一记录结果、统一沉淀样本。这样做的好处是逻辑更稳定、边界更可控，也更适合那些对规则一致性要求较高的业务。
点点agent在线checker_幻觉检测 就是这一路线下比较典型的代表。它体现的是“在线链路 + HTTP 评估器”的组合方式：样本来自真实线上请求，判分逻辑由业务侧服务承载，评估结果则统一沉淀回 X-RAY 的评估实验和实验报告体系里。对于已经在业务内部有较成熟 checker 能力的团队来说，这种方式往往比从头写一个新的 LLM 评估器更直接，也更容易与现有生产规则保持一致。

### 2.2.5 在线评估的推荐使用路径
在线评估的使用路径，通常可以理解成“先看得见，再筛得准，再沉得下”。第一步是确认 AI 链路已经稳定接入 X-RAY，并且你能够在链路里看到后续评估所需的输入输出、observation 或最终结果；第二步是创建评估器，先定义好系统到底准备从哪些维度判分；第三步是在 自动评估任务 中选择 在线链路数据 作为评估来源，并进一步配置评估方式、筛选条件和采样范围；第四步是让任务实际运行，生成 评估实验，然后在 实验报告 中查看整体结果和单条异常样本；最后一步，则是把真正有价值的 bad case 或重点样本回流到数据集，为下一轮离线评估或人工复盘做准备。

### 2.2.6 在线评估结果怎么看
在线评估的结果，不应该只被当成“跑完一次任务后的列表”。更重要的是把它当成一份持续观察线上效果的实验报告来读。首先要看的是整体趋势，例如最近一个周期内某个评估器的分数是否稳定，某次版本上线后是否出现整体下滑；其次要看的是异常样本，也就是哪些样本持续低分、哪些链路最近频繁命中某种问题；最后还要结合单条详情判断，这些低分到底是应用本身的真实问题，还是评估器配置、字段映射或规则边界导致的误判。只有把整体趋势、分布变化和单条诊断结合起来，在线评估才能真正帮助团队做日常监控，而不是停留在“任务跑过了”的层面。

## 3. 专题：如何做好 AI 应用质量闭环

在开始了解各个模块的具体配置之前，有必要先建立一个整体认知：数据集、评估器、自动评估任务、评估实验这几个模块不是孤立的工具，它们共同构成了一条完整的质量闭环。理解了这条闭环之后，再去看后面各章节的操作细节，会更容易明白每一步在做什么、为什么这样做。

### 3.1 如何识别 bad case
识别 bad case 最直接的方法，通常不是人工翻完整个实验列表，而是先在实验明细中按低分筛选，再结合得分理由和单条详情来确认这条样本是否真的存在问题。这里要特别注意的是，低分并不一定总是代表真实 bad case，它也可能来自评估器边界、字段映射错误或输入信息不足。因此，更稳妥的方式通常是“先筛分数，再看理由，最后结合链路或上下文确认”。

### 3.2 如何把问题样本回流到数据集
当前产品里，问题样本主要可以通过两种方式继续沉淀：一种是在 自动评估任务 中直接开启 数据回流，让符合条件的样本自动写入目标数据集；另一种是在 数据集 模块中通过 链路回流数据 从线上链路主动抽样沉淀。前者更适合在评估完成后自动沉淀命中条件的结果，后者更适合围绕某个时间范围或筛选条件做定向抽样。两者结合使用，才能把线上问题真正沉淀为后续可复测的数据资产。

### 3.3 如何持续优化评估器
评估器优化最常见的入口，不是重新从头设计，而是围绕真实结果持续校准。对 LLM 评估器 来说，通常要优先看 Prompt 是否足够清晰、字段映射是否正确、score 和 reason 是否能稳定解释结果；对 HTTP 评估器 来说，则要优先看服务返回是否稳定、输入输出字段定义是否一致、规则本身是否与业务理解保持同步。一个真正好用的评估器，往往不是第一次就写得完美，而是在持续实验和复盘中逐步校准出来的。

### 3.4 如何建立团队日常评估机制
如果你希望这套产品真正变成团队日常工作流，而不是阶段性试验，最稳妥的方式通常是把在线和离线两条路径结合起来使用。在线任务负责持续监控真实效果，并识别和回流低分样本；数据集负责把这些问题样本沉淀下来；离线数据集评估则负责在版本迭代时做回归验证。这样一来，团队既不会只盯线上临时问题，也不会只在离线环境里自我验证，而是逐步形成“线上发现问题、线下稳定复验、再回到线上继续观察”的完整闭环。

## 4. 模块一：数据集

> **开始前确认**
> 使用数据集模块前，建议先想清楚三件事：准备评估哪个 AI 应用、评估哪类场景、样本主要从哪里来。如果计划使用「链路回流数据」把线上链路沉淀进数据集，需要先确认 AI AI 应用已接入 X-RAY 并完成链路数据上报。→ 参考：[AI 应用链路埋点上报规范](https://docs.xiaohongshu.com/doc/ba8f8236ae557122bcc71af185d1ad82)

### 4.1 什么是数据集
在 AI 应用评估里，数据集 是整套评估体系中的样本层。它承接的并不只是人工整理的离线数据，也包括从线上链路中筛选出来的真实请求，以及从自动评估任务回流出来的 bad case。对产品使用者来说，可以把它理解成评估样本库，或者长期沉淀问题样本和高价值样本的资产池。
很多团队在刚开始搭建评估体系时，会把数据集看成一个普通的存储容器；但实际上，数据集真正重要的地方在于“可复用”。没有数据集时，问题样本大多是一次性被看到、一次性被讨论；有了数据集之后，样本才会变成下一轮离线回归、版本比对和上线前验证的基础。

### 4.2 数据集适合解决什么问题
数据集最适合解决“样本沉淀不下来”和“问题无法重复验证”这两类问题。线上 bad case 即使被发现了，如果没有继续收进数据集，下一次版本迭代时仍然很难稳定复现；同样，一批人工确认过的高质量样本，如果只是停留在聊天记录或表格里，也很难被持续复用。数据集的作用，就是把这些原本零散的样本逐步整理成可以长期复测、长期对比的评估资产。
当你希望做离线回归、积累固定 bad case、构建某个业务场景下的黄金样本集，或者想把线上发现的问题继续变成后续可复验的测试集时，都应该优先考虑先建设数据集。值得注意的是，数据集里的样本不只是人工编造的，也可以通过在线链路回流的方式把真实请求中的 good case / bad case 沉淀进来，这样的样本质量通常更接近真实业务场景。

### 4.3 如何创建数据集
进入 数据集 页面后，可以看到搜索、刷新、新建、编辑、复制和删除等基础操作。真正决定数据集是否好用的，不是这些按钮，而是创建时你是否把这个数据集的边界想清楚了。更稳妥的做法是先回答三个问题：它面向哪个业务场景、样本主要来自哪里、后续准备拿它做什么评估。

### 4.3.1 数据集名称和描述怎么写
数据集名称和描述最好能直接体现业务场景、样本来源和用途。例如，“某条线上链路的 bad case 数据集”“某业务模块的离线回归集”这种命名，会比“测试集 a”“样本集 1”更容易理解和维护。对于多人协作的项目来说，名称本身就是一层最基础的说明文档，它应该帮助后续使用者快速判断：这个数据集为什么存在，后续应该拿来做什么。

### 4.3.2 如何设计数据集字段
字段设计的原则，是既要支撑后续评估，又要保留足够多的问题定位信息。当前产品支持 string、integer、float、boolean、datetime、json 等字段类型，常见字段包括 input、output、messages、query、raw_input、response_content 等。对使用者来说，最重要的不是字段越多越好，而是字段是否和评估对象一致，后续评估器是否能从中拿到真正有用的信息。
如果这个数据集后面要配合离线回放使用，字段最好尽量贴近目标输入结构；如果主要用于 bad case 沉淀，则建议优先保留能解释问题的上下文，而不是只保留一个孤立的输出结果。字段名也应该尽量贴近业务语义，避免只使用团队内部才能理解的缩写。

### 4.3.3 创建时的注意事项
创建数据集时，最容易出现的问题是“先建起来再说”，结果后面发现字段无法承接评估、样本也不具备复用价值。更稳妥的方式，是在创建前先想清楚样本是否覆盖核心场景、异常场景和边界场景，字段是否能帮助后续定位问题，后续评估器是否能顺利拿到输入。数据集设计应该服务于后续评估，而不是只满足一次性的记录需求。

### 4.4 如何维护数据集内容
点击数据集名称进入详情页后，核心页面是 数据项 视图。这里提供了两条主要入口：手动添加 和 链路回流数据。前者适合录入已经整理好的离线样本，后者适合把线上真实链路中的样本持续沉淀进数据集。对大多数团队来说，这两条路径基本对应了数据集建设的两种主要来源：人工整理与线上沉淀。

### 4.4.1 如何手动添加数据
手动添加适合样本量不大、但质量要求较高的场景。你可以一次录入多条数据项，也可以在已有样本基础上继续编辑和补充。页面会根据字段类型展示不同的录入控件，因此录入方式本身并不复杂。真正值得花时间的是样本内容本身：它是否具有代表性，是否能覆盖关键问题，是否足以支撑后续评估。

### 4.4.2 如何通过链路回流数据
链路回流数据 是把真实线上样本沉淀为数据集的重要入口。它本质上是在创建一条“从 trace 中筛选并写入数据集”的配置流程。你需要定义任务名称、执行周期、时间范围，以及 trace 和 observation 的筛选方式，再通过字段映射决定哪些链路字段需要写入到数据集的哪些字段中。这一能力的价值在于，它能把“线上看到的问题”继续变成“后续可以反复验证的样本”。

### 4.4.3 如何查看数据项详情
单条数据项详情页不只是查看一行字段值，它也是理解样本来源的重要入口。你可以看到数据项 id、来源 trace id、是否来自评估任务回流、更新时间以及具体字段内容。如果这条样本本身就是从评估结果回流而来的，还可以继续反查到对应的实验报告。这让数据集不再只是一个孤立样本仓库，而是能与线上链路和评估结果形成可追踪的关系。

### 4.4.4 如何编辑和删除数据项
每条数据项当前都支持查看、编辑、删除和打开来源链路。编辑适合补充上下文、修正字段或完善标注，删除则更适合处理误入样本或已经失效的数据。长期来看，数据集质量比数量更重要，因此做适度清理和维护是非常正常的事情。

### 4.5 如何把数据集用于后续评估
数据集最常见的后续用途有两类。一类是直接作为 离线数据集评估 的输入，用于版本回归、效果对比和上线前验证；另一类是作为问题样本沉淀池，不断承接线上链路和实验回流下来的 bad case。随着评估体系成熟，数据集会逐渐从“一个表”演变成“团队共享的评估资产层”。

### 4.6 常见问题
如果后续准备做离线回放，字段最好尽量与目标输入结构保持一致；如果样本来自线上链路，建议优先保留那些真正能帮助定位问题的上下文信息。最重要的是，数据集设计一定要服务于后续评估和复盘，否则它很容易退化成一个难以复用的临时样本仓库。

## 5. 模块二：评估器

> **开始前确认**
> - 创建 **LLM 评估器** 前，需先在「LLM 配置」模块中完成至少一个模型配置并确认连通性正常，否则评估器将无法正常执行。→ 参考：6. 模块三：LLM 配置
> - 创建 **HTTP 评估器** 前，需已有可对外调用的业务 checker 服务，并明确其输入输出字段结构。

### 5.1 什么是评估器
评估器（evaluator） 可以理解成整套评估流程中的“裁判”。数据集负责存样本，自动评估任务负责组织执行，实验报告负责展示结果，而真正决定“这个结果应不应该得高分、为什么得这个分”的，是评估器本身。一个评估器通常至少会输出 score 和 reason 两部分：前者负责量化，后者负责解释。
因此，评估器并不是生成业务结果的模型，而是对结果进行判断和归因的规则承载体。你希望系统关注相关性、正确性、幻觉、工具调用质量，还是某条业务规则是否命中，最终都要通过评估器来表达。

### 5.2 LLM 评估器与 HTTP 评估器有什么区别
当前产品中的评估器主要分为 LLM 评估器 和 HTTP 评估器。LLM 评估器 通过 Prompt 和大模型来进行语义判断，更适合相关性、正确性、幻觉等需要理解上下文的问题；HTTP 评估器 则通过业务服务承载判分逻辑，更适合已经有现成 checker 服务、规则较稳定、希望逻辑更可控的场景。
如果用更直白的话来区分，LLM 评估器 更像“让模型做裁判”，HTTP 评估器 更像“让业务规则服务做裁判”。前者灵活、上手快，后者稳定、可控性强。两者不是互相替代，而是分别解决不同类型的评估问题。

### 5.3 如何选择合适的评估器类型
如果你的问题很难被明确规则完全覆盖，需要结合上下文和业务语义来判断输出是否合理，那么通常更适合优先选择 LLM 评估器。像 sug粗筛 这类场景，本质上就是典型的在线语义评估路线。相反，如果你的团队已经沉淀出较成熟的 checker 服务，或者某类判分逻辑已经被稳定写成服务，那么更适合选择 HTTP 评估器，例如 点点agent在线checker_幻觉检测 这类路径。
第一次搭建评估体系时，很多团队通常会先从一个目标明确的 LLM 评估器 起步，快速验证评估思路；当业务规则越来越稳定后，再逐步把部分能力沉淀成 HTTP 评估器。这样的演进路径通常更平滑，也更符合实际落地节奏。

### 5.4 如何创建 LLM 评估器
进入 评估器 页面后，点击 新建评估器 -> llm评估器 即可开始创建。页面中需要配置评估器名称、评估方式、描述、使用的 LLM、Prompt 和输出字段。这里最关键的并不是把每个框都填满，而是先想清楚这次到底准备评什么，以及模型需要看到哪些信息才能做出合理判断。
如果你第一次理解评估器模块，先看一眼真实页面会更直观。当前评估器页面本身就是一个“规则中心”，它既承接 LLM 评估器，也承接 HTTP 评估器。因此阅读这一章时，可以先用页面去建立“这里负责定义裁判”的直觉，再回来看不同类型评估器的差异。

### 5.4.1 LLM 评估器的 text 模式
text 模式适合那些规则相对直接、你希望用一段完整模板就表达清楚判断标准的场景。它的优点是结构简单、上手快，非常适合作为第一次搭建某类语义评估时的起点。如果你的评估逻辑本身就是“给模型一段上下文，再让它输出 score 和 reason”，那么 text 模式通常已经足够。

### 5.4.2 LLM 评估器的 chat 模式
chat 模式更适合那些你希望明确区分 system、user、assistant 等角色语境的场景。它不一定比 text 模式更复杂，而是在某些评估逻辑下更容易组织上下文和约束。当你希望让评估器更接近真实对话结构，或者想把规则拆成多条消息表达时，chat 模式通常会更清晰。

### 5.4.3 如何设计 Prompt 与变量映射
在 LLM 评估器里，Prompt 和 字段映射 是最容易混淆的两个概念。Prompt 决定的是“裁判标准”，也就是模型应该按什么规则来判断；字段映射决定的是“把哪份试卷交给裁判”，也就是系统最终把哪些真实字段送进 Prompt 中。当前产品里，映射的统一方式是先选择来源对象，再通过 jsonpath 提取字段。因此，即使 Prompt 写得很完整，如果字段映射错了，评估器看到的也可能不是你真正想评的内容。
更准确地说，字段映射不是一个附属配置，而是“评估输入定义”的一部分。它决定了评估器拿到的是完整问题、局部片段，还是被截断后的信息。很多看起来像“模型判断不准”的问题，最后真正的原因并不是 Prompt 写得不好，而是评估器根本没有看到足够完整的输入。

**变量语法**：在 Prompt 中用 `{{变量名}}` 引用字段映射传入的值，例如 `{{message}}`、`{{query}}`。

**字段映射格式**：使用 [JsonPath](https://github.com/json-path/JsonPath) 从 SPAN 的 input / output / metadata 中提取字段。以 sug 粗筛评估器为例：

| 变量名 | 来源 | JsonPath |
|--------|------|----------|
| `query` | output | `$.response_content` |
| `message` | input | `$.messages[?(@.role == 'user')]` |

其中 `response_content` 是 SPAN output 中 AI 生成的 query 结果（XML 格式），`messages` 是 SPAN input 中的对话消息列表。

### 5.4.4 如何设计输出字段
对绝大多数评估器来说，最关键的输出字段仍然是 score 和 reason。score 让实验结果可以被筛选、聚合和统计，reason 则让结果可以被理解和复盘。如果有更复杂的结构化分析需求，也可以继续补充其他字段，但建议优先保证这两个核心字段的稳定性。

### 5.5 如何创建 HTTP 评估器
进入 评估器 页面后，点击 新建评估器 -> http评估器 即可开始创建。与 LLM 评估器相比，HTTP 评估器更强调服务信息、输入字段和输出字段的定义，因为它的核心目的，是把业务已有的判分逻辑接入平台。你需要告诉系统服务在哪里、应该怎么调用、会收到哪些输入、返回什么结构。

### 5.5.1 HTTP 评估器的适用场景
HTTP 评估器适合幻觉 checker、工具调用规则校验、业务侧已有自定义评估服务，以及复杂业务逻辑判定等场景。它的价值不在于“也能打分”，而在于它能复用业务已经沉淀好的规则能力，让平台统一负责调度、实验记录和结果分析。

### 5.5.2 HTTP 评估器的输入设计
HTTP 评估器不会直接吃整条链路原文，而是先通过输入字段定义，明确告诉服务这次会传哪些数据，以及每个字段代表什么含义。尤其在 结果评估 模式下，输入通常会更聚焦在 input 和 output 这类最终结果字段上。输入设计是否清楚，直接决定了服务后续是否能稳定复用。

### 5.5.3 HTTP 评估器的输出设计
和 LLM 评估器一样，HTTP 评估器也建议至少稳定输出 score 和 reason。这样后续实验报告、单条详情、分数筛选和聚合分析都可以沿用同一套结果消费方式。如果返回结构不稳定，后续整条分析链路都会受到影响。

### 5.5.4 创建和调试时的注意事项
创建 HTTP 评估器时，需要重点关注服务返回是否稳定、输出字段定义是否与返回结构一致，以及所选评估方式是否与服务的实际处理能力匹配。很多“评估结果不对”的问题，最后并不是平台问题，而是服务输入输出约定不清或者服务返回发生了变化。

### 5.6 如何调试评估器
评估器最有效的调试入口，通常不是只盯着配置页反复改字段，而是去看 实验报告 -> 单条详情。在那里，你可以直接看到评估器入参、模型输入、模型输出、得分、得分理由和错误原因。对 LLM 评估器 来说，这能帮助你判断是 Prompt 本身有问题，还是字段映射有问题；对 HTTP 评估器 来说，这能帮助你判断是服务输入不对，还是服务返回本身不稳定。

### 5.7 常见问题
评估器只负责判分，不负责生成业务结果；因此它更像“裁判”，而不是“参赛者”。另外，无论使用哪种评估器，score 和 reason 都应该尽量成为最稳定的统一输出。对于第一次上手的团队，建议先从一个目标明确的评估器开始，把规则、映射和结果解释跑顺，再逐步增加更多评估维度。

## 6. 模块三：LLM 配置

> **开始前确认**
> 需要有可用的模型资源，包括模型名称、API Key 和 Endpoint。建议在正式创建前先确认该模型在当前环境中是否真实可用，避免因名称不一致或权限未开通导致连通失败。

### 6.1 什么是 LLM 配置
LLM 配置 是 LLM 评估器 的底层依赖，可以理解成“给裁判指定用哪一个模型、通过什么方式调用”。如果没有这层配置，LLM 评估器即使 Prompt 写好了，也无法真正运行。因此，这一模块虽然看起来只是模型接入配置，但实际上决定了后续语义评估是否能稳定执行。

### 6.2 什么时候需要新建 LLM 配置
当你准备创建新的 LLM 评估器、希望切换评估模型，或者想对比不同模型在评估场景下的稳定性和效果时，就需要新建或调整 LLM 配置。随着评估体系逐步成熟，模型选择通常不再是一次性动作，而会成为需要持续优化的一层基础能力。

### 6.3 如何配置模型信息
进入 LLM 配置 页面后，点击 添加LLM 配置 即可开始创建。页面中需要填写配置名称、模型提供商、模型名称、api key、api endpoint，以及温度、最大 token 数、top p 和每分钟请求限制等参数。这些配置共同决定了评估器后续调用模型时的稳定性、成本和行为边界。

### 6.3.1 模型提供商
模型提供商决定了配置连接的是哪类模型来源。对使用者来说，最重要的并不是提供商名字本身，而是确保当前环境中确实有可用的模型资源，并且与后续要运行的评估器相匹配。

### 6.3.2 模型名称
模型名称需要与实际申请或接入的模型保持一致。很多连通性或调用失败问题，最后都来自模型名称填写不一致，或者使用了当前环境中并未真正开放的模型标识。

### 6.3.3 api key 与 endpoint
api key 和 endpoint 直接决定模型是否能被正常调用。前者负责权限，后者负责请求入口。这两项一旦配置错误，后续所有依赖该模型的评估器都会受到影响，因此建议在正式挂载到评估器之前先做一次连通性确认。

### 6.4 温度、token、top p、qpm 怎么理解
这些参数控制的是模型调用时的行为边界。温度 主要影响随机性，数值越高，输出通常越发散；最大 token 数 决定生成长度上限；top p 用来控制候选词范围；qpm 则限制请求速率。虽然这些概念更偏模型层，但在评估场景里，它们同样会影响评估器输出是否稳定，因此不建议完全忽略。

### 6.5 如何测试模型连通性
列表页提供 测试链接 能力，它的价值不是判断模型“好不好”，而是确认这条配置链路“通不通”。在正式把某个模型挂到评估器之前，先做一次测试，通常能更快暴露权限、地址或模型标识方面的问题。

### 6.6 常见问题
如果 LLM 评估器 出现不可用、报错或结果异常，首先应该确认的通常不是 Prompt，而是这条 LLM 配置本身是否可连通、模型是否真实可用、温度和 token 设置是否过于激进。对语义评估来说，底层模型配置稳定，往往比 Prompt 写得花哨更重要。

## 7. 模块四：自动评估任务

> **开始前确认**
> 创建自动评估任务前，需要先完成以下准备：
> 1. **至少一个可用评估器**。如果使用 LLM 评估器，还需先确认对应的 LLM 配置可连通。→ 参考：5. 模块二：评估器 / 6. 模块三：LLM 配置
> 2. **明确数据来源**：
>    - 离线数据集路径：需提前准备好数据集，并确认字段结构可以和评估器变量对齐。→ 参考：4. 模块一：数据集
>    - 在线链路路径：需先确认 AI AI 应用已接入 X-RAY，Trace 数据在平台上可正常查看。→ 参考：[AI 应用链路埋点上报规范](https://docs.xiaohongshu.com/doc/ba8f8236ae557122bcc71af185d1ad82)

### 7.1 什么是自动评估任务
自动评估任务 是整套产品的执行中心。前面的数据集、评估器和 LLM 配置，更多是在准备样本和规则；到了自动评估任务这一步，系统才真正把“评什么、用什么评、什么时候跑、跑完之后怎么处理结果”组织成一条可以持续执行的流程。因此，自动评估任务不是单纯的调度器，而是把数据来源、评估方式、评估器、执行周期和回流策略串在一起的核心配置单元。

### 7.2 自动评估任务的整体结构
当前页面中的任务配置固定为 4 个步骤：基本配置、评估数据、评估器、数据回流（可选）。这 4 步不是为了把页面分块展示，而是分别对应任务最核心的四类问题：任务叫什么、准备评什么数据、用哪些评估器来判分、哪些结果应该继续回流沉淀。

### 7.3 如何创建自动评估任务
进入 自动评估 页面后，点击右上角 新建自动评估任务 即可开始创建。建议在正式创建之前，先确认自己已经准备好至少一个可用评估器，以及一份明确的数据来源。如果这次准备走 LLM 评估器 路线，还要先确认相关 LLM 配置 已经可用。这样做的原因很简单：自动评估任务本身不生成规则，也不生产样本，它负责把前面已经准备好的内容真正组织成一条可运行的任务。
自动评估页面是整套产品里最接近“实际开始工作”的地方。前面的数据集、评估器和 LLM 配置更多是在做准备，而到了这一步，系统才真正把数据来源、评估方式和评估器组织成一条任务。下面这张截图更适合配合本章一起看，你会更容易理解为什么这一页会被设计成 4 个连续步骤。

### 7.3.1 创建任务前先做哪些准备
开始建任务前，最少要想清楚三件事：这次是面向在线链路，还是面向离线数据集；这次准备挂哪些评估器；结果出来以后，是否需要继续回流沉淀。如果这些问题还没想清楚，直接进入创建流程通常会在第二步和第三步之间反复来回切换。

### 7.3.2 第一步：基本配置
基本配置 主要负责定义任务身份和调度方式。你需要填写任务名称，选择评估类型是 在线链路数据 还是 离线数据集，并决定执行周期是按周、按天还是按小时。虽然这一页看起来最简单，但它决定了后续整条任务的主路径，因此建议命名时直接体现业务场景和任务目标，而不是只写团队内部缩写。

### 7.3.3 第二步：评估数据
评估数据 是整条任务里最复杂、也最关键的一步，因为它决定了这次任务到底在评什么。如果选择 在线链路数据，你还需要进一步选择 单span、全链路评估 或 结果评估，并配置 trace 筛选、日志模式、observation 筛选以及采样率等信息。这里的关键不是把所有条件都填满，而是尽量让进入任务的样本“问题集中、边界清楚、规模可控”。
如果选择 离线数据集，重点则变成选定目标数据集，并视情况决定是否开启 回放配置。在一些离线场景里，数据集里保存的是原始输入，真正要评估的是回放后的结果，这时就需要配置回放对象、映射关系以及必要的请求信息。也就是说，这一步不仅是在选数据来源，也是在定义最终交给评估器的样本结构。

### 7.3.3.1 评估输入决定了你能评什么
理解 评估数据 这一步时，最重要的不是记住所有表单项，而是先理解一个核心原则：评估输入决定了你能评什么。在线任务之所以能做 单span、全链路评估 和 结果评估，依赖的是链路里已经存在不同粒度的数据结构。整条 trace 更适合回答“这次请求整体表现如何”，单个 SPAN / observation 更适合回答“某个中间环节是否有问题”，最终输入输出则更适合回答“最后给用户的结果是否合理”。如果输入里缺少关键上下文、缺少中间过程，或者字段没有被正确筛出来，那么后面的评估器即使规则正确，最终也只能对一个不完整的问题作判断。
这也是为什么在线评估和离线评估在配置体验上看起来不同，但本质上都在做同一件事：先把评估对象定义清楚，再决定让评估器看到哪些输入。对使用者来说，这一步最值得花时间的不是把所有配置项都填满，而是确保进入任务的样本范围、粒度和字段，真的能支撑你想要的判断。

### 7.3.4 第三步：评估器
在这一步里，你需要把一个或多个评估器挂到任务上，并完成字段映射。字段映射的作用，是告诉系统应该从当前数据来源中取哪些字段，交给评估器的哪些变量。对很多第一次使用的人来说，这一步最容易被低估，但它实际上直接决定评估器最终看到的内容是否正确。尤其在离线评估和单 SPAN 评估场景里，映射是否准确几乎决定了任务能不能正确跑起来。

### 7.3.5 第四步：数据回流（可选）
数据回流 不是每条任务都必须开启，但它是形成评估闭环的重要能力。当你希望把某些低分样本、高价值样本或满足特定条件的样本继续沉淀到数据集时，就应该在这一步配置回流目标、字段映射和回流条件。当前产品支持多个回流目标、多评估器组、多条件组合以及 and / or 逻辑，因此它并不是一个简单的“低于多少分就回流”的开关，而是一套可以承载业务筛选逻辑的沉淀机制。

### 7.4 如何运行和管理任务
任务创建完成后，后续日常使用的重点就从“怎么配”转到了“怎么跑、怎么管、怎么看结果”。列表页支持手动触发任务，用于立即执行一次评估；也支持进入任务详情页查看基础信息和历史实验列表。若任务配置需要调整，可以通过 编辑 重新打开 4 步配置页；若希望暂停或恢复周期执行，则可以直接在列表页控制任务开关。

### 7.4.1 如何手动触发任务
当你不想等定时调度，而是希望立即验证一次当前配置是否可用时，可以直接使用列表页中的 触发 操作。这个入口尤其适合刚创建完任务、刚调整过评估器、或者刚修改完回流条件之后做一次快速验证。

### 7.4.2 如何查看任务详情
任务详情页会展示两类信息：一类是任务本身的基础信息，例如名称、数据类型、执行周期、状态和更新时间；另一类则是该任务下产生的实验列表。前者帮助你确认这条任务到底是如何定义的，后者帮助你判断它最近是否真的稳定运行。

### 7.4.3 如何编辑任务
如果你需要更换评估器、调整筛选条件、修改采样率或补充数据回流规则，可以通过列表页中的 编辑 重新进入创建流程。由于自动评估任务本身就是长期配置，因此持续迭代它是非常正常的使用方式。

### 7.4.4 如何启停任务
列表页中的开关主要用于控制这条任务是否继续参与周期调度。对于还在试验中的任务、阶段性暂停的项目，或者暂时不再需要自动运行的场景，这个能力会非常实用。

### 7.5 常见问题
自动评估任务里最容易混淆的三个概念是：在线 / 离线 是数据来源，单span / 全链路 / 结果评估 是评估方式，评估器 才是具体的判分规则。另一个高频误区，是把字段映射当成“高级选项”；但实际上，字段映射决定了评估器真正拿到什么数据，是任务能否正确运行的关键步骤。

## 8. 模块五：评估实验

### 8.1 什么是评估实验
评估实验 表示自动评估任务一次实际运行后的结果集合。从系统角度看，它是一条实验运行记录；从用户角度看，它对应的其实是一份可以被阅读、分析和继续追问题的 实验报告。保留 评估实验 这个系统对象名，是因为页面中的所有结果都围绕它组织；而在阅读体验上，用户真正消费的是实验结果、实验明细和实验分析。

### 8.2 如何查看实验列表
进入 评估实验 页面后，你可以通过搜索和刷新快速定位某次实验。列表中会展示实验 id、实验名称、所属自动评估任务、状态和触发方式等信息。对日常使用者来说，这一页承担的是“实验索引页”的角色，帮助你先找到想看的那次运行，再进入详情查看真正有价值的结果内容。

### 8.3 如何理解实验状态
实验状态反映的是一次评估运行当前所处的阶段。页面中常见的状态包括 待执行、执行中、准备数据中、待评估、评估中、回放中、待回放、已完成、失败 和 已取消 等。阅读这些状态时，最重要的不是死记每个词，而是理解它们分别对应“数据是否准备好”“评估是否已经开始”“结果是否已经产生”这几个关键阶段。

### 8.4 如何查看实验详情
进入某一条实验后，当前页面会分成 实验信息、实验明细 和 实验分析 三部分。可以把这三层理解成三种不同视角：实验信息 告诉你这次实验是什么、怎么触发的、当前状态如何；实验明细 告诉你每一条样本到底发生了什么；实验分析 则帮助你从整体上判断这次实验的分布和聚合结果。

### 8.4.1 如何看实验概览
查看实验详情时，建议先从 实验信息 开始，先确认这是哪次实验、来自在线还是离线、是自动触发还是人工触发、当前状态是什么。很多排查问题其实在这一步就能缩小范围，因为你会先知道自己看的是否就是正确的任务、正确的时间范围和正确的运行结果。

### 8.4.2 如何使用评估分筛选
在 实验明细 中，评估分筛选是快速找到重点样本的核心入口。你可以按照不同评估器的分数、不同操作符和多条件组合来筛选结果，从而快速定位低分 bad case、高分优质样本，或者某个评估器下的边界样本。样本量一旦变大，这通常会比人工翻列表高效得多。

### 8.4.3 如何查看单条样本详情
单条样本详情是定位问题的核心位置。进入详情后，你通常可以看到评估状态、耗时、token 使用情况、得分、得分理由、评估器入参、模型输入、模型输出以及错误原因等信息。对使用者来说，这一页真正重要的价值在于：它能帮助你判断低分到底来自应用输出本身、评估器规则本身，还是字段映射和输入结构的问题。

### 8.4.4 如何做版本对比

版本对比是评估实验最常见的使用场景之一：在同一批样本上，用不同版本的 Prompt、策略或模型跑两次任务，对比两次实验的得分分布，判断新版本是否真的比旧版本更好。

**操作步骤：**

1. **准备两次实验**：确保两次自动评估任务使用的是同一个数据集（或同一批线上样本）、同一个评估器，只在被对比的变量（如 Prompt、模型、策略）上有差异
2. **进入评估实验列表**：找到这两条实验记录，分别点开查看实验分析中的得分分布和平均分
3. **对比维度**：
   - **整体得分**：对比两次实验的平均分和中位数，判断整体是否有提升
   - **分布变化**：查看低分段（如 0 分）样本数是否减少，高分段是否增加
   - **差异样本**：在实验明细中筛选出得分变化最大的样本，结合 reason 判断变化原因是否符合预期
4. **下结论**：如果新版本在整体分布上更好、低分样本明显减少，且差异样本的 reason 指向的是你真正想优化的问题，才能认为这次迭代是有效的

**注意**：如果两次实验的样本来源不同（一次是离线数据集，一次是在线链路抽样），对比结论通常不可靠，因为样本本身的差异可能掩盖真实效果差异。

### 8.5 如何根据实验结果优化应用
实验报告的价值，不在于跑完之后多了一条记录，而在于它能真正指导后续优化。最常见的动作是先从实验明细中筛出低分样本，再结合单条详情判断问题到底出在哪里；如果确认是应用本身的问题，就继续追业务链路和生成逻辑；如果确认是评估器边界或映射问题，就先修正评估方案本身。对于需要长期优化的团队来说，更进一步的动作通常是把确认过的问题样本继续回流到数据集，再用这些沉淀样本进行下一轮离线评估和回归验证。

### 8.6 常见问题
这里最容易混淆的是三个概念：自动评估任务 是长期配置，决定任务怎样持续运行；评估实验 是某次实际运行的结果对象；实验报告 则是用户真正阅读和消费的结果内容。把这三者区分清楚之后，页面里的结构会更容易理解。

## 9. faq

### 9.1 为什么分数看起来不准
分数看起来不准，最常见的原因并不是“模型太差”，而是评估规则和输入数据之间没有真正对齐。你需要先确认评估器规则是否足够清楚，字段映射是否把正确的数据送进了评估器，LLM 评估器 是否因为本身的概率性导致波动，或者 HTTP 评估器 是否返回了不稳定的结果。很多时候，真正的问题不在分数本身，而在于系统到底让谁、拿着什么信息、按什么标准去判了这个分。

### 9.2 为什么实验失败
实验失败通常可以按“数据 -> 评估 -> 模型/服务”这条顺序去排查。先确认数据来源是否正确，尤其是在线筛选条件、离线数据集和回放配置是否能正常产出样本；再确认评估器是否能被正常调用；如果使用的是 LLM 评估器，还要继续检查底层 LLM 配置 是否可连通。把排查顺序走清楚，通常能比直接盯着报错信息更快定位问题。

### 9.3 为什么结果和人工判断不一致
当结果和人工判断不一致时，不建议马上认为“评估器不靠谱”，而是先回到单条样本详情里确认评估器到底看到了什么。很多不一致，最后都来自 Prompt 或规则本身不够清晰、输入映射和实际业务语境不一致，或者输出字段设计没有真正承接判分逻辑。对评估体系来说，这种差异本身并不是坏事，它往往恰恰暴露了规则需要继续校准。

### 9.4 为什么建议先小规模试跑
因为评估器、字段映射、回放配置、回流条件任何一处出问题，都可能让大规模运行结果整体失真。先用小范围样本跑通一遍，能更快确认规则、输入和输出是否真正对齐，也能在成本和风险都更低的情况下把问题提前暴露出来。

### 9.5 如何判断是数据问题、映射问题还是评估器问题
一个比较实用的顺序是：先看单条样本详情，确认这条样本本身是否合理；再看评估器入参是否真的拿到了预期字段；接着看模型输入输出或 HTTP 返回是否符合预期；最后再判断问题到底来自应用输出本身，还是来自评估器和映射规则。这样排查的好处是能逐层剥离问题，而不是一开始就把所有责任都归到模型或任务上。

## 10. 附录

### 10.1 术语表
数据集 指的是用于评估 AI 应用的一组结构化样本；评估器 是对输出进行打分和归因的规则或程序；LLM 评估器 通过 Prompt 和大模型做语义判断，HTTP 评估器 通过业务服务承载自定义评估逻辑；自动评估任务 是长期运行的评估配置；评估实验 是任务一次实际运行后的结果对象；实验报告 是用户对这次运行结果的查看与分析视角；链路回流 / 数据回流 则表示将线上链路或评估结果继续沉淀到数据集的过程。

### 10.2 页面入口汇总
当前主入口下包含 数据集、自动评估、评估实验、评估器 和 LLM 配置 五个一级模块。它们看起来是并列入口，但在实际使用中分别对应样本准备、规则定义、任务执行、结果查看和模型依赖配置等不同环节。

### 10.3 推荐阅读顺序
如果你是第一次使用这套产品，建议按“先建立心智、再理解规则、再看执行和结果”的顺序来阅读。具体来说，可以先看 认识 AI 应用评估 和 典型使用场景，建立对产品定位和使用路径的整体理解；接着阅读 评估器 和 自动评估任务，理解如何把评估规则和执行流程真正配置起来；最后再去看 评估实验，理解结果应该如何被消费、分析和继续回流。这样会比直接从某个单独页面开始阅读更容易形成完整认知。
