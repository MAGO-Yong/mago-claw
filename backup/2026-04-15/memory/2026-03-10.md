```

- Prompt驱动

## 🔑 关键发现与设计洞察

### 1. 完整的数据评估生命周期
```
LLM配置 → 评估器 → 自动评估任务 → 定时执行 → 评估实验 
      ↓                                    ↓
    (参数) → (规则) → (数据源+方式+选择) → (结果生成)
                                         ↓
                            数据自动回流到数据集 → 新一轮评估
```

### 2. 灵活的评估方式
- **单SPAN评估**：评估单个环节
- **全链路评估**：评估完整调用链
- **结果评估**：仅评估最终结果

### 3. 数据流向的闭环设计
- 自动评估任务生成实验记录
- 满足条件的链路自动回流到数据集
- 该数据集可作为下一轮评估的数据源
- 完全自动化无需人工干预

### 4. 多维度的评估结果分析
- **实验明细**：链路粒度的详细评估数据
- **实验分析**：整体统计和分布分析
- 支持按评估器维度聚合
- 支持数据项分布展示

### 二、关键研究发现

**UModel 本质**：
- **Universal Observability Model** = 通用可观测模型
- 基于图模型（Node + Link）描述 IT 世界
- 阿里云云监控 2.0 核心建模框架，2025 年 9 月发布
- 核心价值：打破数据孤岛 → 构建可推理的运维世界模型

**UModel vs CMDB 关键差异**：
- CMDB：静态配置仓库，人工维护，配置视角
- UModel：动态可观测图谱，自动发现，运行时视角
- UModel 增强 CMDB 的拓扑关系，但不替代资产台账功能

**与 LLM AIOps 的关系**（最重要）：
- UModel 解决 LLM 在运维场景的4大障碍：
  1. 没有上下文 → UModel 提供实时图谱作为 Grounding
  2. 数据孤岛 → EntitySet + DataLink 统一多源数据
  3. 数值幻觉 → 传统算法处理数值，LLM 做高阶推理
  4. 无法行动 → USearch/Graph Query 作为工具调用接口
- 三层工具层：数据访问层 → 拓扑感知层 → 智能洞察层

**阿里云智能运维助手架构**：
- 采用"统一数据平台 + UModel + 传统算法 + 生成式AI"混合处理架构
- 智能会话 + 数字员工（自定义运维智能体）
- 典型场景：自然语言查询、全链路异常诊断、自动化根因分析
