## 触发时机

每次路径诊断完成、主报告已上传后，AI 紧接着生成复盘报告并上传。
不需要用户额外指令，属于路径 SOP 的最后一步。

## 7 章节规范

规范是「方向」，不是「脚手架」——AI 知道每章呈现什么，根据实际诊断内容自由组织，不按字段填表。

### §1 告警背景

呈现：集群名、节点 vm_name/IP/port、故障时间、告警类型、路径标签（如「CPU 高 · 路径 B」）。
一句话描述用户的原始输入。

### §2 诊断流程（Step by Step）

呈现：从 Phase 0 到最终结论的完整执行路线，重点记录：
- 每个阶段做了什么、产出了什么关键数据
- **弯路和异常**：哪些采集失败、哪些指标无数据、中途改变了判断方向
- 探测并发情况（Phase 0 三路）
- 推理矩阵命中过程

格式：时序步骤列表，每步说明「做了什么 → 得到什么 → 决策」。

### §3 工具调用清单

呈现：本次诊断调用的所有原子脚本，表格形式：

| 工具 | 状态 | 产出大小 | 关键发现 |
|------|------|---------|---------|

状态：✅ 成功 / ⚠️ 部分数据 / ❌ 失败（含失败原因）。
「关键发现」不能是空的——每个工具至少写一句它贡献了什么关键信息，或者为什么数据无效。

### §4 采集盲区 & 局限

呈现：本次未能采集到的数据、原因、以及对分析结论的影响程度。
常见盲区：myhub DTS slave 不暴露写入侧指标 / get_raw_slow_log HTTP 400 / Error Log 接口参数不兼容。
每条说明：① 缺什么 ② 为什么缺 ③ 是否影响根因结论（影响 / 不影响 / 待确认）。

### §5 根因结论

呈现：完整证据链，从原始数据到最终结论的逻辑推导过程。
格式：
```
[数据源] 关键数值
  ↓ （推导步骤）
[中间结论]
  ↓
[最终根因]
```
同时列出「已排除假说」及排除依据。

### §6 经验教训 & Skill 改进建议

呈现：本次诊断中发现的坑、改进点，面向「下一次同类告警」。
分两类：
- **坑（Pitfall）**：踩过的弯路，下次怎么避免
- **Skill 改进**：SOP / 脚本 / 文档哪里需要修改，具体到文件和章节

每条写清楚：问题描述 → 根本原因 → 改进建议。

### §7 关联报告

呈现：主报告链接（必须）+ 其他关联报告链接（若有）。

## 发布方式

AI 生成 HTML 后，调用：

```bash
python3 scripts/common/publish_report.py \
    --html_path <html_file> \
    --out_dir   <OUT_DIR> \
    --cluster   <cluster> \
    --time_range "<start> ~ <end>" \
    --path <路径字母> --path_label "<路径标签>" \
    --severity <P0|P1|P2> \
    --root_cause "<根因一句话>" \
    --p0_action "<立即行动项>"
```

publish_report.py 负责：upload + run_meta 写入 review_report_url + webhook 通知。

---

## 风格约定

- 深色科技风 HTML（与主报告视觉一致，参考 ads-ad-core 路径 B 报告样式）
- 复盘报告侧重「过程」，主报告侧重「结论」，两者互补不重复
- §6 经验教训必须有实质内容，不能写「本次无问题」
- 不允许出现 `path=?` / `time_range=N/A` 等空字段
