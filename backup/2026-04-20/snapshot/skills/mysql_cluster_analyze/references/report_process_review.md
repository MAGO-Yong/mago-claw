# 诊断过程复盘报告规范（report_process_review）

> 适用于所有 Layer 2 诊断路径（A/B/B2/C1/C2/D/E/F/G）。
> 在主报告（根因分析报告）完成并推送 Webhook 后，紧接着生成此复盘报告并上传 CDN。
> 复盘报告命名规则：`report_process_review_<RUN_ID>.html`

---

## 报告结构（7 章节，固定顺序）

### §1 · 用户提问（原始输入）

用对话气泡（bubble）形式还原：
- 用户提问气泡（右侧）：包含集群名、节点、时间范围、原始问题描述
- AI 初始响应气泡（左侧）：说明"读取 SKILL.md，构建 Phase 0 分诊计划"
- 用户确认气泡（右侧）："执行"

### §2 · 完整分析流程（Step by Step）

按实际执行顺序，逐 Phase 输出：

**Phase 0 · 分诊探测**
- 每个探测步骤单独列出：工具名称、调用参数（脱敏后）、实际输出结果
- 结果标注 ✅ 成功 / ⚠️ 部分成功 / ❌ 失败 + 失败原因

**Phase 1 · 分诊推理**
- 列出推理矩阵（各信号 → 路径推断）
- 标注最终命中路径及置信度

**Phase N · 数据采集（路径专项）**
- 每个 atomic 脚本的调用参数 + 实际输出摘要
- 关键数字高亮（慢查询总数、query_time 最大值、lock_time、线程数等）

**Phase 最终 · 根因确认**
- 根因类型 + 一句话描述
- 路径修正记录（如初始推理 C1 → 最终确认 G+A，要说明修正原因）

**Phase 输出 · 报告生成 & 推送**
- 主报告 CDN URL、Webhook 推送结果

### §3 · 数据采集流图

横向流程图（flex 布局），节点格式：
```
[用户提问] → [工具1 ✅/❌] → [工具2 ✅/❌] → ... → [根因分析] → [HTML 报告]
```
每个节点标注：工具名 + 状态 + 关键数据一行摘要（如"167,363 条 >60s"）。
失败节点用虚线边框 + 灰色标注，不影响整体流程图连续性。

### §4 · 工具链使用情况

以卡片网格展示（每行 3 个），每张卡片包含：
- 脚本名称（粗体）
- 功能描述（1-2 句）
- 状态标签：`✅ 成功` / `⚠️ 部分成功` / `❌ 失败（原因）`

必须包含本次诊断中所有**被调用过的**脚本（含失败的）。

### §5 · 关键发现总结

2×2 网格，每格一条发现：
- `✅ 确认：<发现标题>` — 正向证据，验证推断
- `⚠️ 局限：<发现标题>` — 本次诊断的数据盲区或工具局限

每条发现需包含具体数字或操作作为依据，不得只写结论。

### §6 · 诊断经验 & Skill 改进建议

2×2 网格，每格一条经验，格式：
- 编号（1-4）
- 标题（粗体，10字以内）
- 描述（3-5 句，说明场景、现象、建议改进方式）

重点关注：
- 接口异常的绕过方式（如 redhub 跳过 CK 路径）
- 路径推理修正的触发条件
- 历史证据 vs 实时采集的使用时机
- 可以沉淀进 SKILL.md 的改进点

### §7 · 关联报告

表格列出：
| 报告名称 | 类型（主报告/参考/复盘） | CDN 链接 |

至少包含：
1. 本次生成的主报告（根因分析报告）
2. 本复盘报告自身（标注"当前页"）
3. 如有历史参考报告，也列入

## 样式规范

- **Banner**：深蓝渐变（`#1a1a2e → #0f3460`），白色文字，KPI 格子（黄色数值）
- **对话气泡**：用户气泡右对齐深蓝底，AI 气泡左对齐白底
- **Phase 步骤卡**：每个 Phase 一张卡，卡头用不同渐变色区分（Phase 0 深蓝、Phase 1 青绿、数据采集暗红、确认橙黄、输出绿色）
- **流程图**：横向 flex，`flow-node` 样式区分 trigger/collect/analyze/output/blocked
- **工具链网格**：3列，卡片顶部 border-top 颜色统一为深蓝
- **整体背景**：`#f0f2f5`，卡片白底，圆角 10px，阴影 `0 2px 10px rgba(0,0,0,.07)`
- 文件名命名：`report_process_review_<RUN_ID>.html`

---

## 生成方式

AI 根据本次诊断实际执行记录，**动态填充**所有章节内容。不得使用占位符，所有数字、路径、工具名、结果均来自本次实际采集数据。

CDN 上传完成后，通过 inline curl 向群推送一条独立 Webhook 消息（消息头：「📋 MySQL 诊断复盘报告」），与主报告推送消息区分。同时向用户回复复盘报告 URL。
**⚠️ 容错**：curl 推送失败时向用户说明原因，不影响主报告推送状态。
