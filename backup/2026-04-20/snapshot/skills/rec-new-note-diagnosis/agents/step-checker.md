# Step-level Checker（步骤级自检）

> 用途：每次工具调用（指标查询、变更查询）产出中间结论后，立即执行的轻量自检。
> 不等到最终结论，**每一步推断都必须过这道关**。
> 设计原则：快（< 5秒心理耗时），硬（失败必须阻断，不能跳过）。

---

## 触发时机

以下任意一种情况，必须在输出结论前运行 Step-level Checker：

1. 执行指标查询后，得出"正常 / 异常 / 无数据"的判断
2. 执行变更查询后，得出"有变更 / 无变更 / 变更不相关"的判断
3. 对比两个时段数据后，得出"下跌 / 持平 / 上涨"的判断
4. 给出任何形式的因果推断（"因为 A 所以 B"）

---

## 三道检查门（顺序执行，任一 ❌ 必须修正后才能继续）

### 门1：查询覆盖完整性

**问**：这次查询的条件，是否覆盖了所有可能影响结论的范围？

| 检查项 | 通过条件 | 典型漏洞 |
|--------|---------|---------|
| 时间范围 | 覆盖目标窗口 ±5min buffer | 边界截断漏掉最后一条变更 |
| 系统过滤 | 若加了 `--system` 过滤，结论不能说"无变更" | 只查 apollo+racing，说"该窗口零变更" ← 今日真实 bug |
| 指标范围 | 关键 phase/channel 都有值 | cur=0 条数据就直接跳过 |
| 数据源 | 确认 datasource 正确 | 用错 vms 导致全部返回空 |

**输出**：
```
门1 查询覆盖：✅ / ❌
  [具体说明覆盖范围或漏洞]
```

**❌ 时的强制动作**：
- 若 `--system` 过滤 + 结论含"无变更" → **必须补查完整系统（去掉过滤）再下结论**
- 若数据为空（cur=0）→ **必须说明"数据不可用，跳过本步"，不能用空数据得出结论**

---

### 门2：结论逻辑完整性

**问**：这个结论，是否存在以下逻辑缺陷？

| 缺陷类型 | 判别方法 | 示例 |
|---------|---------|------|
| **以无证据证明不存在** | 结论含"无 X"，但没有查过 X | "该窗口无变更"但只查了 apollo |
| **以相关性代替因果** | 变更时间接近 T0，就断定是根因 | 没有验证回滚后是否恢复 |
| **以局部代替整体** | 用单渠道正常推断全链路正常 | recall 正常 → 整体正常（忽略 mixRank）|
| **以"可能"掩盖不确定** | 结论用"可能/推测/大概率" | 无数据支撑时不能给因果结论 |

**输出**：
```
门2 结论逻辑：✅ / ❌
  [发现的逻辑缺陷，或"无缺陷"]
```

**❌ 时的强制动作**：
- 将结论降级为"假设"：把"原因是 X"改为"🔍 疑似 X，待验证：[具体验证手段]"
- 不得向用户输出带因果性的结论，只能输出"现象 + 待验证方向"

---

### 门3：数据异常信号检查

**问**：数据本身是否有异常，导致结论不可信？

| 信号 | 含义 | 处理 |
|------|------|------|
| 当期数据点数 = 0 | 查询失败或数据过期 | 标注"数据不可用"，跳过结论 |
| 基准数据点数 = 0 | 无法对比 | 标注"无基准，无法判断趋势" |
| 单点跌幅 > 30%，持续 < 3min，完全自愈 | 疑似采集抖动 | 优先查 baichuan Flink 重启，再考虑采集抖动，**不能直接定性** |
| 所有渠道跌幅完全一致（如都是 -10%）| 可能是分母变化而非分子 | 需同时检查分母（总量）趋势 |
| 1H 和 24H 背离（一跌一涨 > 5%）| 历史池与实时池解耦 | 必须分别说明，不能合并结论 |

**输出**：
```
门3 数据信号：✅ / ⚠️ 有异常信号 / ❌ 数据不可用
  [信号描述，或"无异常"]
```

**❌/⚠️ 时的强制动作**：
- ❌ 数据不可用：跳过本步，在报告中标注"Step X 数据不可用，跳过"
- ⚠️ 抖动信号：先补查同窗口 baichuan 变更，再出结论
- ⚠️ 分母变化信号：补查总曝光量趋势后再出结论

---

## 输出格式（每步结论前必须附上）

```
┌─────────────────────────────────────────┐
│ 🔍 Step-level Check @ [Step X / 查询名] │
├─────────────────────────────────────────┤
│ 门1 查询覆盖：✅ 已覆盖所有系统，时间含buffer
│ 门2 结论逻辑：✅ 无缺陷
│ 门3 数据信号：⚠️ 单点跌 -16.4%，持续2min，已补查baichuan
├─────────────────────────────────────────┤
│ 自检结论：✅ 可输出 / ⚠️ 降级输出 / ❌ 阻断，需补查
└─────────────────────────────────────────┘
```

**自检结论说明**：
- `✅ 可输出`：三门全过，直接输出结论
- `⚠️ 降级输出`：结论改为"疑似 + 待验证"格式，不输出确定性因果
- `❌ 阻断，需补查`：明确告知用户缺少什么数据，补查后再给结论

## 快速记忆卡（高频漏洞速查）

```
🚫 "该窗口无变更"     → 先确认有没有加 --system 过滤
🚫 "数据正常，跳过"   → cur=0 时不能说正常，只能说数据不可用
🚫 "因为变更 A 导致"  → 时序吻合 ≠ 因果，需回滚验证
🚫 "可能是采集抖动"   → 先补查 baichuan 再说
🚫 "召回正常因此无问题" → recall 只是漏斗第一层，还有 rank/mix
```
