# 数据流知识（Step 6 按需加载）

## 一、数据流链路

```
笔记发布
  → mq（消息队列）
  → backbone（跨机房数据同步）
  → kafka（实时消费层）
  → Flink（流式处理）→ 产出特征/正排/倒排数据
  → Lindorm/HIVE（持久化）
  → 索引构建
  → lds-* searcher / omega-* searcher
```

**最后一公里**（最容易出问题）：
- Flink → Kafka → 正排/倒排索引

## 二、关键检查点

| 检查点 | 正常信号 | 异常信号 | 排查思路 |
|--------|---------|---------|---------|
| Flink 延迟 | < 5min lag | > 20min / checkpoint 超时 | 查 Flink UI，检查资源使用率 |
| Kafka lag | < 5min | > 30min | 查对应 topic 消费 group |
| Lindorm P99 | < 100ms | > 500ms，P99 突增 | 联系存储 oncall |
| META CPU | < 70% | > 90%，持续升高 | 扩容 / 降级写入 |

## 三、高频故障模式

### 故障一：zk zxid 耗尽（高危！）
- **现象**：kafka mirror 断流，多集群同时延迟
- **特征**：所有依赖 kafka 的渠道同步下跌，且 Flink 无法消费
- **止损**：
  1. 降级对象存储（减少写入压力）
  2. 重启 zk（需 infra oncall）
  3. 新笔记兜底降级（启用 backup 服务）
- **告警**：需监控 zk zxid 用量

### 故障二：Flink checkpoint 超时
- **现象**：倒排任务 checkpoint 超时 > 20min，实时数据积压
- **特征**：单条链路延迟，其他链路正常
- **止损**：通常可自动追数；若持续，重启 Flink job
- **参考**：2026-01-06 故障案例

### 故障三：Lindorm 版本升级/节点故障
- **现象**：同步延迟，P99 突增
- **特征**：通常发生在 Lindorm 维护窗口后
- **止损**：联系阿里云 oncall，通常自动恢复

## 四、特征数据问题

| 场景 | 特征字段 | 排查方式 |
|------|---------|---------|
| 话题特征缺失 | `topic_v4_id_multi` | 检查上游任务产出，字段填充率 |
| 视频特征缺失 | video 相关字段 | 检查转码任务状态 |

**参考**：2025-12-20 故障：topic_v4_id_multi 字段缺失导致后排打散异常，止损方式：backup 服务产出实时数据
