# 渐进式故障案例（watchlist 变更命中时加载）

> 当变更命中 watchlist 中的视频权重/混排权重类参数时，必须读本文件。
> 渐进式故障的特点是：**发布时正常甚至上涨，数小时后才暴跌**。

## 一、识别特征

```
正常故障时序：    发布 → 下跌 → 发现 → 止损
渐进式故障时序：  发布 → 正常/上涨 → 峰值 → 阴跌 → 加速 → 暴跌（12-17h后）
```

**关键识别信号**：
- 发布后 2-4h 指标正常甚至小幅上涨
- 晚高峰（20:00-22:00）达到峰值后开始缓慢下行
- 凌晨低峰期（00:00-06:00）流量低，指标被放大，出现急跌
- 与 SOP 分类不同：**没有明确的 T0**，而是趋势转折点

## 二、已知案例

### 案例1：2026-03-25 视频粗排权重翻3倍（天城/罗思聪）

**变更**：
- 时间：18:02:30
- 参数：`fp_all_video_fusion_weights_tc`: `1.0,1.0,1.0,1.0 → 3.0,1.5,1.0,1.0`
- 系统：arkfeedx（FirstRank 粗排层）

**时序**：
- 18:02 发布 → 18:00-19:00 指标持平甚至微涨
- 19:00 起开始阴跌（-12%，接近-17%）
- 19:14 叠加视频体裁偏好变更（all_scene_pref_params +0.15）
- 19:23 叠加蛇形合并实验全量
- 19:53 叠加混排广告权重 1→4（`hf_new_mix_ad_power`）
- 多变更叠加导致 20:00 后跌幅扩大至 -18%~-19%

**根因机制**：
1. 粗排（firstRank）视频分提升 → 大量图文/新笔记在粗排即被淘汰
2. 精排（finalRank）阶段损耗放大（被粗排淘汰的不会进入精排）
3. 混排（mixRank）广告权重翻4倍进一步挤压推荐内容空间
4. 效果"叠加"：三个独立变更各自影响不大，叠加后超出临界点

**止损**：回滚 `fp_all_video_fusion_weights_tc` + `hf_new_mix_ad_power`

### 案例2：2026-03-20 arkfeedx 发版（渐进式，17小时后爆发）

**变更**：arkfeedx-1-default 发版，14:45 上线

**时序**：
- 14:45 发版 → 17:00 达到峰值（正常）
- 18:00 初现下降迹象（不显著）
- 22:00 加速下跌，跌破 0.065 警戒线
- 03:00 凌晨暴跌

**关键教训**：

- 凌晨低峰期是问题暴露高危时段（流量低 → 绝对值敏感）
- **监控时间线**：发布后必须持续跟踪 18-24h

## 三、渐进式故障排查要点

### 1. T0 识别困难
渐进式故障没有明显的"急跌点"，Step 2 可能找不到清晰 T0。
- 用"趋势转折点"代替"急跌时刻"
- 在逐分钟数据中找到"从正 vs 基准变为负 vs 基准"的时刻作为 T0

### 2. 变更时间和指标异常时间有时差
- 变更在 18:02，T0 在 19:00——时差 58 分钟
- 这在渐进式故障中正常：需要一定流量才能显现影响
- 变更排查窗口应扩大到 **[T0 - 3h, T0 + 30min]**（比标准2h更宽）

### 3. 多变更叠加场景
- 单个变更可能无法解释全部跌幅
- 需要逐一量化每个变更的影响（通过实验分桶对比）
- 止损优先级：先回滚影响最大的单个变更，再评估是否继续

### 4. 凌晨值班响应
- 凌晨低峰期的绝对值可能比白天更低（流量结构不同）
- 对比基准要用**同时段昨日凌晨数据**，不能用白天均值
- 凌晨发现跌破 0.060 → **不能等天亮**，立即联系 oncall 回滚

## 四、监控 Checklist（渐进式故障专用）

```
发布后 +2h：  指标 vs 峰值是否下跌 >5%？（阴跌信号）
发布后 +4h：  指标是否持续低于昨日同期 >8%？
发布后 +8h：  22:00 指标绝对值是否 <0.065？（止损窗口！）
发布后 +12h：  23:00 是否 <0.060？（立即回滚！）
凌晨：        绝对值是否继续下行？（凌晨低流量会放大跌幅）
```
