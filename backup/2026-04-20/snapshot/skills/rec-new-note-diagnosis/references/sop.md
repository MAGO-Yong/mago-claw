## 零、诊断总控流程（强制，每次诊断必读）

### 0.0 诊断模式选择（强制，每次诊断开始时第一步）

收到诊断请求后，**在做任何查询之前**，必须先问用户选择诊断模式：

```
收到诊断请求时，固定输出以下问题：

「请问需要哪种诊断模式？

① 渐进式（推荐）
   每个 Step 完成后暂停，展示当前结论，等你确认后再继续。
   适合：想逐步了解排查过程、有补充信息要提供、需要中途决策止损方向。

② 一查到底
   全程自动执行 Step 0-6，不中途打断，最后输出完整报告。
   适合：复盘历史故障、批量分析、不需要介入过程。」
```

**两种模式的行为差异**：

| 行为 | 渐进式 | 一查到底 |
|------|--------|---------|
| 每个 Step 后暂停 | ✅ 是，等用户「继续」 | ❌ 不暂停，自动执行下一步 |
| 中途向用户提问 | ✅ 允许（如需澄清） | ❌ 禁止，自主判断 |
| T0 范围扩展确认 | ✅ 询问用户是否往前扩 | ❌ 自动扩展，说明原因 |
| 矛盾判断时停顿 | ✅ 展示矛盾，问用户下一步 | ❌ 按默认路径继续 |
| 最终报告 | 分 Step 逐步呈现 | 汇总为一份完整报告 |

**模式绑定**：用户选择后，当前诊断会话全程遵循该模式，不再重复询问。

## 一、故障快速分类

根据RCA统计，新笔记故障主要分为四类：

| 类型 | 占比 | 典型特征 | 快速判断 |
|------|------|----------|----------|
| **发布/变更类** | 20% | 发布时间点后指标异常 | 检查发布时间和指标趋势 |
| **索引问题** | 35% | 单渠道召回量下降 | 检查各召回渠道quota |
| **数据流问题** | 30% | 多集群同时延迟 | 检查kafka lag、Flink延迟 |
| **特征数据问题** | 15% | 特定字段填充率下降 | 检查特征依赖链路 |

## 三、关键排查细节

### Step 0: 指标采集与异常判定

#### 0.1 必须同时查询的两个指标（强制）

**不需要问用户"查1H还是24H"，两个都查。**

**① 诊断指标类型（强制：两个都要查）**

| 指标 | PromQL 关键词 | 含义 | 异常信号解读 |
|------|-------------|------|------------|
| **1H新笔记曝光占比** | `rt_new_one_hour_new_note_imp_ratio` | 过去1小时内发布的新笔记在曝光中的占比 | 对发布侧、实时索引类异常敏感，反映最新供给 |
| **24H新笔记曝光占比** | `rt_new_note_imp_ratio` | 过去24小时内发布的新笔记在曝光中的占比 | 对召回/排序/数据流类异常更稳定，反映整体健康 |

**两个指标必须同时查询，互相印证**：
- 1H↓ 24H 正常 → 近期发布侧或实时索引问题
- 1H 正常 24H↓ → 历史新笔记消耗慢，召回/排序链路问题
- 1H↓ 24H↓ → 系统性问题，优先级最高
- 1H 正常 24H 正常 → 无异常

**诊断报告开头必须同时写出两个指标的判断结果**，不能只报告其中一个。

> ⚠️ **判定为"无异常"后直接结束，不查变更**。  
> 只有至少一个指标判定为"异常"或"可疑"，才继续进入 Step 1 定位 T0。

**② 时间窗口划分规则（强制）**

用户给定时间范围后，必须按以下规则强制划分分析窗口：

| 时间范围 | 窗口划分 | 每窗口时长 |
|---------|---------|---------|
| ≤ 1小时 | 2个窗口 | 10分钟/窗口 |
| 1-2小时 | **6个窗口** | 10分钟/窗口 |
| 2-4小时 | 6个窗口 | 20分钟/窗口 |
| > 4小时 | 至少12个窗口，可多达24个 | 按实际等分 |

> **为什么强制划窗口**：等间距窗口可以捕捉到"骤跌"型突变（如 20:14-20:17 的5分钟内 -6% 骤跌），如果只看整体均值会完全漏掉。

#### 0.2 对比基准置信度校验（强制，在判断异常前执行）

**必须同时查询多条基准线**：`-1d`、`-7d`、`-14d`、`-21d`（至少前三条，条件允许时查四条）。

**核心原则（来自真实经验）**：
> 与 -7d 比当天是下降的，但与 -1d、-14d、-21d 比是平稳的，就**不一定是异常**。
> 判异常不能只和一条基准比，要看多条基准的一致性。某一天的基准本身就不正常（活动、发版、节假日等），就不应该用它来做对比。

**置信度校验规则（三步法）**：

```
step1：计算每条基准线在同一时间段的均值
    baseline_1d, baseline_7d, baseline_14d, baseline_21d

step2：找出"异常基准"
    将所有基准线排序，若某条基准线与其他基准线的均值偏差 > 15%，
    则该条基准线标记为"不置信"（该天本身异常）
    
    例：baseline_7d = 0.09，其余均在 0.065~0.075
    → -7d 偏高 > 20%，标记为不置信（上周同期可能有大活动/发版拉高指标）

step3：用"置信"基准线的均值与今天对比
    - 若多条置信基准都认为今天正常 → 不是异常
    - 若多条置信基准都认为今天异常 → 才是异常
    - 若置信基准之间也有分歧 → 降级为"可疑"，结合变更和绝对值综合判断
    - 若三条及以上全不置信（极端情况）→ 仅看绝对值是否跌破警戒线 0.06
```

**剔除不置信基准的常见原因**（在报告中注明）：
- 该天有大型运营活动（拉高指标）
- 该天有严重故障（拉低指标）
- 节假日（流量结构与工作日不同）
- 大规模发版回滚（指标波动剧烈）
- 周几差异（周末 vs 工作日的自然波动）

**典型场景**：
- -7d 有节日活动 → baseline_7d 偏高 → 今天与 -7d 比必然大跌，是假异常 → 剔除 -7d，用 -1d/-14d/-21d 均值
- -1d 有大规模发布回滚 → baseline_1d 偏低 → 剔除 -1d，用 -7d/-14d/-21d 均值
- 节假日周期规律 → 检查 -7d 是否也是节假日，若是则 -7d 与 -14d 更有可比性
- **今天与 -7d 比跌幅大，但与 -1d/-14d/-21d 比平稳 → -7d 本身偏高，不判为异常**

**在诊断报告开头必须输出**：
```
基准置信度校验：
  -1d  均值: 0.XXXX  [置信 / 不置信，原因：____]
  -7d  均值: 0.XXXX  [置信 / 不置信，原因：____]
 -14d  均值: 0.XXXX  [置信 / 不置信，原因：____]
 -21d  均值: 0.XXXX  [置信 / 不置信，原因：____]
→ 置信基准均值: 0.XXXX（基于 __d、__d、__d 均值）
→ 今日均值: 0.XXXX，vs 置信基准：____%
→ 判定：[异常 / 可疑 / 正常]
```

**判断异常的阈值**（基于置信基准，三条同时满足才判异常）：
- 置信基准中**多数**（≥2条）认为今天下跌 > 10%
- 绝对值跌破 0.06（此条单独成立，与基准无关）
- 对比置信基准趋势相背（今日持续下行而基准平稳）

**只有一条基准线显示异常时**：降级为"可疑"，继续排查但不轻易下"确认异常"结论。

#### 0.3 置信基准传递（强制）

Step 0.2 校验完成后，得出的**置信基准集合**（如"使用 -7d + -14d 均值"）必须传递给后续所有 Step，贯穿 Step 2 到 Step 6：

```
置信基准确定后：
  - Step 3 阶段定位    → 今日 vs 置信基准（非固定 -1d）
  - Step 3 召回渠道    → 今日 vs 置信基准
  - Step 4 召回根因    → 今日 vs 置信基准
  - Step 5 索引排查    → 今日 vs 置信基准
  - Step 6 内容供给    → 今日 vs 置信基准
```

**操作规则**：
- 若 -1d 置信 → 后续步骤优先用 -1d（实时性最强）
- 若 -1d 不置信、-7d/-14d 置信 → 所有后续步骤改用 -7d/-14d 的均值作为 offset 对比
- 若只剩一条基准置信 → 使用该条，在每个步骤的报告中注明"基准可信度低"
- 若三条全不置信 → 后续步骤只看绝对值，放弃同比判断

> **典型场景**：-1d 不置信（如昨天有活动），Step 2 的阶段占比就必须拉 `-7d` 和 `-14d` 的阶段数据做对比，而不是默认拉 -1d 的阶段数据。

#### 0.4 窗口内急跌扫描（强制，与 Step 0.3 同步执行）

> **背景**：窗口均值会稀释短暂急跌信号。例如 20:00-20:20 窗口内前 14 分钟 +5%、后 4 分钟急跌 -8%，窗口均值变成 +2.3%，在 Step 1.2 中完全看不出异常。因此必须同时对每个窗口做内部逐分钟扫描。

> ⚠️ **查询窗口必须用 `[1m]`，不能用 Step 0 标准的 `[5m]`**。
> - Step 0 总体判断用 `[5m]` 是为了平滑噪声，但会把 2-4 分钟的急跌完全掩盖。
> - 急跌扫描必须用 `[1m]` 精细窗口才能捕获真实的瞬间最低值。
> - PromQL 示例：`avg(avg_over_time(rt_new_one_hour_new_note_imp_ratio{...}[1m]))`
> - 实测案例：2026-03-25 20:15 急跌用 `[5m]` 显示 -7.5%，用 `[1m]` 显示 -17.1%，差距 2 倍。

**执行方法**：对每个时间窗口，找出窗口内所有分钟的 `(今日值 vs 置信基准)` 跌幅，并记录：
- **窗口内最深跌幅时刻**（即使窗口均值正常）
- **是否存在连续 ≥ 2 个分钟跌幅 < -5%**

**触发条件**（满足任一即标记为需关注）：
- 窗口内任意单分钟 vs 置信基准跌幅 < **-5%**（注意：不是 -10%，急跌往往只持续几分钟）
- 窗口内连续 ≥ 2 分钟跌幅均 < -3%

**输出格式**（在 Step 1.2 表格后紧接）：
```
Step 1.3 窗口内急跌扫描（单分钟 vs 置信基准 < -5%）：
  时刻    今日     基准     vs基准     持续分钟数
  20:16   0.3540   0.3790   -6.6%  ⚠️   2分钟（20:16~20:18）
```
若无任何时刻触发 → 输出"（无急跌信号）"

**说明**：Step 1.3 触发后，**无论持续时长长短，一律进入 Step 2 定位 T0 和变更排查**。

理由：
- 真实的流量切换、配置下发、服务重启等事件，完全可以在 2-4 分钟内产生真实影响后自愈
- 短暂但两个指标同步跌（1H + 24H 同时触发），几乎可以排除随机噪声
- 能自愈≠无影响，仍需知道是什么触发了它，为后续类似事件积累经验

**唯一的例外**：若该时刻的原始数据点数极少（< 3 个样本），或监控系统已知有上报延迟，可标记为"数据质量问题"并跳过，但必须在报告中说明原因。

**多指标联合判断**（强制）：Step 1.3 触发后，立即交叉检查另一个指标（1H 触发时查 24H，反之亦然）：
- **两者同步跌** → 真实系统事件，直接进 Step 2
- **仅单指标跌** → 可能是指标计算口径问题，标注后进 Step 2（降低优先级）

### Step 1: 异常起始点定位（T0）

> 前置条件：Step 0 判定为异常或可疑，才执行本步骤。

**关键技巧**：
1. 使用 Step 0.2 确定的置信基准做对比（非固定 -1d）
2. **区分异常类型**，并精确定位起始时间点 T0
3. T0 是后续变更查询的锚点

#### 1.1 异常类型识别（强制）

在确认有异常后，必须先判断异常属于哪种类型：

| 类型 | 特征 | 定义 |
|------|------|------|
| **急跌** | 指标在 ≤ 10 分钟内跌幅 > 5%（单分钟 vs 置信基准） | 突发性单点事件，通常有明确触发时刻 |
| **阴跌** | 指标持续 > 30 分钟缓慢下行，每窗口跌幅 < 5% | 渐进式劣化，通常是配置生效滞后或容量问题 |
| **混合** | 先阴跌后急跌，或多段异常 | 多个原因叠加 |

#### 1.2 精确定位异常起始时间点（强制）

**急跌定位**：在逐分钟数据中找到第一个 vs 置信基准跌幅 > 5% 的时刻，该时刻即为**起始时间点 T0**。

**阴跌定位**：在逐分钟数据中，从查询时间范围左侧往右扫描，找到指标开始**连续3个点单调下行**且对比基准由正转负的时刻，即为 **T0**。

```
急跌示例：
  20:10 +3.2%  →  20:11 +2.8%  →  20:12 -0.5%  →  20:13 -8.4%  →  20:14 -15.1%
  T0 = 20:12（首次跌破 0%，后续加速）

阴跌示例：
  19:00 +5.1%  →  19:20 +3.2%  →  19:40 +1.1%  →  20:00 -1.3%  →  20:20 -3.8%  →  20:40 -7.2%
  T0 = 19:40（连续下行，趋势由正转负临界点）
```

#### 1.3 时间范围对齐提示（强制）

定位到 T0 后，必须判断 **T0 是否接近用户给定时间范围的左边界（前 20 分钟以内）**：

```
若 T0 距左边界 ≤ 20 分钟：
  → 异常可能在查询范围外已经开始

  【渐进式模式】：

    自动往前扩展 2 小时，在报告中说明扩展原因，不询问用户。

若 T0 距左边界 > 20 分钟：
  → 时间范围覆盖完整，继续下一步
```

### Step 2: 变更查询（基于 T0，不是基于用户给的时间范围）

> 前置条件：Step 1 已定位 T0。

**变更查询范围固定为：[T0 - 2小时, T0]**

> ⚠️ **窗口右端是 T0 本身，不延伸到 T0 之后**。变更必须在 T0 前发生才能是根因。

```
示例：T0 = 20:12
→ 变更查询范围：18:12 ~ 20:12
→ 查询 #1：tag=外流推荐，系统=apollo+racingweb（排除内流实验）
→ 查询 #2：系统=autobots，无 tag 过滤，人工筛选外流推荐相关索引变更
→ 查询 #3：redbutton 预案平台，查询 [T0-30min, T0+60min] 内的预案执行记录
           ⚠️ 右端必须延伸到 T0 之后 60 分钟，止损操作通常在故障后几分钟到十几分钟内执行
           （不能和根因查询一样截止于 T0，否则会漏掉止损记录）
→ 内流相关实验（tag 含"内流"/"内推"）一律排除，不纳入根因
→ 查询结果必须与 references/config-watchlist.json 逐一对比（见第五节）
```

**redbutton 预案平台查询说明**：
- redbutton 记录预案执行动作（流量切换、降级开关、限流配置等）
- **查询窗口：[T0-30min, T0+60min]**，注意右端必须超过 T0
- 两类记录含义不同，必须分开处理：

| 时序 | 含义 | 处理方式 |
|------|------|---------|
| **T0 之前**的预案执行 | 可能是根因（预案误触发了流量变更导致故障） | 纳入根因分析，标为🔴 |
| **T0 之后**的预案执行 | 止损操作，故障已发生后的处置 | 记录为「处置记录」，不纳入根因，但说明当时判断的止损方向 |

- redbutton 变更通过 xray_changevent_query 的 `--system redbutton` 参数查询
- ⚠️ **不能把 redbutton 查询和根因查询用同一个右端截止时间（T0）**，这会漏掉所有止损记录

**如何判断是外流还是内流变更**：
- **外流**：涉及 hf_/vf_hf_ 系列参数、外流 arkfeedx、omega-hf-*、外流混排（arkmixrank）、外流召回（Lambdahomefeed-recall）
- **内流**：实验名含"内流"/"内推"/"discovery"，或 application=arkfeedx-inner、arkfeedx-rec 等内流专属服务
- **通用参数**：若参数同时影响内外流，需确认实验的 scene/application 范围

**示例（完整）**：
```
Step 0 发现：20:10-20:20 窗口 -18.3% 🔴，Step 1 定位 T0 = 20:12（急跌）
T0 距查询起点 19:00 → 72分钟，范围覆盖充分
→ 查变更范围：18:12 ~ 20:12（不含 20:12 之后）
→ 过滤：tag=外流推荐，排除 内流新笔记流量腾挪c端 等内流实验
→ 发现 19:53 外流大混排框架重构 hf_new_mix_ad_power: 1→4（命中watchlist high）
→ 标记为可疑，继续 Step 3-6
```

**Step 2 完成后，必须按以下决策节点判断是否继续 Step 3-6（强制，不可跳过）**：

| 条件 | 操作 |
|------|------|
| ① 找到 🔴 高度可疑变更 | 继续 Step 3-6，验证变更对各阶段的影响路径 |
| ② 无变更归因 + 异常已自愈 + 24H 全天 vs 置信基准跌幅 ≤ 5% | 可停止，输出「自愈，无明确根因」结论，注明自愈时间和持续时长 |
| ③ 无变更归因 + 异常持续未自愈 | 必须继续 Step 3-6，先查阶段定位再进入对应深挖步骤 |
| ④ 无变更归因 + 异常已自愈 + 但 24H 全天 vs 置信基准跌幅 > 5% | 不得停止，按③处理 |

> ⚠️ **「自愈」不等于「无需继续」**：只有同时满足"已自愈 + 24H 全天无显著影响"才可以停止。单纯自愈但 24H 受影响，仍需走 Step 3-6 定位影响路径。

| 倒排 | `ES` | - | 零/低分quota |

### Step 4: 召回根因定位

> 对比基准：使用 Step 0.2 确定的置信基准，不固定用 -1d。

**矛盾判断（关键！）**：
- recall阶段新笔记 vs 置信基准下跌
- 但各召回渠道quota正常

### Step 5: 索引层排查

**索引切换检查要点**：
1. **RIS2.0追kafka速度**（正常2-5s，异常10-20min）
2. **内存使用率**（>90%有风险）
3. **构建QPS**（<5K可能延迟）
4. **Kafka消费延迟**

**关键表**：

- `dssm_topk_30d` - DSSM topk天级
- `mllm_cluster_*` - MLLM聚类

### Step 6: 内容供给排查

> ⚠️ **Step 6 执行规则（强制，每次必读）**：
> 1. **先查内容池**（`content_pool_1h` / `content_pool_24h`，datasource: `vms-recommend`）
> 2. **内容池下跌** → 继续查 `publish_qps` 和 `audit_retry`，这两条 query 的 datasource 是 **`vms-shequ`**，不是 `vms-recommend`
> 3. **内容池正常** → 跳过发布/审核，直接进入 Step 6.4 数据流层
> 4. **执行每条 query 前必须核查该 query 自身的 `datasource` 字段**，不得仅使用 step 级默认值；若 query 返回 0 条，第一步检查 datasource 是否正确，不要凭猜测枚举其他 datasource

**检查链路**：
1. **发布侧** - 发布QPS（`datasource: vms-shequ`）、审核积压（`datasource: vms-shequ`）
2. **转码侧** - 转码延迟
3. **特征侧** - 新笔记特征填充率

## 五、变更查询与高风险配置对比

> ⚠️ **SOP 规定**：变更查询结果中，每一条变更的 `change_object`（apollo）或实验参数（racingweb）都必须逐一与 `references/config-watchlist.json` 中的清单对比，不能凭经验/直觉筛选，不能写死关注哪些 key。

### 变更查询步骤

```
1. 用 xray_changevent_query 查询异常时间点前 2 小时内的所有变更
   - system: apollo + racingweb
   - tag: 外流推荐
   - env: prod + staging

2. 加载 references/config-watchlist.json，读取：
   - apollo_configs.critical[]  → 最高风险 key 列表
   - apollo_configs.high[]      → 高风险 key 列表
   - experiment_flags.high[]    → 高风险实验 flag pattern 列表

3. 对每一条变更事件：
   - apollo 变更 → XRay API 只返回 namespace（resource_name 字段），**不返回具体 key**
  - 匹配逻辑：检查 resource_name 是否在 watchlist `apollo_configs` 覆盖的 namespace 范围内（如 arkfeedx、arkfeedxmerchant 等）
  - 命中 namespace 后，需人工前往 redcloud 查具体 key diff：`https://redcloud.devops.xiaohongshu.com/redconf/app/nameSpaceList?appId={namespace}&confEnv=PRO`
  - 在 redcloud 中对比该时间点的 key 变化，与 watchlist critical/high key 列表逐一匹配
   - racingweb 变更 → 取 info.after_value.parasChangeEvents[].flagValueMap 的 key，
     与 watchlist experiment_flags 中的 pattern 逐一 fnmatch 匹配
   - 命中 critical → 立即标记为【高度可疑】，优先确认
   - 命中 high    → 标记为【可疑】，纳入排查
   - 未命中       → 标记为【低风险】，暂时搁置

4. 对每个命中的变更：
   - 展示变更前后值（before_value → after_value）
   - 说明该 key/flag 在 watchlist 中的风险描述
   - 结合异常时间点判断时序相关性

5. watchlist 文件路径固定为：
   skills/rec-homefeed-newnote-diagnosis/references/config-watchlist.json
   每次对比前必须重新读取（不要使用记忆中的旧版本）
```

### 注意事项

- **不要写死任何 key 名或 flag pattern**：watchlist 是动态维护的，直接从文件读取，保证和最新版本一致
- apollo 变更接口不返回具体 key/value 时（info 字段缺失），在报告中注明"需人工查 redcloud 确认"，附上 redcloud 链接
- racingweb 变更若只有实验名无参数详情，在报告中注明"需人工查 racing 确认参数变化"，附上 racing 链接

- 变更按🔴🟠🟡三级分类，每级单独一节

- 根因定位块用 `━━━` 分隔，每个 T0 独立一块
- 多段异常（混合型）时，每段 T0 各有独立根因块
- 「建议下一步」必须列出 2-4 条具体待确认项，不写泛泛的"继续排查"
