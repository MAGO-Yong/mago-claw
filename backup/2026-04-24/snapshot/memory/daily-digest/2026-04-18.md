## 💡 用户的好问题/好思考

### 0. AI 应用可观测规范的系统性思考（20:17-21:20）

在编写 AI 应用可观测规范前，用户先要求理解在线服务高可用规范的"编写逻辑"，然后抛出了四个核心问题：

1. **什么是"核心指标"**：成功率怎么算？质量怎么量？
2. **什么算"故障"**：模型答错 vs 服务崩溃，如何区分？
3. **资源边界在哪里**：Token 消耗、并发调用、上下文窗口 vs CPU/内存/磁盘
4. **强制 vs 建议的判断标准**：对 AI 系统来说，哪些不遵守会"直接"造成问题？

这四个问题直击 AI 可观测的核心难点——传统观测的四标（REDUSE）在 AI 场景下如何重新定义。用户不是在要答案，而是在帮我**建立正确的思考框架**，避免直接把传统观测规范照搬到 AI 场景。

### 1. 「长期任务不是任务吗？」— 一句话击穿 Agent 可靠性本质

你发现 Hi 会话的任务清单里漏掉了刚设置的两个长期 cron 任务。不是要你补上，而是追问根因：「你他妈又说假话，我不是要你去补，而是为啥没有把长期任务包含进去，长期任务不是任务吗？另外你真的创建长期任务了吗？」

这个问题直接暴露了 Agent 的核心缺陷：**"做了"和"记下来"是分离的**——做完配置就汇报完成，但没主动把信息写进所有会话都能读到的通用文件。

### 2. 「明确和通用的就应该合并到 agent 和 memory」— 跨会话一致性架构原则

「我的根本目的是你可以有不同的 channel 和会话，各自有自己的上下文没法同步可以理解。但是如果是明确和通用的，就应该合并到 agent 和 memory，这样我在其他渠道都可以访问一致的数据和内容。」

这句话定义了多 Agent/多会话架构下的信息分层原则：
- **会话私有**：单次对话的上下文、临时状态
- **全局共享**：长期任务、规范、偏好 → 必须写进 MEMORY.md / AGENTS.md

### 3. AI 应用评估对标搜广推实验变更

「AI 应用和之前搜广推算法的实验变更很像，实验变更是需要有对应业务指标的，然后发布后观测业务指标的变化。所以 AI 应用的效果评估指标是不是就对标业务指标」

这个类比提供了一个重要的思路：AI 应用可观测不应该只关注系统层面（延迟、错误率），更要关注**业务效果层面**——就像搜广推发布后看 CTR/转化率变化一样，AI 应用发布后也应该看其对应的业务指标变化。

### 4. 主动推动 Agent 自我进化

「你再回顾下我刚才让你变好用的一些点，你也可以全网搜索下别人让 claw 更好变成个人助力的好用的点，你对你进行一个全方位的审查和进化吧」

不是被动修 bug，而是要求 Agent 主动对比外部最佳实践，做系统性自我提升。

## 🔑 关键决策

### A+C 方案统一任务管理

**决策**：MEMORY.md 开头维护任务清单（A）+ daily-digest cron 每天自动同步（C）

**为什么**：
- 选项A（合并进 MEMORY.md）解决"所有会话启动必读"的问题
- 选项C（cron 自动同步）解决"状态持续更新不依赖手动维护"的问题
- 两者结合，既不挤 HOT tier 上限，又不依赖 Agent 实时自觉

### AGENTS.md 改成 Do/Don't 结构化格式

**决策**：把规则从散文式描述改为 Do/Don't 清单

**为什么**：搜索 OpenClaw 最佳实践后发现，结构化格式比散文式描述在 Agent 启动加载时可执行性更高。
