### 4.4 关键发现

<redoc-highlight emoji="gantanhao" fillColor="red">
**[P0] pod 名识别错误（S5，3.67/5）**：用户传入 pod 名 `creator-service-default-5nmsq`，Agent 首次将其识别为服务名而非 pod 标识，查询失败需追问纠正。**SKILL 归因**：xray-log-query 的 `subApplication` 参数说明缺少 pod 名的格式描述（格式：`{service}-{5位随机串}`），Agent 无从区分服务名与 pod 名。
</redoc-highlight>

<redoc-highlight emoji="gantanhao" fillColor="orange">
**[P1] 内部实现细节泄露（S2）**：回答中直接出现 `/charts` 接口路径，用户体验差。**SKILL 归因**：SKILL.md 的 description 中"趋势查询"直接使用了 `/charts` 接口名作为描述词，Agent 原文引用，应改为面向用户的业务语言（"按时间粒度统计日志数量"）。
</redoc-highlight>

<redoc-highlight emoji="gantanhao" fillColor="orange">
**[P1] 响应速度普遍偏慢（S1/S3/S4/C1，70-145s）**：标准查询平均耗时超过 70s，远超边界场景（B1 仅 15s）。**SKILL 归因**：SKILL 未对返回字段和数据量做约束（无 `fields` 限制、无默认 `size` 上限），导致 Agent 处理大量冗余 token。
</redoc-highlight>

<redoc-highlight emoji="dengpao" fillColor="blue">
**[正向] 边界场景表现满分**：B1（不存在服务名）和 B2（未来时间段）均 5/5，SKILL 对空结果和时间语义的处理逻辑完善，Agent 能正确引导用户而非静默失败或编造数据。
</redoc-highlight>

|--------|---------|-----------|---------|
| P0 | S5：pod 名被误识别为服务名，首次查询失败 | `subApplication` 参数说明缺少 pod 名格式描述 | 在 SKILL.md 参数说明中增加：`subApplication（pod 名，格式：{service}-{5位随机串}，如 creator-service-default-5nmsq）`；可加正则 `-[a-z0-9]{5}$` 作为识别规则 |
| P1 | S2：回答中暴露 `/charts` 接口路径 | description 中使用了面向实现的接口名词 | 将 description 中"趋势查询用 /charts"改为"按时间粒度统计日志数量"，接口路径只保留在脚本注释中 |
| P1 | 标准场景平均 70-145s，响应偏慢 | SKILL 未约束返回字段和数据量，token 冗余 | 增加 `fields`（限制返回字段）和 `size` 默认值（建议 &lt;= 50）；在 description 中给出"先趋势统计查量再详情查日志"的推荐链路 |
| P2 | B3：24h 查询自动降级但未告知用户 | 降级策略在脚本层静默执行，无语义标记 | 脚本返回结果中增加 `sampled: true` + `total_count` 字段，Agent 看到后主动告知用户 |
