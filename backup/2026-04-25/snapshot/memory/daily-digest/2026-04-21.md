## 💡 用户的好问题/好思考

### 1. "除了内容外，更多是样式"——PR 稿的本质是视觉传达

**用户原话：**
> "除了内容外，更多是样式，比如背景块的颜色之类的，让文档更可读和重点明显"

**背景：** V2 PR 稿内容已经写完并发布，但用户看完后意识到：文档的视觉层次感和重点突出程度，直接决定了读者能不能"一眼看懂 V2 的价值"。这不是美化，而是信息设计。

**价值：**
- 用户把 PR 文档的定位从"信息罗列"升维到"视觉传达"
- 痛点块要视觉冲击、CLI 要工具感、分栏要有层次、展望要有收尾——每个颜色选择都有语义
- 这个思路同样适用于其他技术文档：内容正确 ≠ 有效传达，视觉分层是必须的

### CLI 与 Skill 并列为两大入口

V2 文档中 CLI 章节被设计为与 Skill 平级的独立章节：
- **Skill 解决**：人对话 AI 查数据
- **CLI 解决**：Agent/CI/CD 直接调 XRay

这是一个产品定位的明确：CLI 不是 Skill 的附属品，是 Agent 时代的独立入口。

### REDoc 批量编辑最佳实践（经验沉淀）

1. `docs:get --mode blocks` 获取所有 blockId 和 hash
2. 大量 ops 写入 JSON 文件，用 stdin 传入 `--ops -`，避免 shell 转义
3. 每次编辑后 hash 失效，必须重新 `docs:get` 获取新 hash
4. 用 `docs:edit --ops` 做块级编辑比 `--target/--replace` 更稳定
