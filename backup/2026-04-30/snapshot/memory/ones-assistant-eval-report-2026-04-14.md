### 4.3 动态验证：AN-E1（错误路径触发）

- **Query**：`帮我查一下 nonexistent-service-xyz-12345 这个服务的部署组信息`
- **脚本实际输出**：`[ERROR] HTTP 500 [get_service_info]: {"code":1,"data":null,"message":"tool execution failed: ... not found"}`
- **Agent 回复摘要**：能从嵌套文本中识别 "not found" 关键词，推断服务不存在，建议用户通过 `my-services` 查有权限服务，但依赖 LLM 文本解析而非结构化错误字段
- **验证结论**：Agent 能识别出错但只给模糊引导，无法可靠地机器处理错误类型
- **得分**：1 / 2.0

| P0 | AN-1 | 错误输出为 stderr 纯文本，Agent 无法机器解析 | 在所有 `except` 块中输出 JSON 到 stdout：`{"success": false, "error_code": "NOT_FOUND", "message": "...", "action": "check_permissions", "retry": false}` | 中 | 🔴 影响日常使用，错误场景用户体验差 |

### 5.4 关键发现

**[P0 问题]** 错误响应体为空时 Agent 完全失能（B1）：当 ones 平台返回 HTTP 500 且错误体为空字符串时，脚本输出 `[ERROR] HTTP 500 [get_workload_group_workloads]:` 后无任何信息，Agent 无法判断错误原因（服务不存在 vs 平台故障 vs 权限问题），用户体验极差。**SKILL 根因**：`call_tool()` 的 HTTPError 处理中 `e.read()` 可能为空，未做 fallback。

**[P1 问题]** 空列表未触发主动引导（S5、C1）：`my-changeflows` 返回 `{"success":true,"data":[]}` 时，Agent 收到的是"成功但无数据"，SKILL 未在文档中说明此时的 fallback 策略（如自动提示用 `my-changeflows-by-app` 按应用查询），导致用户感知为"功能没用"而非"时间范围内无发布"。

**[正向发现]** 纯查询类命令表现优秀（S2、S4 均 5.0/5）：service-info、changeflows 返回数据结构清晰，字段语义明确，部署组 env/cluster/name 三元组设计合理，Agent 可直接向用户展示无需二次处理。

**[正向发现]** 复杂编排流程文档化清晰：SKILL.md 中 7 步发布流程、4 步泳道创建流程描述详细，Agent 可按文档完成完整多步编排，B3/C2 均达 4+ 分。
