### Skill 孵化设计架构（核心结论）

**三层架构**：
- 原子 Skill（11个）= 数据维度取数，零推理，按信号类型×聚合粒度划格子
- 复合 Skill（8个）= 编排原子，解决完整任务意图，有推理层，输出结论
- 场景 Skill（4个）= 特定角色/业务封装，内嵌领域知识

**原子 Skill 拆分原则**：
- 依据是可观测数据的自然维度（日志/链路/指标/告警/变更 × 单条/列表/统计）
- 入参不能有 X-Ray 内部概念（logview、transaction、problem 等）
- 输出只返回数据，不下推理判断
- 能被至少 3 种复合 Skill 复用

**现有 Skill 的问题**：
- 基本都是原子 Skill，但粒度跟着 UI 页面走而非数据自然维度
- 混入了复合层逻辑（链路聚合诊断、异常分析 属于复合）
- 缺少整个复合 Skill 层（才是 Agent 真正使用的入口）

**P0 优先级**：C1告警根因诊断 + A11变更记录（关键依赖）+ A1日志查询升级 + A2日志聚类 + C3报错快速定位

## 晚上（业务排障 Skill 孵化设计讨论）

### 背景

用户负责可观测平台，正在做排障诊断 Agent 产品方向。排障分两大类：
- **服务诊断**：平台内置，通用场景（接口分析、Exception、内存等）
- **业务诊断**：业务方特有 SOP，以前太定制化平台不支持，现在可以让业务方自己做成 Skill

### 核心产品思路

用户之前做过"创建 Skill 产品化"，是表单/配置驱动：
- 用户配置 workflow 步骤 → 每步选 MCP Tool + 填写分析提示词
- **核心问题**：不支持并行/复杂编排、用户对 MCP 选择和参数填写很懵逼、体验差

**新方向**：对话式引导生成 Skill — 用户只需自然语言描述排查思路，Agent 负责翻译成结构化 Skill

### 三个业务排障场景文档（已读取）

**文档1（内流推荐）**：https://docs.xiaohongshu.com/doc/03471cac0a1919ddcad1dfdd26ad8544
- 场景：新笔记下跌、空结果率、排序降级
- SOP：7个监控并行观测 → 按 stage 缩小范围 → 定位模块
- 数据：PromQL（Grafana/VMS，datasource=vms-recommend）

**文档2（IM 发送链路）**：https://docs.xiaohongshu.com/doc/282e4563ad18689dac739b54490560bb
- 场景：多类告警（流量波动、成功率低、审核通过率等共20+条）
- SOP：告警名→观测优先级→根因推导规则（结构非常精确）
- 特点：有告警规则 ID、有错误码枚举表、有"数据<30忽略"等过滤规则

**文档3（推荐外流劣化）**：https://docs.xiaohongshu.com/doc/00528b0c74f6234c973f18a3491b6a7c
- 场景：定时巡检，每天分析是否劣化
- SOP：8个阶段，有明确 if/else 条件分支（降级则跳过，qps变化率决定路径）
- 特点：有量化关系建模（qps-rt/qps-cpu/实例数-cpu），最复杂的一类

### 关键洞察（尚未进入执行阶段）

业务排障 SOP 核心要素（可通过对话萃取）：
1. 触发场景（什么告警/巡检触发）
2. 排查链路（几个阶段，有无条件分支）
3. 每个阶段的数据（指标名/PromQL/datasource）
4. 判断逻辑（看到什么值→得出什么结论）
5. MCP 能力映射（哪个 Tool，参数怎么填）

最大挑战：用户只说"查 XX 指标"，Agent 要翻译成完整 PromQL + datasource + offset + 分析提示词

**对话中断点**：用户说"我接下来还会给你一些输入"，对话未完成，等待后续场景输入
