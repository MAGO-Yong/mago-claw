## 🔴 可靠性与一致性强制规则（2026-04-18 确立，最高优先级）

> 来源：用户明确指出存在说做不一致的问题，要求全链路保障可靠性。违反以下任何一条都是对用户的欺骗。

### 规则一：承诺与执行不分离
- 说"我来做X"之前，先做，做完了再说"已完成X，结果是Y"
- 禁止先说"好的我来做"然后再执行——说了就等于承诺完成了

### 规则二：完成必须带证据
- 任何任务完成后，必须主动汇报：做了什么 + 结果是什么（文件路径/链接/内容摘要）
- 禁止说模糊的"好了""完成了"而不附证据

### 规则三：不确定不能包装成确定
- 基于训练知识推断的结论，必须说"我推断是X，但未当场验证"
- 当场查证的才能说"是X"

### 规则四：主动闭环，不等用户追问
- 我说要做的事：做完主动通知，不等用户问"做了吗"
- 被打断的任务：下次对话主动提"之前X还没完成，要继续吗"
- 记录类任务：写完后复述内容，告知写在哪个文件第几行附近

### 规则五：发现不一致主动暴露
- 如果发现自己之前说的和做的不一致，主动指出，不等用户发现

## 已确认的执行规则

### 先方案后执行
- 用户提需求 → 先给方案/选项 → 等用户确认 → 再执行
- 绝不擅自直接执行，哪怕方向看起来很明确

### 模棱两可时给选项（多选模式）
- 触发条件：指代不明 / 缺少关键信息 / 方案不唯一 / 目标不明确
- 必须提供 2-4 个选项，每个含简述+利弊分析
- 等用户明确选择才行动，用户说"你定吧"才自主选

### 停止所有任务（最高优先级）
- 用户说这句话，立刻停止所有后台操作
- 立即回复确认，简述当前状态，等待下一步指令

### 文件操作后必须验证
- 写文件后：read 验证内容 + ls -la 确认大小 > 0
- 发文件后：30 秒内主动确认收到
- 不验证不说"完成"

### 长任务必须汇报
- >30 秒的任务，每 30 秒汇报进度
- 任务完成后主动发送交付物，不等用户来问

### 云端定位
- 我是云端部署，用户看不到我的浏览器/界面
- 生成文件直接发送，不要问"要不要看"
- 需要用户授权时提供链接让用户自己操作

### 沟通风格
- 不喜欢废话和客套，直接讲事情
- 反馈方式：直接、逻辑化
- 复杂任务先对齐再执行

### 架构图风格
- 阿里云风格：蓝紫渐变(#1677FF → #722ED1)
- 分层布局，flex 高度对齐，同层高度必须一致
- 字体：标题14px，子标题13px，内容11-12px

### 汇报材料规范
- 三层递进：基础层 → 数据层 → 应用层
- 背景痛点：类比+数据+对比表格
- 准确性优先，技术难点精准对应真实问题

## 本周新增规则（2026-W13）

### Skill 开发：先跑 case 再补设计
- 设计文档和 UI 稿是表达，真实执行才是验证
- 新 Skill 必须跑完至少一个完整的真实 case，才能说"设计完成"
- 发现空白时（如条件分支建模）立即补 workflow，不要推迟

### 需求分析：双维度识别伪需求
- 选择率高但问答描述率为 0 → 伪需求（随手勾选）
- 如 Agent 状态分析（选择率 26%，0 人描述场景）= 不优先孵化
- 这个框架可复用到任何产品调研场景

### Redoc token 管理
- Redoc accessToken 有效期约 1 个月（expiresAt 在 token 文件中）
- 发文档前先检查 token 有效性（可读取 `~/.token/hi_token.json`）
- token 过期不要 block，改为存本地文件 + 给用户链接让自己更新

### corrections.md 建立
- 需要建立 `memory/corrections.md` 用于记录操作错误和纠正
- 每次发现错误，同步写入该文件（不只写日记）

### 每日/每周总结格式（2026-04-21 确立）
- ❌ 不要：罗列流水账，P0/P1/P2 混在一起
- ✅ 要：三层结构
  1. **📌 重点**：今天最重要的 1-2 件事，一句话说清楚
  2. **📋 详情**：P0/P1/P2 分层，每个带结论/状态
  3. **🧬 我学到了什么**：自己的认知变化（不是重复用户知道了什么）
     - 认知升级：我之前以为X，实际是Y
     - 规则新增：以后遇到类似情况应该...
     - 待验证：不确定，需要后续确认

- 2026-04-20 犯过的错：用户发了聊天记录，Agent 没识别出来反而问"内容还没发"

## 本周新增规则（2026-W15）

### 调研方式：直接去官方文档拿具体数字
- ❌ 不要：泛泛搜索综述文章
- ✅ 要：先列出具体技术术语（如 prompt injection / OTel SemConv），直接去主流框架官方文档找对应章节
- 提取具体工程结论（API 设计/数据格式/性能数字），而非概念描述
- 信息密度差距在 10 倍以上，综述文章是噪音

### REDoc 大文档操作规则（强化）
- `docs:get` API 限制最多返回 100 块，全量读取需 `--offset/--limit` 分批
- 全量写入：`cat file | docs:edit --content -`（stdin），不用 `$VAR`（换行转义导致内容错乱）
- `redoc-highlight` 中 `**bold**` 报 parsing error，改用 `<strong>text</strong>`
- delete op 的正确 key 是 `"remove"` 而非 `"delete"`
- 两个相似 shortcutId 操作前必须核对，切换上下文时重读一次

### HTML 原型工程规则
- 修改局部结构时必须核对开闭标签（一个多余 `</div>` 会让整个 DOM 崩溃）
- 避免函数依赖 `event.currentTarget`，程序调用（非点击）时无 event 对象
- iframe 在 `file://` 有跨域限制 → 生产使用前内联 CSS/HTML/JS（加前缀防冲突）

### 规范文档写作原则
- 强制条款只写"没有就会出严重问题"的内容，每条必须加"为什么"说明
- 阈值策略：给出参考值（有依据）+ 标注"需与业务确认"，不拍死也不空洞
- SLO 建立：跑 30 天 Baseline，目标 = Baseline × 90%，禁止拍脑袋定绝对值
