# 数据集

## 1. 数据集是什么

在 AI 应用评估里，数据集是整套评估体系中的**样本层**。它承接的并不只是人工整理的离线数据，也包括从线上链路中筛选出来的真实请求，以及从自动评估任务回流出来的 bad case。对使用者来说，可以把它理解成**评估样本库**，或者长期沉淀问题样本和高价值样本的资产池。

很多团队在刚开始搭建评估体系时，会把数据集看成一个普通的存储容器；但实际上，数据集真正重要的地方在于"**可复用**"。没有数据集时，问题样本大多是一次性被看到、一次性被讨论；有了数据集之后，样本才会变成下一轮离线回归、版本比对和上线前验证的基础。

---

## 2. 数据集适合解决什么问题

数据集最适合解决"**样本沉淀不下来**"和"**问题无法重复验证**"这两类问题。线上 bad case 即使被发现了，如果没有继续收进数据集，下一次版本迭代时仍然很难稳定复现；同样，一批人工确认过的高质量样本，如果只是停留在聊天记录或表格里，也很难被持续复用。数据集的作用，就是把这些原本零散的样本逐步整理成可以长期复测、长期对比的评估资产。

当你希望做**离线回归**、积累固定 bad case、构建某个业务场景下的黄金样本集，或者想把线上发现的问题继续变成后续可复验的测试集时，都应该优先考虑先建设数据集。值得注意的是，数据集里的样本不只是人工编造的，也可以通过在线链路回流的方式把真实请求中的 good case / bad case 沉淀进来，这样的样本质量通常更接近真实业务场景。

> **使用限制**：每个数据集最多存放 **1 万条**数据项。

---

## 3. 步骤一：创建数据集

进入 **AI 应用评估 → 数据集**，点击右上角「**+ 新建数据集**」。

【截图：数据集列表页，重点标注右上角「+ 新建数据集」按钮】

在弹出的面板中填写以下内容：

### 3.1 填写名称和描述

数据集名称和描述最好能直接体现**业务场景、样本来源和用途**。例如「sug粗筛 bad case 集 - 2026Q1」「某业务模块离线回归集」这种命名，会比「测试集A」「样本集1」更容易理解和维护。对于多人协作的项目来说，名称本身就是一层最基础的说明文档，它应该帮助后续使用者快速判断：这个数据集为什么存在、后续应该拿来做什么。

### 3.2 配置数据项字段

字段决定了这个数据集能存哪些数据、后续评估器能拿到什么输入。字段设计的原则，是**既要支撑后续评估，又要保留足够多的问题定位信息**。

【截图：新建数据集弹框，显示名称/描述输入框 + 数据项字段配置区域（含字段类型下拉展开状态，显示全部6种类型）】

支持的字段类型如下：

| 类型 | 说明 | 适用场景 |
|------|------|---------|
| 字符串（string） | 文本内容 | 用户 query、模型回复、笔记文本等 |
| 整数（integer） | 整型数字 | 评分、序号等 |
| 浮点数（float） | 小数 | 置信度、概率分值等 |
| 布尔值（boolean） | true / false | 是否命中、是否正确等标注 |
| 日期（datetime） | 时间戳 | 样本产生时间等 |
| JSON | 结构化对象 | messages 数组、raw_input 复杂结构等 |

常见字段组合包括：`input`（输入）+ `output`（输出）；或 `messages` + `raw_input` + `response_content`（适合 LLM 调用场景）。

对使用者来说，最重要的不是字段越多越好，而是**字段是否和评估对象一致**，后续评估器是否能从中拿到真正有用的信息：

- 如果这个数据集后面要配合**离线回放**使用，字段最好尽量贴近目标输入结构
- 如果主要用于 **bad case 沉淀**，则建议优先保留能解释问题的上下文，而不是只保留一个孤立的输出结果
- 字段名应该尽量贴近业务语义，避免只使用团队内部才能理解的缩写

### 3.3 创建时的注意事项

创建数据集时，最容易出现的问题是"先建起来再说"，结果后面发现字段无法承接评估、样本也不具备复用价值。更稳妥的方式，是在创建前先回答三个问题：

1. 它面向**哪个业务场景**？
2. 样本主要**来自哪里**（人工整理 / 线上回流）？
3. 后续准备**拿它做什么评估**（离线回归 / bad case 沉淀 / 版本对比）？

字段设计应该服务于后续评估，而不是只满足一次性的记录需求。

填写完成后点击「**确定**」完成创建。

## 4. 步骤二：添加数据

创建完成后，点击数据集名称进入详情页，可以看到数据项列表。有两种方式添加数据。

【截图：数据集详情页整体视图，显示顶部面包屑导航、数据项列表（含字段列）、右上角「+ 添加数据」按钮】

### 4.1 手动添加

适合样本量不大、但质量要求较高的场景——比如精心整理的黄金样本集、典型 bad case 复现样本。

点击右上角「**+ 添加数据**」，在表单中逐字段填写数据内容，一次可添加一条或多条。页面会根据字段类型展示不同的录入控件，录入方式本身并不复杂。真正值得花时间的是样本内容本身：它是否具有代表性，是否能覆盖关键问题，是否足以支撑后续评估。

【截图：手动添加数据的表单弹框，显示各字段的输入控件】

### 4.2 通过链路回流自动沉淀（推荐）

这是把线上真实样本沉淀到数据集的主要入口。相比手工编造数据，线上真实请求的质量更接近业务实际。

链路回流是在**自动评估任务**中配置的，不是在数据集页面直接操作。在创建或编辑自动评估任务时，第四步「数据回流」中：
- 选择**目标数据集**（提前创建好）
- 设置**回流条件**（如 `score = 0`，即低质量样本自动回流）
- 配置**字段映射**（把 SPAN 的 input / output 映射到数据集字段）

配置完成后，每次任务运行结束，符合条件的样本会自动写入数据集，无需手动操作。

> **前提**：AI 应用已接入 X-RAY 并有链路数据，且已创建好自动评估任务。

【截图：自动评估任务创建流程中「第四步：数据回流」配置页面，显示目标数据集选择 + 回流条件配置 + 字段映射】

## 5. 步骤三：管理数据项

在数据集详情页，每条数据项右侧都有操作入口：

【截图：数据集详情页，标注每行右侧的「查看 编辑 删除」操作列】

- **查看**：查看该条数据项的完整字段内容，以及来源信息（trace ID、是否来自评估回流等）。如果这条样本本身就是从评估结果回流而来的，还可以继续反查到对应的实验报告，让数据集与线上链路和评估结果形成可追踪的关系。
- **编辑**：补充上下文、修正字段内容、完善标注
- **删除**：处理误入样本或已经失效的数据

长期来看，数据集质量比数量更重要。做适度清理和维护是非常正常的事情。

---

## 6. 步骤四：用于评估

数据集最常见的后续用途有两类：

1. **离线评估**：作为自动评估任务的数据来源，用于版本回归、效果对比和上线前验证。在创建自动评估任务时，选择「离线数据集」，选中对应数据集，再配置字段映射即可。
2. **样本沉淀池**：持续承接线上链路和实验回流下来的 bad case，随着评估体系成熟，逐渐从"一个表"演变成"团队共享的评估资产层"。

详细操作参考 → **模块四：自动评估任务**

## 7. 常见问题

**字段设计好之后还能改吗？**
可以继续添加新字段，但已有字段的类型不支持修改。建议创建前先想清楚字段结构，避免后期重建。

**数据集满了怎么办？**
每个数据集上限 1 万条。建议按时间段或场景拆分成多个数据集（如按月、按业务场景），而不是把所有样本堆在一起。

**样本来源优先级怎么选？**
优先使用线上链路回流的真实样本，其次是人工整理的高价值 bad case，最后才是从零构造的合成样本。真实样本的分布和质量通常最接近业务实际。

**后续准备做离线回放，字段怎么设计？**
字段最好尽量与目标输入结构保持一致；如果样本来自线上链路，建议优先保留那些真正能帮助定位问题的上下文信息。最重要的是，数据集设计一定要服务于后续评估和复盘，否则很容易退化成一个难以复用的临时样本仓库。
