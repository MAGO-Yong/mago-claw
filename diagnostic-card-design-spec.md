# AI 诊断卡片设计说明

> 版本：v4 · 日期：2026-04-20
> 场景：XPILOT AIOps 烽火项目 · 短期方案 · 告警诊断结果独立推送卡片
> 原型文件：`diagnostic-card-redesign.html`

---

## 一、卡片定位

**短期方案背景**：诊断卡片与告警事件卡片**分开推送**（两条独立消息），诊断卡片需自成一体，让 SRE 在群消息场景中 3 秒内完成决策扫读。

**卡片角色**：决策摘要，不是缩略版诊断报告。

> 原则：告诉用户"发生了什么、为什么、影响是什么"，不在卡片里还原完整推理过程。

---

## 二、卡片结构总览

```
┌─────────────────────────────────────────────────────┐
│  🤖  针对 19:57 的 [P2] adwind-service 异常环比上涨   [诊断完成] [引用原消息]  │  ← 标题行
├─────────────────────────────────────────────────────┤
│  诊断结论                                             │  ← 结论区（蓝色底色，突出）
│  adwind-service 19:50 发布后……JSON 反序列化失败。      │
├─────────────────────────────────────────────────────┤
│  关键证据                                             │  ← 证据区
│  异常  ExecutionException → JavaBeanDeserializer...  │
│  变更  19:50:35 adwind-service 发布 [最可疑]          │
│  影响  qcsh4 机房 100%，接口 99.4% 失败               │
│  指标  异常量 1→134，19:51~19:52 critical 级          │
├─────────────────────────────────────────────────────┤
│  触发：19:57:08  推送：19:57:36                       │  ← 时间行
├─────────────────────────────────────────────────────┤
│  [展开完整诊断]  [异常详情]  [异常堆栈]  [事件详情]     │  ← 操作按钮
│  诊断是否有用？  👍 有用    👎 没用                    │  ← 反馈行
└─────────────────────────────────────────────────────┘
```

---

## 三、各区块说明

### 1. 标题行

| 元素 | 内容 | 来源 |
|------|------|------|
| 告警描述 | 「针对 19:57 的 [P级] [服务名] [告警规则名]」 | 告警事件元数据：告警触发时间 + 告警等级 + 服务名 + 规则名，由 Agent 拼接 |
| 「诊断完成」标签 | 绿色 badge，表示 Agent 已完成推理 | Agent 执行状态 |
| 「引用原消息」按钮 | 点击在 Hi 中引用告警事件原消息 | Hi IM 的 quote 能力，关联两条卡片 |

**设计意图**：让 SRE 一眼知道"这是针对哪条告警的 AI 诊断结果"，解决短期方案下两条独立消息的认领对应问题。

---

### 2. 诊断结论（核心）

| 元素 | 内容 | 来源 |
|------|------|------|
| 「诊断结论」标识 | 蓝色小标，强调这是 AI 判断而非原始数据 | 固定文案 |
| 根因一句话 | 服务名 + 触发事件 + 根因描述（人话） | Agent 推理生成，替代现有「异常类型」字段 |

**视觉设计**：浅蓝底 + 蓝色边框，形成独立色块，是卡片中视觉权重最高的区域。

**原有问题**：现有卡片「诊断结论」区只有原始字段（服务名、趋势数组、异常类名），没有 AI 推理出的根因判断，需要 SRE 自己读完所有字段再下判断。

**示例**：
> adwind-service 19:50 发布后，下游 noteVideo 字段返回格式由对象变为数组，导致 JSON 反序列化失败。

---

### 3. 关键证据

按固定维度顺序展示（异常 → 变更 → 影响 → 指标），有则展示，无则不占位。

#### 3.1 维度排序原则

| 顺序 | 维度 | 排在前面的原因 |
|------|------|--------------|
| 1 | **异常** | 描述"发生了什么"，是定性判断的起点 |
| 2 | **变更** | 最常见的根因来源，SRE 第一时间想确认 |
| 3 | **影响** | "影响了什么"，决定响应优先级 |
| 4 | **指标** | 量化数据，辅助确认严重程度 |
| — | **依赖** | 有则插在变更之后，无则不展示 |
| — | **历史** | 有相似故障匹配则追加在最后，无则不展示 |

#### 3.2 各维度内容说明

**异常**
- 内容：异常类型 + 报错位置 + 关键异常信息（合并为一句）
- 来源：告警事件中的 `异常类型` + `报错函数` + `异常信息` 三个字段，由 Agent 提炼合并
- 原则：去掉完整堆栈，只保留定位根因必要的信息；完整堆栈进「展开完整诊断」侧边栏
- 示例：`ExecutionException → JavaBeanDeserializer:503，JSON 反序列化失败：期望 { 但实际是 [，字段 noteVideo`

**变更**
- 内容：与异常时间线相关性最高的变更条目
- 来源：告警事件中的变更列表，Agent 按时间相关性排序，取最可疑一条展示
- 标注：加「最可疑」橙色标签，并说明与异常时间点的时间差
- 其余变更：移至「展开完整诊断」侧边栏的推理过程中，标注已排查原因
- 视觉：橙色左边线 + 浅橙底色，与其他维度区分
- 示例：`19:50:35 adwind-service-default 发布部署（姜德铭），与异常激增时间点（19:51）相差 25s`

**影响**
- 内容：机房分布 + 影响接口/功能 + 影响比例
- 来源：告警事件中的 `机房分布` + `影响范围` 两个字段，由 Agent 合并表达
- 视觉：红色左边线 + 浅红底色，标示影响严重
- 示例：`qcsh4 机房 100%，listSearchTopicAIMaterial 接口 99.4% 请求失败`

**指标**
- 内容：异常量趋势 + 峰值时间 + 异常级别
- 来源：告警事件中的 `请求次数` 数组 + `趋势` 字段，由 Agent 转化为可读描述
- 原则：不展示原始时间点数组，改为"起始值→峰值（倍数）"的量化描述
- 示例：`异常量 1→134（环比 200%+），19:51~19:52 达 critical 级`

---

### 4. 时间行

| 元素 | 内容 | 来源 |
|------|------|------|
| 触发时间 | 告警规则触发的时间戳 | 告警事件元数据 `触发时间` |
| 推送时间 | 卡片实际推送到群的时间戳 | 告警事件元数据 `推送时间` |

**放在最后的原因**：时间是辅助信息，已经在标题行体现了大致时间（19:57），精确时间戳作为参考放在卡片底部，不抢占核心区域的视觉权重。

---

### 5. 操作按钮

| 按钮 | 行为 | 来源/说明 |
|------|------|----------|
| 展开完整诊断 | 打开侧边栏，展示推理过程 + 置信度 + P0/P1 建议 + 追问入口 | 主操作，蓝色强调 |
| 异常详情 | 跳转 XRay 异常详情页 | 保留现有卡片链接 |
| 异常堆栈 | 跳转完整异常堆栈页 | 保留现有卡片链接 |
| 事件详情 | 跳转告警事件详情页 | 保留现有卡片链接 |

---

### 6. 反馈行

| 元素 | 行为 | 说明 |
|------|------|------|
| 👍 有用 | 点击后变绿，记录正向反馈 | 用于 Agent 诊断准确率统计 |
| 👎 没用 | 点击后变红，记录负向反馈 | 用于后续 SKILL 优化和 Case 追溯 |

**设计意图**：轻量收集 SRE 对诊断结果的真实感知，数据用于 AgentOps 评估体系。点击后仅保留已选项，另一项消失，避免误操作。

---

## 四、展开完整诊断（侧边栏）

点击「展开完整诊断」弹出右侧侧边栏，包含卡片层不展示的完整信息：

| 区块 | 内容 |
|------|------|
| 推理过程 | Agent 每一步的工具调用结论（查指标 → 查异常 → 查变更 → 分析分布 → 排除其他变更） |
| 诊断结论 | 根因一句话 + 置信度百分比 |
| 处置建议 | P0（立即执行）+ P1（跟进项） |
| 继续排查 | 快捷追问入口（查看发布变更内容/生成回滚步骤/查看异常堆栈/生成排障报告） |
| 对话输入框 | 自由追问，Agent 实时响应 |

**置信度在侧边栏而非卡片的原因**：置信度是推理过程的产物，适合放在推理结论旁边；卡片层的目标是决策摘要，过多元数据反而分散注意力。

---

## 五、信息分层原则

| 层次 | 位置 | 内容 |
|------|------|------|
| **决策层**（卡片） | 直接可见 | 根因结论、关键证据（各维度最核心 1 条）、影响、操作入口 |
| **推理层**（侧边栏） | 点击展开 | 完整推理步骤、排除项、置信度、P0/P1 建议 |
| **原始层**（跳转链接） | 跳转外部页 | 完整异常堆栈、原始指标曲线、事件详情 |

---

## 六、与现有卡片的核心差异

| 对比项 | 现有卡片 | 新设计卡片 |
|--------|---------|-----------|
| 根因描述 | 原始异常类名（ExecutionException） | AI 推理的根因一句话（发布变更导致格式不兼容） |
| 变更展示 | 3 条平铺，无优先级 | 最可疑 1 条 + 「最可疑」标注，其余进侧边栏 |
| 趋势/指标 | 原始时间点数组 `[102,118,81,161]` | 转化为可读描述「1→134，环比 200%+，critical 级」 |
| 异常信息 | 3 个字段分行展示 | 合并为 1 句，去掉完整堆栈 |
| 机房+影响范围 | 散落在「异常分布」区 | 合并为「影响」维度，置信色块标示 |
| 处置建议 | 无 | 侧边栏提供 P0/P1 建议 |
| 反馈机制 | 无 | 👍/👎 轻量反馈，供 AgentOps 采集 |
