# 评估器

## 1. 评估器是什么

评估器可以理解成整套评估流程中的**"裁判"**。数据集（离线整理的样本或线上链路回流的真实请求）负责提供待评估的内容，自动评估任务负责组织执行，实验报告负责展示结果，而真正决定"这个结果应不应该得高分、为什么得这个分"的，是评估器本身。

一个评估器通常至少会输出两部分：**score**（量化得分）和 **reason**（解释原因）。你希望系统关注相关性、正确性、幻觉、工具调用质量，还是某条业务规则是否命中，最终都要通过评估器来表达。

评估器并不是生成业务结果的模型，而是**对结果进行判断和归因的规则承载体**。

---

## 2. LLM 评估器 vs HTTP 评估器

当前产品中的评估器主要分为两类：

| | **LLM 评估器** | **HTTP 评估器** |
|---|---|---|
| **判分方式** | 通过 Prompt + 大模型进行语义判断 | 通过业务服务接口承载判分逻辑 |
| **适合场景** | 相关性、正确性、幻觉等需要理解上下文的问题 | 已有成熟 checker 服务、规则稳定、逻辑可控的场景 |
| **评估范围** | 单 SPAN 评估 / 结果评估 | 单 SPAN 评估 / 结果评估 / **全链路评估** |
| **特点** | 灵活、上手快 | 稳定、可控性强 |

用更直白的话来说：LLM 评估器是"让模型做裁判"，HTTP 评估器是"让业务规则服务做裁判"。两者不是互相替代，而是分别解决不同类型的评估问题。

**如何选择：**
- 问题难以被明确规则完全覆盖、需要结合上下文语义判断 → 优先选 **LLM 评估器**
- 团队已有成熟 checker 服务、某类判分逻辑已稳定写成服务 → 优先选 **HTTP 评估器**
- 需要把完整调用链路（trace + observations）送给服务判断 → 只有 **HTTP 评估器** 支持全链路评估

**行业落地情况：**

| 场景类型 | 代表案例 | 评估器选型 | 效果 |
|---------|---------|-----------|------|
| 语义相关性评估 | sug 粗筛 query 质量 | LLM 评估器 | 快速验证模型输出是否符合业务语义预期 |
| 幻觉检测 | 点点 Agent 在线 checker | HTTP 评估器 | 复用业务侧成熟 checker，结果稳定可控 |
| 全链路质量 | Agent 工具调用完整链路审计 | HTTP 评估器（全链路模式） | 可访问 trace + observations 完整信息，发现中间步骤问题 |

第一次搭建评估体系时，很多团队通常会先从一个目标明确的 LLM 评估器起步，快速验证评估思路；当业务规则越来越稳定后，再逐步把部分能力沉淀成 HTTP 评估器。

## 3. 步骤一：创建 LLM 评估器

> **开始前确认**：需先在「LLM 配置」模块中完成至少一个模型配置并确认连通性正常，否则评估器将无法正常执行。→ 参考：[模块三：LLM 配置](https://docs.xiaohongshu.com/doc/fe0671ef5a9436eb151f9a115cabbee4)

进入 **AI 应用评估 → 评估器**，点击「**新建评估器 → LLM 评估器**」。

【截图：评估器列表页，标注「新建评估器」按钮及下拉选项（LLM评估器 / HTTP评估器）】

### 3.1 填写基本信息

- **评估器名称**：建议体现评估目标，例如「sug query 相关性评估」
- **评估方式**：LLM 评估器支持「单 SPAN 评估」和「结果评估」，**不支持全链路评估**
  - **单 SPAN 评估**：对某一个具体 LLM 调用 SPAN 进行评估，适合评估某次模型调用的输出质量
  - **结果评估**：对整条链路的最终 input / output 进行评估，不感知中间过程
- **LLM**：选择已在「[LLM 配置](https://docs.xiaohongshu.com/doc/fe0671ef5a9436eb151f9a115cabbee4)」中创建好的模型配置

【截图：LLM 评估器创建页面，显示名称/评估方式/LLM配置等基本信息区域】

### 3.2 选择 Prompt 模式

LLM 评估器支持两种 Prompt 模式，区别在于 Prompt 的组织方式：

**Text 模式**：将所有判断规则和变量写在一段连续的文本模板里，结构简单、上手快。

- **适合场景**：评估逻辑直接，用一段话能说清楚；第一次搭建某类语义评估，想快速跑通
- **示例结构**：`你是一个评估专家，请根据以下内容判断质量：query: {{query}}，参考内容: {{message}}，输出 score（0-1）和 reason`

**Chat 模式**：按 system / user / assistant 多个消息角色组织 Prompt，更接近真实对话结构。

- **适合场景**：需要明确区分角色职责（如 system 定义规则，user 提供待评内容）；评估逻辑较复杂，需要分多条消息拆解表达
- **示例结构**：system 消息定义评估规则和输出格式，user 消息传入 `{{query}}` 和 `{{message}}` 等变量

**如何选**：如果不确定用哪种，从 Text 模式开始。当你发现一段话写不清楚规则，或希望把"角色"和"内容"分开时，切换到 Chat 模式。

### 3.3 设计 Prompt 与变量映射

这是 LLM 评估器最核心也最容易出错的环节。**Prompt** 和**字段映射**是两个不同的概念，要分开理解：

- **Prompt** 决定"裁判标准"——模型应该按什么规则来判断
- **字段映射** 决定"把哪份试卷交给裁判"——系统把哪些真实字段送进 Prompt 中

即使 Prompt 写得很完整，如果字段映射错了，评估器看到的也可能不是你真正想评的内容。很多看起来像"模型判断不准"的问题，最后真正的原因是评估器没有看到足够完整的输入。

【截图：LLM 评估器 Prompt 配置区域，显示 Prompt 输入框 + 字段映射配置表（变量名/来源/JsonPath）】

**变量语法**：在 Prompt 中用 `{{变量名}}` 引用字段映射传入的值，例如 `{{message}}`、`{{query}}`。

**字段映射格式**：使用 [JsonPath](https://github.com/json-path/JsonPath) 从 SPAN 的 input / output / metadata 中提取字段。以 sug 粗筛评估器为例：

| 变量名 | 来源 | JsonPath |
|--------|------|----------|
| `query` | output | `$.response_content` |
| `message` | input | `$.messages[?(@.role == 'user')]` |

其中 `response_content` 是 SPAN output 中 AI 生成的 query 结果（XML 格式），`messages` 是 SPAN input 中的对话消息列表。

### 3.4 设计输出字段

对绝大多数评估器来说，最关键的输出字段是 `score` 和 `reason`：

- **score**：让实验结果可以被筛选、聚合和统计
- **reason**：让结果可以被理解和复盘

如果有更复杂的结构化分析需求，可以继续补充其他字段，但建议**优先保证这两个核心字段的稳定性**。

【截图：输出字段配置区域，显示 score 和 reason 字段的配置】

## 4. 步骤二：创建 HTTP 评估器

> **开始前确认**：需已有可对外调用的业务 checker 服务，并明确其输入输出字段结构。

进入 **AI 应用评估 → 评估器**，点击「**新建评估器 → HTTP 评估器**」。

与 LLM 评估器相比，HTTP 评估器更强调**服务信息、输入字段和输出字段的定义**，核心目的是把业务已有的判分逻辑接入平台。需要配置的信息包括：

- **服务名**：ONES 上发布服务的服务名
- **请求 path**：服务的请求路径
- **authorization**：如果需要鉴权，会在请求 header 中带上此值
- **每分钟最大请求数**：控制调用频率，设为 `-1` 表示不限制

HTTP 评估器适合以下场景：幻觉 checker、工具调用规则校验、业务侧已有自定义评估服务，以及需要访问**完整调用链路**的复杂评估。它的价值在于**复用业务已经沉淀好的规则能力**，让平台统一负责调度、实验记录和结果分析。

【截图：HTTP 评估器创建页面，显示服务名/请求path/authorization/每分钟最大请求数等配置项】

### 4.1 评估方式与输入参数

HTTP 评估器支持三种评估方式，是 LLM 评估器不具备的能力之一：

**全链路评估**：将完整的 trace 信息作为输入，传给你的 checker 服务。适合需要审计整条 Agent 调用链路的场景。

传入数据包含两个字段：
- `trace`：完整的 trace 信息，包含用户原始输入和最终输出
- `observations`：trace 的调用链详情，每一步埋点操作都在这里

**单 SPAN 评估**：只对指定的某个 SPAN 进行评估，输入字段由评估器配置决定，传入格式为：

```json
{
  "input": {
    "key1": "xxxx",
    "key2": { ... }
  }
}
```

**结果评估**：对整条链路的最终 input / output 进行评估，不感知中间过程。

### 4.2 输入字段设计

HTTP 评估器不会直接吃整条链路原文，而是通过输入字段定义，明确告诉服务这次会传哪些数据、每个字段代表什么含义。输入设计是否清楚，直接决定了服务后续是否能稳定复用。

### 4.3 输出字段设计与格式要求

和 LLM 评估器一样，HTTP 评估器要求服务**必须返回 `score` 和 `reason` 两个字段**。平台要求的返回格式如下：

```json
{
  "code": 200,
  "message": "评估成功",
  "output": [
    { "name": "score", "value": 0.451 },
    { "name": "reason", "value": "判断依据说明" }
  ]
}
```

如果返回结构不符合此格式，平台将无法正确解析结果，后续实验报告、分数筛选和聚合分析都会受到影响。

### 4.4 创建时的注意事项

- 服务返回是否**稳定**
- 输出字段定义是否与服务实际返回结构**一致**
- 所选评估方式（全链路/单SPAN/结果）是否与服务的实际处理能力**匹配**

很多"评估结果不对"的问题，最后并不是平台问题，而是服务输入输出约定不清，或者服务返回发生了变化。

## 5. 步骤三：调试评估器

评估器最有效的调试入口，不是反复盯着配置页改字段，而是去看**实验报告 → 单条详情**。

在那里，你可以直接看到：
- 评估器**入参**（实际传进去的字段值）
- 模型**输入**和**输出**
- **得分**、**得分理由**和**错误原因**

【截图：实验明细单条详情页，显示评估器入参/模型输入输出/score/reason 等信息】

- 对 **LLM 评估器**：帮助你判断是 Prompt 本身有问题，还是字段映射有问题
- 对 **HTTP 评估器**：帮助你判断是服务输入不对，还是服务返回本身不稳定

一个真正好用的评估器，往往不是第一次就写得完美，而是在持续实验和复盘中逐步校准出来的。

## 6. 常见问题

**评估器和业务模型有什么区别？**
评估器只负责判分，不负责生成业务结果。它更像"裁判"，而不是"参赛者"。

**第一次做，从哪里开始？**
建议先从一个目标明确的 LLM 评估器开始，把规则、映射和结果解释跑顺，再逐步增加更多评估维度。

**score 和 reason 必须都配置吗？**
强烈建议。score 让结果可以被筛选统计，reason 让结果可以被理解复盘。无论哪种评估器，这两个字段都应该是最稳定的统一输出。

**字段映射配错了怎么排查？**
进实验报告 → 单条详情，看「评估器入参」那一栏，直接能看到系统实际传给评估器的字段值。如果值不对，回去检查 JsonPath 表达式。

**HTTP 评估器和 LLM 评估器最核心的差异是什么？**
除了判分方式不同，HTTP 评估器独有**全链路评估**能力——可以把完整的 trace + observations 数据传给你的服务，而 LLM 评估器只支持单 SPAN 或结果评估。
