# 评估实验

## 1. 评估实验是什么

评估实验是自动评估任务**一次实际运行后的结果集合**。从系统角度看，它是一条实验运行记录；从用户角度看，它对应的是一份可以被阅读、分析和继续追问题的**实验报告**。

三个容易混淆的概念，先区分清楚：

| 概念 | 含义 |
|------|------|
| **自动评估任务** | 长期配置，决定任务以何种规则持续运行（周期触发、样本来源、评估器选择等） |
| **评估实验** | 某次实际运行产生的结果对象（系统侧的记录单元） |
| **实验报告** | 用户真正阅读和消费的结果内容（明细 + 分析） |

---

## 2. 步骤一：查看实验列表

进入 **AI 应用评估 → 评估实验**，可以看到所有实验的运行记录列表。

【截图：评估实验列表页，显示实验ID/名称/所属任务/状态/触发方式等列】

列表中展示实验 ID、实验名称、所属自动评估任务、状态（如已完成、失败等）和触发方式（自动触发、调度触发、手动触发）等信息。这一页承担的是**"实验索引页"**的角色——先找到你想看的那次运行，再进入详情查看真正有价值的结果内容。

---

## 3. 步骤二：查看实验详情

点击某条实验名称进入详情页，页面分为上下两部分：

- **实验信息**：本次运行的基本元信息
- **实验明细 / 实验分析**：两个 tab，分别对应样本级别视角和整体分布视角

【截图：实验详情页整体布局，标注上方「实验信息」区域和下方「实验明细」「实验分析」两个 tab】

**建议阅读顺序**：先看实验信息（确认是不是正确的任务和时间范围）→ 再看实验分析（整体分布如何）→ 最后进实验明细（找到具体问题样本）。

### 3.1 实验信息

实验信息区域展示本次运行的基本元数据：ID、名称、数据类型（在线链路 / 离线数据集）、触发方式、当前状态。

很多排查问题在这一步就能缩小范围——先确认自己看的是正确的任务、正确的时间范围和正确的运行结果。

右上角还会展示**本次实验的执行统计**：
- **已完成**：成功完成评估的样本数
- **失败**：LLM 未能完成评估的样本数（可能是模型负载过高、响应超时或其他原因）
- **正在运行**：当前仍在评估中的样本数

失败的样本需要进入单条详情查看错误原因，针对性排查。

【截图：实验信息区域 + 右上角已完成/失败/正在运行统计数字】

### 3.2 实验明细

实验明细 tab 展示本次实验中每一条样本的评估结果，是找到具体问题样本的核心入口。

#### 3.2.1 筛选样本

可以通过多个维度筛选定位重点样本：

- **评估分筛选**：按评估器、操作符（等于/大于/小于等）、分值范围筛选，快速定位低分 bad case 或高分优质样本
- **状态筛选**：按样本评估状态（已完成 / 失败 / 正在运行）筛选，快速找到评估失败的样本进行排查

样本量一旦变大，筛选比人工翻列表高效得多。

【截图：实验明细页，显示左上角筛选区域（评估分筛选 + 状态筛选）+ 样本列表（TraceId/状态/数据类型/评估器分值/操作列）】

#### 3.2.2 查看单条样本详情

点击样本行右侧的「**详情**」进入单条详情页，这是**定位问题的核心位置**。可以看到：

- 评估状态、耗时、Token 使用情况
- **得分**和**得分理由**（reason）
- **评估器入参**（实际传给评估器的字段值）
- 模型输入和模型输出
- 错误原因（如有）
- **对应的链路信息**（可跳转查看完整的 Trace 调用链路，还原 AI 应用在这条请求上的完整执行过程）

【截图：单条样本详情页，显示得分/reason/评估器入参/模型输入输出/链路跳转入口等信息】

这一页真正重要的价值在于：帮助你判断**低分到底来自哪里**——
- 是应用输出本身的问题？
- 是评估器规则/Prompt 的问题？
- 还是字段映射和输入结构的问题？

### 3.3 实验分析

实验分析 tab 用于从**整体视角**观察这次实验的得分分布和聚合结果。

页面包含两个核心模块：

**评估器聚合得分**：选择评估器后，展示该评估器在本次实验中的得分统计，包括 Min、Avg、P50、P90、P99、Max 等分位数指标，帮助你快速判断整体效果和极端值分布。

**得分明细 - 数据项分布**：以各个离散分值（如 0、1、2、3）的样本占比为维度，展示分数分布情况。可以直观看到低分段（0分）占多少、高分段（3分）占多少，快速判断问题的集中程度。

【截图：实验分析页，显示评估器聚合得分柱状图（Min/Avg/P50/P90/P99/Max）+ 得分明细数据项分布图】

## 4. 步骤三：根据实验结果优化应用

实验报告的价值不在于跑完之后多了一条记录，而在于**真正指导后续优化**。

推荐的优化流程：

```
实验分析 → 判断整体得分分布（有无明显低分集中区间）
    ↓
实验明细 → 筛出低分样本（评估分 = 0 或低于阈值）
    ↓
单条详情 → 判断问题出在哪里
    ↓
┌─────────────────────────────────┐
│ 应用输出问题 → 追业务链路和生成逻辑   │
│ 评估器问题   → 修正 Prompt / 映射   │
│ 已确认问题   → 回流到数据集         │
└─────────────────────────────────┘
    ↓
下一轮离线评估 → 验证优化效果
```

对需要长期优化的团队来说，把确认过的问题样本持续回流到数据集、再用这些样本进行下一轮离线回归验证，是让评估体系真正产生价值的核心闭环。

## 5. 常见问题

**实验失败了怎么排查？**
先看实验信息右上角的失败数量，再进实验明细筛选「状态=失败」的样本，进单条详情查看错误原因。常见原因包括：LLM 配置不可连通、模型负载过高、字段映射取值为空等。

**实验明细里有些样本没有分数怎么办？**
通常是评估器执行失败。进单条详情查看「错误原因」：LLM 类错误先检查 LLM 配置连通性和 QPM 限制，HTTP 类错误先检查服务是否正常返回。

**同一个任务跑了很多次实验，怎么快速找到想看的那次？**
在实验列表页用实验名称搜索，结合触发方式（调度触发/自动触发/手动触发）来定位。

**如何判断这次实验的结果是可信的？**
确认三点：样本来源与你的评估目标匹配、实验分析中失败数量不多（完成率过低的实验结论不可靠）、抽查几条 reason 是否合理。
