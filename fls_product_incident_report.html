3.4 根因归纳

层次原因
业务层 大批量商品上架任务在 00:00 同时触发，product-async-default 对同一超热 item（6538...d108）产生高并发并行写入，没有单 item 并发度限制。
数据层 该 item_id 在 sku_multi_relation_0535 分片关联行数 ≥32,236，远超均值（17.8行），出现严重热点分片。
索引层 item_id 列 cardinality 仅 5.62%，range scan 实际覆盖整个 item_id 的全部行，扫描成本随并发累积线性放大。
参数层 innodb_stats_persistent_sample_pages=20，统计信息不能反映极端数据倾斜，优化器无法感知实际代价。

7 · 改进建议

优先级排序

P0 · 立即
限制单 item 并发写入
product-async-default 批量上架任务中，同一 item_id 的 SKU 关联写入应串行化或限并发度（如 semaphore=1~2），避免对同一行产生大量并发 UPDATE。
P0 · 立即
热点 item 分片路由优化
item_id=6538...d108 在 _0535 分片关联行数 ≥32K，是极端热点。评估是否可对该 item 进行数据拆分（如按 sku_id range 拆分到多分片），或对大 item 走独立写入队列。
P1 · 本周内
提高统计信息采样精度
针对写入量大的关键表执行：
ALTER TABLE sku_multi_relation_XXXX STATS_SAMPLE_PAGES=200;
并定期执行 ANALYZE TABLE，使优化器感知真实数据分布。
P1 · 本周内
大批量任务错峰调度
批量上架任务避开整点（00:00/12:00 等），错开到 00:05/00:15 等业务低谷时段。同时设置最大 QPS 限速（如 500 write/s per task）。
P1 · 本周内
慢查询告警前置
当 rs05 分片 max_lock_time > 30s 时提前告警，不等 CPU 飙高再处置。建议在 DMS 配置"慢查询锁等待时长"监控。
P2 · 迭代内
UPDATE 改为 UPSERT 或批量合并
当前每条 SKU-item 关联更新独立发一条 UPDATE，可考虑合并批次或改用 INSERT ... ON DUPLICATE KEY UPDATE，减少锁竞争次数。
