# LLM 配置

## 1. LLM 配置是什么

LLM 配置是 LLM 评估器的**底层依赖**，可以理解成"给裁判指定用哪一个模型、通过什么方式调用"。如果没有这层配置，LLM 评估器即使 Prompt 写好了，也无法真正运行。

这一模块虽然看起来只是模型接入配置，但实际上有三层影响需要提前知道：

- **评估效果取决于模型能力**：模型的理解能力、指令遵循能力直接决定评估器能不能准确打分和给出有价值的 reason
- **评估性能取决于模型响应速度**：模型延迟越高，自动评估任务整体耗时越长，尤其数据量大时体感明显
- **Token 消耗来自于此**：每次评估任务的 LLM 调用费用都由这里配置的模型承担，模型选型和 QPM 设置会直接影响评估成本

因此，LLM 配置不只是一次性的接入动作。随着评估体系逐步成熟，模型选择通常会成为需要持续优化的一层基础能力。

---

## 2. 什么时候需要新建 LLM 配置

以下情况需要新建或调整 LLM 配置：

- 准备创建新的 LLM 评估器
- 希望切换评估模型（如从一个模型切换到另一个）
- 想对比不同模型在评估场景下的稳定性和效果差异

---

## 3. 步骤一：创建 LLM 配置

> **开始前确认**：需要有可用的模型资源，包括模型名称、API Key 和 Endpoint。如果你使用的是公司内部模型，可以在 [QS 模型仓库](https://qs2.devops.xiaohongshu.com/modelrepo) 查询可用模型及申请方式。建议在正式创建前先确认该模型在当前环境中是否真实可用，避免因名称不一致或权限未开通导致连通失败。

进入 **AI 应用评估 → LLM 配置**，点击「**添加 LLM 配置**」。

【截图：LLM 配置列表页，标注「添加 LLM 配置」按钮】

### 3.1 填写配置名称

建议使用能体现模型来源和用途的命名，例如「GPT-4o 评估用」「Claude-3.5 语义判断」，方便后续在评估器里选择时快速识别。

### 3.2 选择模型提供商

模型提供商决定了配置连接的是哪类模型来源。对使用者来说，最重要的并不是提供商名字本身，而是确保当前环境中确实有可用的模型资源，并且与后续要运行的评估器相匹配。

### 3.3 填写模型名称

模型名称需要与实际申请或接入的模型**保持完全一致**。很多连通性或调用失败问题，最后都来自模型名称填写不一致，或者使用了当前环境中并未真正开放的模型标识。

### 3.4 填写 API Key 与 Endpoint

- **API Key**：负责权限验证，确保有权调用该模型
- **Endpoint**：负责请求入口，指向模型服务地址

这两项一旦配置错误，后续所有依赖该模型的评估器都会受到影响。建议在正式挂载到评估器之前，先做一次连通性测试（见步骤二）。

【截图：LLM 配置创建页面，显示配置名称/模型提供商/模型名称/API Key/Endpoint 等字段】

### 3.5 配置调用参数

这些参数控制模型调用时的行为边界，在评估场景里同样会影响评估器输出的稳定性：

| 参数 | 含义 | 评估场景建议 |
|------|------|------------|
| **温度（Temperature）** | 控制输出随机性，数值越高越发散 | 建议设低（如 0~0.3），让评分更稳定、可复现 |
| **最大 Token 数** | 生成长度上限 | 根据 reason 字段预期长度设置，避免截断 |
| **Top P** | 控制候选词范围 | 通常保持默认即可 |
| **QPM（每分钟请求数）** | 限制请求速率 | 根据模型配额设置，避免触发限流 |

---

## 4. 步骤二：测试连通性

配置完成后，在列表页点击对应配置的「**测试链接**」。

【截图：LLM 配置列表页，标注每行右侧的「测试链接」操作按钮】

测试的目的不是判断模型"好不好"，而是确认这条配置链路"**通不通**"。在正式把某个模型挂到评估器之前，先做一次测试，通常能更快暴露权限、地址或模型标识方面的问题。

---

## 5. 步骤三：挂载到评估器

LLM 配置创建完成并测试通过后，就可以在创建 LLM 评估器时选择使用。

进入 **评估器 → 新建 LLM 评估器**，在「LLM」字段中选择已创建好的配置即可。

→ 详细操作参考：[评估器](https://docs.xiaohongshu.com/doc/0bced567cecfcacf14730e6c3490fcbf)

## 6. 常见问题

**LLM 评估器报错或结果异常，先查什么？**
首先确认的通常不是 Prompt，而是这条 LLM 配置本身：是否可连通、模型是否真实可用、温度和 Token 设置是否过于激进。对语义评估来说，底层模型配置稳定，往往比 Prompt 写得花哨更重要。

**模型名称怎么确认填对了？**
以实际申请接入时提供的模型标识为准，不要使用别名或自行缩写。可以先用「测试链接」验证，失败时通常会返回模型不存在或无权限的错误信息。

**温度设多少合适？**
评估场景里建议设低（0~0.3），目的是让评分结果更稳定、可重复验证。如果温度过高，同一条样本多次评估可能得到不同分数，影响实验结论的可信度。

**一个 LLM 配置可以被多个评估器复用吗？**
可以。建议把稳定可用的模型配置复用到多个评估器，而不是每个评估器都新建一套配置，便于统一管理和切换。
