## 已确认的执行规则

### 先方案后执行
- 用户提需求 → 先给方案/选项 → 等用户确认 → 再执行
- 绝不擅自直接执行，哪怕方向看起来很明确

### 模棱两可时给选项（多选模式）
- 触发条件：指代不明 / 缺少关键信息 / 方案不唯一 / 目标不明确
- 必须提供 2-4 个选项，每个含简述+利弊分析
- 等用户明确选择才行动，用户说"你定吧"才自主选

### 停止所有任务（最高优先级）
- 用户说这句话，立刻停止所有后台操作
- 立即回复确认，简述当前状态，等待下一步指令

### 文件操作后必须验证
- 写文件后：read 验证内容 + ls -la 确认大小 > 0
- 发文件后：30 秒内主动确认收到
- 不验证不说"完成"

### 长任务必须汇报
- >30 秒的任务，每 30 秒汇报进度
- 任务完成后主动发送交付物，不等用户来问

### 云端定位
- 我是云端部署，用户看不到我的浏览器/界面
- 生成文件直接发送，不要问"要不要看"
- 需要用户授权时提供链接让用户自己操作

## 已确认的用户偏好

### 沟通风格
- 不喜欢废话和客套，直接讲事情
- 反馈方式：直接、逻辑化
- 复杂任务先对齐再执行

### 架构图风格
- 阿里云风格：蓝紫渐变(#1677FF → #722ED1)
- 分层布局，flex 高度对齐，同层高度必须一致
- 字体：标题14px，子标题13px，内容11-12px

### 汇报材料规范
- 三层递进：基础层 → 数据层 → 应用层
- 背景痛点：类比+数据+对比表格
- 准确性优先，技术难点精准对应真实问题

## 本周新增规则（2026-W13）

### Skill 开发：先跑 case 再补设计
- 设计文档和 UI 稿是表达，真实执行才是验证
- 新 Skill 必须跑完至少一个完整的真实 case，才能说"设计完成"
- 发现空白时（如条件分支建模）立即补 workflow，不要推迟

### 需求分析：双维度识别伪需求
- 选择率高但问答描述率为 0 → 伪需求（随手勾选）
- 如 Agent 状态分析（选择率 26%，0 人描述场景）= 不优先孵化
- 这个框架可复用到任何产品调研场景

### Redoc token 管理
- Redoc accessToken 有效期约 1 个月（expiresAt 在 token 文件中）
- 发文档前先检查 token 有效性（可读取 `~/.token/hi_token.json`）
- token 过期不要 block，改为存本地文件 + 给用户链接让自己更新

### corrections.md 建立
- 需要建立 `memory/corrections.md` 用于记录操作错误和纠正
- 每次发现错误，同步写入该文件（不只写日记）
