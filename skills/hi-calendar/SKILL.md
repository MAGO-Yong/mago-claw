# 最佳实践指引

## 时间处理

- 使用**ISO 8601**时间格式；
- 所有相对时间表达（今天、明天、昨天、本周、当前、最近）都必须基于当前时间解释；
- 用户表达时间时，先获取用户当前时区；
- `timeZoneIana` 缺失时不得静默假设，须询问用户所在时区。

## 通用策略

- **幂等**：每个创建操作须单独调用 `utils:generate-operate-code` 获取新的 operateCode，不同操作之间不得共用。同一操作失败重试时复用同一 operateCode，最多 3 次
- **参会人确认**：通过 `search:employee` 查找参会人时，若存在多个同名人员，须向用户确认具体是哪一位
- **区域选择**：`areaId` 禁止猜测或默认，必须让用户明确选择。用户仅提供会议室名称/关键词但未指定区域时，先询问具体区域

## 写操作策略

适用于 `create`、`create-recurring`、`edit-schedule`。

- 先补齐执行该操作真正需要的信息；不要为了凑参数而猜测未给出的时间、参会人、会议室、区域或文档信息。
- 若用户明确表示不需要参会人或会议室，可以按无参会人/无会议室场景继续，不必强行补全。

## 冲突检测

### 触发条件

任何写操作（`create`、`create-recurring`、`edit-schedule`）**存在参会人**时，必须执行以下完整的时区检测 + 冲突检测流程。

对于 `edit-schedule`，仅在以下变更时触发：新增参会人、修改时间（beginTime/endTime）。仅修改标题、内容、文档等不触发。

**跳过条件**：仅当用户**明确**表达”不要检查””直接创建””直接改””忽略冲突”等意图时，才允许跳过。

### 硬性门控

1. 时区问题未全部与用户确认解决前，**不得**进入冲突检测
2. 冲突问题未全部与用户确认解决前，**不得**执行写操作

### 参会人时区检测

必须按以下步骤检查，有任何问题都必须停止并向用户确认：

1. 使用 `calendar:get-timezone` 查询**所有参会人 + 创建人**的时区。创建人即使不在参会人列表中也必须包含
2. **创建人时区缺失**（`timeZoneIana` 为空）：必须停下来询问用户所在时区，不得假设
3. **参会人时区缺失**（`timeZoneIana` 为空）：假定与创建人同时区继续流程，但必须向用户提示”某某时区未知，已按创建人时区（XX时区）计算，可能存在风险”
4. 除非用户明确指定其他时区，所有时间以**创建人时区**为基准解释和展示
5. 若参会人时区不一致，必须以**表格形式**展示每个参会人对应的本地时间：

   | 参会人         | 时区     | 本地时间               |
   | -------------- | -------- | ---------------------- |
   | 张三（创建人） | 北京时间 | 2026-04-03 14:00-15:00 |
   | John           | 美东时间 | 2026-04-03 02:00-03:00 |

6. 若某参会人的本地时间不在 `10:00-20:00` 范围内，必须在表格后明确标注提醒
7. 标明时区时，优先使用用户可读名称（”北京时间””美东时间””伦敦时间”），不得仅写 `+08:00`、`-04:00` 等 offset

### 时间冲突检测

必须按以下步骤检查，有任何问题都必须停止并向用户确认：

1. 用 `calendar:get-user-schedules` 检查目标时段**所有参会人（含创建人）**是否都空闲
2. 若存在会议室，用 `calendar:get-room-schedules` 检查会议室是否空闲
3. 与目标时间段有任何重合都视为冲突
4. 发现冲突时须清晰报告：谁有冲突、冲突日程的时间段和标题（`hasDetailPermission=true` 时）；权限不足时注明”有冲突但无权查看详情”

### 循环日程补充

循环日程（`create-recurring`）的冲突检测应用于首次实例的时间窗口。检测完成后必须明确告知用户：”仅检查了首次实例 [日期] 的冲突，后续每周实例未逐一验证”

### 特殊项

- 使用 `--doc-auto-empower-list` 或 `--doc-module-links` 前，向用户确认文档标题和链接正确。
- 循环日程创建前，确认用户期望的持续时间以设置 `sequenceEndTime`；`beginTime` 对应的星期几应包含在 `weekdays` 中。
- `cancel-schedule` 需先确认具体日程；循环实例（`scheduleInstanceType=3`）还要确认是取消单次还是整个系列。取消整个系列时传主日程 ID。

## 查询日程

- `get-user-schedules` 用来获取用户日程，可推导个人空闲或多人共同空闲。
- `get-room-schedules` 用来判断特定区域内会议室在目标时段是否可用；用户只给会议室名字时，先解决区域。
- `query-room-schedule` 适合按会议室维度查更长时间窗；`get-room-schedules` 的单次查询窗口只有 1 天，跨多天要拆分。

### 分页策略

是否自动翻页取决于用户意图：

- **需要完整数据时自动翻页**：判断空闲时段、多人交叉分析、计数汇总
- **展示即可时只取首页**：浏览日程列表、查找会议室（首页已有可用结果时无需翻页）
- **不确定时**：先展示首页，询问用户是否需要更多

## 核心概念

### 循环日程模型

- `scheduleInstanceType`：`1`=普通日程，`2`=循环主日程，`3`=循环子日程
- type=2 时 scheduleId 即主日程 ID
- type=3 时 scheduleId 是子日程 ID，`sequenceScheduleId` 指向主日程 ID
- 取消系列用主日程 ID + type=series，取消单次用子日程 ID + type=occurrence

### 专注时间

专注时间（Focus Time）是一种特殊日程类型，用于屏蔽时间段以进行深度工作。

**通用约束：**

- **无需 operateCode**：与 `create`/`create-recurring` 不同，专注时间不使用幂等键
- **无参会人和会议室**：专注时间是个人事项，无需参会人确认和会议室冲突检测
- **日程冲突提示**：创建/编辑前应使用 `get-user-schedules` 检查用户自身日程冲突，叠加在已有会议上可能是有意为之，需让用户确认
- **API 返回 void**：创建/编辑后需通过 `get-user-schedules` 查询确认并获取 scheduleId
- 不能复用 `cancel-schedule` 取消专注时间，须使用 `cancel-focus-time`
- 取消整个循环系列时，传主日程 scheduleId（即 `sequenceScheduleId`）
