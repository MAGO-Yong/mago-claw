# hi-calendar

先读 CLI help，再用下面这些补充规则。

```bash
bunx @xhs/hi-workspace-cli@0.2.8 calendar --help

# 查看具体命令的参数、示例和输出格式
bunx @xhs/hi-workspace-cli@0.2.8 calendar:<method> --help
```

以下是 `--help` 中没有的约束和指引：

## 时间处理

- 使用**ISO 8601**时间格式；
- 所有相对时间表达（今天、明天、昨天、本周、当前、最近）都必须基于当前时间解释；
- 用户表达时间时，先获取用户当前时区；
- `timeZoneIana` 缺失时不得静默假设，须询问用户所在时区。

## 通用策略

- **幂等**：每个创建操作须单独调用 `generate-operate-code` 获取新的 operateCode，不同操作之间不得共用。同一操作失败重试时复用同一 operateCode，最多 3 次
- **参会人确认**：通过 `search:employee` 查找参会人时，若存在多个同名人员，须向用户确认具体是哪一位
- **区域选择**：`areaId` 禁止猜测或默认，必须让用户明确选择。用户仅提供会议室名称/关键词但未指定区域时，先询问具体区域

## 写操作策略

适用于 `create`、`create-recurring`、`edit-schedule`。

- 先补齐执行该操作真正需要的信息；不要为了凑参数而猜测未给出的时间、参会人、会议室、区域或文档信息。
- 若用户明确表示不需要参会人或会议室，可以按无参会人/无会议室场景继续，不必强行补全。

## 冲突检测

只有当写操作会影响时间、参会人时，才需要做下面的检测；其他修改可以跳过。

### 参会人时区检测(包括创建人)

- 默认按下面顺序进行检查，有任何问题都必须停止并向用户进行确认：
  1. 只要存在参会人，必须先查询**所有参会人+创建人**时区，除非用户有明确指定，否则时间须以创建人时区为基准；
  2. 若参会人时区不一致，判断目标时间换算到各自本地后是否落在工作时间 `10:00-20:00`，若不是需要提示目标用户不在工作时间；
- 如需标明时区，优先写“北京时间”“美东时间”“伦敦时间”这类用户可读名称，不要默认只写 `-04:00`、`+08:00` 这类 offset。

### 时间冲突检测

**必须先完成时区检测，确认无误后才能执行本节检测**

- 默认按下面顺序进行检查，有任何问题都必须停止并向用户进行确认：
  1. 用 `calendar:get-user-schedules` 检查目标时段参会人是否都空闲
  2. 若存在会议室，用 `calendar:get-room-schedules` 检查会议室是否空闲
- 与目标时间端有任何的重合都视为冲突

  注意：

- 只有用户明确表达“不要检查”“直接创建”“直接改”“忽略冲突”等意图时，才允许跳过检查；
- 检测中遇到任何问题都要终止写操作流程，**必须向用户进行二次确认**

### 特殊项

- 使用 `--doc-auto-empower-list` 或 `--doc-module-links` 前，向用户确认文档标题和链接正确。
- 循环日程创建前，确认用户期望的持续时间以设置 `sequenceEndTime`；`beginTime` 对应的星期几应包含在 `weekdays` 中。
- `cancel-schedule` 需先确认具体日程；循环实例（`scheduleInstanceType=3`）还要确认是取消单次还是整个系列。取消整个系列时传主日程 ID。

## 查询日程

- `get-user-schedules` 用来获取用户日程，可推导个人空闲或多人共同空闲。
- `get-room-schedules` 用来判断特定区域内会议室在目标时段是否可用；用户只给会议室名字时，先解决区域。
- `query-room-schedule` 适合按会议室维度查更长时间窗；`get-room-schedules` 的单次查询窗口只有 1 天，跨多天要拆分。

### 分页策略

是否自动翻页取决于用户意图：

- **需要完整数据时自动翻页**：判断空闲时段、多人交叉分析、计数汇总
- **展示即可时只取首页**：浏览日程列表、查找会议室（首页已有可用结果时无需翻页）
- **不确定时**：先展示首页，询问用户是否需要更多

## 核心概念

### 循环日程模型

- `scheduleInstanceType`：`1`=普通日程，`2`=循环主日程，`3`=循环子日程
- type=2 时 scheduleId 即主日程 ID
- type=3 时 scheduleId 是子日程 ID，`sequenceScheduleId` 指向主日程 ID
- 取消系列用主日程 ID + type=series，取消单次用子日程 ID + type=occurrence

### 文档 URL 与 shortcutId

```
URL:        https://docs.xiaohongshu.com/doc/{shortcutId}
shortcutId: URL 末段路径，也是 search:doc 返回的 bizId
```
